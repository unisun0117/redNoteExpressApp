## 1. 背景与目标 (Background & Objectives)

**背景**：平台采用 RBAC（基于角色的访问控制）权限模型，用户通过绑定角色获得对应的菜单和按钮级操作权限。当前系统的菜单维护功能已就绪，需要配套的角色管理功能来完成"菜单 → 角色 → 用户"的权限配置链路。

**目标**：为系统管理员提供角色的全生命周期管理能力，包括角色的新增、编辑、查询，以及角色与菜单（到按钮维度）的权限绑定配置，确保每个角色具备正确的菜单访问权限和操作权限。

**一句话背景**：补齐 RBAC 权限模型中"角色配置"环节，可将维护好的菜单权限按角色进行组装分配。

## 2. 角色与使用场景 (Roles & Scenarios)

### 目标用户

| 角色 | 说明 |
| --- | --- |
| 系统管理员 | 负责角色的创建、编辑、查询，以及为角色配置菜单权限 |
| 普通用户 | 被分配角色后，登录系统时获得对应角色的菜单和按钮权限 |

### 使用场景（User Story）

| # | 场景 | 描述 |
|---|------|------|
| 1 | 新增角色 | 新建一个角色（如"仓库管理员"），填写角色基本信息，并从菜单树中勾选该角色可访问的菜单和按钮，保存后角色即可用于用户绑定 |
| 2 | 编辑角色 | 修改已有角色的名称、状态或菜单权限配置，角色编码不可修改以保证权限体系的稳定性 |
| 3 | 查询角色 | 通过角色编码或角色名称快速搜索定位目标角色 |
| 4 | 绑定用户 | 在用户管理中为用户绑定角色，用户登录后即获得该角色对应的菜单权限（角色管理模块本身不负责用户绑定，仅维护角色定义） |

## 3. 核心业务流程 (Core Business Flow)

### 3.1 整体权限配置链路

```
维护菜单（菜单管理模块）
  → 新增角色，从菜单树勾选权限（角色管理模块）
    → 用户绑定角色（用户管理模块）
      → 用户登录后获得对应菜单和按钮权限
```

### 3.2 新增角色流程

1. 点击列表页的"新增"按钮，进入新增角色页面
2. 填写角色编码（唯一标识）、角色名称，设置状态（默认启用）
3. 在"后台菜单"标签页的菜单树中勾选需要授权的菜单和按钮
4. 切换到"企微菜单"标签页，勾选需要授权的菜单和按钮
5. 点击"保存"，系统校验角色编码唯一性和角色名称唯一性
6. 校验通过后，保存角色信息及菜单绑定关系
7. 自动返回列表页，新角色出现在列表中

### 3.3 编辑角色流程

1. 在列表页点击某行的"编辑"，进入编辑角色页面
2. 页面自动回填当前角色信息，角色编码只读不可修改
3. 菜单树自动回填已绑定的勾选状态
4. 根据需求修改：角色名称、状态、菜单勾选（增删均可）
5. 点击"保存"，系统校验角色名称唯一性（排除当前角色自身），更新角色信息，并以全量覆盖方式更新菜单绑定关系
6. 自动返回列表页，列表刷新

### 3.4 状态流转

| 状态 | 含义 | 触发条件 |
| --- | --- | --- |
| 启用 | 正常使用，可绑定给用户 | 新增时默认；或从禁用手动恢复 |
| 禁用 | 角色不可用，已绑定用户失去该角色权限 | 手动禁用 |

> **说明**：禁用角色后，已绑定该角色的用户将失去该角色对应的菜单权限，但用户-角色绑定关系保留。重新启用后权限自动恢复。

## 5. 数据字典与字段级规则 (Data & Field Rules)

### 5.1 角色表核心字段

| 字段名称 | 字段类型 | 来源/依赖 | 默认值 | 读写权限 | 校验规则与约束 | 说明/占位符 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| 角色编码 | String(60) | 新增时输入 | — | 新增可写/编辑只读 | 必填，唯一，仅允许字母数字下划线 | 逻辑主键，如 ADMIN、SALESMAN |
| 角色名称 | String(60) | 新增时输入 | — | 可编辑 | 必填，最长 60 字符，唯一不可重复 | 显示名，如"系统管理员" |
| 描述 | String(255) | 新增时输入 | — | 可编辑 | 最长 255 字符 | 角色的用途说明 |
| 状态 | Enum | 配置 | 启用 | 可编辑 | 枚举：启用/禁用 | 禁用后已绑定用户失去该角色权限 |
| 创建时间 | DateTime | 系统记录 | 当前时间 | 只读 | 格式 YYYY-MM-DD HH:mm:ss | 自动生成 |
| 创建人 | String(60) | 系统记录 | 当前登录用户 | 只读 | — | 自动记录 |
| 更新时间 | DateTime | 系统记录 | 当前时间 | 只读 | 格式 YYYY-MM-DD HH:mm:ss | 自动更新 |
| 更新人 | String(60) | 系统记录 | 当前登录用户 | 只读 | — | 自动记录 |

### 5.2 角色-菜单绑定

一个角色可以绑定多个菜单（含按钮级），菜单数据来源于菜单管理模块。角色保存时，菜单绑定关系采用**全量覆盖**方式更新。

### 5.3 展示逻辑

- 日期时间格式统一为 `YYYY-MM-DD HH:mm:ss`
- 状态列展示：启用（绿色标签）/ 禁用（灰色标签）
- 菜单树中：目录节点展示文件夹图标，菜单节点展示页面图标，按钮节点展示操作图标
- 菜单树默认展开到第一层目录，用户手动展开子节点
- 已勾选的菜单项在树中高亮展示

### 5.4 保存逻辑

- 新增时：创建角色记录 → 全量写入角色-菜单绑定关系
- 编辑时：更新角色基本信息 → 删除旧的角色-菜单绑定 → 全量写入新的绑定关系（全量覆盖策略）
- 如果用户只修改了基本信息未改动菜单勾选，菜单绑定关系保持不变

## 7. 非功能性需求 (Non-Functional Requirements)

### 7.1 权限与安全

| 维度 | 要求 |
| :--- | :--- |
| 数据权限 | 无限制，所有有权限访问角色管理的用户看到同一套角色列表 |
| 操作权限 | 列表查看为通用权限；新增、编辑需具备角色管理操作权限 |
| 编码安全 | 角色编码一旦创建不可在产品界面修改，防止权限体系混乱 |
| 菜单绑定 | 保存时全量覆盖，防止历史脏数据残留 |

### 7.2 性能要求

| 维度 | 要求 |
| :--- | :--- |
| 列表查询 | 支持分页加载，响应流畅不卡顿 |
| 菜单树加载 | 一次性加载完整树结构，无需分页或懒加载 |
| 角色保存 | 保存操作完成后及时反馈结果 |

### 7.3 业务规则

- 角色编码和角色名称作为权限体系的两个唯一标识，新增后角色编码不可在产品界面上修改，角色名称可修改但不可与其他角色重复
- 禁用角色不影响历史数据关联关系
- 角色-菜单绑定关系在编辑时采用全量覆盖策略，前端需回填当前已绑定菜单的勾选状态
- 一个角色可同时配置后台菜单权限和企微菜单权限，两者互不影响

## 8. 附录

### 8.1 功能清单汇总

| 功能项 | 说明 |
| --- | --- |
| 角色列表查询 | 支持按角色编码、角色名称模糊搜索，分页展示 |

### 8.2 与其他模块的关系

| 关联模块 | 关系说明 |
| --- | --- |
| 菜单管理 | 角色管理的菜单树数据来源于菜单管理模块。菜单管理负责维护菜单结构（目录/菜单/按钮）和权限码，角色管理消费这些数据为角色分配权限 |
| 用户管理 | 用户通过"配置角色"绑定角色，角色列表来源于本模块。角色变更后，绑定该角色的用户权限随之变更 |
| 登录认证 | 用户登录后，系统根据其绑定角色加载对应菜单树和操作权限 |
| 权限控制 | 角色的菜单权限决定用户登录后可访问哪些菜单和执行哪些按钮操作 |

### 8.3 特殊业务场景

**场景一：角色禁用影响**

禁用角色后：
- 该角色在用户管理的"配置角色"弹窗中仍展示，但标记为"已禁用"
- 已绑定该角色的用户失去该角色对应的菜单和按钮权限
- 角色-菜单绑定关系保留，重新启用后权限自动恢复
- 用户-角色绑定关系保留

**场景二：菜单变更的级联影响**

当菜单管理模块中某个菜单被删除或禁用时：
- 角色-菜单绑定关系不会自动清除，角色仍可正常保存和编辑
- 用户登录加载菜单树时，已删除/禁用的菜单自然不展示，不影响系统运行
- 建议在菜单管理中删除菜单时提示"该菜单已被 N 个角色引用"

**场景三：最小权限原则**

新增角色时，菜单树的父节点和子节点独立勾选：
- 勾选父节点（目录）不代表自动勾选其下所有子节点
- 必须逐级展开并显式勾选每个需要的菜单和按钮
- 这确保了角色权限的最小化配置，防止误授权
