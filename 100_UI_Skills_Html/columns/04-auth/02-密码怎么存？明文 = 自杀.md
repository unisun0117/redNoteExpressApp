## 摘要
> 数据库里直接存用户密码明文，一旦泄露所有用户账号瞬间沦陷。哈希、加盐、bcrypt 分别是什么？这篇文章用可运行的 Python 代码带你搞懂密码存储的正确姿势。

## 引言

2016年，某知名互联网公司被曝数据库泄露，其中包含数百万用户的明文密码。这不是什么远古时代的事故——直到今天，仍然有开发者在数据库里直接存明文密码。理由是"方便调试"，或者"反正只有我自己能看到数据库"。

这种想法很危险。数据库泄露的途径比你想象的多：SQL 注入漏洞、备份文件被误传、离职员工恶意导出、甚至运维人员工号被盗。一旦数据库暴露，所有用户的密码就变成了公开信息。更可怕的是，很多人会在不同网站使用相同的密码——你泄露的密码可能同时打开用户的所有账号。

密码存储有一条铁律：从用户注册那一刻起，任何人（包括开发者自己）都不应该能逆向还原出用户的明文密码。而实现这一目标的技术手段，就是哈希（Hash）和加盐（Salt）。

## 基础知识储备

1. **哈希函数的基本概念**：输入任意长度数据，输出固定长度的摘要。相同输入始终产生相同输出，但无法从输出反推输入。
2. **Python 字符串编码**：理解 `"hello".encode()` 将字符串转为字节串，这是哈希函数的标准输入格式。
3. **暴力破解的基本原理**：攻击者预先计算常见密码（如 123456、password）的哈希值，然后和泄露的哈希值做对比——这叫彩虹表攻击。
4. **计算复杂度概念**：某些算法故意设计得"慢"，消耗更多 CPU 时间，以增加暴力破解成本。

## 正文

### 哈希：单向加密，不可逆

哈希函数的本质是"摘要算法"。你把一本小说丢进去，它给你一个 64 位的字符串。这个过程是不可逆的——你拿着摘要，绝对还原不出原始内容。就像你把一只鸡做成鸡汤，你再怎么分析鸡汤，也拼不回那只完整的鸡。

常见的哈希算法有 MD5、SHA-1、SHA-256。它们的特点是"快"——非常适合校验文件完整性、做数据去重。但也正因为快，它们根本不适合存密码。想想看，一张 RTX 4090 显卡每秒能计算数十亿次 SHA-256——攻击者可以用庞大的字典库在几秒内碰撞出原始密码。

存储密码的正确选择是"慢哈希"算法——bcrypt、scrypt、argon2。它们的设计目标恰恰相反：故意消耗大量 CPU 资源和内存，让单次哈希变得很慢。用户登录时多等 100 毫秒毫无感觉，但攻击者的暴力破解成本会被放大数千倍。

```python
import bcrypt

password = "MySecurePass123"

# 注册时：对密码做哈希，存入数据库
# bcrypt.gensalt() 会自动生成随机盐值
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
print(f"存入数据库的值: {hashed.decode()}")

# 登录时：从数据库取出哈希值，调用 checkpw 验证
user_input = "MySecurePass123"
is_valid = bcrypt.checkpw(user_input.encode(), hashed)
print(f"密码验证: {'通过' if is_valid else '失败'}")

# 同样密码每次 hash 结果不同，因为 salt 随机
another_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
print(f"再次哈希：{another_hash.decode()}")
print(f"两次结果相同吗？{'是' if hashed == another_hash else '否——这就是加盐的效果'}")
```

### 盐值（Salt）：让相同的密码产生完全不同的哈希

你可能会想：既然哈希是不可逆的，那只要大家都用 bcrypt 就行了，为什么还要"盐"？

因为很多用户用的密码是相同的。如果两个用户都设了密码"123456"，不加盐的话，他们在数据库里的哈希值会完全一样。攻击者一旦通过字典碰撞出一个，就同时攻破了所有用"123456"的用户。

盐值（Salt）是一个随机字符串，每个用户一份，互不相同。哈希计算时把密码和盐值搅在一起——这样一来，即使两个用户密码相同，最终存储的哈希值也完全不同。bcrypt 会把盐值嵌入到最终的哈希串里，你不需要单独存盐值，验证时它会自动提取。

你可以运行下面这段代码直观感受盐值的作用：

```python
import hashlib
import os

# 不加盐：相同密码产生相同哈希
def simple_hash(password: str) -> str:
    return hashlib.sha256(password.encode()).hexdigest()

p1 = simple_hash("admin123")
p2 = simple_hash("admin123")
print(f"不加盐: p1==p2? {p1 == p2}")  # True —— 攻击者一眼看出相同密码

# 加盐：相同密码产生完全不同的哈希
def salted_hash(password: str) -> tuple:
    salt = os.urandom(16).hex()
    hashed = hashlib.sha256((password + salt).encode()).hexdigest()
    return hashed, salt

h1, s1 = salted_hash("admin123")
h2, s2 = salted_hash("admin123")
print(f"加盐后: hash1={h1[:16]}..., hash2={h2[:16]}...")
print(f"即使密码相同，哈希值相同吗？{'是' if h1 == h2 else '否'}")
```

### 密码存储的最佳实践清单

除了使用 bcrypt/argon2 和加盐，还有几条原则必须遵守：

不要自己发明哈希算法。密码学领域有一句名言——"别自己造轮子"。你设计的算法在专家眼中可能漏洞百出。用被广泛审计过的标准库（bcrypt、argon2-cffi）是最安全的选择。

不要限制密码的最大长度。如果你想设上限，至少给到 128 个字符。用户使用密码管理器生成的随机密码，长度往往超过 20 位。

不要禁止特殊字符。有些系统禁止密码包含 `<script>` 或单引号，这其实是在输出端没有做好转义，反而在输入端限制用户。正确处理方式是：密码里可以有任何字符，但渲染到 HTML 时做转义。

考虑支持密码强度检测。注册时给用户一个直观的强度指示器（弱/中/强），并在密码过于简单时给出建议，这不只是体验优化，也是安全防线的一部分。

## 总结

1. 永远不要在数据库里存储明文密码，一旦数据库泄露，所有用户的账号直接沦陷。
2. 哈希是单向不可逆的，但 MD5/SHA256 计算太快，适合校验文件但不适合存密码。
3. 存储密码必须用慢哈希算法（bcrypt / argon2），通过消耗大量计算资源来对抗暴力破解。
4. 盐值（Salt）确保相同密码产生不同哈希值，防止彩虹表攻击和批量碰撞。
5. 不要自己发明密码存储方案，用经过安全审计的标准库是最基本的安全素养。

## 注意事项

1. bcrypt 会自动处理盐值生成和嵌入，你不需要手动管理盐值。
2. Python 的 bcrypt 库必须使用 4.0.x 版本，5.x 版本与 passlib 等认证库不兼容。
3. 密码强度检测是前端体验优化，真正的安全防线在后端——前端校验可以被绕过。

## 坑点

| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 使用 MD5 或 SHA256 存密码 | 数据库泄露后攻击者用 GPU 几小时内碰撞出大部分密码 | 这些算法速度快，设计目标就不是密码存储 | 用 bcrypt 或 argon2，并设置合适的 cost factor（bcrypt 建议 12） |
| 忘记做编码转换 | bcrypt.hashpw() 报 TypeError | bcrypt 要求输入 bytes 类型，而非 str | 始终用 `password.encode('utf-8')` 将字符串转为字节 |
| 限制密码长度过短 | 用户无法使用密码管理器生成的 32 位随机密码 | 开发者没有考虑密码管理器的使用场景 | 密码长度上限至少 128 字符，或者不设上限 |
| 在日志中打印密码 | 日志系统记录了用户的明文密码，日志泄露 = 密码泄露 | 调试时不小心 print 了请求体 | 日志系统配置脱敏规则，过滤 password、token 等敏感字段 |
