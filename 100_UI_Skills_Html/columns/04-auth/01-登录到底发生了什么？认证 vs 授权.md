# 用户认证从0到1 第1篇：登录到底发生了什么？认证 vs 授权

## 摘要
> 点击登录按钮后，前端、后端、数据库之间到底发生了什么？认证和授权为什么是两回事？这篇文章从零拆解登录的完整流程，帮你建立认证领域的全局认知。

## 引言

你每天都要输入账号密码、扫码登录、或者勾选"记住我"。这些操作看起来稀松平常，但背后涉及一套精密的身份验证体系。很多初学者会把"登录"等同于"认证"，把"认证"和"授权"混为一谈，导致在面试或系统设计时漏洞百出。

这篇文章是整个"用户认证从0到1"专栏的第一篇，我们不写代码，先把概念讲透。理解认证和授权的区别，是后续所有技术方案（Session、JWT、OAuth、RBAC）的基础。

当你理解了 HTTP 是无状态协议，再回头看"为什么每次请求都要带 token"，就会觉得理所当然。让我们从一次完整的登录请求开始。

## 基础知识储备

1. **HTTP 协议基础**：知道 GET/POST 请求的区别，了解请求头和响应头的概念。
2. **客户端-服务器模型**：理解浏览器（前端）和服务器（后端）是两个独立的实体，通过 HTTP 通信。
3. **无状态协议**：HTTP 每次请求都是独立的，服务器默认不知道两个请求是否来自同一个用户。
4. **Cookie 的基本概念**：浏览器能自动存储和发送 Cookie，这是维持登录状态的关键机制。
5. **JSON 格式**：前后端交互最常用的数据格式，token 通常以 JSON 形式返回。

## 正文

### 一次完整登录的7个步骤

当你在登录页输入邮箱"test@example.com"、密码"123456"，然后点击"登录"按钮，下面这些事情会依次发生：

第1步，前端代码阻止表单的默认提交行为（`e.preventDefault()`），取出输入框里的邮箱和密码，构造一个 JSON 对象。

第2步，前端通过 `fetch` 或 `axios` 发送 POST 请求到 `/api/login`，请求体为 `{"email":"test@example.com","password":"123456"}`，Content-Type 设为 `application/json`。

第3步，后端收到请求后，先根据邮箱去数据库查询用户记录。如果查不到，直接返回 401："用户不存在或密码错误"——注意这里不会告诉前端具体"用户不存在"还是"密码错误"，因为这等于帮攻击者摸底。

第4步，后端从数据库取出密码哈希值，调用 bcrypt 或 argon2 的验证函数，将用户输入的明文密码和哈希值进行比对。如果匹配，认证通过。

第5步，后端生成一个凭证——可能是 Session ID 存到 Redis，也可能是签一个 JWT——然后通过响应体或 Set-Cookie 返回给前端。

第6步，前端收到响应，把 token 存到内存、localStorage 或者让浏览器自动管理 Cookie。此后每次请求都在请求头里带上 `Authorization: Bearer <token>`。

第7步，后端在每个需要认证的接口里，先验证 token 是否有效，再根据 token 里的用户信息处理业务逻辑。

可以用下面这段 Python 代码快速感受这个流程：

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

# 模拟数据库
fake_db = {"test@example.com": {"password": "hashed_123456", "name": "张三"}}

class LoginRequest(BaseModel):
    email: str
    password: str

@app.post("/api/login")
def login(req: LoginRequest):
    user = fake_db.get(req.email)
    if not user:
        raise HTTPException(status_code=401, detail="邮箱或密码错误")
    # 实际项目中这里用 bcrypt.checkpw
    if req.password != user["password"]:
        raise HTTPException(status_code=401, detail="邮箱或密码错误")
    # 返回临时 token（仅演示）
    return {"token": f"fake_token_for_{req.email}", "name": user["name"]}

@app.get("/api/me")
def get_current_user(authorization: str = ""):
    if not authorization.startswith("Bearer fake_token_for_"):
        raise HTTPException(status_code=401, detail="未登录")
    email = authorization.replace("Bearer fake_token_for_", "")
    return {"email": email, "name": fake_db[email]["name"]}
```

### 认证 vs 授权：面试必考的概念辨析

这两个词的英文缩写经常被混淆——AuthN（Authentication，认证）和 AuthZ（Authorization，授权）。虽然都缩写为"Auth"，但它们是两个完全独立的环节。

认证回答的问题是"你是谁？"。你刷工卡进公司大门，警卫通过工卡照片确认你是你——这就是认证。在系统里，认证意味着验证用户提供的凭据（密码、短信验证码、指纹）是否有效。

授权回答的问题是"你能干什么？"。进了公司大门不代表你能进财务室。你能去几楼、能访问哪些系统，由你的权限决定——这就是授权。在系统里，授权意味着根据用户角色或权限列表，决定允许或拒绝某项操作。

用一个表格来记忆：

| 维度 | 认证（AuthN） | 授权（AuthZ） |
|------|-------------|-------------|
| 问题 | 你是谁 | 你能干什么 |
| 时机 | 登录时 | 每次操作时 |
| 凭据 | 密码、指纹、验证码 | 角色、权限列表 |
| 失败 | 401 Unauthorized | 403 Forbidden |
| 现实类比 | 刷工卡进门 | 工卡权限等级 |

一个常见误区是：登录成功不代表能做所有操作。认证通过只说明"你是合法用户"，但删除文章、修改系统配置这些操作，需要额外的授权检查。如果你写过 `if user.role != "admin": raise HTTPException(403)`，你已经在做授权了。

### 为什么 HTTP 无状态特性催生了认证系统

HTTP 协议在设计之初就是无状态的——每个请求都是独立的，服务器处理完一个请求就忘了它。这种设计让 HTTP 非常简洁高效，但也带来了一个问题：服务器怎么知道连续两个请求来自同一个人？

想象一个场景：你登录成功后访问 `/api/me` 查看个人信息，服务器收到这个请求时，它看到的就是一个新的 HTTP 请求，和刚才那个登录请求没有任何关联。如果不做额外处理，服务器会说"我不认识你"。

这就是为什么我们需要 token、session、cookie 这些机制——它们本质上都在做一件事：在无状态的 HTTP 上"模拟"出一个有状态的会话。浏览器每次请求自动带上 Cookie，或者前端手动在请求头里塞 token，都是为了让服务器"记住"你。

理解了这个本质，你就不会纠结"为什么要带 token"这种问题了——因为不带的话，服务器根本不知道你是谁。

## 总结

1. 登录流程分为前端发送凭据、后端验证凭据、签发凭证、存储凭证、后续请求携带凭证五个阶段。
2. 认证（AuthN）回答"你是谁"，授权（AuthZ）回答"你能干什么"，两者是独立的环节，不可混淆。
3. HTTP 是无状态协议，服务器默认不记得任何用户，因此需要 token/session/cookie 来维持登录状态。
4. 认证失败返回 401，授权失败返回 403，这两个状态码的含义在面试中经常被问到。
5. 理解认证和授权的区别，是后续学习 Session、JWT、OAuth、RBAC 等所有技术方案的前提。

## 注意事项

1. 登录接口的报错信息不要区分"用户不存在"和"密码错误"，统一返回"用户名或密码错误"可以防止攻击者枚举有效账号。
2. 401 和 403 的语义经常被搞混——401 是"你没有提供有效的认证凭据"（未登录或 token 过期），403 是"你的凭据有效，但你没有权限执行这个操作"。
3. 本文演示代码中的密码比对是明文比较，仅用于理解流程，生产环境必须使用 bcrypt 等哈希算法。

## 坑点

| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 用 GET 请求发送登录凭据 | 密码出现在 URL 参数里，被浏览器历史记录、服务器日志、CDN 日志记录 | GET 请求的参数在 URL 中，没有保密性 | 登录接口必须用 POST，请求体传参 |
| 换设备后直接要求重新登录 | 用户在手机上登录后，电脑端并没有被同步 | 多端登录需要服务端维护统一的会话或 token 管理 | 设计时就考虑多端场景：JWT 天然支持，Session 需要集中存储（Redis） |
| 认证和授权代码混在一起 | 一段代码既做登录验证又做权限检查，后续改动频繁出错 | 两个职责耦合，违反单一职责原则 | 中间件层做认证（提取当前用户），路由层/装饰器层做授权（检查角色权限） |
