## 摘要
> 一个接口如果返回全量数据，轻则响应变慢，重则拖垮数据库。分页、过滤、排序是解决"数据过多"的三板斧。本文介绍四种分页方案和工业级过滤排序实践，附完整可运行代码。
## 引言
假设你做了一个电商系统的商品接口 `GET /api/products`。一开始只有 50 个商品，返回全量数据没问题。三个月后商品增加到 5000 个，接口响应时间从 50ms 变成了 3 秒。用户开始投诉"页面打不开"，DBA 开始投诉"查询全表把数据库拖慢了"。

这就是"数据膨胀"问题。任何实际业务中，数据量都会增长。如果一个接口不做分页，它本质上是一个定时炸弹——在数据量小时一切正常，一旦数据量突破某个阈值就会炸。

分页（Pagination）、过滤（Filtering）、排序（Sorting）是解决这个问题的三板斧。它们经常出现在同一个接口中，组合使用。本篇文章会系统性地介绍这几种方案，并给出可以直接运行的代码示例。

## 正文
### 1. 四种分页方案：从简单到高级

**方案一：偏移分页（Offset-based Pagination）**

这是最常见的分页方式。参数是 `page`（页码）和 `size`（每页条数），后端用 `LIMIT + OFFSET` 实现。

```
GET /api/users?page=2&size=10
→ SELECT * FROM users LIMIT 10 OFFSET 10
```

优点：简单易懂，前端实现分页条很方便。缺点：OFFSET 大时性能差（数据库必须扫描前面所有行），且分页期间如果数据有增删，可能出现重复或遗漏。

**方案二：游标分页（Cursor-based Pagination）**

用数据自身的标识作为"游标"，每次查询时带上"上一页最后一条数据的 id"。

```
GET /api/users?cursor=100&size=10
→ SELECT * FROM users WHERE id > 100 ORDER BY id LIMIT 10
```

优点：性能稳定（无论翻到第几页都一样快），不受数据增删影响。缺点：只能"上一页/下一页"，不能跳页。Twitter、Facebook 都在用这种方案。

**方案三：键集分页（Keyset Pagination）**

游标分页的扩展版，游标不一定是 id，可以是任何排序字段的组合。

```
GET /api/users?created_at_gt=2024-01-01&id_gt=100&size=10
```

**方案四：基于 Link Header 的分页**

GitHub API 使用的方案。在响应头中返回上一页/下一页的完整 URL：

```
Link: <https://api.example.com/users?page=2>; rel="next",
      <https://api.example.com/users?page=10>; rel="last"
```

```python
from flask import Flask, request, jsonify

app = Flask(__name__)

# 模拟数据库：100 个用户
users = [
    {"id": i, "name": f"用户{i}", "email": f"user{i}@example.com"}
    for i in range(1, 201)
]

@app.route('/api/users', methods=['GET'])
def list_users():
    # === 分页参数 ===
    page = request.args.get('page', 1, type=int)
    size = request.args.get('size', 10, type=int)

    # 校验
    if page < 1:
        return jsonify({"error": "page 必须大于 0"}), 400
    if size < 1 or size > 100:
        return jsonify({"error": "size 必须在 1-100 之间"}), 400

    # === 过滤参数 ===
    keyword = request.args.get('keyword', '').strip()
    email_domain = request.args.get('domain', '').strip()

    filtered = users

    # 按关键字过滤名字
    if keyword:
        filtered = [u for u in filtered if keyword in u['name']]

    # 按邮箱域名过滤
    if email_domain:
        filtered = [u for u in filtered if u['email'].endswith('@' + email_domain)]

    # === 排序参数 ===
    sort_by = request.args.get('sort', 'id')      # 默认按 id 排序
    order = request.args.get('order', 'asc')       # 默认升序

    allowed_sort_fields = {'id', 'name'}
    if sort_by not in allowed_sort_fields:
        return jsonify({"error": f"不支持的排序字段: {sort_by}"}), 400

    reverse = (order == 'desc')
    filtered = sorted(filtered, key=lambda u: u[sort_by], reverse=reverse)

    # === 分页结果 ===
    total = len(filtered)
    total_pages = (total + size - 1) // size  # 向上取整
    start = (page - 1) * size
    end = start + size
    items = filtered[start:end]

    return jsonify({
        "code": 0,
        "message": "success",
        "data": {
            "items": items,
            "pagination": {
                "page": page,
                "size": size,
                "total": total,
                "totalPages": total_pages,
                "hasNext": page < total_pages,
                "hasPrev": page > 1,
            }
        }
    })

if __name__ == '__main__':
    app.run(debug=True)
```

### 2. 过滤与排序：让 API 更灵活

过滤和排序让用户可以在前端自己"淘"数据，而不是后端对每种筛选条件都写一个专用接口。

**过滤的常见模式：**
- 精确匹配：`?status=active&role=admin`
- 模糊搜索：`?keyword=张三`（在多个字段中搜索）
- 范围筛选：`?price_min=100&price_max=500`、`?created_at_gte=2024-01-01`
- 枚举多选：`?status=active&status=pending`（同一个 key 传多次，后端解析为数组）

**排序的常见模式：**
- 单字段排序：`?sort=created_at&order=desc`
- 多字段排序：`?sort=priority,created_at&order=desc,asc`

过滤和排序的组合使用可以让同一个接口满足 90% 的前端列表查询需求。

### 3. 前端分页实现技巧

```javascript
// 前端分页工具函数
function buildPaginationUrl(baseUrl, params) {
  const query = new URLSearchParams();
  if (params.page) query.set('page', params.page);
  if (params.size) query.set('size', params.size);
  if (params.keyword) query.set('keyword', params.keyword);
  if (params.sort) query.set('sort', params.sort);
  if (params.order) query.set('order', params.order);
  return `${baseUrl}?${query.toString()}`;
}

// 使用示例
async function loadUsers(page = 1) {
  const url = buildPaginationUrl('/api/users', {
    page,
    size: 20,
    keyword: searchInput.value,
    sort: 'name',
    order: 'asc'
  });
  const { items, pagination } = await fetch(url).then(r => r.json()).then(d => d.data);
  renderList(items);
  renderPagination(pagination); // 根据 hasNext/hasPrev 控制按钮状态
}
```

前端拿到 `pagination` 对象后，可以根据 `hasNext` 控制"加载更多"按钮的显隐，根据 `totalPages` 渲染分页条。不需要自己计算是否有下一页。
## 总结
- 任何涉及列表的接口都应当支持分页，即使是内部使用的小接口也能避免未来数据膨胀的风险。
- 偏移分页最简单，适合数据量小的场景；游标分页性能最好，适合大数据量和实时数据场景。
- 过滤和排序是与分页天然搭配的功能，都应通过 Query 参数实现，使用同一套规范。
- 响应中的 `pagination` 对象应包含足够的元信息（total、totalPages、hasNext、hasPrev），让前端零计算渲染分页组件。
- 前端应当封装一个构建分页 URL 的工具函数，避免在每个列表页面重复写参数拼接逻辑。
## 注意事项
- 分页参数 `page` 的最小值是 1，`size` 要有上限（如 100），防止恶意请求一次拉取全量数据。
- 过滤和排序的字段要做好白名单校验，只允许在白名单中的字段，防止 SQL 注入或无效查询。
- 游标分页对删除和添加数据的容忍度更好，如果数据频繁增删，优先考虑用游标分页。
- 排序字段如果数据量大，一定要建数据库索引，否则性能会很差。
## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 大 OFFSET 查询巨慢 | 翻到第 100 页响应要 5 秒 | 数据库必须扫描 OFFSET 前面所有行 | 改用游标分页（WHERE id > cursor） |
| 分页期间数据重复 | 翻页时看到同一条数据出现两次 | 第一页和第二页之间新增了数据，OFFSET 错位 | 使用游标分页或接受偏移分页的近似性 |
| 过滤字段 SQL 注入 | 恶意请求 `?sort=1;DROP TABLE users` 导致数据库被删 | 直接用用户输入的字段拼 SQL | 排序和过滤字段必须做白名单校验 |
| 不分页直接返回全量 | 接口超时，数据库 CPU 100% | 数据量增长后全量查询拖垮数据库 | All APIs that return lists must support pagination |
