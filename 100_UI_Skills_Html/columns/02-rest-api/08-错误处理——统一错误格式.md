## 摘要
> 错误处理做不好，前端调用接口就像在"猜谜"。本文教你建立一套结构化的错误码体系，包括错误码分段设计、全局异常捕获、前端统一处理策略，让前后端在错误处理上达成"契约"。
## 引言
你肯定遇到过这种场景：调后端接口报错了，response 里只有一个 `{"error": "Internal Server Error"}`，你完全不知道是什么原因——是参数传错了？数据库挂了？还是第三方服务超时了？然后你只能截图发给后端同事，他再去翻日志。

好的错误处理应该是：无论什么错误，前端都能拿到结构化的信息——错误码、错误描述、哪个字段出了问题。这样前端可以针对性地做用户提示，比如"邮箱格式不正确"而不是"请求失败请重试"。

这篇文章会带你建立一套完整的错误处理体系。从错误码的设计到全局异常的捕获，从前端拦截器的处理到用户友好的提示策略，覆盖整个错误处理链路。
## 基础知识储备
- **HTTP 状态码知识**：掌握 400、401、403、404、422、500 等常用状态码。
- **统一响应格式**：了解上一篇文章中 `{code, message, data}` 的信封结构。
- **异常处理机制**：理解 try-catch 和全局异常处理器的概念。
- **前端拦截器**：知道 axios 的 request/response interceptor 如何工作。
- **日志记录概念**：了解"前端看到友好提示，后端记录详细日志"的分层原则。
## 正文
### 1. 错误码分段设计

错误码不是随便编一个数字，而是有结构、有规则的。一个好的错误码体系应该让开发者看到错误码就能大致定位问题所在。

推荐使用 5 位数字的错误码，按模块分段：

| 错误码范围 | 含义 | 示例 |
|------------|------|------|
| 0 | 成功 | - |
| 10000-19999 | 通用错误 | 10001: 参数校验失败 |
| 20000-29999 | 用户模块 | 20001: 用户不存在，20002: 密码错误 |
| 30000-39999 | 订单模块 | 30001: 库存不足，30002: 订单已取消 |
| 40000-49999 | 文件模块 | 40001: 文件过大，40002: 格式不支持 |
| 50000-59999 | 第三方服务 | 50001: 支付接口超时，50002: 短信发送失败 |
| 90000-99999 | 系统级错误 | 90001: 数据库连接失败，99999: 未知错误 |

按模块分段的好处是：前端可以根据错误码范围做差异化处理——2xxxx 的用户错误可以展示给用户，5xxxx 的第三方错误需要提示"请稍后重试"，9xxxx 的系统错误可能需要记录额外的监控日志。

```python
# Python 后端：错误码定义与全局异常处理
from flask import Flask, request, jsonify

app = Flask(__name__)

# 错误码定义
class ErrorCode:
    SUCCESS = 0
    # 通用错误 10xxx
    PARAM_MISSING = 10001
    PARAM_INVALID = 10002
    # 用户模块 20xxx
    USER_NOT_FOUND = 20001
    USER_EMAIL_EXISTS = 20002
    USER_PASSWORD_WRONG = 20003
    # 系统错误 9xxxx
    DB_ERROR = 90001
    INTERNAL_ERROR = 99999

# 错误码对应的消息（统一管理，方便国际化）
ERROR_MESSAGES = {
    ErrorCode.PARAM_MISSING: "必填参数缺失",
    ErrorCode.PARAM_INVALID: "参数格式不正确",
    ErrorCode.USER_NOT_FOUND: "用户不存在",
    ErrorCode.USER_EMAIL_EXISTS: "该邮箱已被注册",
    ErrorCode.USER_PASSWORD_WRONG: "密码错误",
    ErrorCode.DB_ERROR: "数据库异常，请稍后重试",
    ErrorCode.INTERNAL_ERROR: "服务器内部错误",
}

def error_response(code, http_status=400, detail=None):
    """构造统一错误响应"""
    message = ERROR_MESSAGES.get(code, "未知错误")
    response_data = {
        "code": code,
        "message": message,
        "data": None
    }
    # 对参数校验错误，可以额外携带 detail 告诉前端具体哪个字段有问题
    if detail:
        response_data["detail"] = detail
    response = jsonify(response_data)
    response.status_code = http_status
    return response

def success_response(data, message="success"):
    """构造统一成功响应"""
    return jsonify({
        "code": ErrorCode.SUCCESS,
        "message": message,
        "data": data
    })

# 全局异常处理器 —— 捕获所有未被 route 函数处理的异常
@app.errorhandler(Exception)
def handle_unexpected_error(e):
    """任何未捕获的异常都会走到这里"""
    # 记录详细日志供开发者排查
    app.logger.error(f"Unexpected error: {type(e).__name__}: {str(e)}", exc_info=True)
    # 返回给客户端的只有通用的错误信息，不暴露内部细节
    return error_response(ErrorCode.INTERNAL_ERROR, http_status=500)

# 路由示例：展示不同类型错误的处理
users_db = {}

@app.route('/api/users', methods=['POST'])
def register():
    data = request.json
    # 参数缺失 → 10001
    if not data or 'email' not in data:
        return error_response(ErrorCode.PARAM_MISSING, 400,
                              detail={"field": "email", "reason": "必填"})
    # 邮箱格式 → 10002
    if '@' not in data.get('email', ''):
        return error_response(ErrorCode.PARAM_INVALID, 400,
                              detail={"field": "email", "reason": "格式不正确"})
    # 邮箱已注册 → 20002
    if any(u.get('email') == data['email'] for u in users_db.values()):
        return error_response(ErrorCode.USER_EMAIL_EXISTS, 422)
    # 成功
    user_id = str(len(users_db) + 1)
    users_db[user_id] = {"id": user_id, "email": data['email']}
    return success_response(users_db[user_id]), 201

if __name__ == '__main__':
    app.run(debug=True)
```

### 2. 字段级校验错误

对于需要告诉前端"具体哪个字段出了问题"的场景，响应中应该携带结构化的错误详情：

```json
{
  "code": 10002,
  "message": "参数格式不正确",
  "data": null,
  "detail": {
    "fields": [
      {"field": "email", "reason": "格式不正确"},
      {"field": "password", "reason": "长度不能少于 6 位"},
      {"field": "age", "reason": "必须在 1-150 之间"}
    ]
  }
}
```

前端拿到这个 `detail.fields` 数组后，可以逐字段在表单中显示红色错误提示，用户体验非常好。

### 3. 前端统一错误处理策略

```javascript
// 前端 axios 拦截器 —— 按错误码分段处理
import axios from 'axios';
import { message } from 'antd'; // UI 提示组件

const api = axios.create({ baseURL: '/api' });

api.interceptors.response.use(
  (response) => {
    const { code, message: msg, data } = response.data;
    if (code === 0) return data; // 成功，直接返回

    // 业务错误：根据错误码分段处理
    if (code >= 10000 && code < 20000) {
      // 通用错误：通常展示给用户
      message.error(msg);
    } else if (code >= 20000 && code < 50000) {
      // 业务模块错误：展示给用户
      message.warning(msg);
    } else if (code >= 90000) {
      // 系统级错误：展示通用提示 + 上报监控
      message.error('系统繁忙，请稍后重试');
      reportToMonitoring({ code, msg }); // 上报监控系统
    }
    return Promise.reject({ code, message: msg });
  },
  (error) => {
    // HTTP 层错误（网络断开、超时、5xx 等）
    if (!error.response) {
      message.error('网络连接失败，请检查网络');
    } else if (error.response.status >= 500) {
      message.error('服务器开小差了，请稍后重试');
    }
    return Promise.reject(error);
  }
);
```

有了这个拦截器，90% 的错误处理都在拦截器中完成了。业务代码只需要关注正常流程：

```javascript
async function submitForm() {
  try {
    const user = await api.post('/users', formData);
    message.success('注册成功！');
  } catch (err) {
    // 大部分错误已被拦截器处理
    // 这里只处理需要特殊逻辑的错误（如不跳转而是显示在特定位置）
    if (err.code === 20002) {
      emailField.setError(err.message); // 在邮箱输入框下方显示错误
    }
  }
}
```
## 总结
- 错误码应该按模块分段设计（5 位数字），让开发者一看错误码就知道错误来源。
- 全局异常处理器捕获未处理的异常，向客户端返回通用错误，向服务器日志记录详细信息。
- 字段级校验错误应携带 `detail.fields` 数组，让前端能逐字段展示错误提示。
- 前端拦截器按错误码范围分段处理：通用错误弹窗、业务错误就地展示、系统错误上报监控。
- 错误信息对用户友好（不说"NullPointerException"），对开发者详细（日志里记录完整堆栈）。
## 注意事项
- 绝对不要向客户端返回堆栈信息、数据库查询语句、服务器文件路径。这是严重的安全漏洞。
- 错误响应也要遵循统一的响应格式（`{code, message, data}`），不要单独返回 `{"error": "xxx"}`。
- 生产环境的错误日志要记录到持久化存储（文件或日志服务），不要只输出到控制台。
- HTTP 状态码和业务错误码要配合使用：422 配合 `USER_EMAIL_EXISTS`，而不是 200 配合 `USER_EMAIL_EXISTS`。
## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 暴露堆栈信息 | 用户看到满屏 Python 报错 | 没做全局异常捕获 | 全局异常处理器统一捕获，返回通用错误信息 |
| 错误码随意编 | 开发者每次自己编，结果 10001 被用了三次 | 没有错误码规范文档 | 建立错误码分段表，在项目中统一维护 |
| 成功和错误格式不一致 | 前端要写两套解析逻辑 | 成功返回 `{data}`，错误返回 `{error}` | 统一为 `{code, message, data}` 信封，错误时 data 为 null |
| 忽略前端体验 | 后端返回"ORA-00001: unique constraint violated" | 直接把数据库报错透传给前端 | 捕获数据库异常，映射为业务错误码和友好提示 |
