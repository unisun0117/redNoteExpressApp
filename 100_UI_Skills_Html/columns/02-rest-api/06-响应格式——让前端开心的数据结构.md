# REST API 设计从0到1 第6篇：响应格式：让前端开心的数据结构
@[TOC](目录)
## 摘要
> API 返回的数据结构直接影响前端的开发体验。一个好的响应格式应该统一、清晰、易解析、可扩展。本文从前端视角出发，教你设计标准的 JSON 响应结构，让前后端协作丝般顺滑。
## 引言
你有没有遇到过这样的情况：同一个项目的不同接口返回格式五花八门——有的返回 `{data: []}`、有的直接返回数组 `[]`、有的返回 `{result: [], status: 1}`、有的干脆返回 `{code: 0, msg: "ok", payload: {...}}`。前端每接一个新接口都要重新看文档、重新写解析逻辑。

这不是技术问题，而是规范问题。当一个 API 的响应格式是统一的，前端可以写一套拦截器（interceptor）来处理所有接口的返回。可以自动提取数据、自动处理错误提示、自动记录日志。反之，格式不统一意味着前端需要为每个接口写定制化的处理逻辑。

本篇文章将给你一套经过实战检验的 JSON 响应格式规范，包括成功响应、错误响应、列表响应，以及相关的命名约定。
## 基础知识储备
- **JSON 数据格式**：理解对象 `{}` 和数组 `[]` 的区别，知道如何嵌套和组合。
- **前端状态管理基础**：知道前端收到响应后要更新 UI 状态，统一格式可以让状态更新的逻辑复用。
- **axios/fetch 拦截器概念**：了解何为请求拦截器、响应拦截器，以及它们的典型用途。
- **HTTP 状态码基础**：知道 200、201、400、401、404、500 的基本含义。
- **数据类型规范**：理解为什么返回字段的类型应该稳定（不要把数字偶尔返回成字符串）。
## 正文
### 1. 统一响应信封的设计

API 的所有响应都应该包裹在一个"信封"（envelope）中。这个信封包含三个核心字段：**code**（业务状态码）、**message**（人类可读的提示信息）、**data**（实际数据）。

```json
// 成功响应
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1,
    "name": "张三",
    "email": "zhangsan@example.com"
  }
}

// 列表响应
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [...],
    "total": 150,
    "page": 1,
    "size": 10
  }
}

// 错误响应
{
  "code": 40001,
  "message": "邮箱格式不正确",
  "data": null
}
```

这个"信封"结构有几个好处：前端可以统一从 `response.data` 取数据，不需要知道是单个对象还是数组。错误时 `data` 为 null，`code` 和 `message` 提供具体的错误信息。最重要的是，前端可以写一个统一的拦截器来处理所有接口：

```javascript
// 前端统一拦截器示例（使用 axios）
import axios from 'axios';

const apiClient = axios.create({
  baseURL: '/api/v1',
  timeout: 10000,
});

// 响应拦截器：统一处理
apiClient.interceptors.response.use(
  (response) => {
    const { code, message, data } = response.data;
    if (code === 0) {
      // 成功：直接返回 data，业务代码不需要访问 data.data
      return data;
    } else {
      // 业务错误：统一弹窗提示
      alert(message || '操作失败');
      return Promise.reject(new Error(message));
    }
  },
  (error) => {
    // 网络错误或 HTTP 错误
    if (error.response) {
      const status = error.response.status;
      if (status === 401) window.location.href = '/login';
      else if (status === 403) alert('无权限');
      else if (status >= 500) alert('服务器错误，请稍候');
    }
    return Promise.reject(error);
  }
);

// 使用 —— 业务代码超级简洁
const user = await apiClient.get('/users/1');  // 直接拿到 data
const list = await apiClient.get('/users');     // 直接拿到 data（含分页信息）
```

有了统一拦截器后，每个业务接口的调用代码只需要一行，错误处理全部由拦截器统一完成。这就是统一响应格式的价值。

### 2. 字段命名与数据类型约定

**命名风格**：JSON 字段统一使用 camelCase（驼峰命名）。虽然 Python 后端习惯用 snake_case，但前端 JavaScript 的世界里 camelCase 才是主流。如果后端返回 snake_case，前端就得写 `data.user_name` 而不是 `data.userName`，这会让前端同事很不舒服。

**类型稳定**：一个字段的类型在整个 API 生命周期中不应该变化。如果 `id` 今天是 `1`（数字），明天就不能变成 `"1"`（字符串）。数字和字符串在前端 JavaScript 中是两种完全不同的东西：`1 === "1"` 是 false。

**避免 null 滥用**：对于列表字段，返回空数组 `[]` 而不是 `null`。对于字符串字段，返回空字符串 `""` 而不是 `null`。这样前端可以安全地直接调用 `data.items.map()` 而不需要先判空。

```python
# Python 后端响应序列化示例
from flask import Flask, jsonify
from datetime import datetime

app = Flask(__name__)

def success_response(data, message="success"):
    """统一成功响应"""
    return jsonify({
        "code": 0,
        "message": message,
        "data": data
    })

def error_response(code, message, http_status=400):
    """统一错误响应"""
    response = jsonify({
        "code": code,
        "message": message,
        "data": None
    })
    response.status_code = http_status
    return response

def paginated_response(items, total, page, size):
    """分页列表响应"""
    return success_response({
        "items": items,
        "total": total,
        "page": page,
        "size": size
    })

# 序列化函数：将数据库对象转为 JSON 兼容的字典
def serialize_user(user):
    return {
        "id": user["id"],
        "name": user["name"],
        "email": user["email"],
        # 时间统一返回 ISO 8601 字符串格式
        "createdAt": user.get("created_at", datetime.now()).isoformat() + "Z",
    }

@app.route('/api/users/<int:user_id>')
def get_user(user_id):
    user = {"id": user_id, "name": "张三", "email": "test@example.com"}
    return success_response(serialize_user(user))

@app.route('/api/users')
def list_users():
    users = [{"id": 1, "name": "张三"}, {"id": 2, "name": "李四"}]
    return paginated_response(
        items=[serialize_user(u) for u in users],
        total=len(users),
        page=1,
        size=10
    )

if __name__ == '__main__':
    app.run(debug=True)
```

### 3. 时间、金额等特殊字段的处理

时间字段在 API 中有两种常见做法：Unix 时间戳（`1700000000`）和 ISO 8601 字符串（`"2024-01-01T00:00:00Z"`）。推荐使用 ISO 8601，因为它可读性强，前端 `new Date()` 可以直接解析，不需要猜"这个时间戳是秒还是毫秒"。

金额字段务必用整数（单位：分）或字符串（避免浮点数精度问题）。`0.1 + 0.2 = 0.30000000000000004` 是一个经典的 JavaScript 坑，如果金额用浮点数传递，累积计算后就会出现一分钱的差异。所以金额用分为单位的整数，或者用字符串表示小数：`"99.99"`。

枚举字段不要用中文，用英文常量。`"status": "active"` 比 `"status": "已激活"` 更好，因为前端可能需要根据枚举值做逻辑判断，中文容易因为编码不一致导致匹配失败。
## 总结
- 统一"信封"结构 `{code, message, data}`，让前端可以写一个拦截器处理所有接口。
- 业务错误码（code）和 HTTP 状态码各司其职：HTTP 状态码管协议层，code 管业务层。
- camelCase 是 JavaScript 世界的标准，JSON 字段命名应遵循前端习惯。
- 时间用 ISO 8601 格式，金额用整数（分）或字符串，避免浮点数精度问题。
- 列表字段始终返回空数组 `[]` 而不是 null，让前端安全地直接遍历。
## 注意事项
- 不要在不同接口中对同一实体使用不同的字段名。用户的名字统一叫 `name`，不要有的接口叫 `userName`，有的叫 `username`，有的叫 `name`。
- 响应数据不要太"裸"。即使只有一个字段，也放在对象里：`{"name":"张三"}` 而不是直接返回 `"张三"`。保留扩展空间。
- 错误响应中 `data` 应该是 `null`，不要返回空字符串 `""` 或空对象 `{}`。null 的语义最明确。
- 避免在响应中暴露内部数据库字段（如密码哈希、自增 id 的原始值、内部状态标记等）。
## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| snake_case 字段名 | 前端代码中 `data.user_name` 和 `data.userName` 混用 | 后端 Python 习惯和前端 JavaScript 习惯不一致 | JSON 字段统一 camelCase，后端序列化时做转换 |
| 列表返回 null | 前端调用 `data.items.map()` 直接报错崩溃 | 空列表返回了 null | 列表字段永远返回 `[]` 而不是 null |
| 金额用浮点数 | 账单总额总是差几分钱，怎么也查不出原因 | 0.1 + 0.2 != 0.3 的浮点数精度问题 | 金额用分为单位的整数（4999 表示 49.99 元） |
| 时间戳格式不一致 | 有时是秒级有时是毫秒级，前端解析出差一个月的时间 | 没有约定时间传输格式 | 统一使用 ISO 8601 字符串：`"2024-01-01T00:00:00Z"` |
