## 摘要
> 从React组件的Props类型联想到数据库字段类型，手把手教你创建第一张数据库表。涵盖常用数据类型、主键、非空、唯一约束，配有可运行的SQL代码。

## 引言

上一篇我们聊了为什么需要数据库。如果你是一个React开发者，你定义的每个组件都有Props，Props有类型——`string`、`number`、`boolean`、`array`。TypeScript更是把类型写到了脸上。数据库的字段类型，本质上就是这些"Props类型"的数据库版本。

想象一下你在React里定义一个用户组件：

```typescript
interface UserProps {
  id: number;
  name: string;
  email: string;
  age?: number;
  isActive: boolean;
}
```

这个interface其实就是一张"数据库表"的雏形。字段名对应列名，类型对应数据库字段类型。建表就是把TypeScript的interface翻译成SQL的CREATE TABLE语句。

本文带你创建人生中第一张数据库表，理解字段类型和约束的意义——这是数据库设计最基础也最重要的技能。

## 基础知识储备

1. **看懂TypeScript类型定义**：知道`string`、`number`、`boolean`分别代表什么。
2. **理解"唯一标识"的概念**：React用`key`来识别列表中的每个元素，数据库用主键来识别每条记录，思想一模一样。
3. **用过Excel**：每个Sheet就是一个"表"，列有列名（字段名），列有格式（数字、文本、日期）——字段类型就是这个"格式"。
4. **SQLite已就绪**：Python自带sqlite3模块，打开终端输入`python -c "import sqlite3"`不报错就行。

## 正文

### 一、字段类型：给每个"Props"指定种族

数据库的字段类型决定了这一列能存什么数据。以下是SQLite最常用的类型，附前端类比：

| SQLite类型 | 含义 | 前端类比 | 示例 |
|-----------|------|----------|------|
| INTEGER | 整数 | `number`（TypeScript） | `age INTEGER` |
| REAL | 浮点数 | `number`（JS的float） | `price REAL` |
| TEXT | 字符串 | `string` | `name TEXT` |
| BLOB | 二进制（图片、文件） | `Blob` / `ArrayBuffer` | `avatar BLOB` |
| NUMERIC | 精确数字（金额） | `BigDecimal` | `balance NUMERIC` |

SQLite有一个特别之处：它的类型系统是"动态类型"的。也就是说，你在TEXT列里存一个数字，SQLite不会报错——它会自动转换。但**这不代表你应该乱存**，好的表设计一定是类型明确、严格约束的，就像TypeScript的严格模式一样。

### 二、建表实战：CREATE TABLE 全解

假设我们要做一个图书管理系统，建一张`books`表：

```sql
CREATE TABLE books (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    author TEXT NOT NULL DEFAULT '佚名',
    price REAL CHECK(price > 0),
    created_at TEXT DEFAULT (datetime('now'))
);
```

逐行解释：

- `id INTEGER PRIMARY KEY AUTOINCREMENT` — 主键，自增。每条记录的唯一标识，就像React列表的`key`。
- `title TEXT NOT NULL` — 书名不能为空。就像组件Props里写`required: true`。
- `author TEXT NOT NULL DEFAULT '佚名'` — 作者默认值。作者未知时自动填"佚名"。
- `price REAL CHECK(price > 0)` — 价格必须大于0。这是约束条件，防止脏数据。
- `created_at TEXT DEFAULT (datetime('now'))` — 自动记录创建时间。

在Python中执行这段SQL：

```python
import sqlite3

conn = sqlite3.connect('library.db')
cursor = conn.cursor()

cursor.execute('''
    CREATE TABLE IF NOT EXISTS books (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        title TEXT NOT NULL,
        author TEXT NOT NULL DEFAULT '佚名',
        price REAL CHECK(price > 0),
        created_at TEXT DEFAULT (datetime('now'))
    )
''')

conn.commit()
conn.close()
print("表创建成功！")
```

### 三、约束：数据库的"业务规则引擎"

约束（Constraint）是数据库的自我防卫机制。它相当于你在前端写的表单校验，但更加可靠——因为数据库的约束根本无法绕过。

**常用约束一览：**

| 约束 | SQL写法 | 作用 | 前端类比 |
|------|---------|------|----------|
| PRIMARY KEY | `id INTEGER PRIMARY KEY` | 唯一标识每条记录 | React的`key` prop |
| NOT NULL | `name TEXT NOT NULL` | 不允许空值 | `required: true` / `!` |
| UNIQUE | `email TEXT UNIQUE` | 值不能重复 | 注册时检查"用户名已被占用" |
| DEFAULT | `role TEXT DEFAULT 'user'` | 默认值 | `defaultProps` |
| CHECK | `age INTEGER CHECK(age >= 0)` | 条件约束 | `if (age < 0) return error` |
| FOREIGN KEY | `user_id INTEGER REFERENCES users(id)` | 外键关联 | 父子组件的数据联动 |

一个综合示例——用户表：

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,
    email TEXT NOT NULL UNIQUE,
    age INTEGER CHECK(age >= 0 AND age <= 150),
    role TEXT NOT NULL DEFAULT 'user' CHECK(role IN ('user', 'admin', 'moderator')),
    created_at TEXT DEFAULT (datetime('now'))
);
```

注意`role`字段的CHECK约束——用`IN`限制可选值，这跟TypeScript的联合类型异曲同工：

```typescript
type Role = 'user' | 'admin' | 'moderator';  // TypeScript版
CHECK(role IN ('user', 'admin', 'moderator')) -- SQL版
```

## 总结

1. 数据库字段类型和TypeScript类型本质相同——都是描述"这数据长什么样"。
2. CREATE TABLE是数据库设计的基本功，每列的类型和约束都代表一条业务规则。
3. PRIMARY KEY是每张表的必备要素，AUTOINCREMENT让数据库自动生成唯一ID。
4. 约束（NOT NULL、UNIQUE、CHECK、FOREIGN KEY）是数据库的"前端表单校验"，但比前端校验更可靠。
5. 建表之前先想清楚业务规则——就像写React组件之前先设计Props interface一样。

## 注意事项

1. 不要用"预留字段"—— `field1 TEXT, field2 TEXT` 这种设计是技术债务的源头。字段名要有业务含义，一个表就是一个明确的数据实体。
2. TEXT类型可以存任何字符串，但不要把数字、日期一股脑全存TEXT。金额用NUMERIC，日期用TEXT（ISO格式），让类型承担"文档"的职责。
3. CHECK约束不要太复杂——它的作用是简单的边界校验，复杂的业务校验应该放在应用层（Python/JS）。

## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 忘记设PRIMARY KEY | 无法确定更新或删除的是哪条记录 | 没有唯一标识 | 每张表必有主键，强烈推荐用自增INTEGER |
| 过度使用NOT NULL | 插入数据频繁报错，因为总是少传某个字段 | 把所有字段都设成了NOT NULL | 只对真正不能为空的字段设NOT NULL，可选字段保持可为空 |
| SQLite类型不匹配不报错 | 把"hello"存到了INTEGER列，没报错 | SQLite是动态类型，不做严格校验 | 在应用层（Python）做类型校验，或使用PostgreSQL等强类型数据库 |
| CHECK约束写成复杂子查询 | 慢得离谱 | CHECK中的子查询每行都要执行一次，1000行就执行1000次 | CHECK只做简单条件判断，复杂校验放应用层 |
