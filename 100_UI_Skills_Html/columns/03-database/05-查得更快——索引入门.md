## 引言

如果你要在React里从一个10000条数据的数组中快速查找某个ID的用户，你会怎么做？直接`users.find(u => u.id === targetId)`——那是O(n)的遍历，10000次比较。聪明一点的做法是用Map或对象建立索引：

```javascript
// O(n)遍历
const user = users.find(u => u.id === 9999);

// O(1)哈希查找 —— 这就是"索引"的前端版
const userMap = new Map(users.map(u => [u.id, u]));
const user = userMap.get(9999);
```

数据库的索引，原理上跟你用Map建立快速查找结构一模一样。但数据库的索引比Map更复杂、更强大——它基于B+树数据结构，支持范围查询、排序优化，而不仅仅是精确匹配。

本文带你入门数据库索引，理解它的原理、学会什么时候建索引、避免最常见的索引用法错误。

## 正文

### 一、索引是什么？—— 数据库的"目录"

想象一本1000页的工具书，没有目录（索引）。你想找"B+树"这个词，怎么办？从第1页开始一页一页翻，直到找到。这就是全表扫描（Full Table Scan）。

有了索引，数据库维护一个额外的数据结构（通常是B+树），存着"关键字→位置"的映射。查询时先走这个结构快速定位，再直接取数据。

**索引用空间换时间**——它占用额外的磁盘空间，每次INSERT/UPDATE/DELETE还要维护索引，但换来了查询速度几十倍甚至上千倍的提升。

### 二、动手实验：有索引 vs 无索引

我们来做一个100万条数据的对比实验，直观感受索引的威力。

```python
import sqlite3
import time

conn = sqlite3.connect('index_demo.db')
cursor = conn.cursor()

# 创建测试表
cursor.execute('''
    CREATE TABLE IF NOT EXISTS products (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        name TEXT NOT NULL,
        price REAL NOT NULL
    )
''')

# 插入100万条数据（首次运行需要几秒钟）
print("检查数据量...")
cursor.execute("SELECT COUNT(*) FROM products")
count = cursor.fetchone()[0]

if count < 1000000:
    print("插入100万条测试数据中...")
    data = []
    for i in range(count, 1000000):
        data.append((f'商品{i}', round(i * 0.5 + 10, 2)))
        if len(data) >= 100000:
            cursor.executemany(
                'INSERT INTO products (name, price) VALUES (?, ?)', data
            )
            conn.commit()
            data = []
            print(f"  已插入 {i+1} 条...")
    if data:
        cursor.executemany(
            'INSERT INTO products (name, price) VALUES (?, ?)', data
        )
        conn.commit()
    print("数据插入完毕！")

# 对比实验：查询price > 500的商品数量
# 实验1：无索引
start = time.time()
cursor.execute("SELECT COUNT(*) FROM products WHERE price > 500")
result = cursor.fetchone()[0]
no_index_time = time.time() - start
print(f"\n无索引查询：{result} 条，耗时 {no_index_time:.3f} 秒")

# 实验2：建索引
print("创建索引...")
cursor.execute("CREATE INDEX IF NOT EXISTS idx_price ON products(price)")
conn.commit()

# 实验3：有索引
start = time.time()
cursor.execute("SELECT COUNT(*) FROM products WHERE price > 500")
result = cursor.fetchone()[0]
index_time = time.time() - start
print(f"有索引查询：{result} 条，耗时 {index_time:.3f} 秒")

print(f"\n速度提升了 {no_index_time / index_time:.1f} 倍！")

conn.close()
```

在我的机器上，100万数据无索引查询约0.08秒，有索引约0.001秒——**快了80倍**。数据量越大，差距越明显。

### 三、什么时候建索引？

不是每列都需要索引。建索引就像在React里给每个数组都建Map缓存——浪费内存。建索引的黄金法则：

**应该建索引的列：**

1. **WHERE条件里的列** — `WHERE email = ?`，email列应该建索引。
2. **ORDER BY的列** — `ORDER BY created_at DESC`，created_at列应该建索引。
3. **JOIN的外键列** — `JOIN users ON posts.user_id = users.id`，posts.user_id应该建索引。
4. **经常组合查询的多列** — `WHERE status = ? AND category_id = ?`，建联合索引。

**不应该建索引的场景：**

- 表很小（少于1000行）——全表扫描可能更快。
- 频繁更新但很少查询的列——索引维护成本高于收益。
- 高重复率的列（如性别、布尔值）——区分度太低，索引没啥用。

### 四、联合索引与最左前缀原则

当WHERE条件涉及多个字段时，可以建联合索引：

```sql
CREATE INDEX idx_status_category ON orders(status, category_id);
```

使用时有一个重要规则——**最左前缀原则**：联合索引`(A, B, C)`可以加速以下查询：
- `WHERE A = ?` —— 走索引
- `WHERE A = ? AND B = ?` —— 走索引
- `WHERE A = ? AND B = ? AND C = ?` —— 走索引
- `WHERE A = ? AND C = ?` —— 走索引（但只用了A部分）

但以下查询**不走这个联合索引**：
- `WHERE B = ?` —— 跳过了最左边的A
- `WHERE C = ?` —— 跳过了A和B

这就像React中的路由嵌套——你得先有父路由，子路由才能匹配。

```sql
-- 用EXPLAIN QUERY PLAN看是否走了索引
EXPLAIN QUERY PLAN
SELECT * FROM orders WHERE status = 'paid' AND category_id = 3;
-- 输出中看到"USING INDEX"就是用了索引
```

## 总结

1. 索引就像字典目录——先查目录再翻书，比一页页翻快得多。本质是空间换时间的策略。
2. 在WHERE、ORDER BY、JOIN涉及的列上建索引——这几列在查询计划中出现的频率最高。
3. 不要在"所有列"上建索引——索引不是免费的，每次写操作都要维护，建多了反而拖慢写入。
4. 联合索引遵循"最左前缀原则"——列的顺序有讲究，排前面的列要能独立使用这个索引。
5. 用EXPLAIN QUERY PLAN验证索引是否生效——不要猜，要让数据库告诉你它用了哪个索引。

## 注意事项

1. 主键自带索引（PRIMARY KEY自动就是索引），不需要额外再建。
2. SQLite和MySQL的索引行为基本一致，但PostgreSQL的索引类型更多（GIN、GiST等）。本文讲的B+树索引用法在所有关系型数据库里通用。
3. 不要在生产环境频繁创建和删除索引——索引的创建对大表来说是个重操作，会锁表。应该在表设计阶段就规划好索引策略。

## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 对低区分度列建索引 | 建了索引查询也没变快 | 性别只有男/女，数据库觉得用索引不划算，依然全表扫描 | 不要在重复率高的列上单独建索引，可以放在联合索引的后面 |
| WHERE中用函数包裹索引列 | `WHERE LOWER(email) = ?` 不走索引 | 函数破坏了索引数据与查询条件的对应关系 | 存数据时就统一格式，或者在表达式索引上建索引（PostgreSQL支持） |
| 联合索引列顺序不当 | 常用的查询条件不走索引 | 联合索引(A,B)对`WHERE B=?`无效，违反最左前缀 | 最常用、区分度最高的列放在联合索引的最左边 |
| 以为建了索引就万事大吉 | 查询计划显示依然全表扫描 | 数据量差不多、或者查询条件用OR连接导致索引无法使用 | EXPLAIN QUERY PLAN是你的好帮手，查了再说 |
