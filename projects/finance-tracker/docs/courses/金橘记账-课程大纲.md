## Lab 1-12：全栈项目实战

### Lab 1：项目认知 — 我们要做什么，为什么这么做

- **📌 前言**：大四那年我帮室友做毕设，发现大多数人的选题都差不多——电商、论坛、博客。我说"你做个记账App吧，小而美，评审老师一看就觉得实用"。结果他真的拿了优秀。这一章，我们先不写代码，先搞清楚"金橘记账"解决了什么问题，为什么温暖治愈的风格比冷冰冰的财务软件更适合个人使用。
- **📚 基础知识储备**：Web 应用基本架构（前端/后端/数据库）、全栈项目开发流程、什么是 MVP
- **🎯 实战目标**：理解项目定位与核心功能；在脑海中建立完整的系统架构图；知道 15 个 Lab 做完后你能得到什么
- **🛠 动手实战**：浏览项目原型页面 → 理解5个核心模块（认证/账单/预算/统计/个人中心）→ 画出系统架构图 → 注册 GitHub 仓库
- **⚠️ 常见坑点**：新手一上来就写代码，写到一半发现方向错了——先花一小时想清楚再动手，比你写两天又推翻重来快得多
- **📝 总结**：心里有地图，手上才不慌。下一个 Lab 我们把开发环境搭起来。

### Lab 2：环境搭建 — Windows 上的全栈开发环境

- **📌 前言**：我见过最多的新手放弃时刻，不是代码写不出来，是环境配不通。node-gyp 报错、Python 版本不对、pip install 红一片——这些我都经历过。这个 Lab 我把每一步都拆开，你照着做就能跑通。
- **📚 基础知识储备**：Node.js 与 npm、Python 虚拟环境（venv）、Git Bash 基础、VS Code 配置
- **🎯 实战目标**：安装 Node.js 22+ & Python 3.10+；用 Vite 创建 React + TypeScript 项目；用 venv 创建 FastAPI 项目；两个项目都能在本地跑起来
- **🛠 动手实战**：安装 Node.js → `npm create vite@latest` 初始化前端 → 安装 antd-mobile + react-router-dom → 创建 Python venv → `pip install fastapi uvicorn sqlalchemy` → 写第一个 `/api/health` 端点 → 前后端同时启动验证
- **⚠️ 常见坑点**：Windows 上 venv 的激活是 `.\venv\Scripts\activate` 不是 `source venv/bin/activate`；npm 安装慢用淘宝镜像；Python 版本别用 3.13（部分包不兼容）
- **📝 总结**：环境通了，项目就跑了一半。下一个 Lab 我们开始写后端。

### Lab 3：后端基石 — FastAPI 项目结构与配置管理

- **📌 前言**：FastAPI 是我用过最舒服的 Python 框架——自动生成 API 文档、类型提示友好、异步支持好。但我第一次用的时候，项目结构乱成一锅粥，所有代码堆在 `main.py` 里。这个 Lab 我教你正确的目录结构。
- **📚 基础知识储备**：FastAPI 路由机制、pydantic-settings 配置管理、Python 模块导入、RESTful API 设计原则
- **🎯 实战目标**：搭建 FastAPI 标准项目结构；用 pydantic-settings 管理环境变量；配置 CORS 中间件；写出 `/api/health` 和 Swagger 文档
- **🛠 动手实战**：创建 `app/` 目录结构（main / config / database / models / routers / middleware）→ 编写 config.py → 编写 database.py（SQLAlchemy engine + session）→ 注册路由 → 启动并访问 `/docs`
- **⚠️ 常见坑点**：`from app.xxx` 导入失败——确认工作目录在 `backend/` 下；CORS 配置别忘了，否则前端请求全被拦
- **📝 总结**：好的项目结构，让你三个月后回来看代码还能秒懂。下一个 Lab 我们设计数据库。

### Lab 4：数据库设计 — SQLAlchemy ORM 三张表

- **📌 前言**：我第一次设计数据库的时候，把所有字段都设成 VARCHAR(255)，连金额都是字符串。后来做统计功能的时候肠子都悔青了——"为什么我不用 INT 存金额？"这个 Lab 我们一次性把表结构设计对。
- **📚 基础知识储备**：关系型数据库基础（表/字段/主键/外键）、SQLAlchemy 2.0 Mapped 语法、金额单位设计（分 vs 元）
- **🎯 实战目标**：设计 User / Transaction / Budget 三张表；理解"金额存分为单位"的原因；用 `Base.metadata.create_all` 自动建表
- **🛠 动手实战**：编写 User 模型（id/uuid, email, password_hash, nickname, avatar）→ Transaction 模型（金额 int 存分, 外键关联 user）→ Budget 模型（按月 + 分类）→ 在 main.py 启动时自动建表
- **⚠️ 常见坑点**：SQLite 外键默认不开启——需要加 `ForeignKey` 约束；DateTime 用 `datetime.utcnow` 不要用 `datetime.now`（时区问题）；金额千万不要用 float（浮点精度丢失）
- **📝 总结**：数据库设计是地基，地基歪了后面全歪。下一个 Lab 我们实现用户注册登录。

### Lab 5：用户认证（后端）— JWT 双 Token 体系

- **📌 前言**：认证是每个全栈项目的必经之路。我当时搞 JWT 的时候，access_token 和 refresh_token 的关系想了半天：为什么需要两个 token？直接一个不过期不行吗？这个 Lab 不仅教你实现，还讲清楚"为什么"。
- **📚 基础知识储备**：HTTP 无状态 & 鉴权方式（Cookie vs Token）、JWT 结构（Header.Payload.Signature）、bcrypt 密码哈希、Bearer Token 规范
- **🎯 实战目标**：实现 `/api/auth/register` + `/api/auth/login` + `/api/auth/refresh` + `/api/auth/me` 四个接口；理解 access_token（15分钟过期）和 refresh_token（7天过期）的配合逻辑；用 `get_current_user` 依赖注入保护接口
- **🛠 动手实战**：编写 auth middleware（decode_token + get_current_user）→ 实现 register 接口（邮箱去重 + bcrypt 哈希）→ 实现 login 接口（密码验证 + 签发双 token）→ 实现 refresh 接口（用 refresh_token 换新 access_token）→ 实现 me 接口（返回当前用户信息）
- **⚠️ 常见坑点**：`bcrypt==4.0.1` 必须是这个版本，5.x 跟 passlib 不兼容；`jose` 库的 `jwt.decode` 参数顺序；refresh_token 作为 query param 传递时的安全问题（生产环境建议放 body）
- **📝 总结**：认证做完了，你的 API 就有了"门禁系统"。下一个 Lab 我们给前端加上登录页面。

### Lab 6：登录注册页（前端）— React Context + Token 管理

- **📌 前言**：前端认证是我刚学 React 时最头疼的事——token 存哪？怎么自动带？过期了怎么办？这个 Lab 我把整个认证流程拆成三块：存储、请求拦截、UI，你跟着做一次就彻底懂了。
- **📚 基础知识储备**：React Context API、Custom Hooks（useAuth）、localStorage 安全考量、401 拦截 & 自动刷新机制
- **🎯 实战目标**：实现 LoginPage（登录 + 注册双 tab）；封装 `api.ts`（authFetch 自动带 token + 401 自动刷新）；实现 `useAuth` Context（login / register / logout / refreshUser）；实现 PrivateRoute 路由守卫
- **🛠 动手实战**：编写 `api.ts`（BASE_URL + setTokens/clearTokens + authFetch 401拦截）→ 编写 `useAuth.tsx`（AuthProvider + Context）→ 编写 LoginPage（邮箱/密码输入 + 登录/注册切换 + 密码显隐切换）→ 在 App.tsx 中配置 PrivateRoute
- **⚠️ 常见坑点**：localStorage 里的 token 名字要跟 api.ts 里一致；401 刷新时出现死循环（刷新失败不要再重试）；useEffect 里调接口要注意依赖数组
- **📝 总结**：登录页面是用户的第一印象，温暖的设计从登录就要开始。下一个 Lab 我们写最核心的账单接口。

### Lab 7：账单 CRUD（后端）— 金钱数据处理的核心逻辑

- **📌 前言**：CRUD 听起来简单，但涉及钱的时候就不一样了。金额四舍五入、分类 emoji 映射、按月筛选、分页查询——每一个细节都可能是 bug 的来源。这个 Lab 我把交易模块从接口到业务逻辑全部讲透。
- **📚 基础知识储备**：RESTful CRUD 设计（POST/GET/PUT/DELETE）、SQLAlchemy 查询过滤（filter / like / contains）、分页查询（offset / limit）、Python 列表推导式
- **🎯 实战目标**：实现 5 个账单接口（列表/创建/更新/删除/分类列表）；支持按月筛选、按分类筛选、按关键词搜索；金额以"分"为单位，前端显示时转换为"元"
- **🛠 动手实战**：定义分类常量（支出8类 + 收入7类 + emoji）→ 实现 GET /api/categories → 实现 GET /api/transactions（筛选 + 分页 + 收支汇总）→ 实现 POST /api/transactions（自动匹配 emoji）→ 实现 PUT /api/transactions/:id → 实现 DELETE /api/transactions/:id
- **⚠️ 常见坑点**：`Transaction.tx_date.like("2026-06%")` 只能用于 SQLite，PostgreSQL 要用 `cast + between`；金额汇总前确认数据类型是 int 不是 str；分页 page 参数注意从 1 开始
- **📝 总结**：CRUD 是永远的基本功——增删改查写好了，剩下都是锦上添花。下一个 Lab 我们给这些接口配上前端页面。

### Lab 8：记账表单与账单列表（前端）— 用户体验细节

- **📌 前言**：记账 App 用得最多的两个页面就是记账表单和账单列表——每天打开几十次。所以我花了很多时间打磨交互：类型切换要流畅、分类选择要直观、金额输入要顺手。这个 Lab 我把这些细节都还原给你。
- **📚 基础知识储备**：React 受控组件（useState + onChange）、URL SearchParams 传参、日期分组算法、CSS-in-JS 行内样式
- **🎯 实战目标**：实现 FormPage（收支切换 + 分类网格 + 金额输入 + 日期选择 + 备注）；实现 ListPage（按日期分组 + 分类筛选 + 搜索 + 收支汇总卡片）；实现编辑和删除功能
- **🛠 动手实战**：编写 FormPage — 从 URL 读取 `?type=expense` 初始化 → 分类网格选择（高亮选中态）→ 金额输入（元转分提交）→ 日期选择 → 编辑模式（`?edit=id` 回填数据 + 删除按钮）→ 编写 ListPage — 按月加载 → 按日期分组（Object.groupBy 或手动分组）→ 分类筛选按钮组 → 搜索防抖 → 汇总卡片
- **⚠️ 常见坑点**：编辑模式下分类默认值要跟初始 type 一致；`parseFloat("0.01") * 100` 结果是 `0.999999...` 需要用 `Math.round`；日期格式前后端统一用 `YYYY-MM-DD`
- **📝 总结**：表单和列表是记账应用的核心交互，做好了用户每天都离不开。下一个 Lab 我们做数据可视化。

### Lab 9：数据看板 — Dashboard 仪表盘 + 趋势图 + 分类占比

- **📌 前言**：数据摆在那里没用，可视化才有用。我第一版 Dashboard 只显示了余额数字，用户说"不够，我想知道我钱都花哪了"。于是加了周趋势柱状图、分类占比进度条。这个 Lab 让你学会用纯 CSS 做图表——不需要任何图表库。
- **📚 基础知识储备**：SQLAlchemy 聚合查询（func.sum / group_by）、数据可视化基础（柱状图/进度条原理）、CSS 动态宽度 + 渐变
- **🎯 实战目标**：实现 `/api/dashboard`（余额 + 当月收支 + 最近5条记录）；实现 `/api/stats/trend`（最近7天每日支出）；实现 `/api/stats/category`（分类支出占比）；纯 CSS 绘制柱状图和进度条
- **🛠 动手实战**：编写 stats 路由（dashboard 聚合查询 + trend 按天循环 + category group_by）→ 前端 DashboardPage — 余额卡片（琥珀渐变背景）→ 快捷记账按钮 → 周趋势柱状图（CSS 高度动态计算）→ 分类占比进度条（可点击跳转筛选）→ 最近记录列表
- **⚠️ 常见坑点**：SQLite 的 `func.sum` 返回可能为 None 要用 `or 0` 兜底；柱状图高度计算时 max 值为 0 会导致除零——设置最小高度；日期范围用 `date.today() - timedelta(days=i)`
- **📝 总结**：数据可视化不一定要用 ECharts——纯 CSS 做的图表更轻量、更适合移动端。下一个 Lab 我们做预算管理。

### Lab 10：预算管理 — 从设定到追踪的完整闭环

- **📌 前言**：记账是"记录过去"，预算是"规划未来"。我有个朋友靠预算功能从月光族变成了每月有结余——他说"看到进度条变红的那一刻，我剁手的欲望瞬间消失了"。这个 Lab 我们实现按月设置分类预算 + 实时追踪消费进度。
- **📚 基础知识储备**：复杂数据库写入（先删后插策略）、PUT vs PATCH 接口设计、进度条颜色逻辑（绿→黄→红）、React 弹窗/Modal 交互
- **🎯 实战目标**：实现 `/api/budgets` GET（按月份读取总预算 + 各分类预算 + 已消费金额）；实现 `/api/budgets` PUT（全量更新当月预算）；前端 BudgetPage 带底部弹窗的新增/编辑分类预算功能
- **🛠 动手实战**：后端 — 预算查询（`__total__` 特殊分类存总预算）→ 分类预算关联 transactions 表计算 spent → PUT 接口"先删后插"策略 → 前端 — 总预算进度条（渐变色 + 剩余金额）→ 分类预算列表（点击编辑）→ 底部弹窗（分类选择 + 金额输入 + 保存）
- **⚠️ 常见坑点**：PUT 全量更新用"先删后插"要放在事务里；`__total__` 是魔法字符串，前后端要约定好；弹窗出现时背景页面不要滚动（`overflow: hidden`）
- **📝 总结**：预算功能是记账 App 的"用户粘性引擎"——设了预算的人，留存率高3倍。下一个 Lab 我们完善个人中心和导航。

### Lab 11：个人中心 & 导航系统 — 应用框架的最后一环

- **📌 前言**：个人中心是产品的"安静角落"——用户在这里看数据、改设置、做管理。我把个人中心叫做 App 的"底气层"：新用户可能很少打开，但老用户会特别在意。这个 Lab 我们把整个应用导航体系串起来。
- **📚 基础知识储备**：React Router v6（BrowserRouter / Routes / Route / Navigate）、底部 TabBar 导航模式、移动端 Safe Area 适配
- **🎯 实战目标**：实现 ProfilePage（用户信息卡 + 累计数据统计 + 菜单列表 + 退出登录）；实现 TabBar（5个Tab：首页/账单/记账/预算/我的）；完善 App.tsx 路由结构（PublicRoute + PrivateRoute + TabBar 显隐控制）
- **🛠 动手实战**：编写 ProfilePage — 用户头像昵称卡片 → 累计记账笔数/收入/支出三栏统计 → 数据管理菜单（全部账单/导出）→ 设置菜单（编辑资料/提醒/分类管理）→ 关于 & 退出登录 → 编写 TabBar — 固定底部 + 当前路由高亮 → App.tsx 中登录页隐藏 TabBar
- **⚠️ 常见坑点**：TabBar 用 `position: fixed` 要配合底部 padding 避免内容被遮挡；`useLocation()` 判断当前路由时注意根路径是 `/`；退出登录后要同时清除 token 和 user state
- **📝 总结**：到这一步，你的应用框架已经完整了。导航和个人中心看起来简单，但做得好需要很多小细节。下一个 Lab，我们把它部署上线。

### Lab 12：部署上线 — Vercel + PythonAnywhere 免费全栈部署

- **📌 前言**：部署听起来很玄学，但其实就是：把前端文件放到 Vercel，把后端代码放到 PythonAnywhere，配好环境变量，点个按钮。我第一次部署也紧张得要命，后来发现：部署不是技术问题，是流程问题。按步骤来，一次就通。
- **📚 基础知识储备**：Vercel 静态部署原理、PythonAnywhere 免费账户限制（无 ASGI/无外部数据库/Whitelist 审核）、环境变量配置、Git 自动部署、WSGI 桥接
- **🎯 实战目标**：前端部署到 Vercel（自动从 GitHub 拉取 + 配置 `VITE_API_URL`）；后端部署到 PythonAnywhere（WSGI 桥接 + SQLite 路径配置 + Reload 生效）；前后端联调通过
- **🛠 动手实战**：GitHub 推送项目 → Vercel 连接仓库 → 配置 Build Command（`npx vite build`）+ Output Directory（`dist`）+ 环境变量 → PythonAnywhere 创建 Web App → 配置 WSGI 文件（asgiref 桥接）→ 设置 DATABASE_URL（4个斜杠 `/`）→ Reload + 验证健康检查 → 前端改 `VITE_API_URL` 指向生产 API
- **⚠️ 常见坑点**：PythonAnywhere 免费版不支持原生 ASGI 必须 WSGI 桥接；SQLite 路径要用4个斜杠 `sqlite:////home/username/db.sqlite3`；Vercel 构建不能用 `npm run build`（tsc 会报错）要用 `npx vite build`；免费版域名是 `xxx.pythonanywhere.com` 有 Whitelist 审核
- **📝 总结**：恭喜！你的金橘记账已经上线了。把链接发给朋友，让他们扫码注册——这一刻的成就感，比任何课程都珍贵。

## Lab 13-15：进阶拓展

### Lab 13：踩坑大全 — 项目里的 20 个坑一次解决

- **📌 前言**：这个 Lab 不是讲新功能，而是把前面 12 个 Lab 里所有踩过的坑聚合在一起。每一个坑我都亲手摔进去过，希望你能绕过去。
- **📚 内容涵盖**：bcrypt 版本冲突（4.0.1 vs 5.x）、Windows venv 激活路径、金额浮点精度问题、前端 401 死循环、SQLite like 查询跨数据库兼容、PythonAnywhere WSGI 桥接、Vercel TypeScript 构建失败、CORS 配置遗漏、JWT decode 异常处理、npm install 国内网络问题……等 20+ 个坑点
- **📝 总结**：踩坑不是浪费时间，踩坑是积累经验。我把这些坑写出来，就是帮你省下自己去踩的时间。

### Lab 14：知识储备大全 — 面试高频考点 + 技术深挖

- **📌 前言**：做完项目不等于学会了所有知识。这个 Lab 我帮你把项目涉及的技术点往深处挖一层——不是为了炫技，是为了让你面试时被问到这个项目能聊得头头是道。
- **📚 内容涵盖**：JWT vs Session 认证对比（为什么选 JWT）、React Context vs Redux vs Zustand 选型分析、SQLAlchemy ORM vs 原生 SQL 取舍、React 19 新特性（Server Components / Actions）、FastAPI 异步编程（async/await + 并发）、TypeScript 类型体操（泛型/工具类型）、全栈项目的安全 Checklist、常见面试题 20 问
- **📝 总结**：项目是敲门砖，知识是底气。做完项目 + 啃完这章，这个项目你可以在简历上写"精通"。

### Lab 15：总结复盘 + 副业变现指南 — 从做完到赚钱

- **📌 前言**：课做完了，然后呢？这一章不写代码，我们聊聊"然后"。这个项目怎么放进简历？怎么在面试中讲出亮点？能不能变成副业收入？我的经验是：做项目是第一步，把项目变成故事才是变现的关键。
- **📚 内容涵盖**：简历项目描述模板（STAR 法则）、面试中如何讲你的项目（"我遇到了什么问题 → 我怎么解决的 → 结果是什么"）、GitHub 项目包装技巧（README / 截图 / Demo 链接）、副业变现路径（接毕设定制 / 出付费教程 / 做 SaaS 收订阅费）、个人品牌建设（掘金/知乎/公众号写作 + 项目案例）、我的推荐书单和学习路径
