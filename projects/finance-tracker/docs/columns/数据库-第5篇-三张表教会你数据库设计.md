# 第5篇：三张表教会你数据库设计 — 主键、外键、关联关系

> 数据库设计就三件事：找实体、定字段、建关联。PRIMARY KEY = React 的 `key={item.id}`，FOREIGN KEY = 跨数组的 `{ userId: "abc" }` 引用

## 📌 前端为什么要学这个？

我第一版金橘记账把数据全塞一张表里。

怎么塞的？`user_info`、`transactions`、`budgets` 全混在一张 30 多个字段的大表里。查个"用户本月花了多少钱"，SQL 写了 50 多行，还经常漏数据。面试官看了一眼设计图，问我："你学过数据库设计吗？"我当时脸都红了。

那之后我重新学了数据库三范式，突然发现——**设计数据库跟设计 React 组件一模一样**。

你写 React 不会把所有逻辑塞进一个 `<App />` 组件吧？你会拆成 `<Header />`、`<Sidebar />`、`<Content />`，每个组件职责单一、通过 props 通信。数据库设计也是这一套：每个表就是一个"组件"，外键就是组件的 `props`，主键就是 `key={item.id}`。

今天用金橘记账的三张真实表——User、Transaction、Budget——把数据库设计彻底讲清楚。看完你就能设计自己的数据库。

## 🔍 核心原理

数据库设计有一个铁律：**"一个事实一处存"**。翻译成前端话就是——别让同一份数据出现在两个地方，否则改一个忘了另一个，数据就不一致了。

金橘记账的三张表，关系长这样：

```
        ┌─────────────┐
        │    User     │  ← 主表：用户
        │─────────────│
        │ id (PK)     │──┐
        │ email       │  │
        │ nickname    │  │  一对多：一个用户有多笔账单
        │ avatar      │  │
        └─────────────┘  │
                         │
              ┌──────────┼──────────┐
              │          │          │
              ▼          ▼          ▼
    ┌─────────────┐   ┌─────────────┐
    │ Transaction │   │ Transaction │  ← 子表：账单
    │─────────────│   │─────────────│
    │ id (PK)     │   │ id (PK)     │
    │ user_id (FK)│   │ user_id (FK)│  ← 外键，指向 User.id
    │ amount      │   │ amount      │
    │ category    │   │ category    │
    └─────────────┘   └─────────────┘

              ┌──────────┼──────────┐
              │          │          │
              ▼          ▼          ▼
    ┌─────────────┐   ┌─────────────┐
    │   Budget    │   │   Budget    │  ← 子表：预算
    │─────────────│   │─────────────│
    │ id (PK)     │   │ id (PK)     │
    │ user_id (FK)│   │ user_id (FK)│  ← 外键，指向 User.id
    │ month       │   │ month       │
    │ category    │   │ category    │
    │ amount      │   │ amount      │
    └─────────────┘   └─────────────┘
```

看懂了吗？就三个概念：

**1. 主键（PRIMARY KEY）= React 的 `key={item.id}`**

你写列表渲染的时候，React 让你加 `key`，因为 React 要用它来判断"哪个元素变了、哪个没变"。数据库也一样——主键是每一行数据的"身份证号"，数据库用它来精确找到某一条记录。

```jsx
// 你每天写的 React
{users.map(user => <UserCard key={user.id} data={user} />)}
```

```sql
-- 数据库每天都在做
SELECT * FROM transactions WHERE id = 'abc-123';
--                        ↑ 用主键精确找到那一条
```

没有主键？表里 10 万条数据，你想找"一条"就变成全表扫描——和 `array.find()` 遍历 10 万个元素一个速度。有主键呢？数据库直接定位，O(1) 的事。

主键通常用两样东西：`uuid`（像 `'a3b2c1d4-...'` 这种随机字符串）或自增整数 `serial`。金橘记账用 uuid —— 为什么？因为分布式系统里两个服务器同时生成自增 ID 会撞车，uuid 在任何机器上生成都是唯一的。

**2. 外键（FOREIGN KEY）= 跨数组引用 `{ userId: "abc" }`**

你写前端的时候经常这样引用跨数组的数据：

```typescript
const users = [
  { id: "u1", nickname: "阿珊" }
];

const transactions = [
  { id: "t1", amount: 2500, userId: "u1" },  // ← userId 指到 users 数组
  { id: "t2", amount: 3800, userId: "u1" },
];

// "这个账单是谁记的？"
const tx = transactions[0];
const owner = users.find(u => u.id === tx.userId);
```

`userId: "u1"` 就是一个"前端版的外键"。它不存完整用户对象，只存一个 id，需要的时候通过 id 去 `users` 数组里找。数据库的外键做的是一模一样的事——`user_id` 列只存 User 的 id，需要用户信息的时候 `JOIN` 一下就行了。

区别在于：你在前端写 `users.find()` 不会有人拦住你删掉一个还被引用的用户。但数据库的外键有"约束"——如果你试图删一个用户，而这个用户还有 500 条账单，数据库会直接拒绝，除非你设置了级联删除。这个保护在做前端的时候根本不存在。

**3. 一对多关系 = 一个数组元素对应另一个数组里的一堆元素**

一个 User 有多条 Transaction ：一个用户今天记了餐饮、交通、购物三笔账。一个 User 有多条 Budget：一个月给餐饮设了预算、交通设了预算、购物设了预算。

这种"一对多"是数据库里最常见的关系。实现方式极其简单：**在"多"的那张表里加一个外键字段，指向"一"的那张表的主键。**

Transaction 和 Budget 之间没有直接外键关联——它们通过"同一个 user_id"间接联系在一起。要不要给它们加外键？不要。因为一笔账单不一定对应一条预算（你可以先记账，再单独设预算），强行加外键反而会让数据写不进去。

> 阿珊说：设计数据库跟设计 React 组件是同一件事——给每个"东西"单独建表，让它们各自管好自己的数据，用外键代替 props 来通信。你说这是后端知识？不，这就是拆分组件的思想，换了个地方用而已。

## 🛠 动手试试

看完原理，来看金橘记账里这三张表的完整定义。这不是教学 demo，是线上正在跑的代码。

### User 表（主表）

```python
# backend/app/models/user.py
from sqlalchemy import String, DateTime
from sqlalchemy.orm import Mapped, mapped_column
from app.database import Base
import uuid
from datetime import datetime

class User(Base):
    __tablename__ = "users"

    id: Mapped[str] = mapped_column(
        String(36), primary_key=True, default=lambda: str(uuid.uuid4())
    )                                                    # 主键 — uuid 字符串
    email: Mapped[str] = mapped_column(
        String(255), unique=True, nullable=False
    )                                                    # unique=True = 不能重复
    password_hash: Mapped[str] = mapped_column(
        String(255), nullable=False
    )                                                    # 存 hash 不存明文
    nickname: Mapped[str | None] = mapped_column(
        String(100), nullable=True
    )                                                    # 可选字段
    avatar: Mapped[str | None] = mapped_column(
        String(10), nullable=True, default="😊"
    )                                                    # emoji 做头像
    created_at: Mapped[datetime] = mapped_column(
        DateTime, default=datetime.utcnow
    )
```

对应的 SQL（SQLAlchemy 自动生成的）：

```sql
CREATE TABLE users (
    id VARCHAR(36) PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    nickname VARCHAR(100),
    avatar VARCHAR(10) DEFAULT '😊',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### Transaction 表（子表，外键指向 User）

```python
# backend/app/models/transaction.py
from sqlalchemy import String, Integer, Date, DateTime, ForeignKey
from sqlalchemy.orm import Mapped, mapped_column
from app.database import Base
import uuid
from datetime import date, datetime

class Transaction(Base):
    __tablename__ = "transactions"

    id: Mapped[str] = mapped_column(
        String(36), primary_key=True, default=lambda: str(uuid.uuid4())
    )
    user_id: Mapped[str] = mapped_column(
        String(36), ForeignKey("users.id"), nullable=False
    )                                                    # ← 外键！指向 users 表
    type: Mapped[str] = mapped_column(
        String(10), nullable=False
    )                                                    # "expense" | "income"
    amount: Mapped[int] = mapped_column(
        Integer, nullable=False
    )                                                    # 金额存分，不存元
    category: Mapped[str] = mapped_column(
        String(50), nullable=False
    )                                                    # 餐饮/交通/购物...
    category_emoji: Mapped[str | None] = mapped_column(
        String(10), nullable=True
    )                                                    # 🍽️/🚇/🛍️
    note: Mapped[str | None] = mapped_column(
        String(200), nullable=True
    )
    tx_date: Mapped[date] = mapped_column(
        Date, nullable=False
    )
    created_at: Mapped[datetime] = mapped_column(
        DateTime, default=datetime.utcnow
    )
```

对应的 SQL：

```sql
CREATE TABLE transactions (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    type VARCHAR(10) NOT NULL,
    amount INTEGER NOT NULL,
    category VARCHAR(50) NOT NULL,
    category_emoji VARCHAR(10),
    note VARCHAR(200),
    tx_date DATE NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

注意最后一行：`FOREIGN KEY (user_id) REFERENCES users(id)` —— 这就是外键约束。它告诉数据库："transaction 的 user_id 必须在 users 表里存在，你不能给一个不存在的用户记账。"

### Budget 表（也是子表，同样外键指向 User）

```python
# backend/app/models/budget.py
class Budget(Base):
    __tablename__ = "budgets"

    id: Mapped[str] = mapped_column(
        String(36), primary_key=True, default=lambda: str(uuid.uuid4())
    )
    user_id: Mapped[str] = mapped_column(
        String(36), ForeignKey("users.id"), nullable=False
    )                                                    # ← 同样指向 users
    month: Mapped[str] = mapped_column(
        String(7), nullable=False
    )                                                    # "2026-06"
    category: Mapped[str] = mapped_column(
        String(50), nullable=False
    )
    category_emoji: Mapped[str | None] = mapped_column(
        String(10), nullable=True
    )
    amount: Mapped[int] = mapped_column(
        Integer, nullable=False
    )                                                    # 预算金额，单位：分
    created_at: Mapped[datetime] = mapped_column(
        DateTime, default=datetime.utcnow
    )
```

对应的 SQL：

```sql
CREATE TABLE budgets (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    month VARCHAR(7) NOT NULL,
    category VARCHAR(50) NOT NULL,
    category_emoji VARCHAR(10),
    amount INTEGER NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

### 三张表的关系总结

| 关系 | 谁 → 谁 | 怎么实现的 |
|------|---------|-----------|
| User → Transaction | 一对多 | Transaction 表里加 `user_id` 外键指向 `users.id` |
| User → Budget | 一对多 | Budget 表里加 `user_id` 外键指向 `users.id` |
| Transaction ↔ Budget | 无直接关联 | 通过同一个 `user_id` 间接关联，但两者之间没有外键 |

用前端代码来描述这个关系，就是：

```typescript
// 这就是数据库里三张表的关系，用 TS 类型写出来
type User = {
  id: string;
  email: string;
  // ...
};

type Transaction = {
  id: string;
  userId: string;  // ← 外键：指向 User.id
  amount: number;
  category: string;
  // ...
};

type Budget = {
  id: string;
  userId: string;  // ← 外键：指向 User.id
  month: string;
  category: string;
  amount: number;
  // ...
};

// "查用户本月花了多少钱" — 用 ts 写就是两行
const userTransactions = transactions.filter(
  t => t.userId === userId && isCurrentMonth(t.txDate)
);
const totalSpent = userTransactions.reduce((sum, t) => sum + t.amount, 0);

// 数据库里写 SQL 也是两行
// SELECT SUM(amount) FROM transactions
// WHERE user_id = ? AND tx_date LIKE '2026-06%';
```

你看，**结构对了，查询就简单了**。三张表各管各的，想算什么 JOIN 一下就行。跟组件拆得合理，业务逻辑就写在对应的 hook 里——一样的道理。

## ⚠️ 常见坑点

| 坑 | 现象 | 解法 |
|----|------|------|
| **一张大表走天下** | 30 个字段全塞一张表，查"用户本月花了多少钱"要写成 `SELECT ... FROM big_table WHERE user_id=? AND tx_type='expense' AND ...`，一条 SQL 几十行。加个新功能就要改表结构，牵一发动全身 | 按实体拆分：User、Transaction、Budget 各管各的。一个表只存一种实体，字段不超过 15 个。拆表不是多此一举，是给你自己留活路 |
| **外键忘了加约束** | 删了一个用户，但他的 500 条账单还在数据库里，变成"幽灵数据"——谁都不知道这笔账是谁记的。查用户账单的时候 `JOIN` 不上，前端直接报错 | 建表时老老实实写 `ForeignKey("users.id")`。数据库会拦着你删有关联数据的父记录。如果非要删，先删子记录再删父记录，或者设置 `ON DELETE CASCADE` |
| **金额用 float 存** | `19.9` 存进数据库变成 `19.899999618530273`，算了一年的账对不上，差了 3 分钱。用户来找你："我明明记的就是 19.9，你怎么给我少了几分？" | 金额永远用 `Integer` 存分。前端显示时 `/100` 转成元。金橘记账所有 `amount` 字段都是 `Integer`。这和 Canvas 里用 `devicePixelRatio` 避免浮点精度的道理一样 |
| **级联删除设错了方向** | `ON DELETE CASCADE` 配在 User 表上，结果一个用户注销，他所有的账单和预算也自动删了。要是用户想恢复数据，找不回来了 | CASCADE 要谨慎用。大部分业务场景应该用默认的 `RESTRICT`（阻止删除），强制你先手动清理子数据。只有明确的"子数据离开了父数据没意义"的场景——比如"订单项"离了"订单"就没意义——才适合用 CASCADE |
| **用业务字段做主键** | 用 email 做主键，结果用户想换邮箱——主键不能改，一改所有引用它的外键全要跟着改，几十张表联动更新，数据库直接锁死 | 主键用无业务含义的 uuid 或自增整数。业务字段（email、手机号）加 `unique` 约束保证唯一性就够。主键只管"找到这一行"，不要承担业务含义 |

## 🎯 面试会问

- **Q：什么是数据库三范式？用你自己的话讲。**
  **答：**三范式的核心就一句话——"一个事实一处存"。第一范式：每列都不可再分，不能把"餐饮:300元"塞一个字段里，要把 category 和 amount 拆成两列。第二范式：非主键列必须完全依赖主键，不能只依赖主键的一部分（只出现在复合主键场景）。第三范式：非主键列不能依赖其他非主键列，比如不能同时存 `birth_date` 和 `age`，因为 age 可以通过 birth_date 算出来。我在金橘记账项目里，User 表只存用户自己的属性，Transaction 表只存账单属性，通过 `user_id` 外键关联——这是第三范式的实践。本质就是 React 组件拆分的逻辑：一个组件不要管另一个组件的数据。

- **Q：外键约束有什么用？不加行不行？**
  **答：**外键约束做两件事：第一，保证引用完整性——你记的每一笔账，`user_id` 必须对应一个真实存在的用户，不能指向一个被删掉的用户。第二，防止误操作——当你要删一个用户，而他还挂着 500 条账单时，数据库会直接拒绝删除，提醒你先处理子数据。不加外键的话，开发阶段可能不出问题，但一到生产环境就容易出"幽灵数据"——数据还在，但不知道属于谁了。金橘记账里 Transaction 和 Budget 的 `user_id` 都设了 `ForeignKey("users.id")`，这是底线。

- **Q：什么时候用 uuid 做主键，什么时候用自增整数？**
  **答：**自增整数（`SERIAL` / `AUTOINCREMENT`）优点是短、有序、索引效率高。缺点是分布式系统里多个服务器同时插入会生成相同 ID 撞车，而且暴露业务量（用户可以通过 ID 号猜出你有多少数据）。uuid 优点是全局唯一，任何机器上生成都不重复，适合分布式和微服务。缺点是长（36 字符），索引占空间更大，排序效率比整数略低。金橘记账用 uuid 是因为：用户量不大，索引效率差异可以忽略，但以后如果要拆分服务，uuid 天然支持分布式。如果你确定单机部署、数据量大、追求极致性能，用自增整数。

## 📝 小结

数据库设计不是玄学，就三件事：

1. **找实体** — 你系统里有哪些"东西"？用户是一个，账单是一个，预算是一个。每样东西单独建一张表。
2. **定字段** — 每个实体有哪些属性？用户有 email、nickname，账单有 amount、category。一次只存一种东西的字段，别把别的实体的字段混进来。
3. **建关联** — 谁和谁有关系？一个用户有多笔账单，一对多，在账单表里加 `user_id` 外键。一个用户有多条预算，一对多，同理加 `user_id`。

这跟写 React 的思维完全一样：拆组件、定 props、建通信。你以为你在学"后端知识"，其实你只是在用另一种方式做你已经会了的事——**把复杂的东西拆成简单的、独立的、可组合的模块**。

如果你手里有金橘记账的项目代码，现在打开 `backend/app/models/` 目录，三张表就摆在那里。试着改一个字段，重新跑一遍后端，看看数据库有什么变化——改完你就记住了。
