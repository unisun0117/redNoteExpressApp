# 第7篇：数据库的10个坑，我都帮你踩过了

> 数据库的坑80%跟"你以为存进去了但其实没有"有关。金额用float → 精度丢失，commit忘记调用 → 数据没写入，SQLite跟PostgreSQL不兼容 → 本地好好的线上报错。

## 📌 前端为什么要学这个？

先跟你讲几个真实故事。

去年金橘记账刚上线，第二天就有用户在群里反馈："阿珊，我记了一笔39.9的咖啡，账单页面显示39.89元，少了1分钱。"我当时觉得不可能，查数据库、打日志、甚至怀疑是前端toFixed的锅。3个小时后，我盯着数据库里`39.89999961853027`这个数字愣住了 —— **浮点数精度丢失**。从那之后，金橘记账所有金额全部用整数存"分"。

还有一个惨案：我在注册接口里写好了`db.add(user)`，本地测试没问题，部署上去用户反馈"明明注册了但登录不了"。查了半天发现——忘记写`db.commit()`了。数据在内存里转了一圈，从来没落到磁盘上。

另外一个让我抓狂的坑：本地跑得好好的，部署到线上直接500报错。原因是本地用的是SQLite，线上部署的是PostgreSQL，两者对某些SQL语法的处理不一样。最后改了一整天才把兼容性问题修完。

这些坑，每一个我都替你踩过了。今天这篇文章，一次给你讲透。

## 🔍 核心原理：数据库身上的三条铁律

踩了这么多坑之后，我总结出背后三条铁律：

**铁律一：计算机不会猜你在想什么**

你说float存39.9，它就老老实实用IEEE 754标准去近似。你说字符串存密码，它就原样存进去。它不会主动提醒你"这个做法不够好"。数据库是你写什么它就存什么，绝对忠诚，但绝不多嘴。

**铁律二：你不说commit，它就当没发生过**

关系型数据库的设计哲学是：一切操作在事务里进行。你`add()`了数据，它只是放到了当前会话的事务缓冲区里。只有你显式调用`commit()`，它才会真正把数据刷到磁盘。这不是bug，是feature——事务机制让一组操作要么全成功、要么全回滚。但如果你不知道这个机制，就成了坑。

**铁律三：数据库不提醒你，它只是安静地坏掉**

SQLite允许你任何时间任何线程操作（除非你配`check_same_thread`），PostgreSQL有严格的类型检查，MySQL默认静默截断超长数据。不同数据库有不同的"性格"，但它们都有一个共性：出问题的时候不一定会报错，它可能只是悄悄地把错误的数据存了进去。

> **阿珊说：** 数据库不会骗你，但它也不会提醒你。它就像一个沉默的记账本——你写错了小数，它就记错的小数；你忘了锁抽屉，它就让别人也能翻。数据库的每一条数据，都是你代码逻辑的诚实镜像。

## 🛠 10个坑详解

### 坑1：Float存金额 —— 39.9变成39.89999961853027

**现象：** 用户输入39.90元，数据库存的是39.89999961853027，账单显示39.89元，凭空少了1分钱。

**原因：** Python（以及几乎所有语言）的float类型使用的是IEEE 754二进制浮点数标准。十进制里干净的39.9，在二进制里是一个无限循环小数，计算机只能取一个近似值。`0.1 + 0.2 == 0.3`返回`False`，是同一个道理。

**金橘记账里的解法：**

```python
# ❌ 错误做法
amount: Mapped[float] = mapped_column(Float, nullable=False)
# 用户输入39.90 → 数据库里是39.89999961853027

# ✅ 正确做法：金额统一用Integer存"分"
amount: Mapped[int] = mapped_column(Integer, nullable=False)  # 单位：分
# 用户输入39.90 → 数据库里是3990
```

前端配合也很简单：

```typescript
// 提交时：元 → 分
const amountInCents = Math.round(amountInYuan * 100);  // 39.90 → 3990

// 展示时：分 → 元
const displayYuan = (amountInCents / 100).toFixed(2);   // 3990 → "39.90"
```

用`Math.round`而不是`parseInt`，因为`39.90 * 100`在JS里也可能算出`3989.999999`，`parseInt`直接截断变3989，又少了1分。

### 坑2：忘记commit —— 数据在内存里转了一圈就走了

**现象：** 代码里写了`db.add()`，逻辑全对，但刷新数据库一看，数据根本没进去。

**原因：** SQLAlchemy（以及绝大多数ORM）的事务机制：所有增删改操作先暂存在session的identity map里，只有调用`session.commit()`才会生成SQL并发送给数据库执行。

**金橘记账里的标准三步：**

```python
# 数据写入的标准三步，缺一不可
db.add(transaction)       # 第1步：放入待写入队列
db.commit()               # 第2步：真正写入数据库（提交事务）
db.refresh(transaction)   # 第3步：刷新对象（获取数据库生成的默认值、自增ID等）
```

常见的漏写场景：
- 写了`db.add()`和`db.commit()`，但中间抛了异常导致commit没执行到 —— 需要用`try/except`+`db.rollback()`
- 只在循环外写了一个`commit()`，但循环中间出错导致前面的也没了 —— 每批提交或每条提交，看业务需要
- Flask/FastAPI里以为请求结束会自动commit —— 不会，必须显式调用

> **阿珊说：** 养成好习惯：每写完一个写操作的函数，最后一行一定是commit。就像出门前摸一下口袋确认带了钥匙——一句commit=一秒，忘记commit=两小时的debug。

### 坑3：SQLite vs PostgreSQL 不兼容 —— 本地好好的，线上500

**现象：** 本地开发一切正常，部署到线上（PostgreSQL）直接报错。SQLite里能跑的SQL，放到PostgreSQL就炸了。

**原因：** SQLite是个"宽容"的数据库——类型检查很松、支持动态类型、单文件架构。PostgreSQL是"严格"的数据库——强类型检查、严格的SQL标准。两者底层实现差异巨大。

**最常见的三种不兼容：**

| 场景 | SQLite | PostgreSQL |
|------|--------|------------|
| 布尔值 | 存`True`实际存的是`1` | 严格区分`boolean`和`integer` |
| 日期类型 | 没有原生日期类型，存文本 | 有`DATE`/`TIMESTAMP`，序列化要转换 |
| 并发写入 | 单写锁，自动排队 | 需要事务管理，并发冲突要处理 |

**金橘记账里的兼容处理：**

```python
# 数据库连接配置：SQLite特殊处理check_same_thread
connect_args = {"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {}
engine = create_engine(settings.DATABASE_URL, connect_args=connect_args)
```

**最佳实践：** 本地开发和线上部署用同一种数据库。如果SQLite只用于快速原型，那在提交代码前一定在PostgreSQL（或你线上用的数据库）上跑一遍测试。Docker一个本地PostgreSQL只要两行命令：

```bash
docker run --name pg-local -e POSTGRES_PASSWORD=123456 -p 5432:5432 -d postgres:16
# 然后 DATABASE_URL 指向 postgresql://postgres:123456@localhost:5432/mydb
```

### 坑4：N+1查询 —— 循环里查数据库

**现象：** 页面加载越来越慢，100条数据要发101次SQL查询。

**原因：** 先查一次拿到N条记录，然后在循环里逐条去查关联数据，导致总共执行了N+1条SQL。

**金橘记账里的真实代码（budgets.py）：**

```python
# ❌ N+1查询：外层查了所有预算分类，内层每个分类再查一次支出
cat_budgets = db.query(Budget).filter(Budget.user_id == user_id).all()
for b in cat_budgets:       # ← N+1的根源：循环里查数据库
    spent = db.query(func.sum(Transaction.amount))\
              .filter(Transaction.category == b.category)\
              .filter(Transaction.type == "expense")\
              .all()
    total_spent = spent[0][0] or 0
```

这段代码：N个预算分类 → 1条查预算 + N条查支出 = N+1条SQL。100个分类就是101条查询。

**解法：** JOIN一次查出，或分组聚合。

```python
# ✅ 一次性查完
from sqlalchemy import func
results = db.query(
    Transaction.category, 
    func.sum(Transaction.amount).label("total_spent")
).filter(
    Transaction.user_id == user_id,
    Transaction.type == "expense",
    Transaction.category.in_([b.category for b in cat_budgets])
).group_by(Transaction.category).all()
```

### 坑5：事务没做原子性保护 —— 删到一半出错了

**现象：** 删除用户时，先删了订单，再删用户信息时出错。结果订单没了，用户还在，数据处于"半残"状态。

**原因：** 多个操作没有包裹在同一个事务里，一个失败不会触发前面操作的回滚。

**解法：**

```python
# ✅ 事务包裹：要么全成功，要么全回滚
try:
    db.query(Order).filter(Order.user_id == user_id).delete()
    db.query(User).filter(User.id == user_id).delete()
    db.commit()
except Exception as e:
    db.rollback()   # 回滚，订单和用户都恢复原样
    raise
```

### 坑6：没加唯一约束 —— 同邮箱注册两次

**现象：** 用户用同一个邮箱注册了两个账号，第一个的密码废了，用户懵了。

**原因：** 只在应用层做了检查（"先查有没有，没有再插入"），但并发请求同时到达时，两个请求都查到"没有"，于是两个都插入成功。

**金橘记账里的解法：**

```python
# ✅ 数据库层面加唯一约束，双重保险
email: Mapped[str] = mapped_column(String(255), unique=True, nullable=False)
```

应用层检查 + 数据库唯一约束 = 双重保险。数据库约束是最后一道防线，任何应用层检查都可能在并发下失效。

### 坑7：密码明文存储 —— 数据库泄露 = 用户裸奔

**现象：** 数据库被拖了，所有用户的密码看得一清二楚。

**原因：** 密码直接存了明文字符串。

**金橘记账里的解法（bcrypt哈希）：**

```python
# ✅ bcrypt单向哈希，即使数据库泄露密码也无法还原
from passlib.context import CryptContext
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

# 注册时
password_hash: Mapped[str] = mapped_column(String(255), nullable=False)
user.password_hash = pwd_context.hash(password)  # "123456" → "$2b$12$LJ3..."

# 登录时
if not pwd_context.verify(plain_password, user.password_hash):
    raise HTTPException(status_code=401, detail="密码错误")
```

关键点：bcrypt是单向的，你只能"验证"密码，不能"还原"密码。数据库泄露了，攻击者看到的也是$2b$12开头的哈希串，没法反推出明文。

### 坑8：SQL注入 —— 拼接字符串构建SQL

**现象：** 用户输入`' OR '1'='1`，直接绕过了登录验证。

**原因：** 用字符串拼接的方式构建SQL查询。

```python
# ❌ 危险：SQL注入
sql = f"SELECT * FROM users WHERE email = '{email}'"
result = db.execute(sql)
# 如果email = "admin@test.com' OR '1'='1"
# 实际执行: SELECT * FROM users WHERE email = 'admin@test.com' OR '1'='1'
# 返回所有用户！

# ✅ 安全：参数化查询（ORM自动处理）
user = db.query(User).filter(User.email == email).first()
# SQLAlchemy自动转义，恶意输入被当作普通字符串处理
```

用ORM（SQLAlchemy、Django ORM）基本能避免SQL注入，裸SQL必须用参数化查询（`?`占位符），永远不要拼字符串。

### 坑9：JSON序列化丢失类型 —— 日期字段报错

**现象：** API返回数据时报`TypeError: Object of type date is not JSON serializable`。

**原因：** Python的`datetime.date`和`datetime.datetime`对象不能直接被`json.dumps()`序列化。

**金橘记账里的处理：**

```python
# ❌ 直接序列化ORM对象报错
return {"data": transactions}  # 日期字段无法JSON序列化

# ✅ 手动转换日期字段
formatted_tx = {
    **tx.__dict__,
    "tx_date": tx.tx_date.isoformat() if tx.tx_date else "",
    "created_at": tx.created_at.isoformat() if tx.created_at else ""
}
```

或者用Pydantic + FastAPI的`response_model`自动处理：

```python
from pydantic import BaseModel
from datetime import date

class TransactionOut(BaseModel):
    id: int
    amount: int
    tx_date: date  # Pydantic自动处理date类型的JSON序列化
    class Config:
        from_attributes = True  # 兼容SQLAlchemy ORM对象
```

### 坑10：没有索引 —— 全表扫描越来越慢

**现象：** 刚开始飞快，用户多了数据多了，查一次要好几秒。

**原因：** 没有给高频查询字段加索引，数据库每次都要全表扫描。

**解法（以金橘记账为例）：**

```python
# 用户ID是每一条查询都会用到的过滤条件
user_id: Mapped[int] = mapped_column(Integer, ForeignKey("users.id"), index=True)
# 日期字段也经常用于排序和范围查询
tx_date: Mapped[date] = mapped_column(Date, index=True)
```

什么字段该加索引？记住一个简单规则：WHERE后面经常出现的字段、ORDER BY用的字段、JOIN关联的字段。但别每个字段都加索引——索引本身也占空间、拖慢写入速度。

## 🎯 面试会问

**Q1：为什么金额不能用float/float？**
A：IEEE 754浮点数在二进制中无法精确表示十进制小数（如0.1），存入39.9实际是39.899999...，累计误差会导致金额不一致。正确做法是用整数存最小单位（分/厘），或使用数据库的DECIMAL类型。

**Q2：ORM的add和commit有什么区别？**
A：`add()`把对象加入ORM session的待处理队列（内存），`commit()`才真正生成SQL并发送给数据库执行（刷盘）。如果add后不commit，数据只存在于当前session中，其他连接看不到，进程重启就丢失。

**Q3：SQLite和PostgreSQL的主要区别？**
A：(1) SQLite是嵌入式数据库，单文件，零配置，适合本地开发和移动端；PostgreSQL是C/S架构，功能完备，适合生产环境。(2) SQLite类型检查宽松（动态类型），PostgreSQL严格。(3) SQLite并发写用单写锁，PostgreSQL支持真正的并发事务。(4) SQLite没有用户权限系统，PostgreSQL有完整的角色和权限管理。

**Q4：怎么避免N+1查询问题？**
A：使用JOIN或子查询一次性获取关联数据，而不是在循环中逐条查询。ORM中可以用`joinedload()`（SQLAlchemy）或`select_related()`/`prefetch_related()`（Django ORM）来声明式地预加载关联数据。

**加分题：索引建得越多越好吗？**
A：不是。索引虽然加速查询，但每次INSERT/UPDATE/DELETE都要同步更新索引，会拖慢写入速度，而且索引本身占用磁盘空间。只给WHERE、ORDER BY、JOIN中高频出现的字段建索引，避免给频繁变更的字段建索引。

## 📝 小结

这10个坑，每一个我都亲身掉进去过。回头来看，它们的共同点是：**不是数据库的bug，而是我们对数据库行为理解不到位。**

记住三件事就够了：
1. **金额存分不存元**——这是血的教训，39.9变成39.89的bug我永远不会忘。
2. **add、commit、refresh三步缺一不可**——数据库不会自作主张帮你提交。
3. **本地和线上用同一种数据库**——不要到了部署那天才发现兼容性问题。

数据库不会骗你，但它也不会提醒你。你代码怎么写，它就怎么存——不多一分，不少一厘，不提醒你哪里可能不对。这才是数据库的魅力，也是数据库的凶险。

—— 阿珊，前端开发者 & 全栈学习者
