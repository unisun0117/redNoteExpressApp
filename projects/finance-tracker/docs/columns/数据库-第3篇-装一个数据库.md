# 第3篇：装一个数据库 — PostgreSQL环境搭建

> 装数据库比装node_modules还快。真正花时间的是理解它怎么跑——就像一个后台服务，开机自启，监听5432端口

## 📌 前端为什么要学这个？

我第一次装PostgreSQL的时候，一共花了5分钟——下载安装包、下一步下一步、完成。我跟同事说"数据库装完了，挺简单的"，然后开始写连接代码。结果报错了：`could not connect to server: Connection refused`。

接下来的2小时，我干了一件事：查资料、改配置、重启电脑、怀疑人生。最后发现——PostgreSQL是个后台服务，我装完了，但是没有启动它。这就像你 `npm install` 了一个项目，然后什么都没做，就觉得页面应该出来了？不，你还没有 `npm run dev`。

这就是前端同学第一次装数据库最别扭的地方：你习惯了装个包就能 `import` 用，但数据库不是"装"一个库——它是"跑"一个服务。

装数据库本身很快。真正花时间的是搞清楚"它在哪跑"、"怎么启动"、"为什么连不上"。这篇文章就是帮你把这些"看不见的坑"一次性填平。

## 🔍 核心原理

先帮你建立一个类比。你写前端的时候，React 开发服务器的逻辑是：

```
终端输入：npm run dev
实际发生：启动一个 HTTP 服务，监听 localhost:5173
前端代码：fetch("http://localhost:5173/api/xxx") 访问它
```

PostgreSQL 的逻辑几乎一模一样：

```
终端输入：pg_ctl start（或者 brew services start postgresql）
实际发生：启动一个数据库服务，监听 localhost:5432
Python代码：postgresql://user:password@localhost:5432/mydb 连接它
```

说白了，PostgreSQL 就是电脑上的一个"后台常驻程序"。你开机的时候它可能自动启动（如果你配置了），然后就一直在 5432 端口上等着你的应用来问数据。你关电脑，它跟着关。你不关电脑，它就一直跑。

此处配图：一个前端开发者的电脑，"localhost:5173"是 React 开发服务器，"localhost:5432"是 PostgreSQL 数据库，两条线分别连到"前端浏览器"和"后端 FastAPI"，中间标注"都是本地服务，只是端口不同"。

你可能会问：为什么金橘记账现在还用 SQLite，而不是 PostgreSQL？

因为 SQLite 不需要装、不需要"启动服务"。它就是一个文件（`finance.db`），你的 Python 代码直接读写这个文件就行。就像一个本地的 JSON 文件——拿来就能用。开发阶段用 SQLite 非常爽，零配置。但上线之后你肯定要换成 PostgreSQL，因为你会有多个用户同时在写数据，SQLite 扛不住并发。

金橘记账的数据库配置里，连切换逻辑都写好了：

```python
# backend/app/database.py
engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {},
)
```

注意 `if "sqlite" in settings.DATABASE_URL else {}` 这一行——如果 URL 里是 sqlite，就加一个 SQLite 专属参数；如果是 PostgreSQL 的连接字符串，这个参数就不需要。换句话说，从 SQLite 切到 PostgreSQL，只需要改配置文件的 `DATABASE_URL`，代码一行不用动。

> 阿珊说：数据库就像一个你请来记账的会计——不是你把账本买回来就行，你得把人家叫到办公室坐下，人家才能开始干活。

## 🛠 动手试试

下面带你从头装一遍 PostgreSQL，然后用金橘记账的项目代码验证连接。

### 第0步：你是 Windows 还是 Mac？

本篇文章两条路都写，你根据自己的系统选就行。

### 第1步：安装 PostgreSQL

**Mac 用户（推荐 Homebrew）：**

```bash
brew install postgresql@16
brew services start postgresql@16
```

Homebrew 的 `services start` 就是帮你设置"开机自启"——以后你每次开机，PostgreSQL 都会自动在后台跑起来。就像你 Mac 上的微信一样，开机就启动了，不需要你每天手动去开。

**Windows 用户：**

去 https://www.postgresql.org/download/windows/ 下载安装包。安装过程中会问你几个问题：
- **安装目录：** 默认就行
- **数据目录：** 默认就行（这是数据库文件存的地方）
- **密码：** 给 `postgres` 用户设一个密码，**记住它**，后面连接要用
- **端口：** 5432（默认，不用改）

安装完它默认会启动服务。如果没启动，去"服务"（Win+R 输入 `services.msc`）里找到 `postgresql` 服务，右键启动。

### 第2步：验证安装——它真的在跑吗？

安装完第一件事：确认服务在不在监听 5432 端口。

**Mac：**
```bash
lsof -i :5432
# 或者
brew services list | grep postgresql
```

**Windows：**
```bash
netstat -ano | findstr 5432
```

如果看到一个进程在 LISTENING 5432 端口，说明它跑起来了。如果什么都没有——恭喜你，这就是我当年踩的第一个坑：服务没启动。回到第1步，手动启动它。

### 第3步：创建金橘记账自己的数据库

PostgreSQL 自带了 `psql` 命令行工具。我们先连进去，给金橘记账建一个数据库。

**Mac：**
```bash
# 切换到 postgres 用户
sudo -u postgres psql
```

**Windows：**
打开 PowerShell 或者 cmd：
```bash
psql -U postgres
```

进去之后会看到一个 `postgres=#` 的提示符。现在给你的项目建数据库：

```sql
-- 建一个数据库，叫 finance_tracker
CREATE DATABASE finance_tracker;

-- 创建一个普通用户（别用超级管理员 postgres 访问应用数据库）
CREATE USER finance_user WITH PASSWORD 'finance123456';
GRANT ALL PRIVILEGES ON DATABASE finance_tracker TO finance_user;

-- 给用户管理表结构的权限
\c finance_tracker
GRANT ALL ON SCHEMA public TO finance_user;

-- 退出
\q
```

这里有个关键操作：**创建了一个独立的数据库用户，而不是直接用 postgres 超级管理员**。这就像你不会用 root 账户去跑一个普通 Node.js 服务一样——最小权限原则。`finance_user` 只能操作 `finance_tracker` 这个数据库，搞不坏其他东西。

### 第4步：让金橘记账连接到 PostgreSQL

现在改金橘记账的配置文件，把 `DATABASE_URL` 从 SQLite 换成 PostgreSQL。

金橘记账用 `pydantic-settings` 管理配置，支持 `.env` 文件和系统环境变量两种方式。我们在 `backend/` 目录下创建 `.env` 文件：

```bash
# backend/.env
DATABASE_URL=postgresql://finance_user:finance123456@localhost:5432/finance_tracker
JWT_SECRET=finance-dev-secret-2026
ALLOWED_ORIGINS=http://localhost:5173
```

原来的默认值是 `sqlite:///./finance.db`，现在改成了 PostgreSQL 连接字符串。其他代码你不需要改任何东西——`database.py` 里那个条件判断已经自动帮你处理了 SQLite 专属参数：

```python
# backend/app/config.py
class Settings(BaseSettings):
    DATABASE_URL: str = "sqlite:///./finance.db"  # 默认值，会被 .env 覆盖
    # ... 其他配置

# backend/app/database.py
engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {},
    #                          👆 SQLite 才加          👆 PostgreSQL 不需要
)
```

### 第5步：启动后端，验证建表

```bash
cd backend
.\venv\Scripts\uvicorn.exe app.main:app --reload --port 8001
```

金橘记账的 `main.py` 在启动时会自动执行 `Base.metadata.create_all(bind=engine)`，这行代码的作用是：**把你定义的所有模型类（User、Transaction、Budget）自动在 PostgreSQL 里创建对应的表。** 这就像你写了一个 TypeScript interface，然后工具帮你自动生成了数据库建表语句。

启动成功之后，去 psql 里验证一下表是不是真的建好了：

```bash
psql -U finance_user -d finance_tracker
\dt   # 列出所有表
```

你应该看到：
```
           List of relations
 Schema |     Name      | Type  |    Owner
--------+---------------+-------+--------------
 public | budgets       | table | finance_user
 public | transactions  | table | finance_user
 public | users         | table | finance_user
(3 rows)
```

三张表，跟金橘记账的模型定义一一对应——User（用户）、Transaction（账单）、Budget（预算）。

### 第6步：用代码验证——插一条数据试试

打开 Python 交互式环境，直接用项目代码操作数据库：

```bash
cd backend
.\venv\Scripts\python.exe
```

```python
# 这段代码验证：连接通了、表建好了、能写数据
from app.database import SessionLocal
from app.models.user import User

db = SessionLocal()

# 创建一条测试用户
u = User(email="test@example.com", password_hash="not_real_hash", nickname="测试")
db.add(u)
db.commit()
db.refresh(u)
print(f"创建用户成功: {u.id}, {u.email}, {u.nickname}")

# 查出来
found = db.query(User).filter(User.email == "test@example.com").first()
print(f"查询结果: {found.nickname}, 头像: {found.avatar}")

# 清理测试数据
db.delete(found)
db.commit()

db.close()
print("数据库连接正常，增删查都没问题！")
```

如果你看到 `创建用户成功: xxx, test@example.com, 测试` 和 `数据库连接正常，增删查都没问题！`，那你的 PostgreSQL 环境已经完整搭建完成了。

### 可选：装一个可视化工具 DBeaver

写 SQL 不一定非得在黑框里写。DBeaver 是一个免费的可视化数据库工具，长得很像你用的 VS Code 左侧文件树——左边是数据库列表，右边是 SQL 编辑器，中间可以点表名直接看数据。

去 https://dbeaver.io/download/ 下载，装好之后新建连接：
- 类型选 PostgreSQL
- Host: `localhost`，Port: `5432`
- Database: `finance_tracker`
- Username: `finance_user`，Password: `finance123456`

连上之后你会发现——这就是一个"可视化的 `array.filter()`"。你点了哪张表，它就把数据一行行展示出来，就像你在 Chrome DevTools 里看一个 JSON 数组一样。

## ⚠️ 常见坑点

| 坑 | 现象 | 解法 |
|----|------|------|
| 服务没启动 | `psql: error: connection to server at "localhost" (::1), port 5432 failed: Connection refused` | 这是最常见的新手坑——装完了没启动。Mac：`brew services start postgresql@16`。Windows：`services.msc` 找到 PostgreSQL 服务，右键"启动"。验证：`lsof -i :5432`（Mac）或 `netstat -ano \| findstr 5432`（Win） |
| 密码不对 | `FATAL: password authentication failed for user "postgres"` | 你安装时设置的密码跟现在输的不一样。Windows 用户可以去 `services.msc` 里停止服务，然后用 `pg_ctl` 以"不验证密码"模式启动，进去改密码。Mac 用户用 `sudo -u postgres psql` 可以跳过密码，进去 `\password postgres` 改密码 |
| 用户没有建表权限 | `permission denied for schema public` | 你用 `finance_user` 连接，但这个用户没有 `public` schema 的写权限。连到 `finance_tracker` 数据库之后执行：`GRANT ALL ON SCHEMA public TO finance_user;` |
| 端口被占用 | 启动 PostgreSQL 时报 `port 5432 already in use` | 你电脑上可能装了另一个 PostgreSQL 实例（比如 Docker 里的）。查占用：`lsof -i :5432`。解决：停掉旧的，或者装的时候选另一个端口（比如 5433）。然后记得改 `.env` 里的连接字符串端口 |
| Homebrew postgresql@16 和 postgresql 都装了 | 启动了一个，连的时候连到了另一个 | `brew services list` 检查哪个版本在跑。确保只有一个在 5432 端口。如果有两个，`brew services stop postgresql` 停掉旧的，只用 `postgresql@16` |

## 🎯 面试会问

- **Q：PostgreSQL 和 SQLite 有什么区别？项目里怎么选？**
  **答：** SQLite 是一个文件数据库，不需要单独安装服务，读写都是直接操作 `.db` 文件。适合开发阶段、单用户场景、嵌入式应用。PostgreSQL 是一个独立的服务器进程，监听端口，支持多用户并发读写、事务隔离、用户权限管理。适合生产环境。我在金橘记账项目里，开发阶段用 SQLite（零配置，启动就能跑），上线时切 PostgreSQL（改一行 `DATABASE_URL` 就行），因为代码里已经做了兼容处理：`connect_args={"check_same_thread": False} if "sqlite" in DATABASE_URL else {}`。

- **Q：生产环境的数据库需要怎么配？**
  **答：** 至少要做三件事：第一，给应用创建独立的数据库用户，不用超级管理员；第二，密码用强密码，连接字符串通过环境变量注入（金橘记账就是这样，`DATABASE_URL` 存在 `.env` 里，不硬编码）；第三，`pg_hba.conf` 限制只能本地访问，如果需要远程连接就配 SSL。面试的时候有一句话很加分："我不在生产环境用 SQLite，因为 SQLite 写操作是串行的，多用户会排队。"

- **Q：为什么数据库用 5432 端口？**
  **答：** 这只是 IANA 分配给 PostgreSQL 的默认端口，跟 HTTP 用 80、HTTPS 用 443、MySQL 用 3306 一样的道理。可以改，但没必要——因为全行业都用 5432，改了反而让别人连不上。端口本身没什么技术含义，只是一个"门牌号"——操作系统收到一个网络请求之后，看端口号决定交给哪个程序处理。5432 就是"这个请求是找数据库的"。

## 📝 小结

装数据库就是一件"5分钟安装，2小时排查"的事。快的那个是下载和安装，慢的那个是搞清楚它在哪跑、怎么启动、为什么连不上。

你今天搭好的这个 PostgreSQL 环境，后面所有文章都会用到。从下一篇文章开始，我们不再对着 SQLite 看 CRUD，而是直接在金橘记账的 PostgreSQL 上写真正的 SQL 查询——INSERT、SELECT、UPDATE、DELETE，用前端数组操作的语言帮你翻译每一行 SQL。

你会发现：**你每天写的 `array.filter()`、`array.push()`、`array.splice()`，数据库里全都有精确的对应物。** 你只是换了一种写法，干的事情一模一样。
