# 第8篇：你的查询为什么这么慢？— 索引与查询优化入门

> 加一个索引 = 查询速度快 1000 倍。索引就是数据库的 Map——O(1) vs O(n) 的区别。但别一上来就加，等慢的时候再加。

## 📌 前端为什么要学这个？

金橘记账用户量上来之后，我发现一个诡异的现象：账单列表页加载要 3 秒。

我的第一反应是前端问题。于是花了两天优化虚拟列表、懒加载图片、砍 bundle 体积、甚至上了 React.memo——页面还是慢。每次刷新都要白屏好几秒，用户体验非常差。

直到我在后端日志里看到：**一条 SELECT 查询执行了 2800ms。**

我当时就愣住了。我一直在优化前端渲染，结果瓶颈根本不在这——数据库在做全表扫描，几万条记录一条一条比对。那个查询是：

```sql
SELECT * FROM transactions WHERE user_id = 123 ORDER BY tx_date DESC;
```

前端优化再牛，也救不了后端一条没索引的查询。我给 `user_id` 加了一个索引，同样一条查询从 2800ms 降到了 50ms。**60 倍提升，一行 SQL 的事。**

从那之后我学会了一件事：**页面慢，先看数据库，再看前端。**

## 🔍 核心原理

### 索引 = 数据库的 Map

你写前端的时候，如果要从一个 10 万条的数组里找某个 id 的对象，你会怎么写？

```javascript
// 方法一：线性遍历 — O(n)
const item = bigArray.find(item => item.id === 12345);

// 方法二：先建一个 Map — O(1)
const map = new Map(bigArray.map(item => [item.id, item]));
const item = map.get(12345);
```

方法一要遍历 10 万次（最坏情况），方法二只需要 1 次哈希查找。**这就是索引的本质。**

数据库的索引不直接是 Map，它用的是一种叫 **B+ 树** 的数据结构——你可以理解为一个多层级的、自动排序的 Map：

```
             [50 | 100]
            /    |    \
      [10|30]  [60|80]  [120|150]
       /  |  \    ...
    [1] [15] [35]
```

B+ 树的好处是：
- **叶子节点有链表串起来**，范围查询特别快（比如查 `WHERE tx_date BETWEEN '2026-01-01' AND '2026-01-31'`）
- **树的高度很低**，找一条记录只需要 2-4 次磁盘 IO
- **自动保持平衡**，不管怎么增删改，查找速度都稳定

### 无索引 vs 有索引：一张图看懂

```
无索引（全表扫描）：
  数据库把整张表从头读到尾，一行一行比对
  就像你在 10 万张发票里一张一张翻，找"2026年3月的外卖消费"
  时间复杂度：O(n)

有索引（B+ 树查找）：
  数据库先查索引，定位到符合条件的数据在哪几页
  就像你先翻到发票本的目录，"2026年3月"在第 47 页，"外卖"类目在第 12 格
  时间复杂度：O(log n)
```

用一个你熟悉的场景类比：

| 操作 | 无索引 | 有索引 |
|------|--------|--------|
| `WHERE user_id = 123` | 全表扫描，逐行比对 | 走 B+ 树，定位到叶子节点 |
| `ORDER BY tx_date DESC` | 全表扫描 + 内存排序 | 索引本身就是有序的，直接读 |
| `WHERE user_id = 123 AND type = 'expense'` | 全表扫描 | 复合索引一次定位 |

> 阿珊说：索引不是魔法，它就是数据库帮你维护的一个排序好的目录。不加索引，数据库就是一本没有目录的词典——每次查单词都要从 A 翻到 Z。

## 🛠 动手试试

### 第一步：找到慢查询

在金橘记账项目里，我们先看看哪些查询最频繁、哪些最该加索引。

**高频查询 1：用户账单列表（分页 + 排序）**

```python
# backend/app/routers/transactions.py
query = db.query(Transaction).filter(Transaction.user_id == user.id)
items = query.order_by(Transaction.tx_date.desc()) \
    .offset((page-1) * page_size) \
    .limit(page_size) \
    .all()
```

这条查询每次打开账单页都执行。它需要：
- 按 `user_id` 筛选（WHERE 条件）
- 按 `tx_date` 排序（ORDER BY）
- 分页取数据

**高频查询 2：月度收支汇总**

```python
# backend/app/routers/transactions.py
income = db.query(func.sum(Transaction.amount)).filter(
    Transaction.user_id == user.id,
    Transaction.type == "income",
    Transaction.tx_date.like(f"{month}%"),
).scalar() or 0
```

这条查询在仪表盘页执行，需要：
- 按 `user_id` + `type` 筛选
- 按 `tx_date` 做前缀匹配（月度过滤）

**高频查询 3：分类统计**

```python
# backend/app/routers/stats.py
results = db.query(
    Transaction.category, func.sum(Transaction.amount).label("total")
).filter(
    Transaction.user_id == user.id,
    Transaction.type == "expense",
    Transaction.tx_date.like(f"{month}%"),
).group_by(Transaction.category) \
 .order_by(func.sum(Transaction.amount).desc()) \
 .all()
```

这条查询需要 `user_id` + `type` + `tx_date` + `category` + `amount`。

### 第二步：加索引

#### 索引 1：user_id（最重要！）

```sql
-- 几乎所有查询都带 user_id 条件，这是最需要索引的列
CREATE INDEX idx_transactions_user_id ON transactions(user_id);
```

为什么这条最重要？因为金橘记账是**多租户系统**——每个用户只能看到自己的数据。所有查询都以 `WHERE user_id = ?` 开头。没有这个索引，每次查询都是全表扫描。

#### 索引 2：user_id + tx_date 复合索引

```sql
-- 账单列表高频：WHERE user_id = ? ORDER BY tx_date DESC
CREATE INDEX idx_transactions_user_date ON transactions(user_id, tx_date DESC);
```

复合索引的规则：**最左前缀原则**。`(user_id, tx_date)` 这个索引可以覆盖：
- `WHERE user_id = ?` （只用最左列，可以）
- `WHERE user_id = ? ORDER BY tx_date` （两列都用上，最佳）
- `WHERE user_id = ? AND tx_date > ?` （范围查询，可以）

但**不能**覆盖：
- `WHERE tx_date > ?` （跳过了最左列，不行）

就像你翻一本按"省份 → 城市"排序的电话簿——你可以快速找到"广东省的所有人"，但找不到"所有叫广州的城市"（因为广州分布在不同的省份下面）。

#### 索引 3：user_id + type + tx_date 复合索引

```sql
-- 月度收支汇总高频：WHERE user_id = ? AND type = ? AND tx_date LIKE ?
CREATE INDEX idx_transactions_user_type_date ON transactions(user_id, type, tx_date);
```

这个索引让月度汇总查询直接走索引，不需要回表。

#### 索引 4：category（分类统计需要）

```sql
-- 分类统计：GROUP BY category
CREATE INDEX idx_transactions_category ON transactions(category);
```

### 第三步：验证效果

加完索引后，用 `EXPLAIN QUERY PLAN` 验证：

```sql
-- 加索引前
EXPLAIN QUERY PLAN
SELECT * FROM transactions WHERE user_id = 1 ORDER BY tx_date DESC;
-- 输出：SCAN transactions  （全表扫描！）

-- 加索引后
EXPLAIN QUERY PLAN
SELECT * FROM transactions WHERE user_id = 1 ORDER BY tx_date DESC;
-- 输出：SEARCH transactions USING INDEX idx_transactions_user_date (user_id=?)
--      （走索引！）
```

看到 `SCAN` 就说明在扫全表，赶紧加索引。看到 `SEARCH ... USING INDEX` 就说明索引生效了。

### SQLAlchemy 里怎么建索引

如果你用的是 SQLAlchemy（金橘记账就是），直接在模型定义里加：

```python
from sqlalchemy import Index

class Transaction(Base):
    __tablename__ = "transactions"
    __table_args__ = (
        Index("idx_user_id", "user_id"),
        Index("idx_user_date", "user_id", "tx_date"),
        Index("idx_user_type_date", "user_id", "type", "tx_date"),
        Index("idx_category", "category"),
    )
```

或者用 Alembic 写迁移：

```python
# migrations/versions/xxx_add_indexes.py
def upgrade():
    op.create_index("idx_user_id", "transactions", ["user_id"])
    op.create_index("idx_user_date", "transactions", ["user_id", "tx_date"])
    op.create_index("idx_user_type_date", "transactions", ["user_id", "type", "tx_date"])
```

## ⚠️ 常见坑点

| 坑 | 现象 | 为什么 | 解法 |
|----|------|--------|------|
| **无索引全表扫描** | 页面加载越来越慢，用户量上来后尤其明显 | 数据库没有目录，每次查都要扫全表。数据量小时不觉得，超过 1 万行开始明显 | 给 WHERE 和 ORDER BY 列加索引。用 `EXPLAIN` 看到 `SCAN` 就动手 |
| **索引加太多** | 写入变慢，磁盘占用变大 | 索引本质是"以空间换时间"。每个索引都是一棵 B+ 树，INSERT/UPDATE/DELETE 时都要同步更新所有索引。一个表建 10 个索引，写一条数据要更新 11 棵树（1 个主键 + 10 个索引） | 只为高频查询的列加索引。低频查询走全表扫描也无妨。定期用 `SHOW INDEX` 审计，删掉没用的 |
| **最左前缀没搞清楚** | 建了复合索引但查询不走索引 | 复合索引 `(A, B, C)` 相当于建了 `(A)`、`(A,B)`、`(A,B,C)` 三个索引，但不能当 `(B)` 或 `(B,C)` 用。`WHERE B = ?` 不走这个索引 | 把等值条件放左边，范围条件放右边。如 `(user_id, type, tx_date)` — user_id 和 type 是等值查询，tx_date 是范围查询 |
| **LIKE 前缀通配符** | `WHERE name LIKE '%外卖%'` 不走索引 | 索引是有序的，但 `%` 开头意味着"任意开头"，B+ 树无法定位起始位置。就像你查电话簿找"名字里带'珊'的人"——只能从头翻到尾 | 尽量用 `LIKE '外卖%'`（前缀匹配，可以走索引）。全文搜索需求用专门的全文索引（如 SQLite FTS5、PostgreSQL tsvector） |
| **函数包裹列** | `WHERE DATE(tx_date) = '2026-03-01'` 不走索引 | 索引存的是原始值，你套了一个函数，数据库认不出来了。相当于你目录上标的是"第47页"，但你想找"页码/2=23.5"——目录帮不了你 | 改写条件，让列保持裸状态：`WHERE tx_date >= '2026-03-01' AND tx_date < '2026-03-02'` |

## 🎯 面试会问

**Q1：为什么加了索引查询就变快？**
**答：** 无索引时数据库做全表扫描，时间复杂度 O(n)，数据量越大越慢。索引本质是一棵 B+ 树，查找时间复杂度 O(log n)。举个例子：100 万条数据，全表扫描平均要扫 50 万行。有索引的情况下，B+ 树高度通常只有 3-4 层，也就是只需要 3-4 次磁盘 IO 就找到了。用前端类比的话——无索引是 `Array.find()`，有索引是 `Map.get()`。

**Q2：索引建得越多越好吗？**
**答：** 不是。索引有代价：(1) 写入性能下降——每次 INSERT/UPDATE/DELETE 都要更新所有索引；(2) 占用磁盘空间——索引本身也是数据结构。所以索引策略是"按需建"：分析慢查询日志，给高频 WHERE/ORDER BY/JOIN 列加索引，写多读少的表尽量少加。

**Q3：复合索引 `(A, B, C)` 在 `WHERE A = ? AND C = ?` 时能用上吗？**
**答：** 能用上 A 列，但 C 列用不上。因为复合索引遵循最左前缀原则——跳过了 B，索引在 A 之后无法继续利用有序性。实际效果相当于只用了一个 `(A)` 的单列索引。这也是为什么建复合索引时，把区分度高的、常用的等值查询列放前面。

## 📝 小结

索引是数据库优化的第一课，也是最划算的一课。一行 `CREATE INDEX`，查询速度可能提升几十到上千倍。

对你的实际项目来说，记住这几点就够了：

1. **所有带 `WHERE user_id = ?` 的查询，给 `user_id` 加索引**（金橘记账几乎所有查询都有这个条件）
2. **经常排序的列，跟筛选列一起建复合索引**（`user_id + tx_date`）
3. **不要一上来就建一堆索引**——等项目跑起来，找到真正的慢查询再动手
4. **加完索引后用 `EXPLAIN` 验证**——看到 `USING INDEX` 才算生效

数据库优化的路还很长——覆盖索引、查询计划分析、慢查询日志、连接池调优……但从索引开始，你已经掌握了性价比最高的一项技能。
