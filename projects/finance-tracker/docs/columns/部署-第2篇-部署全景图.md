# 第2篇：部署全景图 — 浏览器到服务器到底经过了什么

> 部署不是玄学，就是5站接力：浏览器 → DNS → CDN → 后端 → 数据库。每一站只干一件事，跟快递从下单到签收一模一样。

## 📌 前端为什么要学这个？

我第一次部署金橘记账的时候，脑子里一团浆糊。Vercel、域名、DNS、服务器、数据库——这些词我一个个都认识，但串不起来。我前端写得好好的，`npm run dev` 一启动就能用，为什么上线要折腾这么多东西？

后来我画了一张"一个请求的旅行地图"——从用户在浏览器里输入网址，到页面渲染出数据，中间每一个环节谁干了什么，一步一清。那一瞬间全通了：原来部署不是背命令，是理解**一个请求从出生到死亡走过的所有路**。

你前端天天在写 `fetch("/api/transactions")`，这个请求背后经过了多少道关卡，今天我们就把它从头到尾走一遍。

## 🔍 核心原理

### 一张地图看懂全局

先画一张"请求的旅行地图"。假设金橘记账部署上线了，用户在浏览器里输入 `https://jinqiu.finance`，打开账单页面——下面就是完整的5站接力：

```
你敲下回车
    |
    v
[浏览器] ---> 拆 URL，准备 HTTP 请求
    |
    v
[ DNS ]  ---> 查域名对应的 IP 地址
    |         jinqiu.finance → 76.76.21.21
    v
[ CDN ]  ---> 就近返回静态文件（index.html, main.js, favicon.ico）
    |         如果 CDN 没有，回源到后端服务器
    v
[后端服务器] --> 收到 /api/transactions?month=2026-06
    |         JWT 鉴权 → Router 匹配 → Controller 处理
    v
[数据库]  ---> SELECT * FROM transactions WHERE user_id=...
    |         磁盘上找到数据，返回给后端
    v
[反向路径] --> 数据库 → 后端 → CDN → DNS → 浏览器 → 渲染到屏幕
```

用快递来类比，每一步都对应你生活中熟悉的东西：

| 快递场景 | 部署5站 | 干了什么 |
|----------|---------|----------|
| 你打开购物 App 下单 | **浏览器** | 构造一个 HTTP 请求，比如 `GET /api/transactions` |
| 输入收货地址，系统查邮编 | **DNS** | 把域名 `jinqiu.finance` 翻译成 IP 地址 `76.76.21.21` |
| 就近仓库有货直接发 | **CDN** | 静态文件（HTML/CSS/JS/图片）在离用户最近的节点取，不用跑到源站 |
| 厨房按订单做菜 | **后端服务器** | 接收请求，鉴权，查数据库，拼 JSON，返回响应 |
| 冰箱里取食材 | **数据库** | 从磁盘上读写持久化数据，SELECT/INSERT/UPDATE/DELETE |

> **阿珊说：部署就是给每个请求安排好这几个"快递站点"，站点之间怎么连接、谁先谁后、断了怎么办——这些搞清楚了，部署就通了九成。**

### 第一站：浏览器 — 请求的起点

你在浏览器里做的事，本质上就是一句话：发起 HTTP 请求。

金橘记账前端里这行代码：

```typescript
// frontend/src/services/api.ts
const res = await fetch(`${BASE_URL}/transactions?month=2026-06`, {
  headers: { "Authorization": `Bearer ${accessToken}` }
});
```

这行 `fetch` 就是一切的起点。浏览器把你的请求打包成一个 HTTP 报文，里面装着三样东西：

- **请求行**：`GET /api/transactions?month=2026-06 HTTP/1.1` — 告诉服务器你要干什么
- **请求头**：`Authorization: Bearer eyJhbG...`、`Content-Type: application/json` — 附加信息
- **请求体**：POST 请求才有的数据，GET 请求没有

浏览器在发出请求之前，还得先知道把请求发到哪——这就需要 DNS。

### 第二站：DNS — 互联网的电话簿

你说 `jinqiu.finance`，服务器只认 `76.76.21.21`。DNS（Domain Name System）就是那个翻译的人。

过程比你想象的简单：

```
浏览器问本地 DNS：jinqiu.finance 的 IP 是啥？
本地 DNS 查缓存 → 有就直接返回
               → 没有就问根 DNS → 根 DNS 说"去问 .finance 的"
                               → .finance DNS 说"去问 jinqiu 的"
                               → jinqiu 的 DNS 说"76.76.21.21"
```

前端有个近似的概念：`/etc/hosts` 文件。你在 Windows 的 `C:\Windows\System32\drivers\etc\hosts` 里写 `127.0.0.1 myapp.local`，浏览器访问 `myapp.local` 就直接解析到本地。DNS 就是这个文件的互联网级别版本——只不过不是手写的，是分层的、分布式的、全球同步的。

你把金橘记账部署到 Vercel，Vercel 自动分配一个域名 `jinqiu.vercel.app`。后来你买了个独立域名 `jinqiu.finance`，加了一条 CNAME 记录指向 `jinqiu.vercel.app`——这就是你在 DNS 里做的事情。后面第5篇会详细讲。

### 第三站：CDN — 就近取货

DNS 解析出 IP 之后，浏览器拿到 IP，开始建立连接。如果这个 IP 指向的是 CDN 节点（比如 Vercel 的全球边缘网络），那么：

- 静态文件（`index.html`、`main-abc123.js`、`favicon.ico`）直接从 CDN 返回，就近访问，几十毫秒搞定
- 没有缓存或者 API 请求，CDN 做一层代理，转发到源站（你的后端服务器）

你在前端写的这行就是 CDN 存在的意义：

```html
<!-- index.html -->
<script type="module" src="/assets/main-abc123.js"></script>
```

这个 JS 文件有 500KB，如果全世界用户都从你一台美国服务器拉，慢了不说，流量费也扛不住。CDN 把它缓存在全球几百个节点上，东京用户从东京拿，伦敦用户从伦敦拿。

### 第四站：后端服务器 — 厨房做菜

API 请求绕过 CDN 缓存（或者 CDN 回源），到达你的后端服务器。金橘记账的后端是 FastAPI，看它怎么处理一个请求：

```python
# backend/app/main.py
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI(title="金橘记账 API", version="0.1.0")

# 第一步：CORS 中间件 — 检查请求来源是不是允许的域名
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.ALLOWED_ORIGINS.split(","),
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# 第二步：Router 路由 — 匹配 URL 到处理函数
app.include_router(transactions.router, prefix="/api", tags=["transactions"])

# 第三步：Controller 处理函数 — 鉴权 + 查数据库 + 返回 JSON
# backend/app/routers/transactions.py
@router.get("/transactions")
def list_transactions(
    month: str | None = None,
    user: User = Depends(get_current_user),  # JWT 鉴权
    db: Session = Depends(get_db),            # 数据库连接
):
    query = db.query(Transaction).filter(Transaction.user_id == user.id)
    if month:
        query = query.filter(Transaction.tx_date.like(f"{month}%"))
    return query.order_by(Transaction.tx_date.desc()).limit(50).all()
```

后端做的事说穿了就三步：**收请求 → 查数据 → 返回 JSON**。但每一步都有细节——CORS 不配好前端跨域报错、JWT 鉴权没通过返回 401、数据库连接用完没关服务器内存泄漏。后续文章会逐个拆开。

### 第五站：数据库 — 冰箱取食材

后端处理函数调用 `db.query(Transaction).filter(...)` 的时候，SQLAlchemy 把它翻译成 SQL，发给数据库：

```sql
SELECT * FROM transactions 
WHERE user_id = 'abc-123' 
  AND tx_date LIKE '2026-06%' 
ORDER BY tx_date DESC 
LIMIT 50;
```

数据库从磁盘上找到数据，返回给后端，后端拼成 JSON，沿着来时的路反向传回去：

```json
[
  { "id": "1", "type": "expense", "amount": 3500, "category": "餐饮", ... },
  { "id": "2", "type": "expense", "amount": 1500, "category": "交通", ... }
]
```

这条 JSON 经过服务器 → CDN → DNS → 浏览器，最终变成你眼里的一行行账单。

### 前端类比：每个部署概念都有你熟悉的影子

| 部署概念 | 前端概念 | 说明 |
|---------|---------|------|
| DNS 解析 | `import` 路径解析 | Vite 把 `import "@/components/TabBar"` 中的 `@` 解析成 `src/`，DNS 把域名解析成 IP。都是"别名找实体" |
| CDN 缓存 | `Cache-Control: max-age=3600` | 前端可以用 HTTP 缓存头控制浏览器缓存，CDN 是缓存在服务端边缘节点 |
| Nginx / 反向代理 | Vite dev server proxy | 你在 `vite.config.ts` 里配 `proxy: { "/api": "http://localhost:8000" }`，跟 Nginx 把请求转发到后端一模一样 |
| CORS | `Content-Security-Policy` | 浏览器安全策略——CORS 管"谁可以跨域请求"，CSP 管"谁可以加载资源" |
| 环境变量 | `.env` 文件 | 开发用 `http://localhost:8000`，生产用 `https://api.jinqiu.finance`，同一个变量名不同环境不同值。金橘记账里 `VITE_API_URL` 就是这个模式 |
| HTTPS / SSL | `fetch` 走 `https://` | 没有 HTTPS 浏览器标记"不安全"，你的 JWT token 明文裸奔在公网上 |
| 服务器进程 | `npm run dev` | 一个长期运行的程序，监听端口，等待请求。前端 `vite` 也是服务器，只是它只跑在本地 |
| 数据库连接 | `localStorage.getItem()` | 都是持久化存取——localStorage 存在浏览器里，数据库存在服务器磁盘上 |

## 🛠 动手试试

### 用 DevTools Network 面板看一次完整的请求旅行

打开你的浏览器（Chrome/Edge），访问任意一个线上网站，这里我们用金橘记账本地版做演示。

**第1步：打开 Network 面板**

按 `F12` 打开 DevTools → 切换到 **Network** 标签页 → 勾选 "Preserve log"（防止页面刷新丢数据）。

**第2步：发起一个请求**

打开金橘记账前端 `http://localhost:5174`，登录后进入账单页面。你会看到 Network 面板里刷出一排请求。

**第3步：点开一个请求，看完整环节**

点击 `/transactions?month=2026-06` 这条请求，你会看到几个重要分区：

- **Headers 标签** — 看 Request Headers：
  ```
  GET /api/transactions?month=2026-06 HTTP/1.1
  Host: localhost:8002
  Authorization: Bearer eyJhbGciOi...  ← JWT token，后端鉴权靠它
  Content-Type: application/json
  Origin: http://localhost:5174          ← 后端 CORS 检查这个值
  Referer: http://localhost:5174/        ← 从哪个页面发起的
  ```
  请求头里最关键的三个：`Host` 告诉服务器要找哪个站点（一台服务器可能托管多个网站），`Authorization` 证明你是谁，`Origin` 帮你通过 CORS 检查。

- **Response 标签** — 看 Response Headers + 响应体：
  ```
  HTTP/1.1 200 OK
  content-type: application/json
  access-control-allow-origin: http://localhost:5174  ← CORS 放行
  content-length: 2847
  ```
  响应体就是后端返回的 JSON 数据，所有账单记录都在里面。

- **Timing 标签** — 看请求到底在哪个阶段花了时间：

  把鼠标悬停在 Timing 的每个色块上：

  | 时间段 | 含义 | 部署对应环节 |
  |--------|------|------------|
  | **Queueing** | 浏览器排队等待 | 浏览器太忙，请求在队列里等 |
  | **DNS Lookup** | 查域名对应 IP | DNS 查地址 |
  | **Initial connection** | TCP 三次握手 + SSL 握手 | 和服务器建立加密连接 |
  | **Request sent** | 请求发出去了 | 数据在去服务器的路上 |
  | **Waiting (TTFB)** | 服务器处理中，等待第一个字节回来 | 后端查数据库、拼 JSON 的时间 |
  | **Content Download** | 收到第一个字节后下载完整响应 | 数据在回来的路上 |

  想一想：DNS Lookup 对应快递的"查邮政编码"，Waiting 对应"厨房做菜的时间"，Content Download 对应"快递在路上"。

**第4步：感受"请求的旅行"**

同一个页面刷新，第二次加载你会发现 Timing 里 DNS Lookup 和 SSL 握手时间几乎为 0——浏览器做了 DNS 缓存和连接复用。这就是为什么线上要配 CDN 和 HTTP/2：把重复的代价减到最小。

### 补充实验：用 ping 和 nslookup 看 DNS

打开终端，跑两条命令感受 DNS 的存在：

```bash
# 查域名对应的 IP
nslookup vercel.com
# 输出：Address: 76.76.21.21

# 看延迟 — 不同地理位置到服务器的延迟
ping vercel.com
# 输出：Reply from 76.76.21.21: bytes=32 time=48ms TTL=56
```

你把 `vercel.com` 换成你部署的域名，`ping` 的 `time=xxms` 就是用户到服务器的物理延迟。为什么 CDN 重要？因为如果你的服务器在新加坡，北京用户 ping 过去 200ms，CDN 节点在北京只要 10ms——差距就是用户体验。

## ⚠️ 常见坑点

| 坑 | 现象 | 解法 |
|----|------|------|
| **域名买了没解析** | 浏览器 "ERR_NAME_NOT_RESOLVED" — DNS 查不到 IP，就像寄快递地址写了个不存在的地名 | 去域名服务商（阿里云/Cloudflare）后台加 A 记录或 CNAME 记录。加完之后等几分钟到 48 小时（DNS 传播延迟） |
| **HTTPS 证书没配** | 浏览器大红锁，"您的连接不是私密连接" | 用 Vercel/Netlify 这种平台自动签发 Let's Encrypt 免费证书。自建服务器用 `certbot` 一键搞定 |
| **CORS 拦请求** | 前端控制台红字：`Access to fetch at 'https://api.xxx.com' from origin 'https://xxx.com' has been blocked by CORS` | 后端加上 CORS 中间件，`allow_origins` 把前端域名加进去。金橘记账的 `settings.ALLOWED_ORIGINS` 就是这个作用 |
| **环境变量没改** | 上线了前端还在请求 `localhost:8000`，线上环境调本地地址当然连不上 | 前端 `.env.production` 配 `VITE_API_URL=https://api.jinqiu.finance`，后端 `.env` 配 `ALLOWED_ORIGINS=https://jinqiu.finance`。部署平台也要配 |
| **数据库连不上** | 后端 500 错误，日志里 `Connection refused` | SQLite 是文件数据库，随便放就行。PostgreSQL 需要配用户名、密码、Host、Port、数据库名——少一个就连不上。另外云数据库通常有 IP 白名单，你的服务器 IP 要加进去 |
| **端口没开** | `curl localhost:8000` 通了，公网访问不了 | 云服务器安全组/防火墙默认只开 80（HTTP）和 443（HTTPS）。你的后端跑在 8000 端口，要么配 Nginx 反向代理，要么去安全组加规则 |

## 🎯 面试会问

- **Q：从输入 URL 到页面展示，经过了哪些步骤？越详细越好。**
  **答：** 这道题面试官想听的是一条完整路径，不是零散的单词。标准答案是：
  1. **浏览器解析 URL**：提取协议（https）、域名（jinqiu.finance）、路径（/）、端口（默认 443）。
  2. **DNS 解析**：浏览器缓存 → hosts 文件 → 本地 DNS → 根 DNS → 顶级域 DNS → 权威 DNS，逐级查询到 IP 地址。
  3. **TCP 三次握手**：SYN → SYN-ACK → ACK，建立可靠连接。HTTPS 还要加一步 TLS 四次握手（协商加密算法 + 交换证书 + 生成对称密钥）。
  4. **发送 HTTP 请求**：请求行 + 请求头 + 请求体（POST 时有），沿着网线/光纤到达服务器。
  5. **服务器处理**：Nginx 收到请求 → 如果是静态文件直接返回，API 请求转发给 FastAPI → 中间件（CORS、JWT 鉴权）→ Router 匹配 → Controller 处理 → 查数据库 → 拼 JSON。
  6. **响应返回**：HTTP 响应（状态码 + 响应头 + 响应体）沿着网络原路返回浏览器。
  7. **浏览器解析渲染**：解析 HTML 构建 DOM 树 → 解析 CSS 构建 CSSOM → 合并成渲染树 → 布局（Layout）→ 绘制（Paint）→ 合成（Composite）。遇到 `<script>` 标签暂停解析，下载并执行 JS。
  8. **JS 发起二次请求**：React 应用 hydration 之后，`useEffect` 里 `fetch("/api/transactions")`，再次发起请求，重复步骤 1-6，拿到数据后 setState 更新 DOM。
  
  面试加分的说法："在金橘记账项目里，第 1 步到第 4 步是 Vercel 帮我做的（CDN + HTTPS），第 5 步是 PythonAnywhere 上的 FastAPI 做的，第 6-8 步是 React + antd-mobile 做的。我理解部署要解决的就是第 1-5 步，让第 6-8 步的前端代码能跑到用户手里。"

- **Q：CDN 是什么？什么场景该用？**
  **答：** CDN（Content Delivery Network）是内容分发网络，在全球各地部署边缘节点，把静态文件缓存到离用户最近的节点。用户请求时直接从最近节点取，不用跑到源站。三种典型场景：（1）前端部署——Vercel/Netlify 自带全球 CDN，`main-abc123.js` 全球访问都快；（2）图片/视频——OSS + CDN，用户上传的图片自动分发；（3）API 加速——CDN 可以缓存 GET 请求的结果，读多写少的接口加几秒 CDN 缓存，后端压力骤降。不需要 CDN 的场景：实时性要求高的数据（银行余额、股票行情），或者纯内网系统。

- **Q：HTTP 请求头和响应头里有哪些跟部署相关的字段？**
  **答：** （1）`Host`：一台服务器可能托管多个域名（虚拟主机），服务器靠这个头区分你是要找哪个站；（2）`Origin`/`Referer`：CORS 靠 Origin 判断是否放行；（3）`Authorization`：JWT token 放在这里，后端鉴权；（4）`access-control-allow-origin`（响应头）：后端告诉浏览器哪些域名可以跨域请求；（5）`Cache-Control`（响应头）：`max-age=31536000` 告诉 CDN 和浏览器缓存一年；（6）`Content-Encoding: gzip`（响应头）：服务器做了压缩，浏览器自动解压，传输体积减 70%。

## 📝 小结

部署说起来复杂，拆开就是5站接力。你现在闭上眼睛应该能说出这一条链路：

**浏览器（发起请求）→ DNS（域名翻译）→ CDN（静态文件就近拿）→ 后端（处理业务逻辑）→ 数据库（读写数据）→ 原路返回。**

每一站只干一件事，跟快递从下单、查地址、就近仓库、厨房做菜、冰箱取食材——一模一样。以后遇到部署问题，不要慌，问自己一句："哪个环节出了故障？"——连不上是 DNS，证书报错是 HTTPS，跨域是 CORS，500 是后端，响应慢可能是数据库没建索引。
