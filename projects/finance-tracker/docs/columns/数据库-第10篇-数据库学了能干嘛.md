# 第10篇：数据库学了能干嘛？— 全栈/数据分析/爬虫三条路

> 数据库不是终点，是基础设施。学会数据库之后，三条路全通了：全栈开发（FastAPI+PG）、数据分析（Pandas+SQL）、爬虫工具（采集+存储）

## 📌 前端为什么要学这个？

上个月我在做金橘记账项目的时候，突然在电脑前愣了一下——以前觉得"好难"的东西，怎么不知不觉都变简单了？

后端不就是读数据库+写数据库吗？数据分析不就是 SELECT + GROUP BY 吗？爬虫不就是采集数据 + INSERT 吗？

这感觉特别像你第一次学 CSS 的时候。学之前觉得"哇，网页怎么画出来的"，学完之后发现：原来网页能做，小程序也能做，甚至做张海报、调个界面配色，用的都是同一套 CSS 思维。**一通百通。**

数据库就是这样——它不是终点，它是一把钥匙。打开之后，三条技术路线的门同时开了。

## 🔍 核心原理

先回到前端视角帮你理解这个事。

你写 React 的时候，组件的状态要么是本地 `useState`，要么是全局 Context。不管放哪，本质都是在"存"东西，然后"取"出来用。

数据库就是把这个逻辑搬到服务端：**数据不再存在浏览器的内存里，而是存在硬盘上的一个专业仓库里。** 这个仓库不仅帮你存，还帮你查、帮你算、帮你保证数据不会丢。

此处配图：一个圆分成三块，分别标注"全栈开发""数据分析""爬虫工具"，中间是"数据库"，箭头从中心指向三块，表示数据库支撑三条路。

三条路具体是什么思路呢？

**第一条路：全栈开发。** 这条路最直接。你学会用 SQLAlchemy 定义模型（就是 Python 类，每个类对应数据库里一张表），然后通过 FastAPI 把数据暴露给前端。前端发一个 GET 请求，后端去数据库查一圈，把结果 JSON 化再丢回去——一条完整的数据链路就通了。

**第二条路：数据分析。** 这条路对于写过 SQL 的人来说几乎零门槛。你用 Pandas 读数据的时候，脑子里想的其实是 `SELECT * FROM table WHERE ...`。数据库里的聚合函数 `SUM`、`COUNT`、`GROUP BY`，在 Pandas 里全有对应。等于说你的 SQL 知识直接平移过去了。

**第三条路：爬虫工具。** 爬虫的本质分两步：从网上抓数据（requests/httpx），然后把数据存起来。第二步"存起来"就是你写 `INSERT INTO` 的本事。存进数据库之后，你想怎么查就怎么查，想怎么分析就怎么分析。

> 阿珊说：数据库不是学了才有用，是用着用着你会发现——妈的，我以前怎么没学。

## 🛠 动手试试

我们用金橘记账项目的真实代码，看看同一套数据库知识怎么支撑三条路线。

### 路线一：全栈开发（FastAPI + SQLAlchemy + React）

这是金橘记账的数据库配置，SQLite 开发和 PostgreSQL 上线用的同一套代码，切换只需要改一行 `DATABASE_URL`：

```python
# backend/app/database.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, DeclarativeBase
from app.config import settings

engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {},
)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

class Base(DeclarativeBase):
    pass

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

模型就是一个 Python 类，继承 `Base`，每个属性映射到表的一列。账单表长这样：

```python
# backend/app/models/transaction.py
class Transaction(Base):
    __tablename__ = "transactions"

    id: Mapped[str] = mapped_column(String(36), primary_key=True, default=lambda: str(uuid.uuid4()))
    user_id: Mapped[str] = mapped_column(String(36), ForeignKey("users.id"), nullable=False)
    type: Mapped[str] = mapped_column(String(10), nullable=False)  # "expense" | "income"
    amount: Mapped[int] = mapped_column(Integer, nullable=False)    # 单位：分
    category: Mapped[str] = mapped_column(String(50), nullable=False)
    category_emoji: Mapped[str | None] = mapped_column(String(10), nullable=True)
    note: Mapped[str | None] = mapped_column(String(200), nullable=True)
    tx_date: Mapped[date] = mapped_column(Date, nullable=False)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow)
```

有了模型，增删改查就是语义化的 Python 代码，而不是写字符串 SQL：

```python
# 查：列表查询 + 筛选 + 分页
query = db.query(Transaction).filter(Transaction.user_id == user.id)
if month:
    query = query.filter(Transaction.tx_date.like(f"{month}%"))
if category:
    query = query.filter(Transaction.category == category)
items = query.order_by(Transaction.tx_date.desc()).offset((page-1)*page_size).limit(page_size).all()

# 增
tx = Transaction(user_id=user.id, type=req.type, amount=req.amount, category=req.category, ...)
db.add(tx)
db.commit()
db.refresh(tx)

# 聚合查询：本月收入和支出
income = db.query(func.sum(Transaction.amount)).filter(
    Transaction.user_id == user.id,
    Transaction.type == "income",
    Transaction.tx_date.like(f"{month}%"),
).scalar() or 0
```

前端消费这些数据也直观得不行——就是调接口，拿数据，渲染：

```typescript
// frontend/src/services/api.ts — 前端 API 封装
getTransactions: (params: Record<string, string>) => {
  const qs = new URLSearchParams(params).toString();
  return authFetch(`/transactions?${qs}`).then(r => r.json());
},
getDashboard: (month?: string) =>
  authFetch(`/dashboard${month ? `?month=${month}` : ""}`).then(r => r.json()),
```

你看，一条链路走通：React 组件调 `api.getDashboard()` → FastAPI 用 SQLAlchemy 查数据库 → JSON 返回 → 前端渲染。**这就是全栈开发的核心数据流，没你想的那么神秘。**

### 路线二：数据分析（SQL → Pandas）

假设你想分析金橘记账的数据：这个月钱都花哪了？哪几天花得最多？

你的 SQL 知识直接平移过来。先看数据库里怎么查分类占比：

```python
# backend/app/routers/stats.py — 金橘记账分类统计接口
results = (
    db.query(
        Transaction.category,
        Transaction.category_emoji,
        func.sum(Transaction.amount).label("total")
    )
    .filter(
        Transaction.user_id == user.id,
        Transaction.type == "expense",
        Transaction.tx_date.like(f"{month}%"),
    )
    .group_by(Transaction.category)
    .order_by(func.sum(Transaction.amount).desc())
    .all()
)

# 计算每个分类的占比
for r in results:
    pct = round(r.total / expense_total * 100) if expense_total > 0 else 0
    # r.category, r.emoji, r.total, pct → 扔给前端画图
```

如果你把数据导出成 CSV，用 Pandas 做同样的分析，思路完全一样：

```python
import pandas as pd

df = pd.read_csv("transactions.csv")
# 过滤支出
expenses = df[df["type"] == "expense"]
# 按分类聚合
result = expenses.groupby("category").agg(
    total=("amount", "sum"),
    count=("id", "count")
).sort_values("total", ascending=False)
# 加一个占比列
result["pct"] = (result["total"] / result["total"].sum() * 100).round(1)
```

你能明显看出来：Pandas 的 `groupby().agg()` 就是 SQL 的 `GROUP BY + SUM/COUNT`；Pandas 的 `sort_values()` 就是 `ORDER BY`。**你学的不是 SQL 语法，你学的是数据处理思维。** 这个思维在数据库里能用，在 Pandas 里也能用，在 Excel 透视表里也能用。

### 路线三：爬虫工具（采集 + 存储）

爬虫的逻辑出奇简单：抓数据，存数据库。存的部分你已经会了：

```python
import requests
from bs4 import BeautifulSoup

# 抓取某个网站的数据
resp = requests.get("https://example.com/articles")
soup = BeautifulSoup(resp.text, "html.parser")

for item in soup.select(".article-item"):
    title = item.select_one(".title").text
    link = item.select_one("a")["href"]

    # 存进数据库 — 这部分和写账单没区别！
    article = Article(title=title, url=link)
    db.add(article)

db.commit()
print(f"存入了 {count} 条数据，可以用 SQL 随便查了")
```

数据存进数据库之后，你就拥有了一个私人数据仓库。想查什么写 SQL 就行：`SELECT * FROM articles WHERE title LIKE '%数据库%'`。这比存 JSON 文件高级在：你可以跨表关联、可以建索引加速查询、可以做聚合统计。

三条路都走通了。底层都是同一套数据库能力，只是上层的应用场景不同。

## ⚠️ 常见坑点

| 坑 | 现象 | 解法 |
|----|------|------|
| SQLite 切换 PostgreSQL 后 `check_same_thread` 报错 | 上线后发现服务起不来，日志里一堆连接错误 | 金橘记账 `database.py` 里已经处理了：`connect_args={"check_same_thread": False} if "sqlite" in DATABASE_URL else {}`。SQLite 需要这个参数，PG 不需要，所以用条件判断自动区分 |
| 金额用 float 存导致精度丢失 | 记账应用里 3.50 + 2.30 算出来等于 5.799999... | 金橘记账用**分**做单位，`amount: int`，存整型。前端显示时除以 100。金融、记账类应用一定不要用浮点数存钱 |
| `.all()` 和 `.scalar()` 混用 | 聚合查询明明只有一条结果，却返回一个列表，取数据时报错 | 聚合函数（`func.sum`、`func.count`）用 `.scalar()` 拿标量值；多行查询用 `.all()` 拿列表。金橘记账的 stats 路由里就是这样做的：`db.query(func.sum(...)).filter(...).scalar() or 0` |
| 前端直接拼 SQL 字符串 | 搜索功能被人输入 `'; DROP TABLE users; --`，整个表没了 | 永远不要拼字符串。SQLAlchemy 的 filter 方法是参数化的：`Transaction.tx_date.like(f"{month}%")`，这里的 `month` 你虽然拼接了字符串，但 SQLAlchemy 底层会用参数绑定。更安全的做法是用 `params` 传参 |
| Pandas 日期比较不生效 | `df[df["date"] > "2026-06-01"]` 要么全 True 要么全 False | Pandas 里字符串和日期不能直接比。先用 `df["date"] = pd.to_datetime(df["date"])` 转成日期类型再比较，这和 SQL 里写 `WHERE tx_date > '2026-06-01'` 需要 date 类型是一个道理 |

## 🎯 面试会问

- **Q：说说你对数据库在前端开发中的角色的理解？**
  **答：** 数据库是"后端的大脑"，负责持久化存储和高效查询。前端不直接操作数据库，而是通过 REST API 或 GraphQL 间接访问。我做金橘记账的时候，前端 React 组件调用 `api.getDashboard()`，后端用 SQLAlchemy 做聚合查询（`func.sum` + `group_by`），把当月余额、周趋势、分类占比一次性算好返回。前端的职责是拿到数据后渲染出好看的图表，后端的职责是用 SQL 高效地把数据算出来。这个分工是理解全栈架构的起点。

- **Q：SQL 和 Pandas 有什么区别？什么时候用哪个？**
  **答：** SQL 适合数据存数据库里的场景——大数据量、多表关联、需要索引加速。Pandas 适合数据已经在内存里（CSV、API 返回的 JSON）的探索性分析——灵活、可视化友好、和 Python 生态无缝衔接。我在金橘记账的 stats 路由里用 SQL 直接算出分类占比（避免把几万条数据从数据库传到 Python 再算），但如果数据是外部 CSV 文件，我就用 Pandas 的 `groupby` 做同样的逻辑。底层思维是一样的，选哪个看数据在哪。

- **Q：爬虫爬回来的数据怎么存？存文件还是存数据库？**
  **答：** 少量临时数据可以存 JSON 文件，但一旦数据量上千条、或者需要按条件查询，必须用数据库。存数据库之后你可以写 SQL 做统计分析、建索引加速查询、把不同来源的数据做关联。比如爬了多个电商平台的商品数据，存进同一个数据库，用 `SELECT category, AVG(price) FROM products GROUP BY category` 就能横向对比——存文件的话做这个分析得写一堆脚本。

## 📝 小结

数据库从来不是你学习路线的终点，它是一个跳板。

前端学会了数据库，后端就没那么神秘了——你知道数据怎么存的、怎么查的，写 API 的时候脑子里有完整的链路。数据分析师学会了数据库，SQL 到 Pandas 的迁移基本无缝——聚合函数、分组、排序，概念是同一套。爬虫工程师学会了数据库，就解决了最头疼的"数据放哪"问题——放数据库里，随便查、随便分析。

从 CSS 到网页/小程序/海报，是一通百通；从数据库到全栈/分析/爬虫，也是一通百通。你学的不是某一个框架的 API，你学的是数据在整个技术栈里流动的底层逻辑。
