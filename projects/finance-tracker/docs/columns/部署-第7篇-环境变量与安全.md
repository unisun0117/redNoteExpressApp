# 部署从0到1 第7篇：环境变量与安全 —— .env 文件的正确姿势

> 把 API Key 写在代码里 push 到 GitHub，等于把家门钥匙挂在门口。

---

## 📌 前端为什么要学

有次我用 JWT 做登录功能，图省事把 secret 直接写在了 `config.py` 里：

```python
JWT_SECRET = "my-super-secret-key-2026"
```

然后 `git add .`、`git commit`、`git push`——一套行云流水的操作。

晚上躺床上刷手机，突然坐起来——我刚才是不是把 secret push 上去了？

打开 GitHub 一看，`config.py` 明晃晃地躺在仓库里，JWT_SECRET 一字不差地展示给全世界。还好是 private 仓库，但后背还是一阵凉。如果是 public 仓库，任何 clone 了项目的人拿着这个 secret 就能伪造 JWT，假装是你 App 的任何一个用户。

从那天起，**.env 必加 .gitignore** 就成了我的肌肉记忆。

但这件事让我意识到另一个问题：**环境变量这门课，前端很少系统学过。**

前端写 `API_BASE_URL = "http://localhost:8000"` 写惯了，改个地址就 `Ctrl+H` 替换。上了生产才知道——开发环境的地址、生产环境的地址、测试环境的地址，全是同一个变量，但值不一样。你不可能每次上线前手动改代码。

> 阿珊说：把配置写死在代码里，就像把 `font-size: 16px` 写在几千行 CSS 里——当时爽，改的时候哭。环境变量就是把所有"会变的东西"提到一个地方统一管理。

---

## 🔍 核心原理

### 环境变量 = React Context

如果你写过 React，理解环境变量只需要一个类比：

**环境变量就是 App 层的 Context Provider。**

```tsx
// ❌ 把值写死在组件里
function App() {
  return <div>API地址：http://localhost:8000</div>;
}

// ✅ 通过 Context 注入
function App() {
  const apiUrl = useContext(ConfigContext);
  return <div>API地址：{apiUrl}</div>;
}
```

你的组件（业务代码）不关心 API 地址具体是什么——它只从 Context 里取值。Context 里配什么，它就用什么。开发环境配 `localhost`，生产环境配 `api.yourdomain.com`——**改配置，不改代码。**

环境变量做的事一模一样：

```
┌─────────────────────────────────────────┐
│              环境变量层                   │
│  .env (开发)    Vercel Env (生产)        │
│  ┌──────────┐   ┌──────────────────┐    │
│  │API_URL=  │   │API_URL=          │    │
│  │localhost │   │api.jinju.com     │    │
│  └──────────┘   └──────────────────┘    │
│         ↓               ↓                │
│  ┌──────────────────────────────────┐    │
│  │        你的应用程序               │    │
│  │   只读 process.env / import.meta │    │
│  │   不关心值从哪来的               │    │
│  └──────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

代码只写 `process.env.API_URL`（后端）或 `import.meta.env.VITE_API_URL`（前端），**不写具体值**。具体值从外部注入——`.env` 文件在开发时注入，平台环境变量面板在生产时注入。

### 三句话理解本质

1. **环境变量是配置，不是代码。** 代码提交给 Git，配置注入给进程。
2. **操作系统原生支持。** 你在终端敲 `echo $PATH`，看到的就是环境变量——程序启动时，操作系统会把环境变量传给进程。
3. **12-Factor App 第三条。** 业界公认的应用开发方法论中，第三条就是"在环境变量中存储配置"——你的代码与配置严格分离，同一份代码可以部署到任意环境。

### .env 文件是什么

`.env` 是一个纯文本文件，格式就是 `KEY=VALUE`：

```bash
# .env
DATABASE_URL=sqlite:///./finance.db
JWT_SECRET=finance-dev-secret-2026
OPENAI_API_KEY=sk-xxxxxxxxxxxx
ALLOWED_ORIGINS=http://localhost:5173
```

它是**环境变量的本地载体**——只在你的电脑上存在，不上传到 Git。程序启动时，框架或库读取这个文件，把每一行变成一个环境变量注入到进程里。

```python
# Python (pydantic-settings)
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    DATABASE_URL: str  # 自动从 .env 或系统环境变量读取

    class Config:
        env_file = ".env"  # 告诉它去找 .env 文件
```

```javascript
// Vite 前端
const API_URL = import.meta.env.VITE_API_URL;  // Vite 从 .env 注入
```

.env 是 dev Context，平台环境变量面板是 prod Context。同一套取值逻辑，不同环境不同值。

## 🛠 动手试试

### 金橘记账的环境变量实战

金橘记账是一个全栈项目，前端用 Vite + React，后端用 FastAPI。两边的环境变量处理方式不一样，我们分别来看。

#### 第一步：创建 .env 文件

**后端 .env**（`backend/.env`）：

```bash
DATABASE_URL=sqlite:///./finance.db
JWT_SECRET=finance-dev-secret-2026
JWT_ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=15
REFRESH_TOKEN_EXPIRE_DAYS=7
ALLOWED_ORIGINS=http://localhost:5173
```

**前端 .env**（`frontend/.env`）：

```bash
VITE_API_URL=http://localhost:8000/api
```

> Vite 必须用 `VITE_` 前缀，普通的 `REACT_APP_` 不认。后面坑点细说。

#### 第二步：.gitignore 加上 .env

打开项目根目录的 `.gitignore`，确保有这一行：

```gitignore
.env
```

也可以更精确一点，排除所有以 `.env` 开头的文件（有的项目会分 `.env.development`、`.env.production`）：

```gitignore
.env*
```

但是！你可能会加一个 `.env.example` 文件传上去——它不含真实 key，只是告诉队友"你需要配哪些变量"：

```bash
# .env.example（可以提交到 Git）
DATABASE_URL=sqlite:///./finance.db
JWT_SECRET=your-secret-here
JWT_ALGORITHM=HS256
ALLOWED_ORIGINS=http://localhost:5173
```

**规则很简单：有真实值的 .env 不上传，模板 .env.example 可以上传。**

#### 第三步：后端用 pydantic-settings 读取

金橘记账后端用的是 FastAPI + pydantic-settings，这是业界最优雅的配置管理方式。

```python
# backend/app/config.py
import os
from pydantic_settings import BaseSettings


class Settings(BaseSettings):
    DATABASE_URL: str = "sqlite:///./finance.db"      # 默认值
    JWT_SECRET: str = "finance-dev-secret-2026"        # 默认值（仅开发用）
    JWT_ALGORITHM: str = "HS256"
    ACCESS_TOKEN_EXPIRE_MINUTES: int = 15
    REFRESH_TOKEN_EXPIRE_DAYS: int = 7
    ALLOWED_ORIGINS: str = "http://localhost:5173"

    class Config:
        env_file = ".env"       # 告诉它去找 .env 文件


settings = Settings()           # 全局单例，别处 import 就行
```

这段代码的精妙之处有四点：

1. **类型安全**：每个字段都标注了类型（`str`、`int`），读错了 pydantic 会直接报错。
2. **默认值**：每个字段后面都有 `= "默认值"`——没找到环境变量就用这个。开发时不用配也能跑。
3. **优先级**：系统环境变量 > `.env` 文件 > 默认值。部署时平台环境变量自动覆盖 .env
4. **单例模式**：`settings = Settings()` 是模块级单例，其他文件 `from app.config import settings` 直接拿

在鉴权中间件里用起来是这样的：

```python
# backend/app/middleware/auth.py
from app.config import settings

def decode_token(token: str) -> dict:
    payload = jwt.decode(
        token,
        settings.JWT_SECRET,           # 从环境变量来的 secret
        algorithms=[settings.JWT_ALGORITHM]
    )
    return payload
```

你的代码里永远不会出现 `"my-secret-2026"` 这种硬编码字符串——它只出现在 `.env` 里，而 `.env` 永远不会上传。

#### 第四步：前端用 Vite 环境变量

金橘记账前端也很简洁，只有**一个**环境变量——API 地址。

```typescript
// frontend/src/services/api.ts
const BASE_URL = import.meta.env.VITE_API_URL || "http://localhost:8000/api";
```

注意 `|| "http://localhost:8000/api"` 这个兜底值——当环境变量没配时，自动 fallback 到默认值。保证代码在任何时候都不会因为缺一个环境变量就炸。

有了 `BASE_URL`，所有接口都在这个基础上拼接：

```typescript
async function authFetch(path: string, options: RequestInit = {}): Promise<Response> {
  const headers: Record<string, string> = { ...(options.headers as Record<string, string>) };
  if (accessToken) headers["Authorization"] = `Bearer ${accessToken}`;

  let res = await fetch(`${BASE_URL}${path}`, { ...options, headers });
  // ... token 刷新逻辑
  return res;
}
```

`BASE_URL` 的值决定了 API 请求打到哪——本地开发打到 `localhost:8000`，生产环境打到 `api.jinju.com`。**同一行 `import.meta.env.VITE_API_URL`，在不同环境读出不同值。**

#### 第五步：Vercel 配环境变量

前端部署到 Vercel 后，`.env` 文件不会上传（被 `.gitignore` 屏蔽了）。你需要在 Vercel 项目设置里手动配：

1. 打开 Vercel Dashboard → 你的项目 → Settings → Environment Variables
2. 添加一条：
   - **Key**：`VITE_API_URL`
   - **Value**：`https://jinju-api.onrender.com/api`
   - **Environment**：勾选 Production 和 Preview
3. 点 Save

配完之后，**重新触发一次部署**（随便 push 个 commit 或点 Redeploy）。Vite 在 build 时会把 `VITE_API_URL` 的值**内联编译进 JS 文件**。所以你改了环境变量后不重新部署，读到的还是旧值——这是最常见的坑。

#### 第六步：Render 配环境变量

后端部署到 Render 也同理：

1. Render Dashboard → 你的 Web Service → Environment
2. 添加环境变量：
   - `DATABASE_URL` = Neon 连接字符串
   - `JWT_SECRET` = 生产环境的随机 secret
   - `ALLOWED_ORIGINS` = `https://jinju.vercel.app`
3. 点 Save Changes，Render 会自动重启服务

> 生产环境的 `JWT_SECRET` 一定要用随机强密码——打开终端跑 `openssl rand -hex 32` 拿一个 64 位十六进制字符串，别用"my-secret"这种。

#### 第七步：验证环境变量是否生效

部署完怎么确认环境变量配对了？

**后端验证**——FastAPI 加一个调试接口（只给管理员用）：

```python
@router.get("/debug/config")
def debug_config():
    return {
        "database": settings.DATABASE_URL[:30] + "...",  # 只显示前30个字符
        "jwt_secret_set": bool(settings.JWT_SECRET),      # 不暴露具体值
        "origins": settings.ALLOWED_ORIGINS,
    }
```

访问 `https://jinju-api.onrender.com/api/debug/config`，看到 `jwt_secret_set: true` 就说明配上了。

**前端验证**——打开浏览器控制台，看看实际发出的请求打到了哪个地址。Network 面板里看 XHR 请求的域名——如果是 `jinju-api.onrender.com` 而不是 `localhost`，说明环境变量生效了。

## ⚠️ 常见坑点

### 坑1：.env 没加 .gitignore

**症状**：GitHub 仓库里明晃晃躺着 `.env` 文件，所有 secret 暴露。

**排查**：`git ls-files | grep .env`——如果输出不为空，说明 .env 已经被跟踪了。

**修复**：

```bash
git rm --cached .env          # 从 Git 跟踪中移除，但保留本地文件
echo ".env" >> .gitignore      # 加到 .gitignore
git add .gitignore
git commit -m "chore: remove .env from git tracking"
```

**如果已经 push 到 public 仓库**——泄露的所有 key、secret、token **立刻去对应平台吊销换新的**。GitHub 的 commit 历史删不掉——即使你新 commit 删了文件，旧 commit 里还是能看到。唯一的正确姿势是让旧的 secret 失效。

**预防**：在项目刚开始时就加 `.gitignore`，**先加后写代码**。金橘记账的 `.gitignore` 模板：

```gitignore
# 环境变量
.env
.env.local
.env.*.local

# Python
__pycache__/
venv/
*.pyc

# Node
node_modules/
dist/
```

### 坑2：Vite 环境变量忘了 `VITE_` 前缀

**症状**：在 `.env` 里写了 `API_URL=http://localhost:8000`，但在代码里 `import.meta.env.API_URL` 拿不到，值是 `undefined`。

**原因**：Vite 出于安全考虑，**只暴露 `VITE_` 前缀的环境变量给客户端代码**。这是有意为之——防止后端 secret 被不小心暴露到前端 bundle 里。如果你在 `.env` 里写了 `DATABASE_URL`，Vite 不会把它暴露给浏览器——这恰恰是好事。

**正确写法**：

```bash
# ❌ 错误——Vite 不暴露
API_URL=http://localhost:8000

# ✅ 正确——必须 VITE_ 前缀
VITE_API_URL=http://localhost:8000
```

```typescript
// ✅ 能拿到
const url = import.meta.env.VITE_API_URL;

// ❌ undefined
const url = import.meta.env.API_URL;
```

**前端类比**：Vite 的 `VITE_` 前缀机制就像 `prop-types`——只暴露你显式声明过的变量，没声明的一律过滤掉。防止你一不小心把 `SECRET_KEY` 泄露到前端 bundle。

### 坑3：改了环境变量没重新部署

**症状**：在 Vercel 面板改了 `VITE_API_URL`，刷新页面还是连的旧地址。

**原因**：Vite 在 **build 阶段**把环境变量的值内联编译进 JS 文件了。你在 Vercel 面板改的值，必须**重新触发 build** 才能生效。

```
VITE_API_URL 的值 —> Vite build 时读 —> 硬编码进 dist/assets/index-xxx.js
                         ↑
                  只有重新 build 才会重新读
```

**修复**：改了环境变量后，Vercel 上点 "Redeploy"，Render 上点 "Manual Deploy" → "Deploy latest commit"。

**Render 的情况好一点**——后端是运行时读取环境变量，重启服务就能读到新值。但 Vercel 是 build 时就把值编译进静态文件了，所以必须重新 build。

### 坑4：前端环境变量不是秘密

**最重要的一条**：Vite 的环境变量会被**编译进前端 JS bundle**。任何人在浏览器里打开 DevTools → Sources → 你的 JS 文件，搜索 `VITE_`，你的环境变量值一览无余。

```bash
# ❌ 绝对不要在前端 .env 里放这些
VITE_DATABASE_PASSWORD=xxx         # 数据库密码暴露给所有人
VITE_OPENAI_API_KEY=sk-xxx         # API Key 暴露给所有人
VITE_JWT_SECRET=xxx                # JWT Secret 暴露给所有人
```

**前端环境变量只能放"不介意被人看到"的配置**：API 地址、站点名称、Google Analytics ID——这些东西本身就是公开的。

**所有 secret 只配在后端环境变量里。** 这是铁律。

### 坑5：.env 值里有空格或引号

**症状**：写了 `JWT_SECRET = " my-secret "`，结果运行时读到的值带着引号和空格。

**原因**：`.env` 文件不是 JSON，不是 Python，不是 JavaScript——它就是纯 `KEY=VALUE`：

```bash
# ❌ 错误——引号会被当作值的一部分
JWT_SECRET="my-secret"
# 实际读到的是: "my-secret"（带着双引号）

# ❌ 错误——等号两边的空格也会被吃掉或保留，行为不一致
JWT_SECRET = my-secret

# ✅ 正确——无引号、无空格
JWT_SECRET=my-secret
```

## 🎯 面试会问

### Q1：前端环境变量安全吗？

**不安全。** 所有前端的"环境变量"在 build 后都变成了 JS bundle 里的明文字符串。任何人打开浏览器 DevTools → Sources 就能看到。

**但这不是问题**，因为前端环境变量本来就不应该放 secret。放的是 API 地址、Google Analytics ID、站点标题——这些信息本身就是"前端运行时需要知道"的公开信息。

真正的 secret（API Key、数据库密码、JWT Secret）全放在后端环境变量里——用户浏览器永远接触不到你的后端进程内存。

> 一句话：**前端的"环境变量"是"配置"不是"秘密"，后端的"环境变量"才是"秘密"。**

### Q2：为什么不用代码写配置，非要搞环境变量？

本质原因：**同一份代码要跑在多个环境里。**

```
同一个 main.py 文件：
- 开发者A的电脑 → .env 配 localhost
- 开发者B的电脑 → .env 配 localhost:8001
- Render 生产环境 → 环境变量配 Neon PostgreSQL
- CI/CD 测试环境 → 环境变量配 SQLite in-memory
```

如果写死在代码里，你要维护四份代码。用环境变量，**一份代码，四套配置**。

另一个原因：**容器化和微服务。** Docker 容器启动时注入环境变量比改配置文件方便得多。Kubernetes 的 ConfigMap 和 Secret 底层也是环境变量。这已经成为云原生时代的基础设施标准。

### Q3：.env.example 和 .env 的关系是什么？

`.env` 是你本地的真实配置——有你的 secret，不上传。`.env.example` 是配置模板——告诉大家"这个项目需要哪些变量"，不含真实值，上传到 Git。

```bash
# .env（不上传）
JWT_SECRET=6a8f3c2e1b0d...（你的真实 secret）

# .env.example（上传）
JWT_SECRET=your-jwt-secret-here    # 占位符，告诉队友"你得配这个"
```

新成员 clone 项目后，`cp .env.example .env`，然后填上自己的值——像前端的 `npm install` 一样是标准化的项目初始化步骤。

## 📝 小结

这篇文章的核心只有一句话：**代码归 Git，配置归环境变量。**

- **环境变量 = React Context**：Provider（.env 或平台面板）给值，Consumer（你的代码）取值，Consumer 不关心值从哪来。
- **.env 是开发环境专用**：本地用 `.env` 文件配 → `.gitignore` 屏蔽 → 传到 Git 的就一个 `.env.example` 模板。
- **生产环境在平台配**：Vercel → Settings → Environment Variables；Render → Environment 面板。配完记得重新部署。
- **pydantic-settings 是真优雅**：类型安全 + 默认值 + 自动读 .env + 环境变量优先覆盖，Python 配置管理的最佳实践。
- **Vite 必须 VITE_ 前缀**：故意的安全机制，防止后端 secret 意外泄露到前端 bundle。
- **前端环境变量不是秘密**：所有 `VITE_` 开头的值都会编译进 JS bundle，任何人都能看到。secret 只配在后端。

到今天你的部署全链路已经通了：

- 前端：Vercel 自动部署
- 后端：Render 长期运行
- 数据库：Neon PostgreSQL 云数据库
- **配置：环境变量统管一切**

环境变量就是你的"线上配置中心"——不用它的时候觉得这是个玄学，用了之后才发现：**最好的配置管理，是让代码不知道自己在哪个环境运行。**
