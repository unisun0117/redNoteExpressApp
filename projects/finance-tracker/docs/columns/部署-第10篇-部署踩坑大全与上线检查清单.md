## 🔍 15 个真实部署坑

以下 15 个坑按照"部署顺序"排列——从域名配完开始，到打开浏览器看到正常页面为止。

### 坑 1：CORS 没更新 —— `ALLOWED_ORIGINS` 还写着 localhost

**症状：** 前端页面打开了，所有 API 请求都报 `blocked by CORS policy`。浏览器 Console 里一片红。

**原因：** 浏览器有同源策略——`vercel.app` 发请求给 `onrender.com`，属于跨域请求。后端必须明确告诉浏览器"我允许 `vercel.app` 的请求"，否则浏览器直接拦截。

金橘记账的 CORS 配置在 `backend/app/main.py`：

```python
# main.py
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.ALLOWED_ORIGINS.split(","),
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

而 `ALLOWED_ORIGINS` 在 `config.py` 里的默认值是：

```python
ALLOWED_ORIGINS: str = "http://localhost:5173"   # ← 只有本地地址！
```

**如果你只配了本地地址，部署后 Vercel 的域名 (`https://xxx.vercel.app`) 不在列表里，请求直接被浏览器拦截。** 现象是 Network 面板里请求显示红色 `(blocked)`，Response 是空的，连 500 都看不到——因为请求根本没发到后端。

**修复：**

在 Render 的环境变量面板里，把 `ALLOWED_ORIGINS` 改成包含你前端域名的列表：

```bash
# Render 环境变量
ALLOWED_ORIGINS=https://arashi-finance-tracker.vercel.app
```

如果要允许多个域名，用英文逗号分隔：

```bash
ALLOWED_ORIGINS=https://arashi-finance-tracker.vercel.app,https://www.yourdomain.com
```

改完要**重启服务**让变量生效（Render 上点 Manual Deploy → Restart）。

### 坑 2：环境变量没配 —— 线上忘配 `DATABASE_URL`

**症状：** 本地跑得飞起，一上线就 `500 Internal Server Error`。日志里写着 `Can't connect to database` 或者数据库路径找不到。

**原因：** `DATABASE_URL` 在 `.env` 文件里写的是本地 SQLite 路径：

```bash
# backend/.env（本地开发用）
DATABASE_URL=sqlite:///./finance.db
```

但 `.env` 文件被 `.gitignore` 屏蔽了，不会上传到 Render。Render 上找不到 `.env` 文件，就用 `config.py` 里的默认值。如果你 Render 上**没有手动配** `DATABASE_URL`，后端就拿着 `sqlite:///./finance.db` 去 Render 服务器上找 SQLite 文件——当然找不到。

**更糟的是**，即使找到了（比如你第一次没过脑随手创建了一个），Render 的免费版文件系统**不是持久的**——服务重启后文件就没了（见坑 5）。

**修复：**

在 Render Dashboard → Environment 里配：

```bash
DATABASE_URL=postgresql://neon_user:password@ep-xxx.us-east-2.aws.neon.tech/finance?sslmode=require
```

**Render 配环境变量的步骤：**

1. Render Dashboard → 你的 Web Service → Environment 标签
2. 点 Add Environment Variable
3. Key 填 `DATABASE_URL`，Value 填 Neon 的连接字符串
4. 保存 → 服务自动重启

**前端同理：**

```bash
# Vercel Settings → Environment Variables
VITE_API_URL=https://arashi-finance-api.onrender.com/api
```

别忘了：Vite 改环境变量后必须**重新 build**（点 Vercel 的 Redeploy），否则 JS bundle 里还是旧值。

### 坑 3：Render 15 分钟休眠 —— 用户第一次打开白屏

**症状：** 网站刚上线那几天正常，过几天发现用户说"第一次打开很慢，甚至白屏"。你自己试了一下——确实，第一次打开要等 30-60 秒。再次刷新就秒开了。

**原因：** Render 免费版有一个"隐藏设定"——你的 Web Service 如果 **15 分钟没有收到任何 HTTP 请求**，Render 就会把它休眠（spin down）以节省资源。下一次请求到来时，Render 需要把容器重新启动起来。对于 Python + FastAPI 这个冷启动过程大约需要 **30 到 60 秒**。

30 秒什么概念？用户在浏览器前等了 5 秒没反应，90% 的人会关掉。

```
用户访问 → Render 唤醒容器 → pip install 依赖 → uvicorn 启动 → 健康检查 → 返回响应
                                ↑ 这中间可能要 30-60 秒
```

**修复：UptimeRobot 保活**

[UptimeRobot](https://uptimerobot.com/) 是一个免费服务，每 5 分钟自动访问一次你指定的 URL。有了它，你的 Render 服务就永远不会触发"15 分钟无请求"的休眠条件。

1. 注册 UptimeRobot 账号
2. Add New Monitor → HTTP(s)
3. URL 填 `https://arashi-finance-api.onrender.com/api/health`
4. Monitoring Interval 选 5 minutes
5. 保存

现在，你的 `/api/health` 端点每隔 5 分钟就会被访问一次——Render 看到持续有请求，就不会休眠了。

**金橘记账的 health 端点代码：**

```python
# backend/app/main.py
@app.get("/api/health")
def health():
    return {"status": "ok"}
```

最后别忘了部署完验证一下：直接访问 `https://你的域名.onrender.com/api/health`，看返回的是不是 `{"status": "ok"}`。

### 坑 4：Vercel SPA 路由 404 —— `/login` 直接访问报错

**症状：** 从首页点"登录"按钮，Router 跳转到 `/login`——正常。但**直接刷新页面**或**直接访问 `https://xxx.vercel.app/login`**——Vercel 返回 404。

**原因：** Vercel 是一个静态文件服务器。你访问 `https://xxx.vercel.app` 时，Vercel 去 `dist/` 文件夹里找到 `index.html`，返回给你。React Router 接管页面后，你在页面上点击<kbd>登录</kbd>，Router 在**前端**拦截了 URL 变化，不会真的发 HTTP 请求——所以你看到了 `/login` 页面。

但当你直接在浏览器地址栏输入 `https://xxx.vercel.app/login` 时，这变成了一个**真正的 HTTP 请求**——浏览器向 Vercel 服务器要 `/login` 这个路径的文件。但 Vercel 的服务器上根本没有 `/login` 这个文件夹或文件——它只有一个 `/index.html`。

```
直接访问 /login    → Vercel 去 dist/ 找 login 文件 → 没有 → 404
通过 Router 跳转   → React Router 在前端拦截 URL → 显示 LoginPage 组件 → 正常
```

**修复：`vercel.json` 配 SPA Rewrites**

在 `frontend/` 目录下创建 `vercel.json`：

```json
{
  "rewrites": [
    {
      "source": "/((?!api|assets|_next|favicon.ico).*)",
      "destination": "/index.html"
    }
  ]
}
```

这条配置的意思是：**所有路径（除了静态资源）都返回 `index.html`**。浏览器拿到 `index.html` 后，React Router 会根据当前 URL 渲染对应的页面组件。

它本质就是把"路由匹配"的工作从服务器移交给了前端——**这就是 SPA 的核心工作方式。**

### 坑 5：SQLite 数据丢失 —— Render 重启清文件

**症状：** 上线那几天数据正常，用户可以注测、记账。过了两天重新打开——**数据全没了**，用户也要重新注册。

**原因：** Render 的免费版提供的是**临时文件系统（Ephemeral Filesystem）**。你的服务可以写文件、读文件，完全正常——但只要 Render 重启了你的实例（24 小时至少一次，部署也会触发重启），**所有文件全部清空。**

SQLite 的数据存在 `.db` 文件里。文件在 Render 的临时文件系统上。重启 = 文件消失 = 数据消失。

```
部署时 → Render 创建新容器 → 临时文件系统空白 → SQLite 创建 finance.db → 用户注册、记账
重启后 → Render 创建新容器 → 临时文件系统空白 → finance.db 没了 → 用户数据全丢了
```

**这就是为什么生产环境必须用外部数据库（Neon / Supabase / PlanetScale）。**

**修复：用 Neon PostgreSQL 替代 SQLite**

金橘记账早就想到了这一步——`database.py` 的代码已经同时兼容 SQLite 和 PostgreSQL：

```python
# backend/app/database.py
engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {},
)
```

注意 `if "sqlite" in settings.DATABASE_URL else {}`——当 `DATABASE_URL` 是 PostgreSQL 地址时，不传 `check_same_thread` 参数。同一份代码，本地用 SQLite 开发，生产用 PostgreSQL 运行。

**切换到 Neon 只需要三步：**

1. Neon 建项目 → 拿到连接字符串：`postgresql://user:pass@ep-xxx.us-east-2.aws.neon.tech/finance`
2. Render 环境变量改 `DATABASE_URL` 为 Neon 地址
3. 重启 Render 服务 → FastAPI 启动时 `Base.metadata.create_all()` 自动建表

代码不用改一行——`create_engine` 自动根据连接字符串识别数据库类型。

### 坑 6：PostgreSQL `LIKE` 不兼容 —— 大小写敏感差异

**症状：** SQLite 上通过的关键字搜索，切换到 PostgreSQL 后搜不出任何东西。

**原因：** SQLite 的 `LIKE` 默认**不区分大小写**，而 PostgreSQL 的 `LIKE` 是**区分大小写的**。

```sql
-- SQLite：能搜到
SELECT * FROM transactions WHERE note LIKE '%apple%';  -- "Apple" 也能匹配

-- PostgreSQL：搜不到
SELECT * FROM transactions WHERE note LIKE '%apple%';  -- 只匹配 "apple"，不匹配 "Apple"
```

**修复：** PostgreSQL 用 `ILIKE` 进行不区分大小写的匹配，或者用 `LOWER()` 转成小写再比较：

```sql
SELECT * FROM transactions WHERE note ILIKE '%apple%';
-- 或
SELECT * FROM transactions WHERE LOWER(note) LIKE LOWER('%apple%');
```

SQLAlchemy 里对应的写法：

```python
# .ilike() 方法 = PostgreSQL 的 ILIKE，SQLite 上自动 fallback 到 LIKE
from sqlalchemy import func
Transaction.note.ilike(f"%{keyword}%")
```

### 坑 7：bcrypt 4.0.1 —— 版本的学问

**症状：** `pip install -r requirements.txt` 正常。用户注册时报 500，日志显示 `ValueError: bcrypt: password exceeds 72 bytes` 或者 `AttributeError: module 'bcrypt' has no attribute '__about__'`。

**原因：** `passlib` 库自带的 bcrypt 支持是可选依赖。当你 `pip install passlib[bcrypt]` 时，pip 会安装 bcrypt 的最新版（目前是 5.x）。但 **bcrypt 5.x 与 passlib 不兼容**——5.x 去掉了 passlib 依赖的一些内部属性。

```python
# passlib 内部代码（简化示意）
from bcrypt import __about__        # ← bcrypt 5.x 里这个没了
```

很多部署教程写的是 `passlib[bcrypt]`，这会在 Render 上自动拉 bcrypt 5.x，然后注册就炸了。

**修复：** 在 `requirements.txt` 里**显式锁定 bcrypt 版本为 4.0.1**：

```txt
fastapi>=0.110.0
uvicorn[standard]>=0.27.0
sqlalchemy>=2.0.0
pydantic>=2.0.0
pydantic-settings>=2.0.0
python-jose[cryptography]>=3.3.0
passlib[bcrypt]>=1.7.0
python-multipart>=0.0.9
bcrypt==4.0.1               # ← 必须锁定！5.x 与 passlib 不兼容
```

**passlib 依赖。** 你说 `passlib[bcrypt]` 已经带了 bcrypt 为什么还要单独写？因为 `passlib[bcrypt]` 对 bcrypt 的版本约束是 `>=`——它会装最新版，而不是锁定 4.x。单独写 `bcrypt==4.0.1` 就是告诉 pip：你不许装 5.x。

> 阿珊说：依赖管理最难受的 bug，不是版本太老，是“刚好装了个不兼容的新版本”。锁定版本不是为了保守，是为了你下次部署时跟你上次部署的结果一模一样。

### 坑 8：端口写死 8000 —— `$PORT` 的故事

**症状：** Render 部署日志显示 `Service started successfully`，但访问 URL 一直转圈。Render 的健康检查报了 `Connection refused`。

**原因：** Render 不像你本地——它不会默认用 8000 端口。Render 启动你的容器时，会通过环境变量 `$PORT` 传一个随机端口号给你。你的代码必须读取这个环境变量，用 Render 分配给你的端口启动服务，**不能写死 8000**。

```bash
# ❌ 写死端口——本地能跑，Render 不行
uvicorn app.main:app --host 0.0.0.0 --port 8000

# ✅ 用环境变量——本地和 Render 都能跑
uvicorn app.main:app --host 0.0.0.0 --port $PORT
```

**Render 上配置的 Start Command 正确写法：**

```bash
cd backend && uvicorn app.main:app --host 0.0.0.0 --port $PORT
```

金橘记账的 `$PORT` 在 Render 上自动存在（Render 注入的环境变量），不需要你在自己的 `.env` 或代码里额外配置。

**$PORT 在你本地不存在怎么办？** 加个 fallback：

```bash
# 本地开发时 PORT 不存在，fallback 到 8001
uvicorn app.main:app --host 0.0.0.0 --port ${PORT:-8001}
```

### 坑 9：HTTPS Mixed Content —— 前端 HTTPS，后端 HTTP

**症状：** 前端页面能打开，API 请求也能发，但浏览器 Console 里有一条黄色警告：
```
Mixed Content: The page at 'https://xxx.vercel.app' was loaded over HTTPS,
but requested an insecure XMLHttpRequest endpoint 'http://xxx.onrender.com/api/health'.
This request has been blocked; the content must be served over HTTPS.
```

然后，所有 API 请求都被浏览器**直接拦截**了——前端空白，Network 面板里请求显示 `blocked:mixed-content`。

**原因：** Vercel 默认给你 `https://`。如果你的前端在 HTTPS 下，浏览器就**不允许**它发 HTTP 请求。这是浏览器安全策略——一个"安全"页面不能去请求一个"不安全"的接口。

Render 免费版**默认也是 HTTPS**，但你的 `VITE_API_URL` 里可能写的是 `http://`：

```bash
# ❌ 写了 http，浏览器直接 blocked
VITE_API_URL=http://arashi-finance-api.onrender.com/api

# ✅ 必须是 https
VITE_API_URL=https://arashi-finance-api.onrender.com/api
```

**修复：** Vercel 环境变量里 `VITE_API_URL` 确保以 `https://` 开头。

### 坑 10：Vite 环境变量没 `VITE_` 前缀

**症状：** `.env` 文件里写了 `API_URL=https://xxx.onrender.com/api`，但 `import.meta.env.API_URL` 拿不到值（`undefined`）。

**原因：** Vite 出于安全考虑，**只暴露 `VITE_` 开头的环境变量给客户端代码**。目的是防止后端 secret 不小心被暴露到前端 JS bundle 里。这不是 bug，是**故意设计的安全机制**。

```bash
# ❌ 错误 — Vite 不暴露
API_URL=http://localhost:8000

# ✅ 正确 — VITE_ 前缀
VITE_API_URL=http://localhost:8000
```

金橘记账前端的正确写法：

```typescript
// frontend/src/services/api.ts
const BASE_URL = import.meta.env.VITE_API_URL || "http://localhost:8000/api";
```

### 坑 11：requirements.txt 漏依赖

**症状：** `pip install -r requirements.txt` 成功。启动时报 `ModuleNotFoundError`。

**原因：** 你用 `pip install python-jose` 装了一个依赖，但忘了 `pip freeze > requirements.txt` 更新文件。本地的 `venv/` 里装了这个包，能跑，但 Render 是从零开始装的——`requirements.txt` 里没有就不会装。

**修复：** 每次装了新依赖，养成习惯更新 requirements.txt：

```bash
# 本地装新包
pip install python-dotenv

# 同步到 requirements.txt
pip freeze > requirements.txt
```

或者更精确的方式——手动写 `requirements.txt`（金橘记账用的方式）：

```txt
fastapi>=0.110.0
uvicorn[standard]>=0.27.0
sqlalchemy>=2.0.0
pydantic>=2.0.0
pydantic-settings>=2.0.0
python-jose[cryptography]>=3.3.0
passlib[bcrypt]>=1.7.0
python-multipart>=0.0.9
bcrypt==4.0.1
```

手动写可以精确控制版本，但容易漏。**个人推荐：开发时随意装，上线前用 `pip freeze > requirements.txt` 锁死版本。**

### 坑 12：`.pyc` 文件冲突

**症状：** 同样的代码，本地跑正常，Render 上报了一堆奇怪的 import 错误。

**原因：** Python 编译后会在 `__pycache__/` 下生成 `.pyc` 字节码文件。这些文件是**与 Python 版本强绑定的**。你本地 Python 3.12 生成的 `.pyc`，推到 Git 传到 Render。Render 上可能是 Python 3.10 或 3.11——读到不相容的 `.pyc` 就可能报莫名其妙的错误。

**修复：** `.gitignore` 加 `__pycache__/`：

```gitignore
# Python
__pycache__/
*.pyc
*.pyo
```

如果已经上传了：

```bash
git rm -r --cached __pycache__/
git commit -m "chore: remove __pycache__ from git tracking"
```

### 坑 13：`.gitignore` 没加 `.env`

**症状：** 打开 GitHub 仓库，发现 `.env` 文件明晃晃地躺在项目根目录下，JWT Secret 和数据库连接字符串一字不差地展示给全世界。

**后果：** 对于 public 仓库，任何 clone 了代码的人都能看到你的密钥。拿着 JWT_SECRET 就能伪造 JWT，假装是你 App 的任何一个用户。

**记住：GitHub 的 commit 历史删不掉。即使你新 commit 删了 .env，旧 commit 里还是能看到。唯一的正确姿势是让旧的 secret 失效——去对应平台（Neon / Render / Vercel）重置密钥。**

**预防：先加 `.gitignore` 再写 `.env`。** 金橘记账推荐的 `.gitignore` 内容：

```gitignore
# 环境变量
.env
.env.local
.env.*.local

# Python
__pycache__/
*.pyc
*.pyo
venv/
*.db

# Node
node_modules/
dist/
```

检查是否已经有 `.env` 被跟踪了：

```bash
git ls-files | grep .env
```

如果有输出，立刻处理：

```bash
git rm --cached .env
git commit -m "chore: remove .env from git tracking"
```

### 坑 14：Monorepo Root Directory —— Vercel 找不到 `package.json`

**症状：** Vercel 构建时报错：`Unable to find package.json` 或 `Cannot find module 'react'`。但你的仓库里明明有 package.json。

**原因：** 金橘记账是一个 monorepo 项目——根目录放文档和配置，`frontend/` 和 `backend/` 下才各有各的 `package.json` / `requirements.txt`。

```
arashi-finance-tracker/
├── backend/
│   ├── app/
│   └── requirements.txt        ← Python 项目在这里
├── frontend/
│   ├── src/
│   ├── package.json             ← 前端项目在这里
│   └── vite.config.ts
└── docs/
```

Vercel 默认在**仓库根目录**找 `package.json`——找不到就报错。Render 同理。

**修复：**

- **Vercel：** Settings → General → Root Directory → 填 `frontend`
- **Render：** Settings → Root Directory → 填 `backend`

两个平台都有这个配置项，只是叫法可能不同（Root Directory / Project Directory / Base Directory）。

**Render 上 Build Command 和 Start Command：**

- Build Command：`pip install -r requirements.txt`
- Start Command：`cd backend && uvicorn app.main:app --host 0.0.0.0 --port $PORT`

注意 Start Command 开头多了一个 `cd backend`——如果你没配 Root Directory，用 `cd backend` 也能搞定。

### 坑 15：免费额度超限

**症状：** 某天发现网站打不开，Vercel 面板显示 `Bandwidth Limit Exceeded`，Render 显示 `Free tier limit reached`。

**Vercel 免费额度：**

| 项目 | 免费额度 | 超限后果 |
|------|---------|---------|
| 带宽 | 100 GB/月 | 超出后站点暂停服务 |
| 构建次数 | 6000 分钟/月 | 超出后无法触发新构建 |
| 同时构建数 | 1 个 | 多个 push 会排队 |

**Render 免费额度：**

| 项目 | 免费额度 | 超限后果 |
|------|---------|---------|
| Web Service 运行时间 | 750 小时/月 | 刚好够一个实例跑一个月 |
| PostgreSQL 数据库 | 1 个免费实例 | 30 天后过期（需手动续期或付费） |
| 并发构建 | 1 个 | 多个服务会排队 |

**现实情况：** 750 小时刚好是一个月（约 730 小时）——如果你只跑一个 Web Service，免费额度刚好够用。但 90 天后 Render 免费数据库会过期，到期前你会收到邮件提醒。

**应对方案：**

1. **监控用量：** Vercel Dashboard → Usage 页面看带宽消耗
2. **Neon 替代 Render PostgreSQL：** Neon 免费版没有过期限制，比 Render 内置 PostgreSQL 更稳定
3. **图片走 CDN：** 不要把大图放在 Vercel 的 `public/` 目录里——每加载一张图就消耗带宽。用 Cloudinary 或 imgur 托管图片
4. **非高峰期请求合并：** 减少不必要的 API 轮询

**个人建议：** 个人项目用免费版完全够——金橘记账跑了一个月，带宽消耗不到 100MB。超限主要发生在公之于众的站点——如果你的项目日活 100 人以上，这个量级也该考虑付费了。

## 🎯 面试会问

### Q1：你部署的项目遇到过什么问题？怎么解决的？

面试官问这个，不是要考你技术细节——他想看你**有没有真正把项目推到线上给别人用过**。一个只在 localhost 跑过的项目和一个上线的项目，差了一整套"环境适配"的经验。

**按"两个最难忘的坑"来回答：**

1. **Render 15 分钟休眠**：免费版 Web Service 15 分钟没请求就休眠，冷启动 30-60 秒，用户第一次打开白屏。解决：UptimeRobot 每 5 分钟请求 `/api/health` 保活。

2. **bcrypt 5.x 与 passlib 不兼容**：部署后注册功能报 500，日志里是 passlib 调用 bcrypt 内部 API 报错。排查半天发现 bcrypt 5.x 去掉了 `__about__` 属性——那加就是 `passlib[bcrypt]` 自动装的。解决：`requirements.txt` 锁死 `bcrypt==4.0.1`。

这两点一说，面试官就知道：**这个人是真的部署过，不是看了教程就完事。** 每个答不上来的坑，背后都是一次深夜排查的血泪。

### Q2：你觉得部署最难的是什么？

"部署最难的不是技术，是'环境差异'。你本地装了一堆东西——Python 3.12、node 22、全局安装了 `nodemon` 和 `pm2`、浏览器里有一堆插件——但部署环境是**裸的**。它没有你的全局工具，没有你的 node_modules 缓存，没有你的 .env 文件。”

"所有部署问题都是'本地能跑，线上不行'的变种。要解决它，不是靠一次配好——是靠整理检查清单，确保**每次部署都经过同样的核查步骤**。”

面试官想听的不是"部署很简单"或"部署很难"——他想听的是**你对"环境差异"有认知，并且养成了系统化排查的习惯**。
