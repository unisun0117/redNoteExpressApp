# 第9篇：面试官会怎么问你数据库？— 10道高频题+答题思路

> 面试官不在乎你背了多少概念，在乎你"有没有真正用过"。答题公式：概念一句话 + 我的项目里怎么用的 + 遇到过什么问题

## 前端为什么要学这个？

我面全栈岗位的时候，被问了一个看起来很简单的问题："你项目里用过事务吗？"

我当时可自信了，上来就背："事务是数据库操作的最小工作单元，具有 ACID 特性——原子性、一致性、隔离性、持久性……"

面试官打断了我："你项目里真正用过吗？什么场景？代码怎么写的？"

我哑口无言。不是我不知道 ACID，是我从来没想过"背下来"和"写出来"之间差了十万八千里。

后来我反思，其实面试官根本不在乎你背了多少概念——你背得过一个调了 API 文档的 GPT 吗？他们想看的是：**你遇到问题的时候，知不知道用数据库的哪个特性去解决。** 这就像面试官问你 React，他不会问"虚拟 DOM 的定义是什么"，他会问"你项目中哪个场景虚拟 DOM 没帮上忙，你怎么处理的"。

所以我后来把所有面试常问的数据库问题，都用金橘记账这个真实项目的场景重新过了一遍。结果你猜怎么着——下一轮面试，一个关于事务的追问，我直接说"我记账项目里，删一条账单同时要更新预算和统计，这三个操作必须一起成功或一起失败"，然后打开代码给他看。面试官说"好，这个我信了。"

下面这 10 道题，每一道我都用"概念一句话 + 我项目里怎么用的 + 遇到过什么问题"这个公式来答。

## 核心原理

我们先不刷题，先搞清楚一个底层问题：**数据库到底解决什么？**

你写前端的时候，`useState` 存个变量，页面刷新就没了。`localStorage` 可以持久化，但没法查"本月餐饮花了多少钱"。JSON 文件存硬盘上可以，但两个人同时写就乱了。

数据库本质上就是一个**不会丢数据、能快速查、多人同时用不乱套**的存储引擎。它的所有特性都围绕这三个目标：持久性、查询效率、并发安全。

打个前端类比：

- **索引**像 React 的 Virtual DOM —— 不是查原始数据，而是查一份"排好序的快照"，B+ 树的结构就跟 Fiber 树的 diff 思路很像，都是"分层、只走必要路径"。
- **事务**像 `useReducer` 的 dispatch —— 多个状态变更必须原子化，要么全生效，要么全回滚，不能出现中间态。
- **锁**像 `useSyncExternalStore` —— 多个组件同时读写同一个外部状态时，必须有一套规则保证不乱。
- **连接池**像浏览器的 HTTP/2 连接复用 —— 不会每次请求都重新三次握手，而是维护一个连接池，用完归还。

> 阿珊说：数据库不是用来背的，是用来解决"数据丢了怎么办、查得太慢怎么办、两个人同时改一条数据怎么办"这三个问题的。你对着这三个问题去学，所有概念都有根了。

## 10 道高频面试题

### Q1：你项目里索引是怎么用的？为什么能加速查询？

**概念一句话：** 索引就像书的目录，数据库不用翻全表，直接定位到数据页。

**我的项目里：** 在金橘记账里，用户查账单是最高频操作。每个用户只能查自己的账单，所以 `user_id` 是查询的必带条件。而且账单按月份筛选，再加上 `tx_date` 排序。

```python
# 金橘记账 transactions.py — 这是项目里每时每刻在跑的查询
query = db.query(Transaction).filter(Transaction.user_id == user.id)

if month:
    query = query.filter(Transaction.tx_date.like(f"{month}%"))
if category:
    query = query.filter(Transaction.category == category)
if type:
    query = query.filter(Transaction.type == type)

items = query.order_by(Transaction.tx_date.desc()).offset((page-1)*page_size).limit(page_size).all()
```

你看看这段代码：`user_id`、`tx_date`、`category`、`type` 四个字段都在 `WHERE` 里出现。如果没有索引，每一页查询数据库都要扫全表。用户只有 100 条账还好，但假设 10 万条，每次翻页都要扫 10 万行。

所以我给 `transactions` 表建了组合索引 `(user_id, tx_date)` ——因为这两个字段总是一起出现，组合索引就是电梯直达，省掉了回表查询。

**遇到过的问题：** 早期我没给 `category` 单独建索引，用户按分类筛选时发现 1000 条账就开始卡。因为组合索引 `(user_id, tx_date)` 最左前缀匹配之后，`category` 没法复用这个索引。后来补了一个 `(user_id, category)` 索引才解决。

### Q2：项目里什么时候用事务？怎么写？

**概念一句话：** 事务把多个 SQL 操作打包成一个原子操作，要么全成功，要么全回滚。

**我的项目里：** 金橘记账"更新预算"这个接口，就是一个标准的事务场景。用户设置月度预算的时候，需要：1）删掉旧预算；2）插入新总预算；3）循环插入各分类预算。三步中任何一步失败，都会导致数据不一致。

```python
# 金橘记账 budgets.py — 更新预算是一个隐性事务
@router.put("/budgets")
def update_budgets(req: BudgetUpdate, user=Depends(get_current_user), db=Depends(get_db)):
    # 步骤1：删除该月旧预算
    db.query(Budget).filter(
        Budget.user_id == user.id,
        Budget.month == req.month,
    ).delete()

    # 步骤2：插入总预算
    db.add(Budget(user_id=user.id, month=req.month, category="__total__", amount=req.total_budget))

    # 步骤3：插入分类预算
    for c in req.categories:
        db.add(Budget(user_id=user.id, month=req.month, category=c.category, amount=c.amount))

    db.commit()  # 三条操作一起提交
```

这里 `db.commit()` 是关键——前面所有 `delete` 和 `add` 都在一个事务里，commit 的时候才真正写入。如果中途 `req.categories` 里有一条数据格式不对导致插入失败，整个 commit 都不会生效，旧预算也不会被删掉。

**遇到过的问题：** 有一次我在"删除 + 插入"之间加了一个外部 API 调用（查汇率），结果网络超时导致事务长时间挂起，锁住了整个 budgets 表。教训：**事务里不要放网络请求，不要放耗时计算，越短越好。**

### Q3：你的数据库用的是 SQLite 还是 PostgreSQL？为什么选它？有什么区别？

**概念一句话：** SQLite 是文件数据库，PostgreSQL 是客户端-服务器架构的关系型数据库。

**我的项目里：** 金橘记账开发阶段用 SQLite，因为不需要装任何东西——一个 `.db` 文件丢在项目里就能跑。团队里任何人 clone 下来都能立刻 `uvicorn` 启动，零配置。

```python
# 金橘记账 config.py — 默认 SQLite，一行改 PostgreSQL
class Settings(BaseSettings):
    DATABASE_URL: str = "sqlite:///./finance.db"  # 开发环境
    # 生产环境只需改环境变量：DATABASE_URL=postgresql://user:pass@host/db
```

```python
# 金橘记账 database.py — 一行代码适配两种数据库
engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {},
)
```

这个 `check_same_thread=False` 只在 SQLite 下需要，因为 FastAPI 是多线程处理请求的，SQLite 默认不允许跨线程用同一个连接。换成 PostgreSQL 后这个参数自动跳过。

**遇到过的问题：** SQLite 的 `ALTER TABLE` 能力很弱，不支持直接 `DROP COLUMN`。有一次我重构模型想删一个字段，SQLite 直接报错。后来学乖了：开发阶段用 SQLite 可以，但一旦模型稳定了，尽快切到 PostgreSQL 做测试，避免上线前发现 SQL 不兼容。

### Q4：你项目里有没有用到外键？外键约束和代码里手动校验选哪个？

**概念一句话：** 外键保证引用的数据一定存在，删用户时自动处理关联数据（级联删除或拒绝删除）。

**我的项目里：** transaction 和 budget 都通过 `user_id` 外键关联到 users 表。

```python
# 金橘记账 transaction.py 模型 — user_id 外键约束
class Transaction(Base):
    __tablename__ = "transactions"
    user_id: Mapped[str] = mapped_column(String(36), ForeignKey("users.id"), nullable=False)
    # ...
```

```python
class Budget(Base):
    __tablename__ = "budgets"
    user_id: Mapped[str] = mapped_column(String(36), ForeignKey("users.id"), nullable=False)
    # ...
```

外键的好处是：如果有人写了个 bug 要插入一条 `user_id` 不存在的账单，数据库直接拒绝，连插入都插不进去。这比你代码里“查一下用户存不存在再插入”要可靠得多——代码校验可能被跳过，外键约束绕不过去。

**遇到过的问题：** 测试的时候我想清空 users 表重新造数据，结果 SQLite 报错说"transactions 表引用了这些用户"。因为没设 `CASCADE`，必须先删 transactions 再删 users。后来我加上了 `ondelete="CASCADE"` 才解决。这个问题让我记住：外键的级联策略（`RESTRICT` / `CASCADE` / `SET NULL`）要在建表时就想清楚。

### Q5：ORM 和原生 SQL 你更倾向用哪个？为什么？

**概念一句话：** ORM 用对象操作数据库（`db.query(User).filter(...)`），原生 SQL 直接写 `SELECT * FROM users WHERE ...`。

**我的项目里：** 大部分查询用 SQLAlchemy ORM 就够了。但聚合统计的时候，ORM 表达起来累赘，我直接用了 `func.sum` ——这算是 ORM 和 SQL 的混合模式。

```python
# 金橘记账 stats.py — 聚合查询混用 ORM 和 func
income = db.query(func.sum(Transaction.amount)).filter(
    Transaction.user_id == user.id,
    Transaction.type == "income",
    Transaction.tx_date.like(f"{month}%"),
).scalar() or 0

# 分组统计 —— 这个用纯 ORM 写就很自然
results = (
    db.query(Transaction.category, Transaction.category_emoji, func.sum(Transaction.amount).label("total"))
    .filter(Transaction.user_id == user.id, Transaction.type == "expense")
    .group_by(Transaction.category)
    .order_by(func.sum(Transaction.amount).desc())
    .all()
)
```

我的原则是：**CRUD 用 ORM，复杂报表查询用原生 SQL 或 func。** 别学那些非要"纯 ORM"或"纯 SQL"的极端派——工具是帮你干活的，不是帮你站队的。

### Q6：token 和 session 有什么区别？你项目里怎么存登录状态？

**概念一句话：** Session 把状态存服务端，token 把状态编码到字符串里，服务端无状态。

**我的项目里：** 金橘记账用 JWT（token 的一种），access token 过期后用 refresh token 换新的。

```python
# 金橘记账 auth.py — JWT 双 token 机制
def create_tokens(user_id: str) -> TokenResponse:
    access_payload = {
        "sub": user_id,
        "exp": datetime.utcnow() + timedelta(minutes=15),  # 15分钟过期
        "type": "access",
    }
    refresh_payload = {
        "sub": user_id,
        "exp": datetime.utcnow() + timedelta(days=7),  # 7天过期
        "type": "refresh",
    }
    return TokenResponse(
        access_token=jwt.encode(access_payload, settings.JWT_SECRET),
        refresh_token=jwt.encode(refresh_payload, settings.JWT_SECRET),
    )
```

为什么不用 session？因为 session 需要把 session_id 存数据库或 Redis，每次请求都要查一次。JWT 不需要——token 本身就带了用户 ID 和过期时间，服务端解密一下就行。而且金橘记账后端部署在单机，不需要共享 session。

**但这里有个坑：** JWT 一旦签发，在过期前无法主动失效。用户点了"退出登录"，token 仍然有效。所以我前端 `clearTokens()` 只是删了本地存储，后端没有黑名单机制。这对于一个记账应用来说是可以接受的，但如果是金融交易系统，就必须加 token 黑名单。

```typescript
// 金橘记账 api.ts — 前端 token 管理
function clearTokens() {
  accessToken = null;
  refreshToken = null;
  localStorage.removeItem("access_token");
  localStorage.removeItem("refresh_token");
}
```

### Q7：数据库连接池是什么？你项目里怎么配的？

**概念一句话：** 预先创建 N 个连接保持存活，请求来了直接拿，用完归还，避免反复 TCP 握手。

**我的项目里：** SQLAlchemy 的 `sessionmaker` 内部维护连接池，FastAPI 里每个请求通过 `get_db` 依赖拿一个 session，请求结束就归还。

```python
# 金橘记账 database.py — 连接池配置（隐式的，SQLAlchemy 默认管理）
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()  # 归还连接到池，不是真正关闭
```

这里 `db.close()` 不是真的断开 TCP 连接——是把连接归还给连接池。连接池的大小默认是 5（SQLAlchemy 默认 `pool_size=5`），对于金橘记账这种个人应用足够了。

**遇到过的问题：** 有一天我快速连续发了几十个请求测并发，报错"QueuePool limit reached"。因为我没有手动设置 `pool_size` 和 `max_overflow`，默认超出 5 个连接 + 10 个溢出的上限就拒绝新请求了。后来我调大到 `pool_size=10, max_overflow=20`，并且确保每个请求都正确调用了 `db.close()`。

### Q8：数据库范式是什么？你项目里做到了第几范式？

**概念一句话：** 范式是消除数据冗余的设计规范，从 1NF 到 5NF 越来越严格，实践中做到 3NF 就够了。

**我的项目里：** 金橘记账设计上做到了 3NF。

- **1NF（每列不可拆分）：** `amount` 用 `int` 存分而不是存"12.5元"字符串，`category_emoji` 和 `category` 分开存。
- **2NF（消除部分依赖）：** transactions 表的每条记录完全依赖自己的主键 `id`，不存在某个字段只依赖 `user_id` 的情况。如果有，就应该拆到 users 表。
- **3NF（消除传递依赖）：** 用户昵称 `nickname` 放在了 users 表，没有在 transactions 表里重复存。如果你在 transactions 里存了 `user_nickname`，用户改名时就得同时更新 transactions 表里的所有记录——这就是违反 3NF 的传递依赖。

```python
# 金橘记账 user.py & transaction.py — 3NF 设计
# users 表：存用户自己的信息
class User(Base):
    __tablename__ = "users"
    nickname: Mapped[str | None] = mapped_column(String(100), nullable=True)

# transactions 表：只存 user_id 引用，不冗余存 nickname
class Transaction(Base):
    __tablename__ = "transactions"
    user_id: Mapped[str] = mapped_column(String(36), ForeignKey("users.id"))
```

**遇到过的问题：** 一开始我想偷懒，在 transactions 表里加了个 `user_name` 字段，心想反正前端要展示。结果用户改了昵称之后，历史账单里显示的还是旧名字。这就是典型的"反模式"——为了少一次 JOIN 而冗余数据，最后数据不一致了才知道疼。

### Q9：你项目里有没有用到聚合查询？怎么写的？

**概念一句话：** 聚合查询（`SUM`、`COUNT`、`AVG`、`GROUP BY`）把多行数据汇总成一行统计结果。

**我的项目里：** 仪表盘和分类统计大量使用聚合查询。比如仪表盘首页需要展示当月收入、支出、结余：

```python
# 金橘记账 stats.py — 仪表盘聚合查询
income = db.query(func.sum(Transaction.amount)).filter(
    Transaction.user_id == user.id,
    Transaction.type == "income",
    Transaction.tx_date.like(f"{month}%"),
).scalar() or 0

expense = db.query(func.sum(Transaction.amount)).filter(
    Transaction.user_id == user.id,
    Transaction.type == "expense",
    Transaction.tx_date.like(f"{month}%"),
).scalar() or 0

balance = income - expense  # 结余直接算，不必再查一次数据库
```

分类占比统计里用到了 `GROUP BY`，这是面试必问的：

```python
# 金橘记账 stats.py — GROUP BY 聚合
results = (
    db.query(
        Transaction.category,
        Transaction.category_emoji,
        func.sum(Transaction.amount).label("total")
    )
    .filter(Transaction.user_id == user.id, Transaction.type == "expense")
    .group_by(Transaction.category)
    .order_by(func.sum(Transaction.amount).desc())
    .all()
)
```

这个查询的逻辑是：把该用户所有支出记录按分类分组，算每个分类的总金额，按金额从高到低排序。最后在业务层算百分比：

```python
for r in results:
    pct = round(r.total / expense_total * 100) if expense_total > 0 else 0
    categories.append({"name": r.category, "emoji": r.category_emoji, "amount": r.total, "percentage": pct})
```

**遇到过的问题：** `func.sum()` 在没有匹配行时返回 `None` 而不是 0，Python 里 `None + 其他值` 会直接报 TypeError。所以我每个 `.scalar()` 后面都加了 `or 0` ——这是个血泪教训。

### Q10：数据量大了怎么办？你考虑过哪些优化？

**概念一句话：** 数据量上去了，优化主要从索引、查询优化、读写分离、缓存、分库分表这些方向入手。

**我的项目里：** 金橘记账目前数据量不大，但我在设计上已经为未来做了准备：

1. **所有列表查询都加了分页** —— 哪怕现在一次只返回 50 条，也用了 `.offset().limit()`：
   ```python
   # 金橘记账 transactions.py — 分页
   items = query.order_by(Transaction.tx_date.desc()).offset((page-1)*page_size).limit(page_size).all()
   ```

2. **前端按需加载** —— 账单列表页默认只加载当月数据，切换月份时才重新请求：
   ```typescript
   // 金橘记账 ListPage.tsx
   const load = () => {
     const q: Record<string, string> = { month, page: "1", page_size: "100" };
     api.getTransactions(q).then(data => { /* ... */ });
   };
   ```

3. **统计结果可以在前端做增量** —— 账单列表接口在校验 token 时已经查了一次用户表，返回的数据里收入/支出汇总就是前端 `sum()` 算的，不需要后端再跑一条 SQL。

**如果数据量再大一个量级，我会考虑：**
- 给 monthly stats 单独建一张预聚合表，定时任务去更新，不用每次实时查。
- 冷数据归档：超过 3 年的账单迁移到归档表，活跃表只保留近期的。
- 读写分离：统计类的查询走只读副本，写操作走主库。

## 常见坑点

| 坑 | 现象 | 解法 |
|----|------|------|
| `func.sum()` 返回 None | `None + 0` 报 TypeError，直接崩接口 | 每个 `.scalar()` 后加 `or 0`：`db.query(func.sum(...)).scalar() or 0` |
| SQLite 并发连接报错 | FastAPI 多线程下 SQLite 抛 "objects created in a thread can only be used in that thread" | `connect_args={"check_same_thread": False}` |
| 事务忘了 commit | 数据插入了但查询不出来，以为代码有 bug | SQLAlchemy 设置 `autocommit=False` 的话，增删改后必须调 `db.commit()` |
| 金额用 float 存 | 0.1 + 0.2 = 0.30000000000000004，账对不上 | 金额存 int（分），前端 `/100` 显示，永远不做浮点运算 |
| 分页忘了过滤用户 ID | 用户 A 看到了用户 B 的账单 | 所有查询第一行都是 `.filter(Model.user_id == user.id)` |
| JWT token 无法主动失效 | 用户点退出登录后 token 仍然可用 | 轻度场景接受这个缺陷；重度场景加 token 黑名单或缩短过期时间 |

## 小结

你会发现这 10 道题，每一道我都在用同一个模板回答：**先一句话说清楚概念，然后告诉你我项目里怎么写的，再坦白我踩过什么坑。** 这个模板之所以好用，是因为它把面试官最想知道的三个信息点全覆盖了：

1. 你真的理解概念，不是死记硬背
2. 你在真实项目里用过了，代码拿得出手
3. 你碰到过问题、解决了、有反思——这说明你有工程经验

下次面试，别背八股文了。打开你的 GitHub，把你项目的代码读一遍，用这个公式把每个数据库概念套一遍。**你讲自己的项目，不用背，不会忘，而且面试官能听出来是真的。**

下一期是专栏的最后一篇：《从单体到分布式——架构师视角的数据库演进之路》，我会用金橘记账的架构演变（SQLite→PostgreSQL→读写分离→缓存层）来串联整个专栏学过的所有概念。数据库从 0 到 1 到头了，我们看看"从 1 到 100"要怎么走。

—— 阿珊，前端开发者 & 全栈学习者
