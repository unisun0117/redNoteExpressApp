# 第2篇：数据库怎么存你的数据？

> 数据库表 = Array\<Object\>，一行 = 一个Object，一列 = 一个key。你每天都在操作"表"——users.map(u =\> u.email)就是一次SELECT

## 📌 前端为什么要学这个？

我第一次听到"表"的时候，脑子里浮现的是Excel——以为可以随便合并单元格、加个背景色、这行填数字下两行写段话。后来发现数据库的"表"比Excel严格得多：每一列的类型是定死的，你不能这一行是字符串、下一行塞个数字进去。

但转念一想，这跟我们在前端天天打交道的东西其实一模一样——**Array\<Object\>**。

你写业务的时候，接口返回一个列表，就是一个Array，里面每个元素是一个Object，每个Object的key就是字段。你用 `users.map(u => u.email)` 拉出所有用户的邮箱，这就等于数据库里的 `SELECT email FROM users`。你用 `users.filter(u => u.vip)` 筛选VIP用户，就等于 `SELECT * FROM users WHERE vip = true`。

数据库表没有你想象的那么神秘，它就是用一种更高效、更安全的方式，把你熟悉的那些数组成到了硬盘上。今天我们就来把这个"黑盒子"拆开看看。

## 🔍 核心原理

### 一张表就是一个 Array\<Object\>

在金橘记账项目里，我们有一个"用户表"和一张"账单表"。你用 TypeScript 来想象的话，它们长这样：

```typescript
// 这就是一张"users"表
const users: Array<{
  id: string;
  email: string;
  password_hash: string;
  nickname: string | null;
  avatar: string | null;
  created_at: Date;
}> = [];

// 这就是一张"transactions"表
const transactions: Array<{
  id: string;
  user_id: string;
  type: "expense" | "income";
  amount: number;        // 单位：分
  category: string;
  category_emoji: string | null;
  note: string | null;
  tx_date: Date;
  created_at: Date;
}> = [];
```

数据库的表跟这两个 TypeScript 类型定义几乎一模一样。唯一的区别是，TypeScript 的类型只是编译时的标注，而数据库的类型约束是跑在运行时的——你敢塞错类型它就敢报错。

看一下金橘记账项目里真实的数据库定义（用的是 SQLAlchemy）：

```python
# backend/app/models/user.py
class User(Base):
    __tablename__ = "users"

    id: Mapped[str] = mapped_column(String(36), primary_key=True, default=lambda: str(uuid.uuid4()))
    email: Mapped[str] = mapped_column(String(255), unique=True)
    password_hash: Mapped[str] = mapped_column(String(255))
    nickname: Mapped[str | None] = mapped_column(String(100), nullable=True)
    avatar: Mapped[str | None] = mapped_column(String(10), nullable=True, default="😊")
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow)
```

你仔细看这个 Python 代码，它做的事情跟你定义 TypeScript interface 一模一样：声明字段名、声明类型、声明哪个字段可空。`__tablename__ = "users"` 就是给这个表格起个名字，仅此而已。

```python
# backend/app/models/transaction.py
class Transaction(Base):
    __tablename__ = "transactions"

    id: Mapped[str] = mapped_column(String(36), primary_key=True, default=lambda: str(uuid.uuid4()))
    user_id: Mapped[str] = mapped_column(String(36), ForeignKey("users.id"), nullable=False)
    type: Mapped[str] = mapped_column(String(10), nullable=False)  # "expense" | "income"
    amount: Mapped[int] = mapped_column(Integer, nullable=False)    # 单位：分
    category: Mapped[str] = mapped_column(String(50), nullable=False)
    category_emoji: Mapped[str | None] = mapped_column(String(10), nullable=True)
    note: Mapped[str | None] = mapped_column(String(200), nullable=True)
    tx_date: Mapped[date] = mapped_column(Date, nullable=False)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow)
```

> **阿珊说：数据库不是魔法，它就是把你前端脑子里的"数组+对象"那一套，换了个更靠谱的存储方式——内存里的数组一刷新就没了，磁盘上的表你关机一年它还在。**

### 增删改查 = 数组操作

数据库四大操作 CRUD，你每天都在前端写：

| 数组操作 | 数据库操作 | 金橘记账的真实代码 |
|----------|-----------|-------------------|
| `arr.filter(...)` | `SELECT ... WHERE ...` | `db.query(Transaction).filter(Transaction.type == "expense")` |
| `arr.push(...)` | `INSERT INTO ...` | `db.add(tx); db.commit()` |
| `arr[i] = newVal` | `UPDATE ... SET ...` | `tx.amount = 1000; db.commit()` |
| `splice / pop` | `DELETE FROM ...` | `db.delete(tx); db.commit()` |

来看看金橘记账项目里真实的查询怎么写。这段代码是账单列表接口，包含了筛选、排序、分页，几乎覆盖了你日常会写的所有查询：

```python
# backend/app/routers/transactions.py
@router.get("/transactions")
def list_transactions(
    month: str | None = None,
    category: str | None = None,
    type: str | None = None,
    q: str | None = None,
    page: int = 1,
    page_size: int = 50,
    user: User = Depends(get_current_user),
    db: Session = Depends(get_db),
):
    # 1. 基础查询 — 相当于 transactions.filter(t => t.user_id === currentUser.id)
    query = db.query(Transaction).filter(Transaction.user_id == user.id)

    # 2. 条件叠加 — 相当于 .filter(t => t.tx_date.startsWith("2026-06"))
    if month:
        query = query.filter(Transaction.tx_date.like(f"{month}%"))
    if category:
        query = query.filter(Transaction.category == category)
    if type:
        query = query.filter(Transaction.type == type)
    if q:
        query = query.filter(Transaction.note.contains(q))  # 模糊搜索

    # 3. 分页 — 前端最熟悉的 .slice((page-1)*size, page*size)
    total = query.count()
    items = (
        query.order_by(Transaction.tx_date.desc())
        .offset((page - 1) * page_size)
        .limit(page_size)
        .all()
    )
```

用前端思维来读这段代码：`db.query(Transaction)` 等于拿到了整个 transactions 数组，`.filter(...)` 就是不停套数组的 filter 方法，`.order_by(...)` 就是 sort，`.offset().limit()` 就是 slice。每一步都是你熟悉的逻辑。

连聚合查询也没区别——前端用 reduce，数据库用 sum：

```python
# backend/app/routers/stats.py — 当月收支汇总
income = db.query(func.sum(Transaction.amount)).filter(
    Transaction.user_id == user.id,
    Transaction.type == "income",
    Transaction.tx_date.like(f"{month}%"),
).scalar() or 0
```

这跟 JavaScript 的 `transactions.filter(t => t.type === "income").reduce((sum, t) => sum + t.amount, 0)` 逻辑完全一致，只是数据库用 SQL 在底层跑，处理百万级数据也毫秒返回。

### 到这里，你可以把 SQL 当外语学了

SQL 不是天书，它就是一种"跟数据库表对话"的语言。当你写惯了 `.filter().sort().slice()`，你会发现 SQL 的语法只是换了种写法，逻辑一模一样。这篇文章我们先用 ORM（SQLAlchemy）写，后面的文章会让你直接面对 SQL——放心，你现在看到的这些链式调用，到时候一眼就能翻译回去。

## 🛠 动手试试

### 第一步：创建表结构

金橘记账的建表逻辑很简单。你只需要定义了 Python 的 Model 类，SQLAlchemy 通过 `Base` 这个基类就能自动在数据库里建好对应的表。

```python
# backend/app/database.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, DeclarativeBase
from app.config import settings

engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {},
)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

class Base(DeclarativeBase):
    pass

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

`create_engine` 是数据库的连接引擎，`SessionLocal` 是每次操作数据库的"会话"。`get_db()` 这个函数被 FastAPI 用依赖注入的方式管理——每个请求进来拿一个数据库会话，请求结束自动关闭，连接池不会漏。

`connect_args={"check_same_thread": False}` 这行是 SQLite 特有的坑（后面坑点表会讲），换成 PostgreSQL 就不需要了。

### 第二步：用前端方式模拟数据库操作

打开浏览器的 Console，我们来模拟一下金橘记账的数据操作：

```javascript
// === 模拟数据库 ===
// 假设这是后端返回的原始数据
const transactions = [
  { id: "1", type: "expense", amount: 3500, category: "餐饮", category_emoji: "🍽️", note: "午餐外卖", tx_date: "2026-06-30" },
  { id: "2", type: "expense", amount: 1500, category: "交通", category_emoji: "🚇", note: "地铁月卡", tx_date: "2026-06-29" },
  { id: "3", type: "income", amount: 500000, category: "工资", category_emoji: "💰", note: null, tx_date: "2026-06-28" },
  { id: "4", type: "expense", amount: 8900, category: "购物", category_emoji: "🛍️", note: "买了双鞋", tx_date: "2026-06-27" },
  { id: "5", type: "expense", amount: 2200, category: "餐饮", category_emoji: "🍽️", note: "火锅", tx_date: "2026-06-27" },
];

// === 查（SELECT + WHERE）===
// 六月的所有支出，按日期降序排列
const juneExpenses = transactions
  .filter(t => t.type === "expense" && t.tx_date.startsWith("2026-06"))
  .sort((a, b) => new Date(b.tx_date) - new Date(a.tx_date));
// 等于：SELECT * FROM transactions WHERE type='expense' AND tx_date LIKE '2026-06%' ORDER BY tx_date DESC

// === 筛选（WHERE）===
// 所有餐饮类的支出
const foodOnly = transactions.filter(t => t.category === "餐饮");
// 等于：SELECT * FROM transactions WHERE category='餐饮'

// === 聚合（SUM + GROUP BY）===
// 每个分类花了多少钱
const byCategory = transactions
  .filter(t => t.type === "expense")
  .reduce((acc, t) => {
    acc[t.category] = (acc[t.category] || 0) + t.amount;
    return acc;
  }, {});
// 结果：{ 餐饮: 5700, 交通: 1500, 购物: 8900 }
// 等于：SELECT category, SUM(amount) FROM transactions WHERE type='expense' GROUP BY category

// === 增（INSERT）===
const newTx = {
  id: "6",
  type: "expense",
  amount: 5000,
  category: "娱乐",
  category_emoji: "🎬",
  note: "看电影",
  tx_date: "2026-06-30"
};
transactions.push(newTx);
// 等于：INSERT INTO transactions (...) VALUES (...)

// === 改（UPDATE）===
const target = transactions.find(t => t.id === "2");
if (target) target.amount = 2000;
// 等于：UPDATE transactions SET amount=2000 WHERE id='2'

// === 删（DELETE）===
const idx = transactions.findIndex(t => t.id === "4");
if (idx !== -1) transactions.splice(idx, 1);
// 等于：DELETE FROM transactions WHERE id='4'

console.log("当前数据：", transactions);
console.log("分类汇总：", byCategory);
```

直接把这段代码贴到浏览器 Console 里跑，结果立即可见。你看，`filter` 就是 `WHERE`，`sort` 就是 `ORDER BY`，`push` 就是 `INSERT`，`splice` 就是 `DELETE`。每次你在前端写这些数组操作，背后数据库都在写几乎一样的 SQL。

金橘记账的前端 `api.ts` 里也是这么调接口的：

```typescript
// frontend/src/services/api.ts
// 查账单列表 — 注意月份、分类、类型、页码都拼在 URL 参数里
getTransactions: (params: Record<string, string>) => {
  const qs = new URLSearchParams(params).toString();
  return authFetch(`/transactions?${qs}`).then(r => r.json());
},

// 创建一条账单 — POST 请求带着 JSON body
createTransaction: (data: any) =>
  authFetch("/transactions", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(data)
  }).then(r => r.json()),

// 删除一条 — DELETE 请求
deleteTransaction: (id: string) =>
  authFetch(`/transactions/${id}`, { method: "DELETE" }).then(r => r.json()),
```

前端的增删改查，和后端的增删改查，一一对应。你只要搞懂了 Array\<Object\> 的操作，数据库的玩法你已经在用了。

## ⚠️ 常见坑点

| 坑 | 现象 | 解法 |
|----|------|------|
| **金额用浮点数** | 0.1 + 0.2 = 0.30000000000000004，记账记出几分钱的误差，月底对账对不上 | 金橘记账的做法：数据库统一存"分"（int），前端显示除以 100。`amount: Mapped[int] = mapped_column(Integer)` |
| **SQLite 多线程锁** | FastAPI 是多线程的，不加 `check_same_thread: False` 就报 `SQLite objects created in a thread...` 错误 | `connect_args={"check_same_thread": False}` 告诉 SQLite "放行，我知道你在多线程用"。但生产环境还是换 PostgreSQL |
| **字符串拼 SQL 被注入** | 你写 `f"SELECT * FROM users WHERE email = '{email}'"`，别人输入 `' OR 1=1 --`，整个表都给你查出来了 | SQLAlchemy 的参数化查询自动防注入：`.filter(Transaction.category == category)` 底层用的是参数绑定，不是拼字符串 |
| **N+1 查宝宝** | 查 100 条账单，每条账单再去查一次用户名，结果发了 101 次 SQL 请求 | 用 `joinedload` 或 `selectinload` 做预加载。金橘记账目前数据量小还没暴露出这个问题，但分表关联多了就要注意 |
| **忘记 commit** | `db.add(tx)` 之后忘了 `db.commit()`，程序跑完数据没了 | SQLAlchemy 默认事务模式是手动提交。写完数据必须 `db.commit()`，或者把 `autocommit=True` |

## 🎯 面试会问

- **Q：数据库的表和 Excel 有什么区别？**
  **答：** Excel 是自由格式——合并单元格、一行多个标题、颜色随便加，面向的是人的眼睛。数据库表是严格结构的——每列有固定类型、每行字段数一致、不允许单元格合并，面向的是程序的读写。在金橘记账里，金额字段定死是 Integer，你不可能不小心存一个 "三十块" 进去。

- **Q：为什么不用 JSON 文件存数据，非要用数据库？**
  **答：** 100 条数据用 JSON 没问题。10 万条数据你用 JSON 文件，查一条 `fs.readFileSync('data.json')` 整个文件读进内存，筛选完再写回去——每次操作都是全量读写。数据库有索引、有查询优化、有事务保证，查一条只需要读几 KB，而不是整个文件。金橘记账现在用 SQLite（一个单文件数据库），你觉得它和 JSON 文件长得像，但它内部有完整的 SQL 引擎。

- **Q：ORM 和直接写 SQL 怎么选？**
  **答：** ORM（如 SQLAlchemy）让你用对象的方式操作表，不用手拼 SQL 字符串，安全、类型友好，适合 90% 的场景。直接写 SQL 适合复杂报表、性能敏感的大批量操作。金橘记账的账单增删改查用 ORM 完全够了，而统计页面的 `func.sum()` `GROUP BY` 本质上就是 SQL，只是用 ORM 语法包了一层。

## 📝 小结

数据库的表没有那么吓人，把它理解成"存在硬盘上的 Array\<Object\>"就够了。你今天在 JavaScript 里写的每个 filter、sort、slice、reduce，在数据库那边都有一一对应的操作，只是换了一套语法。

下一篇文章，我们会从 ORM 往下挖一层——直接面对 SQL。《第3篇：三句话搞定增删改查》会告诉你，其实 SQL 就三个句式，学会了就能搞定 80% 的工作。
