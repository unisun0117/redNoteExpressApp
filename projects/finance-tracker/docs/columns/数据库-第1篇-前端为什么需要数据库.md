# 第1篇：前端为什么需要数据库？

> 你已经在用数据库了，只是没意识到——localStorage就是一个单表的NoSQL数据库。

## 📌 前端为什么要学这个？

我第一个正经项目，用户数据全存在 localStorage。

上线没几天，有个用户跑来问我："我的数据呢？"我一看后台——啥也没有。他用的是公司电脑，Chrome 里清了个缓存，我的"数据库"就灰飞烟灭了。

那一刻我才知道，**持久化存储不是开玩笑的**。

你写前端天天跟数据打交道：`fetch` 拿数据、state 存数据、props 传数据。但你有没有想过，**这些数据在服务器那端是怎么存的？**

想象你打开金橘记账 App，点开"本月支出占比"——餐饮花了 3000 块。这 3000 是哪来的？不是 localStorage 存的，是数据库里几十条 `INSERT` 语句一毛一毛垒起来的。前端的 `.filter()` `.reduce()` 只是在展示结果，**真正帮你"记住"数据的是数据库**。

学数据库不是因为你"转后端"了。是因为你想知道：**你的数据从哪来，到哪去，凭什么在你关了浏览器后还能活着**。

## 🔍 核心原理

先说结论：**数据库就是个更靠谱的 localStorage。**

回忆下你用 localStorage 的时候：

```javascript
// 存
localStorage.setItem("transactions", JSON.stringify([
  { id: "1", type: "expense", amount: 2500, category: "餐饮" },
  { id: "2", type: "income", amount: 15000, category: "工资" }
]));

// 取
const items = JSON.parse(localStorage.getItem("transactions") || "[]");

// "查"
const foodItems = items.filter(t => t.category === "餐饮");
```

不就是个 `Array<Object>` 嘛。新增 push、删除 splice、查找 filter、修改 find + 赋值 — CRUD 一套搞定。

但你发现了没——它只能存一个"数组"。如果你有用户表、账单表、预算表，全塞进一个 key 里面？结构一复杂，localStorage 就成了灾难现场。

数据库做的事情，本质上和你用 localStorage 一模一样：**存数据、取数据、查数据、改数据**。区别在于它做了三件事你做不到：

**第一件：它帮你管好多"张"表。** 用户一张、账单一张、预算一张，各管各的，互不打架。localStorage 自己顶多是个单表的 NoSQL，表一多你就得自己写逻辑切分，而且每个 key 之间没法做关联。

**第二件：它真的持久。** 不是"浏览器关了就没了"那种持久法，是"服务器重启了数据还在"那种。localStorage 活在用户的硬盘上，数据库活在服务器的磁盘上——服务器不关机、不清缓存、不卸载浏览器。

**第三件：它能做复杂查询。** 你不是只能靠 `filter()` 傻傻遍历。数据库有索引——和在 JS 里用 `Map` 做 O(1) 查找一样。没索引时找一条数据要扫全表（和 `.filter()` 一个速度），有索引后直接定位（和 `Map.get()` 一个速度）。

> 阿珊说：localStorage 是你桌面上的便签纸，数据库是你公司的档案室——都是记录信息，但一个吹口气就没了，一个锁着铁柜子。

看一下金橘记账里数据库和前端的关系就清楚了：

```
你的手机屏幕
    ↓ 点"记一笔支出"
前端 React 组件 (DashboardPage.tsx)
    ↓ api.createTransaction({ type: "expense", amount: 2500, category: "餐饮" })
API 层 (authFetch → POST /api/transactions)
    ↓ FastAPI 路由
后端 (transactions.py)
    ↓ db.add(tx); db.commit()
数据库 SQLite
    ↓ 写入磁盘
数据持久保存 ✅
```

前端发起请求 → 后端路由接收 → 操作数据库 → 返回结果。这四步走通了，你就明白了"前端拿到数据之前发生了什么"。

## 🛠 动手试试

来看看金橘记账里一张真实的数据表是怎么定义的。这是 Transaction 模型，它定义了你每笔账单的结构：

```python
# backend/app/models/transaction.py
class Transaction(Base):
    __tablename__ = "transactions"  # 表名

    id: Mapped[str] = mapped_column(
        String(36), primary_key=True,
        default=lambda: str(uuid.uuid4())
    )  # 主键 — 每条记录的"身份证号"
    user_id: Mapped[str] = mapped_column(
        String(36), ForeignKey("users.id"), nullable=False
    )  # 外键 — 指向 users 表，"这笔账是谁的"
    type: Mapped[str] = mapped_column(
        String(10), nullable=False
    )  # "expense" 还是 "income"
    amount: Mapped[int] = mapped_column(
        Integer, nullable=False
    )  # 金额，单位：分（不是元！）
    category: Mapped[str] = mapped_column(String(50), nullable=False)  # 餐饮/交通/购物
    category_emoji: Mapped[str | None] = mapped_column(String(10), nullable=True)  # 🍽️/🚇/🛍️
    note: Mapped[str | None] = mapped_column(String(200), nullable=True)  # 备注
    tx_date: Mapped[date] = mapped_column(Date, nullable=False)  # 交易日期
    created_at: Mapped[datetime] = mapped_column(
        DateTime, default=datetime.utcnow
    )  # 创建时间
```

如果你只用过前端，这个看起来就像 TypeScript 的类型定义加了一些装饰器。没错——**ORM 就是帮你把 Python 类翻译成数据库表的工具**，就像 JSX 帮你把组件翻译成 DOM。你不用手写建表 SQL，SQLAlchemy 会自动帮你生成：

```sql
CREATE TABLE transactions (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL REFERENCES users(id),
    type VARCHAR(10) NOT NULL,
    amount INTEGER NOT NULL,
    category VARCHAR(50) NOT NULL,
    category_emoji VARCHAR(10),
    note VARCHAR(200),
    tx_date DATE NOT NULL,
    created_at DATETIME
);
```

再看数据库连接是怎么建立的——就三行：

```python
# backend/app/database.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, DeclarativeBase

engine = create_engine(
    settings.DATABASE_URL,
    connect_args={"check_same_thread": False} if "sqlite" in settings.DATABASE_URL else {},
)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
```

`create_engine` 建立数据库连接，`sessionmaker` 创建会话工厂。每次请求来了，FastAPI 就"借"一个 session，用完"还"回去。这个模式和 `fetch` 的 keep-alive 连接复用一模一样——不断的连接断开太浪费资源。

最妙的是第 6 行那个判断：如果是 SQLite，就加 `check_same_thread: False`。为什么？因为 SQLite 默认不让多线程同时写，但在 Web 应用里，同时可能有多个请求进来，不加这个参数会直接报错。懂了前端 event loop 的人看一眼就明白：**数据库也有并发问题**。

再看实际的 CRUD——增删改查长这样：

```python
# backend/app/routers/transactions.py

# 查：列表 + 筛选 + 分页
query = db.query(Transaction).filter(Transaction.user_id == user.id)
if month:
    query = query.filter(Transaction.tx_date.like(f"{month}%"))
if category:
    query = query.filter(Transaction.category == category)
items = query.order_by(Transaction.tx_date.desc())\
    .offset((page - 1) * page_size).limit(page_size).all()

# 增
tx = Transaction(user_id=user.id, type=req.type, amount=req.amount,
                 category=req.category, note=req.note, tx_date=req.tx_date)
db.add(tx)
db.commit()
db.refresh(tx)

# 改
tx = db.query(Transaction).filter(
    Transaction.id == tx_id, Transaction.user_id == user.id
).first()
tx.amount = req.amount
tx.category = req.category
db.commit()

# 删
tx = db.query(Transaction).filter(
    Transaction.id == tx_id, Transaction.user_id == user.id
).first()
db.delete(tx)
db.commit()
```

帮你在脑子里翻译成前端：

| SQLAlchemy 操作 | 前端等价 |
|----------------|---------|
| `db.query(Transaction).filter(...)` | `array.filter(item => item.type === "expense")` |
| `.order_by(desc()).offset().limit()` | `.sort().slice()` |
| `db.add(tx); db.commit()` | `array.push(obj); localStorage.setItem(...)` |
| `db.query(...).first()` | `array.find(item => item.id === id)` |
| `db.delete(tx); db.commit()` | `array.splice(idx, 1); localStorage.setItem(...)` |

一模一样的增删改查，只是语法不同。**你其实早就会用数据库了，你只是没用过 SQL 语法而已。**

最后看一眼，前端那边也不裸调 fetch，封装了一个带自动刷新 token 的 `authFetch`：

```typescript
// frontend/src/services/api.ts
async function authFetch(path: string, options: RequestInit = {}): Promise<Response> {
  const headers: Record<string, string> = {
    ...(options.headers as Record<string, string>)
  };
  if (accessToken) headers["Authorization"] = `Bearer ${accessToken}`;

  let res = await fetch(`${BASE_URL}${path}`, { ...options, headers });
  if (res.status === 401 && refreshToken) {
    const refreshed = await refreshAccessToken();
    if (refreshed) {
      headers["Authorization"] = `Bearer ${accessToken}`;
      res = await fetch(`${BASE_URL}${path}`, { ...options, headers });
    } else {
      clearTokens();
      window.location.href = "/login";
    }
  }
  return res;
}
```

这就是前后端之间数据流通的完整链路。你点"记一笔支出"→ `authFetch` 发请求 → FastAPI 路由接收 → SQLAlchemy 操作数据库 → 数据写进磁盘。每一环你都看得见。

## ⚠️ 常见坑点

| 坑 | 现象 | 解法 |
|----|------|------|
| localStorage 存敏感数据 | 把 JWT token、用户密码存 localStorage，任何插件都能读到 | token 存 httpOnly cookie 或至少用内存变量管理，不要裸放 localStorage |
| 金额用浮点数存 | `19.9` 存数据库变成 `19.899999…`，算多了账对不上 | 金额一律用整数存，单位用分。金橘记账所有 `amount` 字段都是 `Integer`，前端 `v/100` 显示为元 |
| 不分表全塞一个数组 | 用户、账单、预算全混在一个 JSON 里，查一个用户的所有账单要嵌套过滤三次 | 每类数据单独建表，用 `user_id` 外键关联。表多了不丢人，数据脏了才丢人 |
| SQLite 默认拒绝多线程 | 后端一并发请求就报 `SQLite objects created in a thread can only be used in that same thread` | 连接字符串加 `check_same_thread: False`，或者直接上 PostgreSQL |
| 不清数据库就改模型 | 改了 Transaction 模型，加了新字段，启动直接报错 `no such column` | 要么删库重建（本地开发），要么用 Alembic 做数据库迁移（生产环境） |

## 🎯 面试会问

- **Q：localStorage 和数据库的核心区别是什么？**
  **答：**localStorage 是客户端浏览器的临时存储，用户清缓存就没了，容量 5-10M，只能存字符串；数据库是服务器端的持久化存储，写进磁盘，容量几乎无限，支持结构化查询和事务。金橘记账项目里，JWT token 虽然暂时借用了 localStorage 存取，但核心业务数据（用户账单、预算）全部存在后端 SQLite 数据库里，服务和用户设备无关。

- **Q：ORM 是什么，为什么不用纯 SQL？**
  **答：**ORM 就是你写 Python 类，它帮你生成 SQL。类比前端：你写 JSX，React 帮你生成真实 DOM——你不用手写 `document.createElement`。SQLAlchemy 的 `Base` 类就相当于 JSX 的组件基类。ORM 的好处是不用写拼接字符串的 SQL，不容易写错，而且自动防注入。但复杂查询还是要懂 SQL，就和复杂组件你还是要理解 DOM 一样。

## 📝 小结

数据库没那么吓人。你今天写的每一个 `localStorage.setItem`，每一行 `array.filter`，本质上都是在手动模拟数据库做的事。区别只是：数据库帮你把"存"和"查"做到了生产级——持久、可靠、能并发。

这一篇我们先建立认知：数据库 = 更靠谱的 localStorage。下一章我们扒开看看，数据库到底怎么在磁盘上存数据的——SQL 语句长什么样，索引为什么能查这么快，事务怎么保证"要么全成功要么全失败"。
