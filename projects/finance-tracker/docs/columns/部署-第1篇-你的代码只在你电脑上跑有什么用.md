# 第1篇：你的代码只在你电脑上跑有什么用 — 从 localhost 到公网

> localhost:5173 只有你自己能打开。把代码变成一个别人能访问的网址，才叫"做出来了"。

## 📌 前端为什么要学部署？

去年冬天，我花了三周做完第一个 React 项目。在 `localhost:5173` 上跑得飞起，页面丝滑，功能齐全。我截图发到闺蜜群里说"你们看我做的！"朋友们说"发链接来玩玩"。

我复制了地址栏：`http://localhost:5173`。发过去。

对方回："打不开啊。"

那一瞬间我愣住了。然后才反应过来——**我的电脑不是服务器，她们连不上我的网络**。我写了一个别人看不见的东西。就像一个餐馆，厨具齐全、菜都做好了，但是没有门让客人进来。

从那之后我就养成了一个习惯：**每个项目最后一个 commit 之前，先让它上线**。没部署 = 没做完。

如果你现在的状态是：能 `npm run dev`，但不知道怎么让远在老家的朋友打开你写的页面——这篇就是为你写的。

## 🔍 核心原理

先说结论：**部署就是把你的前端代码，从你的电脑搬到一台公网能访问的服务器上。**

你写前端对下面这个概念肯定不陌生：

```
npm run dev → Vite 启动一个开发服务器 → 监听 localhost:5173
```

这是什么意思？翻译成人话就是：Vite 在你电脑上开了一个"临时的后门"，只有你自己能用浏览器敲开。

为什么别人打不开？因为 `localhost` 是一个特殊地址，它永远指向"你这台电脑自己"。别人的电脑上输 `localhost`，指向的是**他们自己的电脑**，不是你电脑上的 Vite 服务。

来看一张对比：

```
你在自己的机器上开发：
  你的浏览器 ──→ localhost:5173 ──→ Vite 开发服务器（你的电脑上）
  朋友的浏览器 ──X  localhost:5173 ──→ 他自己的电脑（啥也没有）

部署到线上后：
  你的浏览器 ──→ https://finance-tracker.vercel.app ──→ Vercel 服务器
  朋友的浏览器 ──→ https://finance-tracker.vercel.app ──→ 同一台 Vercel 服务器 ✅
```

这就是核心区别。部署之后，所有用户访问的是**同一台公网服务器**，不是你桌面上的那台电脑。

还是不太明白？用前端最熟悉的类比：**`npm run dev` 等于你在家炒菜，部署等于你开了一家餐厅让客人来吃。** 你做的菜再好吃，没人能进你家厨房夹一筷子。餐厅就不一样——有招牌（域名）、有门牌号（公网 IP）、有开放式厨房（服务端口）。你换了两个术语，但本质没变，后厨还是那个后厨。

> 阿珊说：代码跑在 localhost 上就像日记锁在抽屉里——写得再好也只有自己知道。

把这条链路拆开来看，部署实际上跨了三层：

```
第 1 层：你的代码
  ├── index.html
  ├── main.js (被 Vite 打包)
  └── style.css (被 Vite 处理)
          ↓ 构建 (npx vite build)
第 2 层：静态文件产物
  ├── dist/index.html
  ├── dist/assets/index-abc123.js
  └── dist/assets/index-def456.css
          ↓ 部署到服务器
第 3 层：公网可访问的 URL
  └── https://你的项目.vercel.app
```

这三层是你做部署永远绕不过的认知底座。第一层是你写的东西，第二层是 Vite 帮你"编译"出来的东西（浏览器不认得 TypeScript 和 JSX，需要先翻译成原生 JS + CSS 才行），第三层是你部署的终点——一个活生生的网址。

如果你用过 Vercel，你可能觉得部署就是 `git push` 然后自动上线 —— 没错，那是因为 Vercel 帮你把三层全包了。但你至少要知道这三层是存在的，不然出了 bug 你都不知道去哪一层排查。

## 🛠 动手试试

接下来带你体验"从 localhost 到公网"的最短路径。三分钟就够了。

### 第一步：确认你的 localhost 是跑着的

打开金橘记账的前端（或者随便一个你做过的项目）：

```bash
cd E:\AI_Workspace\projects\finance-tracker\frontend
npm run dev
```

看到 Vite 输出这个，说明你的开发服务器是活的：

```
  VITE v5.x.x  ready in xxx ms
  ➜  Local:   http://localhost:5174/
  ➜  Network: http://192.168.x.x:5174/
```

记下 `http://localhost:5174` 这个地址。你现在在浏览器里打开它——没问题，能访问。

### 第二步：用 ngrok 把它暴露到公网

[ngrok](https://ngrok.com) 是一个临时隧道工具，它给你一个公网地址，指向你本地的端口。不需要注册账号（免费版即可），不用改一行代码。

先去 [ngrok.com](https://ngrok.com) 注册一个免费账号，下载安装。然后在终端跑：

```bash
# 开一个新的终端窗口，不要关掉 npm run dev
ngrok http 5174
```

你会看到类似这样的输出：

```
Forwarding   https://abc123.ngrok-free.app -> http://localhost:5174
```

把这个 `https://abc123.ngrok-free.app` 发给你的朋友——或者你自己用手机打开。你能看到你自己的 localhost 项目，**在任何一台联网的设备上都能打开**。

30 秒前，你的代码还窝在你的电脑里。现在，全世界任何一个联网的设备都能访问你的项目了。这就是部署的本质。

### 第三步：体验完了，想想 ngrok 做了什么

ngrok 做的事情就是：它在公共互联网上租了一个地址，然后把所有访问这个地址的请求，通过一个加密隧道转发到你的电脑端口上。

```
用户的浏览器
    ↓ https://abc123.ngrok-free.app
ngrok 公网服务器（中转站）
    ↓ 加密隧道
你的电脑:5174（Vite dev server）
```

这样做当然不是真的"部署"——你的电脑一关机，链接就断了。真正的部署是让你的代码跑在**一个永不停机的服务器**上。但概念是一模一样的：**给你的代码一个公网地址。**

这个体验的目的不是让你用 ngrok 当生产环境，而是让你亲手摸到"公网"两个字长什么样。摸了之后，后面讲 Vercel、讲 nginx、讲 Docker，你都能回到这个基础认知上：无非就是"把文件放到一台公网服务器，配好端口和域名"。

## ⚠️ 常见坑点

我刚接触部署时踩过的坑，一个个说给你听。

| 坑 | 现象 | 为什么 | 怎么解 |
|---|---|---|---|
| 把 localhost 发给别人 | 别人打不开 | localhost 指向自己的电脑 | 用 ngrok、Vercel、或者买服务器 |
| 公司/校园网络有防火墙 | ngrok 连不上 | 企业网络经常封掉非标准端口 | 换手机热点试一下，或者连家里 Wi-Fi |
| Vite 只绑定了 localhost | 同 Wi-Fi 下别人也打不开 | Vite 默认 `host: "localhost"` | 改成 `host: "0.0.0.0"` 或 `host: true` |
| `npm run build` 报错 | `tsc -b` 卡住 | TypeScript `verbatimModuleSyntax` 不兼容 | 用 `npx vite build` 而不是 `npm run build` |
| 端口被占用 | `Error: port 5173 is already in use` | 另一个 Vite 进程没关 | `npx kill-port 5173` 或者在 `vite.config.ts` 里换端口 |
| 环境变量丢了 | 部署后 API 请求全 404 | `VITE_API_URL` 在 `.env` 里配了 localhost，部署后没改 | 在 Vercel 环境变量面板设置 `VITE_API_URL` 指向你的后端线上地址 |

这几个坑我帮你们踩过了。尤其是第一个——**别把 localhost 发给别人**。这条是新手入门部署的必修课，我花了整整一个晚上才悟出来的。

最值得多说一句的是第四坑：**`npm run build` vs `npx vite build`**。很多前端项目（包括金橘记账和红薯快写）都在 `package.json` 配了 `"build": "tsc -b && vite build"`。问题是 `tsc -b` 会在 TypeScript 5.x + `verbatimModuleSyntax` 配置下报错。如果你部署的时候一直 build 失败，试试直接用 `npx vite build` 绕过 TypeScript 编译。`tsc -b` 做的是类型检查，不是构建必需的步骤。

## 🎯 面试会问

**Q：当你输入一个网址回车后发生了什么？从前端角度答。**

这道题面试官真的会问。而且最妙的是，学会部署之后，你可以不在"DNS 解析 → TCP 三次握手"这个层面卡住，而是给出一个**有前端开发视角的完整答案**：

```
用户输入 https://finance-tracker.vercel.app
  ↓
1. 浏览器解析 URL — 协议(HTTPS) + 域名 + 路径
  ↓
2. DNS 查询 — 把域名翻译成 IP 地址
   （就像查通讯录：finance-tracker.vercel.app → 76.76.21.21）
  ↓
3. 建立 HTTPS 连接 — TCP 三次握手 + TLS 加密握手
  ↓
4. 浏览器发送 HTTP GET 请求到 Vercel 服务器
  ↓
5. Vercel 服务器收到请求，返回打包好的三件套：
   - dist/index.html
   - dist/assets/index-abc123.js
   - dist/assets/index-def456.css
  ↓
6. 浏览器收到 HTML，开始解析：
   - 遇到 <link rel="stylesheet"> — 下载 CSS，阻塞渲染
   - 遇到 <script type="module"> — 下载 JS，执行 React 代码
  ↓
7. React 挂载到 #root，组件开始渲染
   - useEffect / useLayoutEffect 触发
   - 如果有 API 调用（authFetch → VITE_API_URL）
   - 发起 XHR/Fetch 请求到后端（https://api.xxx.com/api/auth/me）
  ↓
8. 页面渲染完成，用户看到你的 App
```

如果你能在面试中把第 5 步到第 8 步**用你自己的项目举例**，你已经赢了。比如"我部署在 Vercel 上，Vite 构建产出的 `dist/` 被上传到 Vercel 的 CDN 节点，用户请求时从最近的 CDN 节点返回静态文件"——这就是一个能让人记住的回答。

**加分回答：** 追问"你还知道什么相关的东西？"你可以接上"前端部署的安全配置"：HTTPS 必须开、环境变量不能提交到 Git、CORS 要配 `ALLOWED_ORIGINS`、构建产物的 Source Map 不要发到生产环境。

## 📝 小结

这一篇我只想让你记住一件事：**把代码跑在 localhost 上不算做完，跑在一个别人能打开的网址上才算。**

你不需要在这一篇就学会怎么配 Nginx、怎么买服务器、怎么建 CI/CD 流水线。那些是后面的事情。这一篇要的是认知转变——你要从"我的项目在本地能跑"切换到"我的项目上线了吗"。

下一篇我们讲 Vercel——为什么它是前端的部署初恋。你不需要配服务器、不需要装系统、不需要学 Linux，一个 `git push` 就能把项目上线。下期见。
