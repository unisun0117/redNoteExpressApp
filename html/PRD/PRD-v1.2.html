0. 全局业务规则

⏰ 截单机制

  • 截单时间：每日 22:00:00
  • 22:00 前下单 → 状态"已创建"，可退款
  • 22:00 系统定时任务 → 所有"已创建"订单 → "已锁定"
  • 已锁定后不可退款，进入供应链流程
  • 次日 00:00 起为新的下单日

📦 供应链时序

  • 22:00 截单 → 系统汇总订单 → 通知各供应商
  • 供应商送货到中央仓 → 仓库分拣称重
  • 自有冷链物流凌晨取货 → 6:00-7:00 配送到客户冰箱
  • 配送员拍照签收 → TMS回传POD
  • 全程当日达，无在途损耗

🔢 偏差规则

  • 生鲜商品出库重量与订购量允许微量偏差
  • 客户已知晓并接受此规则
  • 偏差不做百分比限制，如实记录
  • 差异金额通过退款/待收款处理

👥 企业与用户

  • 一个企业(Customer) 可有多名采购员(User)
  • 采购员账号由后台/BD创建，不支持自主注册
  • 登录：微信授权 + 手机号绑定
  • 收货地址BD录入时确定（含GPS坐标），不可修改

1.1 完整状态流转

已创建

采购员提交订单。微信支付已扣款/月结已通过校验。截单前(22:00前)可退款。

→ 已锁定(22:00) | 已取消(主动退款)

已锁定

22:00截单后自动变更。不可退款。系统汇总通知供应商。

→ 分拣中(供应商到仓)

分拣中

供应商已送货到仓，仓库人员逐项称重填入实发量。差异自动计算。

→ 已分拣(确认分拣)

已分拣

分拣完成，实发金额锁定。等待推送TMS创建配送任务。

→ 配送中(推送TMS)

配送中

自有物流取货配送中。TMS回传GPS(仅BD可见)。小程序显示物流信息+预计送达。

→ 已签收(TMS回传POD)

已签收

客户签收确认。结算量=实发量(无在途损耗)。月结:正式扣减额度。微信:差异退款/生成待收款。

→ 已完成(无异议) | 退货处理中(申请售后)

已完成

订单锁定不可修改。所有金额已清算。月结金额计入对账单。

→ 终态

退货处理中

采购员申请售后，运营审核中。含退货退款和仅退款两种子路径。

→ 已完成(退款处理完毕)

已取消

截单前采购员主动退款。微信原路退回。月结释放额度校验。

→ 终态

2. 支付与额度引擎

2.1 微信支付全链路

下单时： 签收后： 订购金额 = ¥232.00 实发金额 = ¥231.20 │ │ ▼ ▼ 微信支付扣款 ¥232.00 ┌─ 实发 < 订购(少0.80) → 原路退回 ¥0.80 │ │ │ └─ 实发 > 订购(多3.50) → 生成待收款 ¥3.50 │ │ ▼ ▼ 订单"已创建" 下次下单时 CHECK: │ 有未清待收款 → 阻断下单 │ 必须先支付待收款 ▼ 22:00截单 → 供应链 → 分拣 → 配送 → 签收
⚠️ 待收款逻辑：
  • 实发 > 订购时，差额自动生成一条PendingPayment记录，挂在客户企业名下
  • 采购员下次提交订单时，系统校验该企业是否有未清待收款
  • 有待收款 → 弹窗提示"您有待收款 ¥X.XX 需先支付"，阻断下单，引导支付
  • 待收款支付成功后，方可继续下单
  • 待收款也支持微信支付（小程序端支付）

2.2 月结额度全链路

下单时： 签收后： CHECK(可用额度 ≥ 订购金额) 实发金额 = ¥231.20 │ │ ├─ 充足 → 下单成功 ▼ │ 仅校验，不扣减 正式扣减额度: │ credit_used += ¥231.20 └─ 不足 → 阻断下单 可用额度 = 总额度 - credit_used 提示"额度不足，联系BD"
额度计算公式：
可用额度 = credit_limit(总额度) - credit_used(已挂账总额)
额度状态阈值：
状态条件行为
正常可用额度 > 订购金额正常下单
预警已挂账 > 总额度 × 80%可下单，BD端显示🟡预警
超额可用额度 < 订购金额阻止下单，提示联系BD
冻结BD手动冻结阻止下单，提示"月结已冻结"

2.3 支付方式判断

企业月结状态可用额度采购员可选默认
已开通 + 正常充足月结挂账 / 微信支付月结挂账
已开通 + 正常不足微信支付（月结置灰+"额度不足"）微信支付
已开通 + 预警充足月结挂账 / 微信支付月结挂账（+提示）
已开通 + 超额不足微信支付（月结不可选）微信支付
已开通 + 冻结-微信支付（月结不可选）微信支付
未开通-微信支付微信支付

2.4 退款渠道对应

原支付方式退款方式触发场景
微信支付原路退回微信截单前退款 / 实发<订购差额 / 退货审核通过
月结挂账额度冲销(credit_used -= 退款金额)截单前退款 / 实发<订购差额 / 退货审核通过

  • 差异 > 0：红色（待收款）
  • 差额金额 = 差异 × SKU价格
  • 实发金额 = 订购金额 + 差额合计

确认签收

  • 点击弹出签收确认弹窗
  • 列出每件商品实发量，采购员确认
  • 正常签收：确认 → 状态→已签收 → 触发结算
  • 有问题：点击"申请售后"→ 跳转退货申请页
  • 签收后不可撤销

分拣操作逻辑

  • 实发量输入框默认预填订购量
  • 运营修改实发量 → 差异自动计算（实发 - 订购）
  • 差异在 ±10% 以内 → 显示黄色
  • 差异超过 ±10% → 显示红色
  • 实发量 > 订购量 → 系统弹出确认框"已电话沟通客户确认?"
  • 实发量 = 0 → 该商品标记为缺货

确认分拣后触发

  1. 订单状态 → "已分拣"
  2. 实发金额锁定 = Σ(实发量 × SKU单价)
  3. 微信支付订单：自动计算差额 → 实发<订购?发起原路退款 : 生成待收款记录
  4. 月结订单：不做任何扣款（等待签收后）
  5. 记录操作人和时间戳

暂存逻辑

  • 保存当前填写进度，不改变订单状态
  • 其他运营可继续分拣
  • 订单状态保持在"分拣中"

标记缺货

  • 被标记缺货的商品：实发量强制置0
  • 订单确认分拣时：该商品金额自动全额退款
  • 微信支付：原路退该商品金额
  • 月结：不扣减该商品金额
