从 0 到 1
设计一个 Agent

— 不是调 API，不是写 Prompt，是设计一个能自主行动的 AI 系统 —

你可能是这样想的："Agent 不就是让 AI 多调几个工具吗？"
大错特错。设计 Agent 和写 Prompt 的差距，
就像搭积木和设计一座城市——完全不是一个量级的事。

🤖 到底什么是 Agent？

先对齐概念——不然聊半天发现你说的 Agent 和我说的不是一个东西。

🧠 推理引擎
LLM — 负责"想"
👁️ 感知层
读文件 · 搜代码 · 调 API · 查数据库
✋ 行动层
写文件 · 执行命令 · 发请求 · 调工具
🧭 记忆层
上下文 · 向量库 · 文件记忆 · 对话历史

❌ Agent 不是

  • 一个写得更长的 Prompt
  • 一个只会一问一答的聊天机器人
  • 一个固定的自动化脚本
  • 调几次 API 就算 Agent

✅ Agent 是

  • 能自主感知环境状态
  • 能独立决策下一步做什么
  • 能主动调用工具改变环境
  • 能从反馈中学习和调整
🔑 核心区别Chatbot 等你说完才动
Agent 自己决定什么时候动、动什么
🎭 一句话定义Agent = LLM + Tools + Memory + Control Loop
📐 工程本质你设计的是"决策框架"
不是"对话模板"

📐 设计 Agent 的七步法

这不是教科书上的理论——这是实战踩坑后总结的设计框架。每一步都有明确的交付物。

Step 1

🎯 定义边界

Agent 要解决什么问题？不解决什么问题？输入什么、输出什么？成功的标准是什么？

Step 2

🧬 设计人格与系统提示词

Agent 的"性格"是什么？它遵循什么行为准则？它的知识边界在哪？遇到不知道的事怎么办？

Step 3

🔧 设计工具矩阵

Agent 需要哪些"手"和"眼"？每个工具的输入输出是什么？权限边界怎么划定？

Step 4

📚 设计上下文策略

每次调用 Agent 时它能看到什么？长对话怎么压缩？哪些信息持久化、哪些用完就丢？

Step 5

🔄 设计控制流

Agent 的执行模式是什么？ReAct 循环？Plan-then-Execute？多 Agent 协作？单步还是多步？

Step 6

🛡️ 设计异常与边界

工具调用失败了怎么办？Token 超了怎么办？死循环了谁来喊停？用户中断怎么优雅退出？

Step 7

🧪 测试与迭代

怎么验证 Agent 的行为符合预期？回归测试怎么做？如何系统化地评估 Agent 质量？

🎯 第一步：定义边界

1

最难的不是"做什么"，是"不做什么"

新手常犯的错误：想让 Agent 什么都能干。结果是——什么都能干 = 什么都干不好。

好的 Agent 设计从一个清晰的边界声明开始。问自己四个问题：

1. 它解决什么问题？

用一句话说清楚。比如："自动将 Figma 设计稿转成 React 组件代码"

2. 它不解决什么？

明确拒绝范围。比如："不管设计评审、不管后端 API 生成、不管部署"

3. 输入输出是什么？

输入：Figma 链接 + 组件库选择
输出：可运行的 .tsx + .css 文件

4. 成功的标准？

可量化。比如："生成的代码 ESLint 0 error，视觉还原度 > 90%"

经验之谈：边界越清晰，Agent 越可靠。模糊的边界 = 模糊的行为 = 不可预测的结果。宁可先做小做精，也不要一上来就做大而全。

🧬 第二步：设计人格与系统提示词

2

System Prompt 是 Agent 的"宪法"

它决定了 Agent 的所有行为底色。不是写一段"你是一个 helpful assistant"就完事了——那等于没设计。

🎭 角色定义

"你是谁"——不是泛泛的"助手"，而是具体的角色。比如"你是一个拥有 10 年 React 经验的资深前端工程师"

📏 行为准则

"你怎么做事"——遇到不确定的事先问还是先做？代码风格是什么？注释写到什么程度？

🚫 硬约束

"你绝对不能做什么"——不能删除文件、不能 push 代码、不能修改配置文件。这是安全底线。

📐 质量标准

"做到什么程度才算好"——测试必须过、Lint 零报错、代码风格匹配项目、改动要解释原因。

// 好的 System Prompt 结构（简化示意） ## 你是谁 你是一个专门做 React 组件重构的 Agent。 ## 你怎么工作 1. 先读完整文件，理解现有逻辑 2. 拆解重构步骤，列出改动清单 3. 逐个文件修改，每次改完检查 lint 4. 全部改完跑一遍测试 ## 你不能做什么 - 不能改后端 API 接口 - 不能删掉还在用的代码（先确认引用） - 改动超过 5 个文件先出方案让用户确认 ## 质量标准 - ESLint 0 error 0 warning - 每个组件必须有 PropTypes / TypeScript 类型 - 改动说明写在 commit message 里

🔧 第三步：设计工具矩阵

3

工具 = Agent 的能力边界

给 Agent 什么工具，它就能做什么事。工具设计有三个核心问题：选哪些工具、怎么描述工具、权限怎么控。

🛠️ 工具选择原则

  • 最小充分：够用就好，别堆砌
  • 原子化：一个工具只做一件事
  • 可组合：工具之间能串联协作
  • 有反馈：每次调用返回明确结果
  • 可降级：出错时有 fallback 策略

📝 工具描述（Function Schema）

  • name：动词+名词，如 search_files
  • description：一句话说清功能 + 使用时机
  • parameters：类型、必填/可选、默认值
  • 返回格式：让 Agent 知道调用完会拿到什么
⚠️ 常见坑工具描述写得太模糊
Agent 不知道该什么时候用
→ 要么不用，要么乱用
💡 最佳实践每个工具的 description 里写上"when to use"——给 Agent 明确的触发条件
🔒 安全要点危险工具加确认机制
delete / force push / drop table
必须先过用户这关

📚 第四步：设计上下文策略

4

上下文是 Agent 的"视野"和"记忆"

Agent 每做一次决策，能看到的全部信息 = 上下文。视野太窄 → 瞎猜。视野太宽 → 被无关信息干扰。记忆太短 → 金鱼脑。记忆太长 → Token 爆炸。

📋 静态上下文

每次调用都带的"底座"

  • System Prompt（人格 & 规则）
  • CLAUDE.md（项目规范）
  • 工具定义（可用能力）
  • Memory 文件（持久偏好）

🔄 动态上下文

根据当前任务注入

  • 相关文件内容（Read 结果）
  • 搜索结果（Grep/Glob）
  • Git 状态（当前分支 & diff）
  • 环境信息（OS / Shell）

🗜️ 压缩策略

长对话不爆 Token 的关键

  • 对话摘要（总结旧消息）
  • 语义去重（删重复信息）
  • 滑动窗口（只保留最近 N 轮）
  • 分级存储（热/温/冷数据）
System 18%
Memory 12%
Tools 15%
Files 20%
History 35%

▲ 典型 Agent 调用的 Token 分配 | 历史对话往往占大头，压缩策略直接影响成本

🔄 第五步：设计控制流

5

Agent 怎么"跑"起来——四种经典模式

控制流决定了 Agent 的执行节奏。选错模式，要么效率极低（来回问），要么失控（改飞了）。

🔁

ReAct 模式

Thought → Action → Observation → 循环

每走一步都"先想、再做、再看结果"。最经典、最稳定。适合需要动态调整方向的任务。

适用：代码修改、Debug、探索性任务

📋

Plan-then-Execute

先出完整计划 → 用户确认 → 一口气执行

降低不确定性。用户看到完整方案后再动手，避免"改到一半发现方向错了"。

适用：重构、多文件改动、架构级变更

👥

Multi-Agent 协作

多个 Agent 分工协作，各司其职

一个负责分析、一个负责编码、一个负责审查。Pipeline 或 Parallel 方式组织。

适用：大型项目、跨领域任务、需要多重验证

🔁

自主循环模式

Agent 自己决定什么时候停

设定结束条件（如"测试全部通过"或"连续3轮无新发现"），Agent 自己循环直到满足。

适用：批量处理、持续优化、自动化审计

选型建议：80% 的场景用 ReAct 就够了。复杂任务加上 Plan Mode 作为前置步骤。等你熟练了，再考虑 Multi-Agent 和自主循环。不要一上来就追求复杂的控制流——简单的模式跑通了，复杂的自然水到渠成。

🛡️ 第六步：设计异常与边界

Agent 不出错的概率是 0%。
区分好 Agent 和差 Agent 的不是"会不会出错"，而是"出错了之后怎么办"。

🚨 常见异常场景

  • 工具调用失败（文件不存在、权限不够）
  • Token 超限（对话太长，塞不下了）
  • 死循环（Agent 反复做同一件失败的事）
  • 幻觉（生成了不存在的 API/函数名）
  • 用户中断（中途想换个方向）
  • 超时（一个工具调用卡住太久）

✅ 应对策略

  • 自动重试：工具失败 → 换个参数再试一次
  • 优雅降级：做不到完美 → 做到 80% 然后标记
  • 熔断机制：连续失败 3 次 → 停下来问用户
  • 状态检查点：每 N 步检查一次"方向对吗"
  • 超时控制：每个工具调设置 timeout
  • 可恢复：中断后能接着之前的状态继续
🔑 黄金法则宁可停下来问用户
也不要自己瞎猜着改
⏱️ 超时设计每个工具调用都要有
合理的 timeout
别让 Agent 卡住不动
🔄 重试智慧重试不是机械重复
要带着新信息重试
（换个参数 / 换个策略）

🧪 第七步：测试与迭代

7

Agent 的测试比传统软件难一个数量级

传统软件：输入 A → 断言输出 B。确定性的。

Agent：同一个输入可能产生不同的决策路径。因为 LLM 本质上是概率性的。

🧪 单元级

单个工具调用是否返回正确格式？Schema 校验是否通过？权限控制是否生效？

这部分可以自动化——传统测试方法完全适用

🔗 集成级

多步工具调用串联是否顺畅？上下文传递是否完整？中间步骤失败时恢复是否正常？

需要 mock 工具，模拟真实场景

🎯 行为级

Agent 的行为是否符合预期？面对模糊指令是否会主动澄清？边界情况处理是否合理？

最难的层级——需要人工评估或 LLM-as-Judge

🏗️

设计 Agent，
本质上是设计一个"决策系统"

回到最开始那句话：

Agent = LLM + Tools + Memory + Control Loop

这四个要素，LLM 是最容易换的——今天用 Claude，明天换 GPT，后天用开源模型。

但 Tools 的设计、Memory 的策略、Control Loop 的架构——这些才是你真正的护城河。

不要只做一个"会写 Prompt 的人"。
做一个会设计 Agent 系统的工程师。

✏️ 优秀的 Agent 不是"调"出来的——是"设计"出来的
