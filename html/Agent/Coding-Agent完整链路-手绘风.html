Coding Agent
全链路揭秘

— 从你输入一句话，到代码真正被修改 · 每一个字节的旅程 —

你以为：我说"帮我修个 bug"→ AI 直接改了代码
实际上：这条链路上有 6 大环节、数十次工具调用、上万 token 的上下文编排——每一个环节做不好，结果都会崩。

🔗 全链路流水线 · 六大环节

这不是一个"黑盒"——每一个环节都有明确的输入、处理和输出。理解这条链路，你才能真正驾驭 Coding Agent。

环节 1

📥 任务输入与预处理

用户输入自然语言任务 → 系统提取关键意图 → 确定是否需要 Plan Mode → 判断任务复杂度 → 决定策略（直接执行 or 先出方案）

环节 2

📚 上下文组装

加载 System Prompt → 注入 CLAUDE.md / Memory → 附上 Git Status → 拼入相关文件 → 工具定义注入 → Token 预算分配

环节 3

🧠 模型推理与规划

LLM 理解任务全貌 → 拆解为子任务 → 决定调用哪些工具 → 生成执行计划 → 可能先进入 Plan Mode 等用户确认

环节 4

🔍 环境探索

Glob 搜文件结构 → Grep 搜代码内容 → Read 读关键文件 → 建立项目心智模型 → 定位需要修改的代码位置

环节 5

✏️ 代码修改与落地

生成修改方案 → Edit 精确替换 or Write 全量覆写 → 保持代码风格一致 → 处理边界条件 → 多文件协同修改

环节 6

✅ 验证与闭环

自我审视修改结果 → Lint 检查 → 运行测试 → 错误时自动回到环节 4 重试 → 向用户汇报 → 等待反馈进入下一轮

📥 环节一：任务输入与预处理

1

你的第一句话，经历了什么

输入形式：自然语言（"帮我加一个登录功能"、"修一下这个 bug"、"重构 UserService"）

系统预处理：

1. 意图识别 → 这是"新增功能"还是"修bug"还是"重构"？ 2. 复杂度评估 → 单文件改动 or 多文件联动？ 3. 策略选择 → 直接上手改 or 先进 Plan Mode 出方案？ 4. 权限检查 → 需要用户确认吗？文件在可操作范围内吗？ 5. 上下文预热 → 加载 Git Status、最近提交记录、项目结构概览
意图分类复杂度判定策略路由 权限门控上下文预热
🎯 关键设计Coding Agent 不会"你说啥它就改啥"——它会先判断这件事能不能做、怎么做
⚠️ 常见坑任务描述太模糊 → Agent 瞎猜 → 改错地方 → 返工
💡 最佳实践说清楚：改哪个文件、达到什么效果、有什么约束条件

📚 环节二：上下文组装

2

决定 AI "看到什么"的关键一步

这是 Coding Agent 最被低估的环节。很多人以为 AI 厉害是因为模型强——其实一半的功劳在上下文组装。

每次调用前，系统要精心编排 AI 眼前的"视野"：

📋 必装内容

  • System Prompt — 角色定义、行为准则
  • CLAUDE.md — 项目规范、架构说明
  • Memory 文件 — 用户偏好、历史决策
  • Git Status — 当前分支、变更状态
  • 工具定义 — 所有可用工具 + 参数 Schema
  • 对话历史摘要 — 长对话压缩后的精华

🧩 按需注入（动态）

  • 相关文件内容 — 代码库搜索到的匹配文件
  • 目录结构 — Glob 扫描的项目骨架
  • 最近提交 — 相关文件的 git log
  • 外部文档 — RAG 检索到的技术文档
  • 运行环境信息 — OS、Shell、平台版本
  • Skills 定义 — 可用的斜杠命令/技能
一句话总结：上下文组装 = 给 AI 配一副"眼镜"。镜片越清晰（信息越精炼），AI 看得越准。垃圾信息塞多了，再强的模型也会"眼花"。

🧠 环节三：模型推理与规划

3

LLM 的"思考"过程

有了上下文，模型开始"想"。这不是一个简单的"输入→输出"——现代 Coding Agent 会经历一个多步推理链。

1. 理解

解析用户意图，理解任务全貌。"他真的想要什么？这个项目是什么结构？"

2. 拆解

把大任务切成小步骤。"先读哪个文件？改哪里？改完之后要验证什么？"

3. 决策

选择工具和策略。"用 Edit 精确替换还是 Write 重写？需不需要先进 Plan Mode？"

4. 生成

产出具体的工具调用序列。"Read 第 42 行 → Edit 替换 → Bash 跑测试"

🧠 Thinking 模式部分 Agent 支持"思考链可见"——你可以在终端看到 AI 的推理过程
🎭 Plan Mode复杂任务先出方案，用户确认后再动手——避免"一顿操作猛如虎，一看结果全白费"
⚡ 推理 ≠ 执行模型只负责"想"和"决定调用什么工具"——真正的文件操作在下一步

🔍 环节四：环境探索与工具调用

4

Coding Agent 的"手"和"眼睛"

模型不能直接操作文件。它通过工具调用（Tool Call / Function Call）来与外部世界交互。这是一套精密的 API 协议。

👀 读类工具（探索世界）

  • Read — 读取文件内容（最常用）
  • Glob — 按模式匹配文件路径
  • Grep — 正则搜索代码内容
  • Bash — 执行 shell 命令收集信息
  • WebFetch — 查阅在线文档

✏️ 写类工具（改变世界）

  • Edit — 精确字符串替换（不改错）
  • Write — 创建/覆写整个文件
  • Bash — 执行命令（npm install 等）
  • NotebookEdit — Jupyter 单元格操作
关键机制：每次工具调用的结果会追加回上下文，作为模型下一轮推理的输入。这就是 Agent 能"边看边改"的秘密——感知→行动→再感知的循环。

✏️ 环节五：代码修改与落地

5

从"想法"到"改好了"的最后一公里

真正改代码的方式比你想象的讲究得多。不是"AI 写一段代码贴进去"——有一套精确的操作协议。

🎯 Edit（精确替换）

最常用方式。给一个 old_string 精确匹配原文，再给 new_string 替换。必须 Read 过才能 Edit——防止瞎改。

old_string 必须唯一且完全匹配（包括缩进），否则调用失败

📝 Write（全量覆写）

创建新文件或完全重写。必须 Read 过才能覆写已有文件。适合新建文件或大范围重构。

保持代码风格一致是关键——match the comment density, naming, and idiom

🔧 Bash（命令执行）

运行 lint、测试、安装依赖等。可以修改文件（如 sed），但现代 Agent 更倾向用 Edit/Write 保证精确性。

git 操作也通过 Bash——但 push 通常需要用户确认

🔒 安全机制危险操作需要用户确认——删除文件、force push、修改配置等
🎨 风格匹配好 Agent 会匹配项目现有的代码风格——注释密度、命名惯例、缩进方式
📐 多文件协同一个任务可能改 5 个文件——Agent 按依赖顺序逐个修改

✅ 环节六：验证与闭环

"写完就跑"的 Agent 是废的。
真正靠谱的 Coding Agent 会在改完代码后自己检查一遍——甚至错了自己改。
✏️修改代码
→
👀自我审视
→
🧪Lint / 测试
→
❌发现错误？
→
🔁重新修改

这是一个自动循环——直到验证通过，或者 Agent 确认无法自动修复，才会向用户求助

🟢 闭环成功时

  • Edit 操作全部成功（无匹配失败）
  • Lint 检查通过，无新增错误
  • 关联测试全部绿灯
  • 代码风格与项目一致
  • 向用户清晰汇报改动内容

🔴 闭环失败时

  • old_string 匹配失败 → 重新 Read → 重新 Edit
  • Lint 报错 → 分析错误 → 修正代码
  • 测试挂了 → 分析失败原因 → 调整实现
  • 超出能力范围 → 诚实告知用户
  • 超过最大重试次数 → 标记为需要人工介入

💰 Token 经济学：一次修改烧了多少 Token

每个环节都在消耗 Token。理解 Token 分配，你才知道怎么省钱、怎么提效。

系统 8%
上下文 25%
推理 15%
工具 12%
生成 30%
验证 10%

▲ 典型的一次"修改 3 个文件"任务的 Token 消耗分布（总计约 15K-40K tokens）

📊 输入 Token

主要由上下文组装决定。System Prompt ~2K、CLAUDE.md ~1-3K、对话历史 ~2-10K、文件内容 ~3-15K。

💡 精简 CLAUDE.md 和 Memory 能显著省钱

📤 输出 Token

模型思考 + 工具调用参数 + 代码内容。每次 Edit 的输出约 500-2000 tokens，Write 可能 3000+。

💡 用 Edit 代替 Write 更省 Token

🔄 缓存命中

System Prompt 和静态文件会被缓存。缓存命中时输入价格打 1 折。好的上下文设计能显著提高缓存命中率。

💡 稳定不变的头部内容放前面
