OpenSpec · 管"做什么"

结构化的项目规范管理系统。每一个功能变更都是一份完整的提案文档，包含需求、设计、任务清单和规格说明。

  • 📋 Proposal — 为什么要做、改什么
  • 🏗️ Design — 技术方案和架构决策
  • ✅ Tasks — 可执行的任务清单
  • 📝 Specs — 功能规格说明文档
🦸

Superpowers · 管"怎么做"

14 个Claude Code 技能组成的 AI 辅助开发方法论。覆盖从需求到交付的完整开发流程。

  • 🧠 Brainstorming — 想法 → 设计
  • 🧪 TDD — 先写测试再写代码
  • 🔍 Debugging — 系统化排查
  • 👀 Code Review — AI 审查代码

🦸 Superpowers 14 大技能

按使用顺序排列，覆盖从构思到交付的全流程：

🧠

brainstorming 头脑风暴

任何创意工作前必须先用的技能。通过 9 步流程把模糊想法变成完整设计：探索上下文 → 可视化 → 提问澄清 → 提2-3方案 → 展示设计 → 写设计文档 → 自审 → 用户审阅 → 转实施。

📝

writing-plans 编写计划

把设计文档转成可执行的实施计划。拆解任务、排优先级、预估工作量。复杂任务先出计划再动手，确认后 AI 按计划逐步执行。

🧪

test-driven-development TDD

铁律：先写失败的测试，再写代码让它通过。新功能、Bug修复、重构——全部适用。不写测试先写代码？删掉重来。

⚡

executing-plans 执行计划

按照已确认的计划逐步实施。每一步有明确的输入输出和验收标准，像流水线一样把计划变成代码。

🔍

systematic-debugging 系统化调试

bug 找不到原因时不要瞎猜。这个技能引导你走系统化排查流程：复现 → 隔离 → 定位根因 → 修复 → 验证。

👀

code-review 代码审查

写完代码让 AI 审一遍。检查逻辑漏洞、安全隐患、代码质量。分为请求审查和接收审查两个技能，双向视角。

✅

verification-before-completion 完成前验证

功能做完了 ≠ 可以交了。这个技能确保实现和设计文档一致，所有验收标准通过，没有遗漏的边界情况。

🚀

finishing-a-development-branch 收尾分支

功能开发完成后的标准收尾流程：检查是否有未提交的改动 → 决定合并还是提 PR → 清理开发痕迹。

🤖

subagent-driven-development 子智能体

派发多个 AI 子智能体并行执行任务，像微服务一样拆分工作，然后汇总结果。

🌳

using-git-worktrees 工作树

用 git worktree 并行开发多个功能，互不干扰。每个 worktree 独立工作区，切分支不用 stash。

🧠 高手技巧

🎯

原则一：先 Spec 后 Skill

先跑 /opsx:explore → propose 把需求写成规范文档，再用 Superpowers 技能按规范实施。规范是"合同"，技能是"施工队"。

🔗

原则二：命令链式调用

熟练后可以一口气串联：
/brainstorming → /opsx:propose → /opsx:apply → /code-review → /opsx:archive

📦

原则三：规范即文档

OpenSpec 的 specs/ 目录就是你的活文档。每次 /opsx:sync 把增量合并进主规范，项目文档永远不会过时。

🚫

原则四：不过度使用

修一个 typo、改一行 CSS 不需要走完整流程。小任务直接做，中大型功能才启动 OpenSpec + Superpowers 工作流。
