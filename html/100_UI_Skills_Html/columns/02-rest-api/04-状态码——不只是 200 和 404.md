# REST API 设计从0到1 第4篇：状态码：不只是 200 和 404
@[TOC](目录)
## 摘要
> HTTP 状态码是服务器和客户端之间的"暗号"。很多开发者只用 200 和 404 打天下，但标准状态码有 60 多种，选对状态码能让 API 更加专业。本文带你分类掌握最常用的状态码，并给出实战代码。
## 引言
你有没有调接口时遇到过这种情况：请求明明是失败的，后端却返回了 200 OK，然后你只能在响应体的 JSON 里查 `code !== 0` 来判断是否成功。这种"全部 200"的做法在中小公司非常普遍。

为什么不应该这样做？因为 HTTP 状态码是协议层的"元数据"，它和 JSON 里的业务状态码处在不同的层次。当你正确使用 201、400、401、403、404、500 这些状态码时，监控系统、负载均衡器、浏览器缓存、前端拦截器都能直接利用这些信息，不需要解析响应体。

这篇文章将帮你建立一个清晰的状态码分类体系，让你知道什么场景该返回什么状态码，以及如何在代码中正确实现。
## 基础知识储备
- **HTTP 响应结构**：知道响应由状态行（含状态码）、响应头、响应体组成。
- **五个状态码大类**：1xx 信息、2xx 成功、3xx 重定向、4xx 客户端错误、5xx 服务器错误。
- **JSON 响应格式基础**：能设计包含状态信息的 JSON 响应结构。
- **前端拦截器概念**：知道 axios/fetch 的拦截器可以根据状态码做统一处理，比如 401 自动跳转登录页。
- **REST 资源操作概念**：了解增删改查分别对应的典型响应格式。
## 正文
### 1. 状态码五大家族：一张表全览

HTTP 状态码按百位数字分为五类，每一类都有明确的语义：

| 大类 | 范围 | 含义 | 典型示例 |
|------|------|------|----------|
| 1xx | 100-199 | 信息提示 | 101 协议切换（WebSocket 升级） |
| 2xx | 200-299 | 成功 | 200 OK、201 Created、204 No Content |
| 3xx | 300-399 | 重定向 | 301 永久重定向、304 Not Modified |
| 4xx | 400-499 | 客户端错误 | 400 Bad Request、401 Unauthorized、403 Forbidden、404 Not Found、422 Unprocessable Entity |
| 5xx | 500-599 | 服务器错误 | 500 Internal Server Error、502 Bad Gateway、503 Service Unavailable |

对于 REST API 开发者来说，最应该掌握的是 2xx、4xx 和 5xx。1xx 通常由底层协议自动处理，3xx 主要用于网页跳转场景。

```javascript
// 前端拦截器中按状态码分类处理
fetch('/api/xxx')
  .then(res => {
    if (res.status === 401) {
      // 未登录，跳转登录页
      window.location.href = '/login';
      return;
    }
    if (res.status === 403) {
      // 没权限，提示用户
      alert('您没有权限执行此操作');
      return;
    }
    if (res.status === 404) {
      // 资源不存在
      console.warn('请求的资源不存在');
      return;
    }
    if (res.status >= 500) {
      // 服务器错误，统一提示
      alert('服务器开小差了，请稍后重试');
      return;
    }
    return res.json();
  });
```

### 2. 最常用的 8 个状态码详解

**200 OK**：请求成功。GET 获取到数据、PUT/PATCH 更新成功、DELETE 删除成功并返回内容时用。

**201 Created**：新建资源成功。POST 创建新数据时使用。最佳实践是在响应头中带上 `Location` 字段，指向新资源的 URL。

**204 No Content**：操作成功但没有需要返回的内容。常见于 DELETE 删除成功时，或者 PUT 更新后不需要返回更新后的数据。

**400 Bad Request**：请求格式错误。比如必填字段缺失、JSON 格式不正确、参数类型不对。这是最常见的客户端错误。

**401 Unauthorized**：未认证。用户没有登录或 token 过期。注意跟 403 的区别：401 是"你是谁我都不知道"，403 是"我知道你是谁，但你不能做这个操作"。

**403 Forbidden**：已认证但无权限。比如普通用户试图访问管理员接口。

**404 Not Found**：请求的资源不存在。注意这个"不存在"不只是数据没找到，也可以是"路由不存在"。

**422 Unprocessable Entity**：请求格式正确但语义有问题。比如创建用户时邮箱格式正确但已被注册。这个状态码在 REST API 中非常实用，很多开发者不知道它。

**500 Internal Server Error**：服务器内部错误。程序抛异常、数据库连接失败等。注意不要向客户端暴露错误堆栈信息。

```python
# Flask 中不同状态码的实现
from flask import Flask, request, jsonify

app = Flask(__name__)
users = {}

@app.route('/api/users', methods=['POST'])
def create_user():
    data = request.json
    # 400: 参数缺失
    if not data or 'email' not in data:
        return jsonify({"error": "email 字段必填"}), 400
    # 422: 邮箱已被注册（格式正确但业务逻辑不通过）
    if any(u.get('email') == data['email'] for u in users.values()):
        return jsonify({"error": "该邮箱已被注册"}), 422
    user_id = str(len(users) + 1)
    users[user_id] = {"id": user_id, "email": data['email'], "name": data.get('name', '')}
    # 201: 创建成功，带上 Location 头
    response = jsonify({"data": users[user_id]})
    response.status_code = 201
    response.headers['Location'] = f'/api/users/{user_id}'
    return response

@app.route('/api/users/<user_id>', methods=['GET'])
def get_user(user_id):
    user = users.get(user_id)
    if user:
        return jsonify({"data": user})  # 200 OK
    return jsonify({"error": "用户不存在"}), 404  # 404 Not Found

@app.route('/api/users/<user_id>', methods=['DELETE'])
def delete_user(user_id):
    if user_id in users:
        del users[user_id]
        return '', 204  # 204 No Content
    return jsonify({"error": "用户不存在"}), 404

if __name__ == '__main__':
    app.run(debug=True)
```

### 3. 为什么不应该"全部返回 200"

这是一种令人遗憾的实践：不管成功还是失败，HTTP 状态码都是 200，然后在响应体 JSON 里用一个 `code` 字段区分：

```json
// 反模式：全部返回 200
// 成功
{"code": 0, "message": "success", "data": {...}}
// 失败
{"code": 10001, "message": "参数错误", "data": null}
```

这种做法的三个问题：

第一，**破坏了 HTTP 协议的语义**。监控系统、负载均衡器看不到错误（它们只看状态码），所有请求都是 200，怎么判断服务是否健康？

第二，**前端多写一层判断**。正常只需 `if (res.ok)`，现在每次都要解析 JSON 再判断 `if (data.code === 0)`。

第三，**浏览器 DevTools 看不出问题**。在 Network 面板里所有请求都是绿色对勾，排查问题时少了关键线索。

正确做法：HTTP 状态码用于协议层的成功/失败，JSON body 里可以额外带业务错误码和详细信息。
## 总结
- HTTP 状态码分五类，REST API 开发者重点掌握 2xx、4xx、5xx。
- 201 用于创建成功、204 用于无返回内容的成功、400 用于参数错误、422 用于业务校验失败。
- 401 和 403 的区别：401 是"没登录"，403 是"没权限"。前端拦截器可以据此分别处理。
- 不要"全部返回 200"，这破坏了 HTTP 协议的语义，增加了前后端的隐性成本。
- 状态码 + JSON body 两层错误信息：状态码管协议层，body 管业务层，各司其职。
## 注意事项
- 5xx 错误时，响应体中不要暴露数据库的 stack trace 或服务器内部路径。这既是不安全的，也是不专业的。
- 302 重定向在 REST API 中要慎用——大多数 API 客户端不会自动跟随重定向。
- 注意 200 的滥用：POST 创建成功后应该返回 201 而不是 200，DELETE 成功后推荐返回 204。
- 状态码和语义要匹配：不要用 400 来处理认证问题（应该用 401），也不要用 500 来处理用户输入错误（应该用 400 或 422）。
## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 全部返回 200 | 监控面板永远全绿，出了问题无法自动告警 | 所有请求都是 200，包括错误 | HTTP 状态码如实反映请求结果，JSON body 补充业务细节 |
| 混淆 401 和 403 | 登录过期跳转和权限不足提示混为一谈 | 不清楚两者的认证层级区别 | 401 未认证(没登录)，403 已认证但无权限 |
| 500 暴露堆栈 | 用户看到满屏 Python/Java 报错信息 | 没做全局异常捕获和格式化 | 500 时只返回通用错误信息，详细日志记录在服务器日志文件 |
| POST 成功返回 200 | 创建成功后前端不知道去哪个 URL 获取新资源 | 没设置 Location 响应头 | 201 Created + Location 头指向新资源 URL |
