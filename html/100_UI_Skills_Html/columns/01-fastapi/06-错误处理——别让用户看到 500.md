# FastAPI 从0到1 第6篇：错误处理：别让用户看到 500

@[TOC](目录)

## 摘要
> 本文教你如何在 FastAPI 中优雅地处理各种错误，让用户看到友好的提示而不是冷冰冰的 500 Internal Server Error。适合已经写出接口但还没处理异常场景的开发者，学完你就能写出健壮的后端接口。

## 引言

你有没有遇到过这种体验——手机上点一个按钮，弹出来的是"网络错误，请稍后重试"，重试了八百次还是不行。这时候你会不会想：到底是哪里出错了？是我操作有问题还是服务器崩了？

前端的错误处理通常就是 `try-catch` 包住 `fetch`，然后 `alert('请求失败')`。但问题是——这个错误信息是从后端返回的，如果后端只返回一个 500 状态码，前端只能干瞪眼。好的错误处理需要后端先给出清晰的错误信息，前端才能准确地展示给用户。

这篇文章教你如何在 FastAPI 中分层处理错误：从单个接口的 `HTTPException`，到全局异常捕获，再到自定义异常体系。让你的 API 即使出错了，也是有尊严地出错。

## 基础知识储备

- **HTTP 状态码**：200（成功）、400（客户端错误，如参数不对）、404（找不到）、422（数据校验失败）、500（服务器内部错误）。
- **Python 异常处理**：`try-except-finally` 机制，`raise` 抛出异常，`except` 捕获异常。
- **前端错误处理对比**：React 的 Error Boundary 捕获渲染错误，和后端全局异常处理器的思路完全一样。
- **Fail Fast 原则**：尽早发现问题、尽早报错，不要掩盖错误让它到后面炸得更惨。

## 正文

### 基础款：HTTPException

```python
from fastapi import HTTPException

@app.get("/api/users/{user_id}")
def get_user(user_id: int):
    if user_id <= 0:
        raise HTTPException(status_code=400, detail="用户 ID 必须为正整数")
    
    user = find_user_by_id(user_id)
    if not user:
        raise HTTPException(status_code=404, detail=f"用户 {user_id} 不存在")
    
    return user
```

前端收到 400 或 404 时，可以用 `response.status` 判断状态码，用 `response.json()` 拿到 `detail` 字段里的错误信息：

```javascript
const res = await fetch(`/api/users/${id}`);
if (!res.ok) {
    const error = await res.json();
    alert(error.detail); // "用户 123 不存在"
}
```

这就是前后端协同错误处理的基础——状态码告诉前端"是什么类型的错误"，detail 告诉用户"具体出了什么问题"。

### 进阶款：自定义异常类

```python
class UserNotFoundException(HTTPException):
    def __init__(self, user_id: int):
        super().__init__(
            status_code=404,
            detail=f"用户 {user_id} 不存在"
        )

class PermissionDeniedException(HTTPException):
    def __init__(self, message: str = "你没有权限执行此操作"):
        super().__init__(status_code=403, detail=message)
```

使用起来非常干净：

```python
@app.get("/api/users/{user_id}")
def get_user(user_id: int, current_user = Depends(get_current_user)):
    user = find_user_by_id(user_id)
    if not user:
        raise UserNotFoundException(user_id)  # 清晰表达意图
    if user.id != current_user.id:
        raise PermissionDeniedException()
    return user
```

自定义异常类的最大好处是**语义化**——`raise UserNotFoundException(123)` 比 `raise HTTPException(status_code=404, detail="用户 123 不存在")` 更清晰地表达了"是什么问题"。而且如果你要统一修改 404 错误的格式，只要改 `UserNotFoundException` 一个地方就行。

### 终极大招：全局异常处理器

```python
from fastapi import Request, HTTPException
from fastapi.responses import JSONResponse

@app.exception_handler(HTTPException)
async def http_exception_handler(request: Request, exc: HTTPException):
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "success": False,
            "error": {
                "code": exc.status_code,
                "message": exc.detail,
            }
        }
    )

@app.exception_handler(Exception)
async def unexpected_exception_handler(request: Request, exc: Exception):
    # 记录日志，方便排查
    import logging
    logging.error(f"未捕获的异常: {type(exc).__name__}: {exc}")
    
    return JSONResponse(
        status_code=500,
        content={
            "success": False,
            "error": {
                "code": 500,
                "message": "服务器内部错误，请稍后重试",
                # 线上环境不要暴露 traceback！
            }
        }
    )
```

全局异常处理器的价值：
1. **统一错误格式**：每个错误都返回 `{ success: false, error: { code, message } }`，前端解析起来不用判断多种格式。
2. **兜底**：任何你没想到的异常，都会被 `Exception` 处理器捕获，返回 500 而不是直接崩掉。
3. **日志记录**：在全局处理器里统一记日志，不用在每个接口里写 `logger.error(...)`。

这个模式和 React Error Boundary 的用法几乎一样——在顶层设置一个边界，捕获所有子组件（子路由）抛出的错误，统一展示。

### 前端协作的最佳实践

前后端约定好统一错误格式以后，前端可以封装一个通用的 fetch 函数：

```javascript
async function apiFetch(url, options = {}) {
    const res = await fetch(url, options);
    const data = await res.json();
    
    if (!data.success) {
        // 统一错误处理
        if (data.error.code === 401) {
            // 跳转登录页
            window.location.href = '/login';
        }
        throw new Error(data.error.message);
    }
    
    return data;
}
```

这样前端的错误处理逻辑集中在一处，不用在每个组件里重复写判断逻辑。

## 总结

1. `HTTPException` 是 FastAPI 内置的最简单错误返回方式，一行代码搞定状态码和错误信息。
2. 自定义异常类让错误抛出的语义更清晰，方便统一管理和修改。
3. 全局异常处理器是生产环境必备——统一错误格式、兜底未知异常、集中记日志，三步到位。
4. 前后端约定好统一的错误格式（`{ success, error: { code, message } }`），可以大幅简化前端的错误处理逻辑。
5. 线上环境一定不要把 Python 的 traceback 暴露给客户端——那是安全漏洞。

## 注意事项

1. `HTTPException` 只能被 `@app.exception_handler(HTTPException)` 捕获到。如果你自定义了异常但没有继承 `HTTPException`，它会走到 `Exception` 处理器。
2. 全局异常处理器的注册顺序有讲究：先注册具体的（`HTTPException`），再注册通用的（`Exception`），和 `try-except` 一样。
3. 在全局处理器里不要又 `raise` 一个异常——那会导致递归，最后服务崩溃。

## 坑点

| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 异常被静默吞掉 | 请求返回 200 但数据不对 | `try-except` 里写了 `pass` | 至少记一条日志再继续，不知道怎么办就 `raise` 让全局处理器兜底 |
| 生产环境暴露 traceback | 用户看到了 Python 的堆栈信息 | 全局 Exception 处理器里直接输出了 `str(exc)` | 500 错误统一返回"服务器内部错误"，详细错误只记日志 |
| 校验错误格式不一致 | Pydantic 校验失败返回的格式和自己写的错误格式不同 | FastAPI 自带的 422 校验错误有独立的格式 | 也用全局异常处理器统一 `RequestValidationError` 的格式 |
| 异步函数里异常没被捕获 | 全局处理器没生效，返回原始 500 | 全局处理器对同步和异步要分别注册 | 如果你的路由都是 `async def`，处理器也要用 `async def` |
