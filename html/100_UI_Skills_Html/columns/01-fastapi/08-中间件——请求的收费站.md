# FastAPI 从0到1 第8篇：中间件：请求的收费站

@[TOC](目录)

## 摘要
> 本文带你掌握 FastAPI 中间件的原理和实战用法，实现请求日志记录、跨域处理、性能监控等通用功能。适合想进一步提升后端工程能力的开发者，学完你就能写出解耦且可复用的横切关注点逻辑。

## 引言

想象一下：你有一条高速公路，每个请求就是一辆车。中间件就是高速上的收费站——每辆车都得经过，检查有没有交费（鉴权），记一下车牌（日志），看看是不是超宽了（请求体大小校验）。过了收费站才能到达真正的目的地（路由函数）。

前端同学最熟悉的中间件应该就是 `cors` 了——本地开发时 Vite 的 proxy 本质上也是一种中间件，把所有 `/api` 请求转发到后端。Express.js 里到处是中间件：`app.use(cors())`、`app.use(express.json())`。FastAPI 的中间件机制和 Express 非常像，掌握了就是一个思维迁移。

这篇文章从最实用的日志中间件讲起，到 CORS 配置、请求耗时统计，最后手写一个简单的限流中间件。这些都是任何一个上线项目必须用到的东西。

## 基础知识储备

- **洋葱模型**：请求从外层中间件进入，经过所有中间件到达路由，响应再从路由原路返回，经过所有中间件。就像剥洋葱，一层一层进去，一层一层出来。
- **ASGI 中间件**：FastAPI 基于 Starlette（一个 ASGI 框架），它的中间件是 ASGI 协议层面的，能拦截所有请求，包括 WebSocket。
- **Express 对比**：Express 中间件用 `(req, res, next)` 三个参数，FastAPI 中间件用 `(request, call_next)`。
- **横切关注点**：和核心业务无关但每个请求都需要的东西——日志、认证、压缩、CORS——放在中间件里统一处理最合适。

## 正文

### 最实用的中间件：请求日志

```python
import time
import logging
from fastapi import FastAPI, Request

app = FastAPI()
logger = logging.getLogger("api")

@app.middleware("http")
async def log_requests(request: Request, call_next):
    # 请求进来时
    start_time = time.time()
    logger.info(f"--> {request.method} {request.url.path}")
    
    # 执行真正的路由处理
    response = await call_next(request)
    
    # 响应返回时
    duration = time.time() - start_time
    logger.info(
        f"<-- {request.method} {request.url.path} "
        f"{response.status_code} {duration:.3f}s"
    )
    
    return response
```

每次请求都会在控制台看到这样的输出：

```
--> GET /api/users
<-- GET /api/users 200 0.023s
--> POST /api/users
<-- POST /api/users 201 0.156s
```

这个日志中间件是最实用的——开发时帮你追踪每个请求，生产环境帮你排查性能瓶颈。哪个接口慢了，一目了然。

### CORS 中间件：前后端联调的救星

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173", "https://你的域名.com"],
    allow_credentials=True,
    allow_methods=["*"],     # 允许所有 HTTP 方法
    allow_headers=["*"],     # 允许所有请求头
)
```

前端同学一定见过这个红色报错：

```
Access to fetch at 'http://localhost:8000/api/users' from origin 
'http://localhost:5173' has been blocked by CORS policy
```

这就是浏览器的同源策略——前端和后端跑在不同的端口上，浏览器默认拦截。CORS 中间件就是告诉浏览器："我允许这个前端域名访问我"。开发环境通常用 `allow_origins=["*"]`（允许所有来源），但生产环境一定要写成具体的域名。

### 请求耗时监控：给慢请求打标记

```python
@app.middleware("http")
async def slow_request_alert(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    duration = time.time() - start
    
    # 超过 1 秒的请求标记为慢请求
    if duration > 1.0:
        logger.warning(
            f"慢请求！{request.method} {request.url.path} "
            f"耗时 {duration:.2f}s"
        )
    
    # 在响应头里加耗时信息，前端也能看到
    response.headers["X-Process-Time"] = f"{duration:.3f}s"
    return response
```

前端的 `Network` 面板里就能看到每个请求的 `X-Process-Time` 响应头，不用看后端日志也能知道后端花了多久。

### 手写一个简单限流

```python
from collections import defaultdict
from datetime import datetime, timedelta

# 简陋但有效的内存限流器（生产环境请用 Redis）
rate_limit_store: dict[str, list[datetime]] = defaultdict(list)

@app.middleware("http")
async def rate_limit(request: Request, call_next):
    client_ip = request.client.host
    now = datetime.now()
    window = now - timedelta(seconds=10)  # 10 秒窗口
    
    # 清理过期记录
    rate_limit_store[client_ip] = [
        t for t in rate_limit_store[client_ip] if t > window
    ]
    
    if len(rate_limit_store[client_ip]) >= 5:
        from fastapi.responses import JSONResponse
        return JSONResponse(
            status_code=429,
            content={"success": False, "error": {"code": 429, "message": "请求太频繁，请稍后重试"}}
        )
    
    rate_limit_store[client_ip].append(now)
    return await call_next(request)
```

这个限流中间件的意思是：同一个 IP 在 10 秒内最多发 5 个请求，超出就返回 429 Too Many Requests。虽然简陋（服务重启数据就没了），但思路很清晰——给每个 IP 维护一个时间戳列表，只统计最近 10 秒内的请求。

生产环境应该换成 Redis 实现，这样多个服务实例之间共享计数，而且 Redis 有现成的滑动窗口方案。

### 中间件的执行顺序

```python
# 注册顺序 = 执行顺序（先进后出，像洋葱）
app.add_middleware(CORSMiddleware)    # 最外层
app.add_middleware(RateLimit)         # 中间层
# @app.middleware("http")             # 最内层（默认最后）
```

洋葱模型：请求先经过 CORS -> 限流 -> 你的日志中间件 -> 路由函数。响应原路返回：路由函数 -> 日志中间件 -> 限流 -> CORS。这就是为什么中间件的 `call_next` 之前是"请求阶段"，之后是"响应阶段"。

## 总结

1. 中间件是处理请求和响应的管道，所有请求都会经过它，适合放日志、CORS、鉴权、限流等横切逻辑。
2. FastAPI 的 `@app.middleware("http")` 装饰器定义一个中间件函数，接收 `request` 和 `call_next`。
3. CORS 中间件是开发环境必装的，让前端能跨域访问后端 API。生产环境要指定具体域名，别用通配符。
4. 请求耗时监控中间件帮你发现性能瓶颈，`X-Process-Time` 响应头让前端也能看到每个请求的耗时。
5. 中间件执行顺序遵循洋葱模型，注册早的在外面，越晚注册的越接近路由。

## 注意事项

1. 中间件里的 `await call_next(request)` 是必须调用的——不调用就等于把请求吞了，路由函数永远不会执行。
2. 中间件是同步阻塞的——如果你在中间件里做了耗时的同步操作（比如读大文件），整个请求都会被阻塞。能异步的尽量异步。
3. CORS 的 `allow_origins=["*"]` 不能和 `allow_credentials=True` 同时使用，这是浏览器的安全策略。要用 cookies 就必须指定具体域名。

## 坑点

| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| CORS 配了但前端还是报错 | 跨域报错依旧 | 配置中间件后没重启服务 | 中间件只影响新请求，改完配置必须重启 |
| 中间件改了响应体 | 后续读取 body 失败 | 响应体是流，只能读一次 | 别在中间件里读响应体，要改响应用 `response.body` 或创建新 Response |
| 路由函数抛异常后中间件没执行 | 异常导致中间件的 call_next 后面的代码被跳过 | 响应阶段的代码在 call_next 之后，异常会中断它 | 用 `try-except` 包住 `call_next` |
| 打印不到 request body | `await request.body()` 返回空 | FastAPI 的 body 只能读一次，路由函数里读过了 | 中间件里不要读 request body，用日志中间件只记录 method 和 path |
