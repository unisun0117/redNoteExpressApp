@[TOC](目录)

## 摘要
> 本文教你如何使用 FastAPI 内置的 TestClient 为接口编写自动化测试，覆盖基本请求、数据库隔离、认证模拟等场景。适合已经写了多个接口但还没写测试的开发者，学完你就能为每个接口配上可靠的测试用例。

## 引言

你有没有过这种体验——改了一行代码，觉得"这么小的改动应该没问题吧"，上线后用户反馈说某个功能崩了。这就是没有测试的代价。每一次改动都是一次赌博，赌自己没有破坏已有的功能。

前端同学可能接触过 Jest 或 Vitest，写组件测试、快照测试。后端的测试思路完全一样——给一个输入，验证输出是否符合预期。只不过输入是 HTTP 请求（GET、POST 等），输出是 HTTP 响应（状态码、JSON 数据）。

FastAPI 内置了基于 `httpx` 的 `TestClient`，写 API 测试只需要三步：创建测试客户端、发请求、断言响应。这篇文章用一个完整的例子带你走通整个流程，让你以后每次改代码都底气十足。

## 基础知识储备

- **测试金字塔**：单元测试（最底层，最多份）→ 集成测试 → E2E 测试（最顶层，最少份）。本文讲的是集成测试——测试整个 API 请求链路。
- **pytest**：Python 最流行的测试框架，测试文件以 `test_` 开头，测试函数也以 `test_` 开头。`assert` 断言和 Jest 的 `expect()` 是同一回事。
- **测试数据库**：测试时不能用生产数据库——每次测试要独立隔离，互不干扰。通常用 SQLite 内存数据库或临时文件。
- **arrange-act-assert**：测试的黄金模式——准备数据（arrange）、执行操作（act）、验证结果（assert）。

## 正文

### 安装和基本用法

```bash
pip install pytest httpx
```

FastAPI 的 TestClient 来自 Starlette，不需要额外安装。

```python
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)

def test_root():
    response = client.get("/")
    assert response.status_code == 200
    assert response.json() == {"message": "Hello, FastAPI!"}
```

在终端运行：`pytest -v`，就能看到测试结果。是不是和前端用 `npm test` 差不多？一个 `test_` 文件几个 `test_` 函数，框架自动发现和执行。

### 测试 POST 接口

```python
def test_create_user():
    response = client.post(
        "/api/users/",
        json={
            "name": "测试用户",
            "email": "test@example.com",
            "age": 25,
        }
    )
    assert response.status_code == 201
    data = response.json()
    assert data["name"] == "测试用户"
    assert data["id"] is not None  # 数据库应该生成了 ID
```

然后写一个测试"缺少必填字段"的用例——测试异常场景和测试正常场景一样重要：

```python
def test_create_user_missing_name():
    response = client.post(
        "/api/users/",
        json={"email": "test@example.com", "age": 25}  # 故意缺少 name
    )
    assert response.status_code == 422  # Pydantic 自动触发校验错误
```

### 数据库隔离：让测试不影响真实数据

```python
import pytest
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker
from app.database import Base, get_db
from app.main import app

# 测试用数据库（SQLite 内存模式，测试结束就消失）
TEST_DATABASE_URL = "sqlite:///./test.db"

engine = create_engine(TEST_DATABASE_URL)
TestingSessionLocal = sessionmaker(bind=engine)

# 覆盖依赖注入，让路由用测试数据库
def override_get_db():
    db = TestingSessionLocal()
    try:
        yield db
    finally:
        db.close()

app.dependency_overrides[get_db] = override_get_db

@pytest.fixture(autouse=True)
def setup_database():
    # 每个测试前创建表
    Base.metadata.create_all(bind=engine)
    yield
    # 每个测试后清理数据
    Base.metadata.drop_all(bind=engine)

def test_create_and_get_user():
    # 创建用户
    create_res = client.post("/api/users/", json={"name": "小明", "email": "xm@test.com", "age": 20})
    assert create_res.status_code == 201
    user_id = create_res.json()["id"]
    
    # 用 GET 验证用户确实创建成功了
    get_res = client.get(f"/api/users/{user_id}")
    assert get_res.status_code == 200
    assert get_res.json()["name"] == "小明"
```

这段代码里最关键的概念是 `dependency_overrides`——FastAPI 允许你替换掉任意的 `Depends()` 函数。测试时把 `get_db` 替换成测试数据库的版本，所有接口就会自动连接测试数据库，完全隔离。

这和前端用 Mock Service Worker (MSW) 拦截 fetch 请求返回假数据的思路是一样的——不改变业务代码，只替换底层依赖。

### 带认证的测试

```python
def test_protected_endpoint():
    # 没带 token → 401
    response = client.get("/api/me")
    assert response.status_code == 401
    
    # 带了有效 token → 200
    response = client.get(
        "/api/me",
        headers={"Authorization": "Bearer fake-jwt-token"}
    )
    # 注：需要你的认证依赖允许测试 token
    
    # 带了过期/无效 token → 401
    response = client.get(
        "/api/me",
        headers={"Authorization": "Bearer invalid-token"}
    )
    assert response.status_code == 401
```

如果你用了复杂的 JWT 认证，建议在测试环境提供一个"一键创建带 token 的用户"的 fixture：

```python
@pytest.fixture
def auth_headers():
    # 注册并登录，拿到 token
    res = client.post("/api/auth/register", json={...})
    token = res.json()["access_token"]
    return {"Authorization": f"Bearer {token}"}

def test_me(auth_headers):
    # 直接用 fixture 拿到的 token
    response = client.get("/api/me", headers=auth_headers)
    assert response.status_code == 200
```

## 总结

1. FastAPI 内置 `TestClient`，基于 `httpx`，写测试就是 `client.get()` / `client.post()` 然后 `assert`，语法极其简洁。
2. `app.dependency_overrides` 是测试神器——把真实数据库替换成测试数据库，完全隔离，测完就扔。
3. 测试要覆盖正常场景和异常场景——至少测"应该返回 200"、"应该返回 422"、"应该返回 404"三种情况。
4. `pytest.fixture` 用来复用测试前置逻辑，比如数据库初始化、获取认证 token、创建测试数据。
5. 接口多了以后，可以给每个 router 配一个对应的 test 文件：`routers/users.py` -> `tests/test_users.py`。

## 注意事项

1. TestClient 在同一个进程中运行，不需要真的启动服务器。不要用 `requests` 库去请求 `localhost:8000`——那测的是运行中的服务，不是代码本身。
2. 测试文件放在 `tests/` 目录下，和 `app/` 同级。pytest 默认会扫描 `tests/` 目录下的所有 `test_*.py` 文件。
3. `dependency_overrides` 是模块级的全局状态。如果多个测试文件用了不同的 override 配置，要注意相互影响的问题。

## 坑点

| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 测试用的生产数据库 | 测试跑完后，线上用户数据被清空了 | 数据库配置没覆盖，直接用了生产库 | 必须用 `dependency_overrides` 把数据库换成测试库或内存库 |
| 测试间数据污染 | 测试 A 创建的用户影响了测试 B 的结果 | 没有在测试间清理数据 | 用 `pytest.fixture(autouse=True)` 在每个测试后 `drop_all` |
| TestClient 不能测试文件上传 | `UploadFile` 参数传不进去 | TestClient 对文件上传的写法不同 | `client.post("/upload/", files={"file": ("name.txt", b"content", "text/plain")})` |
| 测试通过了但实际环境报错 | 测试用的 SQLite，生产用的 PostgreSQL，SQL 不兼容 | SQLite 和 PostgreSQL 的 SQL 语法有差异 | CI 里增加一个 PostgreSQL 环境跑一遍相同的测试 |
