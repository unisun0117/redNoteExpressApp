# 用户认证从 0 到 1（JWT/OAuth）— 10 篇专栏大纲

> 面向人群：会写登录页面但不知道 token 怎么生成的开发者。每篇有原理 + 代码 + 安全提醒。

## 第 1 篇：登录到底发生了什么？认证 vs 授权

**核心内容：**
- 你点"登录"按钮之后，前端和后端分别干了什么
- 认证（Authentication）：你是谁？
- 授权（Authorization）：你能干什么？
- 为什么 HTTP 是无状态的？每次请求怎么知道是你
- Cookie、Session、Token 三兄弟的江湖故事

**前端类比：**
- 认证 = 进公司大门刷工卡（验证身份）
- 授权 = 工卡只能进你自己部门（权限控制）
- HTTP 无状态 = 每次进大门都要重新刷卡，服务器不认识你
- Token = 给你发了一张临时通行证，有效期内不用反复验证

**【注意】** "认证"和"授权"是两个不同概念，面试经常被问："请解释 Authentication 和 Authorization 的区别"。一句话：AuthN 验证你是谁，AuthZ 决定你能干什么。

**Lab 1：** 打开任意网站的登录页面，F12 → Network，输入账号密码登录。观察：
- 登录请求发了什么数据？（username + password 在 Body 里？还是 Form Data？）
- 返回了什么？（有没有 token？set-cookie header？）
- 之后的请求带了什么？（Authorization header？Cookie？）
把整个流程画成一张时序图。

## 第 2 篇：密码怎么存？明文 = 自杀

**核心内容：**
- 为什么不能明文存密码？数据库泄露 = 全完蛋
- Hash 是什么：单向加密，不可逆（MD5 → SHA256 → bcrypt）
- 加盐（Salt）：同样的密码，不同的 hash
- bcrypt 为什么是行业标准：慢哈希 = 暴力破解成本极高
- 密码存储最佳实践

**前端类比：**
- Hash = `btoa()` 的单向不可逆版
- Salt = 给每个用户加一个随机前缀，`hash(salt + password)`
- bcrypt 的"慢" = 故意让它慢，用户登录多等 100ms 没感觉，但黑客爆破成本 ×1000 倍

**【注意】** MD5 和 SHA1/256 不适合存密码——它们设计目标就是"快"，而密码哈希需要"慢"。永远用 bcrypt 或 argon2。

**Lab 2：** 用 Python 的 `bcrypt` 库做以下实验：
- 将密码 "123456" hash 3 次，观察每次结果是否不同（因为每次 salt 随机）
- 用 `bcrypt.checkpw()` 验证密码正确性
- 对比 bcrypt 和 hashlib.sha256 的速度差异
- 写一个 `User` 类，`set_password()` 时存 hash，`check_password()` 时验证——密码原文绝不存储

## 第 3 篇：Session 认证——最"古老"但最稳的方案

**核心内容：**
- Session 认证流程：登录 → 服务器存 session → 返回 session_id → 浏览器 cookie
- Cookie 的属性：HttpOnly、Secure、SameSite、Max-Age
- Session 存哪里：内存、Redis、数据库
- Session 的优缺点：安全但需要服务器端存储

**前端类比：**
- Session = 你去健身房办卡，前台给你一个带编号的柜子钥匙
- Session ID = 柜子钥匙编号，真正的数据（你的衣服）在柜子里
- Cookie = 浏览器每次自动带上钥匙，你不用手动掏

**【注意】** Cookie 一定要设 `HttpOnly`（JS 读不到，防 XSS 偷 cookie）和 `Secure`（只在 HTTPS 下传输）。SameSite=Lax 可以防 CSRF。

**Lab 3：** 用 FastAPI 实现 Session 认证：
- 不依赖任何第三方包，只用 Python 字典做简易 SessionStore
- 用户登录 → 生成 session_id（uuid） → 存到字典 → 通过 Set-Cookie 返回
- 后续请求 → 从 Cookie 取 session_id → 查字典找到用户 → 返回数据
- 实现"退出登录"：从字典中删除 session
- 写一个 `/me` 接口，返回当前登录用户信息（未登录返回 401）

## 第 4 篇：JWT 入门——不需要服务端存储的认证

**核心内容：**
- JWT 是什么：JSON Web Token，三部分（Header.Payload.Signature）
- JWT 认证流程：登录 → 服务器签发 token → 前端存 token → 每次请求带 token
- JWT 的结构：Header（算法）、Payload（数据）、Signature（签名），用 `.` 分隔
- JWT 的优势：无状态，不需要服务端存储；跨域友好

**前端类比：**
- JWT = 给你一张"防伪身份证"，不需要每次都去数据库查你是谁
- Header = 身份证上的"居民身份证"字样（说明这是什么证）
- Payload = 身份证上的姓名、出生日期（公开信息）
- Signature = 身份证上的防伪水印（别人伪造不了）

**【注意】** JWT 的 Payload 是 **Base64 编码，不是加密**。任何人都能解码看到内容。不要把密码、信用卡号等敏感信息放在 Payload 里。

**Lab 4：** 手动"解剖"一个 JWT：
- 去 [jwt.io](https://jwt.io) 看 JWT 的在线解析
- 用 Python 的 `pyjwt` 库签发一个 token：
  ```python
  import jwt
  token = jwt.encode({"user_id": 42, "role": "admin"}, "secret-key", algorithm="HS256")
  print(token)
  ```
- 用 `base64` 库手动解码 Payload（注意加 padding）
- 试着修改 Payload 然后验证 → 观察签名验证如何失败
- 理解：JWT 的安全性完全依赖 Signature 部分

## 第 5 篇：JWT 实战——Access Token + Refresh Token 双 Token 方案

**核心内容：**
- 为什么需要两个 token？Access Token 短有效期 + Refresh Token 长有效期
- Access Token：15 分钟过期，用来调接口
- Refresh Token：7 天过期，用来换新的 Access Token
- 刷新流程：Access Token 过期 → 用 Refresh Token 换 → 都过期就重新登录
- 前端怎么"无感刷新" token

**前端类比：**
- Access Token = 电影票，2 小时内有效，过期作废
- Refresh Token = 会员卡，用会员卡换新的电影票，不用重新注册
- 无感刷新 = axios interceptor 检测 401 → 自动拿 Refresh Token 换新的 → 重发请求

**【小提示】** 前端实现无感刷新最容易犯的错：多个请求同时返回 401，触发多次 refresh。用 Promise 缓存解决——第一个请求去刷新，后续请求等同一个 Promise。

**Lab 5：** 实现完整的双 Token 认证系统：
- `POST /register`：注册用户，密码 bcrypt 加密存数据库
- `POST /login`：验证密码，返回 `access_token`（15 分钟）+ `refresh_token`（7 天）
- `POST /refresh`：用 refresh_token 换新的 access_token
- `GET /me`：需要有效的 access_token，返回用户信息
- 写一个前端页面测试完整流程，包括 token 过期后的自动刷新

## 第 6 篇：OAuth 2.0 概念——"用微信登录"到底做了什么

**核心内容：**
- 什么是 OAuth 2.0：授权框架，让第三方应用访问你的资源但不拿你的密码
- OAuth 的 4 种授权模式：授权码模式（最常用）、隐式模式、密码模式、客户端模式
- 角色介绍：Resource Owner（你）、Client（App）、Authorization Server（微信）、Resource Server（微信用户数据）
- 授权码流程：跳转 → 授权 → 拿 code → 换 token → 调接口

**前端类比：**
- OAuth = 你去酒店，不用在前台交身份证原件，而是给前台看一个临时的"电子身份证"
- 授权码模式 = 你先去公安局网站认证，拿到一个 code，再用 code 去酒店换房卡
- 为什么比直接给密码好 = 酒店拿不到你的身份证，只拿到一个临时的"已验证"标记

**【注意】** OAuth 是用来**授权**的，不是用来**认证**的。但实际开发中大家用 OAuth "顺便"做了认证（有微信 token = 登录成功）。OpenID Connect 才是 OAuth 基础上的认证标准。

**Lab 6：** 模拟 OAuth 2.0 授权码流程（不需要真的对接微信/GitHub）：
- 写一个"授权服务器"（FastAPI），提供 `/authorize` 和 `/token` 接口
- 写一个"客户端应用"（HTML 页面）
- 完整走一遍：用户点"GitHub 登录" → 跳授权页 → 确认授权 → 拿 code → 换 token → 拿到用户名
- 画一张流程图标注每一步发生了什么

## 第 7 篇：OAuth 实战——接入 GitHub/微信登录

**核心内容：**
- 以 GitHub OAuth 为例（免费、无需审核、5 分钟配好）
- 注册 OAuth App → 拿到 Client ID 和 Client Secret
- 前端：跳转 GitHub 授权页 → 回调拿到 code
- 后端：用 code 换 access_token → 用 access_token 获取用户信息
- 微信登录的特殊之处：需要服务号/开放平台、回调域名限制

**前端类比：**
- Client ID = 你 App 在 GitHub 的"身份证号"（公开的）
- Client Secret = 你 App 在 GitHub 的"密码"（绝密，只放后端）
- 回调 URL = GitHub 授权完要跳回哪个页面，类似于 `window.location.href`

**【注意】** Client Secret **绝对不能**出现在前端代码中。浏览器端拿不到也拿不到 GitHub 的 access_token——只能拿到授权码 code，后端用 code + secret 去换 token。

**Lab 7：** 实现 GitHub OAuth 登录：
- 注册 GitHub OAuth App（Settings → Developer settings → OAuth Apps）
- 写后端 `/auth/github/login`（重定向到 GitHub）和 `/auth/github/callback`（处理回调）
- 用户用 GitHub 登录后，后端自动创建或绑定本地用户账号
- 完整流程走通后，你可以在自己的 App 上看到"GitHub 登录"按钮了 🎉

## 第 8 篇：权限管理——RBAC 入门（谁可以干什么）

**核心内容：**
- RBAC：Role-Based Access Control，基于角色的权限控制
- 经典三层：User（用户）→ Role（角色）→ Permission（权限）
- 角色设计：admin、editor、viewer（不要为每个人都定制权限）
- 代码实现：FastAPI 的 Depends 做权限校验
- 权限粒度：粗粒度（整个接口）vs 细粒度（只能看自己的数据）

**前端类比：**
- RBAC = React 的条件渲染：`{user.role === 'admin' && <AdminPanel />}`
- 角色 = User Group，admin 自动拥有所有权限
- 权限检查 = Route Guard / Protected Route
- 细粒度权限 = "只能编辑自己的文章" = `if(post.authorId !== currentUser.id) return 403`

**【小提示】** 一开始设计角色时，先从 2-3 个角色起步（管理员、编辑、普通用户）。不要过度设计，等业务真的需要了再加。Facebook 一开始也只有一种角色。

**Lab 8：** 给你的项目加上 RBAC：
- 定义 3 种角色：`admin`、`editor`、`viewer`
- `admin`：可以增删改查所有资源
- `editor`：可以编辑和查看，但不能删除
- `viewer`：只能查看
- 实现一个 FastAPI 的 `Depends(require_role("admin"))` 权限装饰器
- 用不同角色登录测试：editor 尝试 DELETE → 返回 403

## 第 9 篇：安全防护——前 5 大 Web 安全漏洞

**核心内容：**
- XSS（跨站脚本攻击）：用户输入 `<script>alert('xss')</script>` 怎么办
- CSRF（跨站请求伪造）：别人伪造你的身份发请求
- SQL 注入：`' OR '1'='1` 登录绕过
- 暴力破解：限制登录尝试次数
- 敏感信息泄露：不要在前端代码里写密钥

**前端类比：**
- XSS = `dangerouslySetInnerHTML` 的风险
- CSRF = 别人用你的 fetch 发请求（因为 cookie 自动带上了）
- SQL 注入 = 前端表单不校验，后端直接拼 SQL
- 暴力破解 = `while(true) { tryLogin(i++) }`
- 密钥泄露 = 把 `.env` 文件 commit 到 GitHub

**【注意】** 安全不是一个功能，是一种习惯。你可以不是安全专家，但至少要知道这些攻击的原理和基本防御。

**Lab 9：** "攻击"你自己的应用（安全测试）：
- 在输入框输入 `<script>alert(1)</script>` → 看后端/前端是否防护
- 用错误的密码连续登录 20 次 → 看是否有登录限制
- 将 JWT token 修改 Payload 后请求 → 看签名验证是否拦截
- 检查前端代码和 GitHub 仓库 → 有没有密钥泄露
- 每发现一个问题，写出修复方案并实施

## 第 10 篇：认证选型指南 + 面试高频题

**核心内容：**
- 什么时候用 Session？什么时候用 JWT？什么时候用 OAuth？
  - 单体应用 → Session 就够了
  - 前后端分离 + API → JWT
  - 第三方登录 + 开放平台 → OAuth

  5. OAuth 授权码模式流程
  6. Cookie 和 Token 的区别？

  8. RBAC 怎么设计？
  9. 前端怎么安全存储 Token？
  10. 使用 HTTPS 为什么还不够？

**前端类比：**
- 选型 = 技术选型，没有银弹，看场景
