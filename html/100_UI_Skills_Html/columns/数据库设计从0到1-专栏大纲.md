# 数据库设计从 0 到 1 — 10 篇专栏大纲

> 面向人群：会写前端 CRUD 但数据库靠 ORM 糊弄过去的开发者。每篇有前端类比 + SQL 习题 + 真实场景。

---

## 第 1 篇：为什么要数据库？localStorage 不够用吗？

**核心内容：**
- 前端数据存哪？localStorage、IndexedDB、内存变量的局限
- 数据库解决什么问题：持久化、并发、查询、一致性
- 关系型 vs 非关系型：什么时候用 MySQL/PostgreSQL，什么时候用 MongoDB
- 一张"用户表"长什么样

**前端类比：**
- 数据库表 = 一个巨大的 JSON 数组 `[{id:1, name:"张三"}, {id:2, name:"李四"}]`
- 数据库 = 这个数组存在服务器硬盘上，不会因为关浏览器而消失
- 并发访问 = 100 个人同时往数组里 push 数据，数据库保证不乱

**【小提示】** 如果你写过后端，大概率用过 ORM 直接操作数据库。这个专栏让你知道 ORM 背后到底干了什么。

**Lab 1：** 在你的电脑上安装 SQLite（零配置，一个文件就是数据库）。创建一张 `students` 表（id、name、age、email），用命令行插入 3 条数据，查询全部出来。感受一下：这不就是一个持久化的 JSON 数组吗？

## 第 2 篇：表怎么建？字段类型和约束

**核心内容：**
- 常用字段类型：INTEGER、TEXT、REAL、BOOLEAN、DATETIME
- NOT NULL：这个字段必须填
- DEFAULT：不填就自动给默认值
- UNIQUE：不能重复（用户名、邮箱）
- PRIMARY KEY：每条数据的唯一身份证

**前端类比：**
- 字段类型 = TypeScript 的类型标注
- NOT NULL = `required` 属性
- DEFAULT = 表单的 `defaultValue`
- UNIQUE = 注册时提示"该用户名已被使用"
- PRIMARY KEY = `Array.find(item => item.id === xxx)` 的那个 `id`

**【注意】** 实际开发中最常见的 bug 之一：该加 UNIQUE 的字段没加，导致重复数据，后续清理火葬场。

**Lab 2：** 创建一张 `products` 商品表：
```sql
CREATE TABLE products (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  price REAL NOT NULL CHECK(price > 0),
  stock INTEGER DEFAULT 0,
  category TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(name)
);
```
试着插入两条同名商品 → 看 UNIQUE 约束如何报错。试着插入 price = -10 → 看 CHECK 如何拦截。

## 第 3 篇：关系——表与表怎么"牵手"

**核心内容：**
- 一对多：一个用户有多篇文章（最常见的关系）
- FOREIGN KEY：外键是什么，为什么要有
- 多对多：一个学生选多门课，一门课有多个学生（需要中间表）
- 一对一：一个用户有一个 profile（直接外键 + UNIQUE）

**前端类比：**
- 一对多 = 一个 `user` 对象里有个 `posts: Post[]` 数组
- FOREIGN KEY = `post.userId` 这个字段指向 `user.id`
- 多对多中间表 = 两个数组的交叉引用：`student_courses = [{studentId:1, courseId:3}, ...]`

**【小提示】** 设计表关系的时候，想想前端拿到数据要怎么展示。如果前端要展示一个"用户的文章列表"，那后端就该是一对多关系。

**Lab 3：** 创建三张表：`authors`（id、name）、`posts`（id、title、content、author_id FOREIGN KEY → authors.id）、`tags`（id、name）、`post_tags`（post_id、tag_id 中间表）。然后：
- 插入 2 个作者，每人 3 篇文章
- 给文章打标签
- 用 JOIN 查询：列出每篇文章的作者名和标签

## 第 4 篇：增删改查——SQL 入门必学

**核心内容：**
- INSERT：插入一条/多条数据
- SELECT：查询数据（最常用）
- UPDATE：修改已有数据（⚠ 别忘了 WHERE）
- DELETE：删除数据（⚠ 先 SELECT 确认再 DELETE）
- WHERE 条件：`=`、`>`、`<`、`!=`、`LIKE`、`IN`、`BETWEEN`

**前端类比：**
- INSERT = `array.push()`
- SELECT = `array.filter()` + `array.find()`
- UPDATE = `array[index] = newValue`
- DELETE = `array.splice(index, 1)`
- WHERE = `filter(item => item.age > 18)`

**【注意】** 生产环境执行 `UPDATE` 或 `DELETE` 之前，**一定**先跑一遍 `SELECT` 看要改哪些数据。`DELETE FROM users` 不加 WHERE = 全删，没有撤销键。

**Lab 4：** 给 Lab 3 的数据库做以下操作：
- 插入 5 个新标签
- 把作者"张三"的所有文章标题前加上"[精选]"
- 查询所有"包含标签 'Python' 且作者年龄大于 25"的文章
- 删除所有没有标签的文章（先 SELECT 确认，再 DELETE）
- 统计每个作者的文章数

## 第 5 篇：查得更快——索引入门

**核心内容：**
- 全表扫描 vs 索引查找：100 万条数据找一条要多慢？
- 索引原理：像书的目录，不读全书也能定位
- 什么时候加索引：WHERE/JOIN/ORDER BY 的字段
- 索引不是越多越好：写入变慢、占空间

**前端类比：**
- 索引 = 前端用 `Map` 做 O(1) 查找，不用 `Array.find` 的 O(n)
- 加了索引的查询 = `map.get(id)` 瞬间返回
- 不加索引的查询 = `array.find(item => item.id === xxx)` 遍历全部

**【小提示】** 一个简单判断：如果你经常 `WHERE email = 'xxx'`，那就给 email 加索引。用户登录接口大概率会用到。

**Lab 5：** 用 Python 脚本生成 10 万条用户数据插入 `users` 表。分别测试：
- 不加索引：`SELECT * FROM users WHERE email = 'user99999@test.com'` → 记录耗时
- 加索引：`CREATE INDEX idx_email ON users(email)` → 同样查询 → 记录耗时
对比两个耗时，感受"索引到底快了多少倍"。

## 第 6 篇：聚合查询——统计的魅力

**核心内容：**
- COUNT、SUM、AVG、MAX、MIN：五个聚合函数
- GROUP BY：按分类统计
- HAVING：筛选聚合后的结果
- ORDER BY + LIMIT：排序 + 分页

**前端类比：**
- COUNT = `array.length`
- SUM = `array.reduce((sum, item) => sum + item.amount, 0)`
- GROUP BY = `Object.groupBy(array, item => item.category)`
- ORDER BY + LIMIT = `array.sort().slice(0, 10)`

**【注意】** `WHERE` 和 `HAVING` 的区别：WHERE 在聚合前过滤行，HAVING 在聚合后过滤组。面试必考。

**Lab 6：** 创建一个 `orders` 表（id、user_id、amount、category、created_at）。插入 50 条随机订单数据。写出以下 SQL：
- 每月的总销售额（GROUP BY 月份）
- 每个分类的平均订单金额（AVG + GROUP BY）
- 找出总消费金额超过 ¥1000 的用户（HAVING）
- 最近 30 天的 Top 10 消费用户（ORDER BY + LIMIT）

## 第 7 篇：事务——要么全成功，要么全失败

**核心内容：**
- 什么是事务：一组操作，要么全部执行，要么全部回滚
- 经典场景：转账（A 扣钱 + B 加钱，两步必须同时成功）
- ACID 四特性：原子性、一致性、隔离性、持久性
- `BEGIN` / `COMMIT` / `ROLLBACK`

**前端类比：**
- 事务 = `Promise.all([a, b, c])`，一个失败全部 reject
- 转账 = `await Promise.all([deductA(100), creditB(100)])`
- ROLLBACK = `try/catch` 中的回滚操作

**【小提示】** 后端面试高频题："转账功能怎么实现？"—— 答案就是事务。A-100 和 B+100 放在同一个事务里。

**Lab 7：** 模拟银行转账：
```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```
然后写一个"会失败的转账"：A 扣 10000（余额不够），观察事务如何自动回滚。再写一个手动 `ROLLBACK` 的例子。

## 第 8 篇：表设计实战——从需求到数据库

- ER 图怎么画（用文字描述就够了）

**前端类比：**

## 第 9 篇：范式与反范式——什么时候该"冗余"

**核心内容：**
- 1NF、2NF、3NF：三大范式是什么（用人话）
- 范式的目的：消除数据冗余、避免更新异常
- 反范式：有时候故意冗余，为了查询更快
- 实际开发：大多数场景 3NF 就够了，别强迫症

**前端类比：**
- 范式化 = 你的 Redux store，数据只有一份，改了到处更新
- 反范式 = 缓存计算结果，用空间换时间
- 更新异常 = 改了 store 里的用户名，但某个组件里存了一份旧的

**【小提示】** 面试问范式，答出"3NF 就是每个字段都直接依赖主键"就够了。实际写代码时，别为了范式把一个简单的查询搞成 5 次 JOIN。

**Lab 9：** 给 Lab 8 的选课系统做以下分析：
- 检查每张表是否符合 3NF
- 找一处可能"查询太慢"的场景，故意加一个冗余字段（反范式）
- 说明加冗余字段的利弊：查询快了多少 vs 更新麻烦了哪些

## 第 10 篇：ORM 入门——写 Python 就能操作数据库

**核心内容：**
- ORM 是什么：Object Relational Mapping，把表映射为类
- SQLAlchemy ORM 入门：定义 Model、CRUD 操作
- ORM 的优缺点：方便 vs 性能损耗
- 原生 SQL vs ORM：什么时候该自己写 SQL

**前端类比：**
- ORM = JSX 编译成 DOM 操作，你写 `<div>`，React 帮你操作 DOM
- ORM 就是：你写 `User.query.filter_by(name="张三").first()`，ORM 帮你生成 SQL
- 需要手写 SQL 的场景 = 需要 `useRef` 直接操作 DOM 的场景

**【注意】** 会用 ORM 不代表懂数据库。调试慢查询、优化索引、理解 EXPLAIN——这些只能靠原生 SQL 知识。

**Lab 10：** 用 SQLAlchemy ORM 重写 Lab 8 的选课系统：
- 定义 ORM 模型（Student、Course、Teacher、Enrollment）
- 用 ORM 实现：学生选课、查询成绩、统计每个老师的学生数
- 打开 SQLAlchemy 的 SQL 日志，观察 ORM 到底生成了什么 SQL
- 找一个"ORM 生成的 SQL 不太合理"的场景（比如 N+1 查询问题）
