# 数据库设计从0到1 第9篇：范式与反范式——什么时候该冗余
@[TOC](目录)
## 摘要
> 范式教你减少冗余，反范式教你适当冗余。用前端DRY原则和性能缓存策略做类比，详解1NF到3NF的核心规则，以及什么场景下该故意打破范式。

## 引言

前端开发中有一个著名的DRY原则——Don't Repeat Yourself，不要重复代码。重复代码是万恶之源，改一处要同步改N处。数据库设计中也有类似的原则——**范式（Normalization）**，它的核心目标就是消除数据冗余。

但DRY是绝对真理吗？不一定。有些情况下，重复一小段代码（或数据）可能比抽象一个过于复杂的共享模块更清晰。React 自定义Hook就是一个例子——有时候直接在组件里写几行逻辑，比为了复用而引入一个复杂的Hook更易读。

数据库设计也一样。范式在大多数情况下是对的，但有时**故意增加冗余**可以换取巨大的查询性能提升。这个叫**反范式（Denormalization）**。

本文带你从1NF到3NF理解规范化的核心思想，然后告诉你什么时候可以"违规"——前端开发者对这类权衡应该特别有共鸣。

## 基础知识储备

1. **理解DRY原则**：知道为什么要避免重复代码，也知道过度抽象的代价。
2. **用过React/Vue的性能优化**：memo、useMemo、缓存策略——知道有时"冗余的计算结果"可以避免重复计算。
3. **掌握表设计和JOIN**：会建表、设外键、写JOIN查询。
4. **理解冗余的问题**：改了一个地方的数据，另一个地方没同步→数据不一致。

## 正文

### 一、第一范式（1NF）—— 列不可再分

**规则：每个字段必须是原子值，不能是数组、集合或嵌套结构。**

违反1NF的例子：

```sql
-- 错误：一个字段存了多个值
CREATE TABLE student_courses (
    student_id INTEGER,
    student_name TEXT,
    courses TEXT  -- 存的是 "数学,英语,物理" —— 逗号分隔，违反1NF！
);
```

符合1NF的设计：

```sql
-- 正确：每门课单独一条记录
CREATE TABLE student_courses (
    student_id INTEGER,
    student_name TEXT,
    course_name TEXT
);
-- student_id=1, "张三", "数学"
-- student_id=1, "张三", "英语"
-- student_id=1, "张三", "物理"
```

**前端类比：** React组件的Props不应该是`courses="数学,英语,物理"`这样的大杂烩，而应该是`courses={['数学', '英语', '物理']}`——数组的每个元素是独立可操作的。1NF说的就是把"打包数据"拆成"独立条目"。

### 二、第二范式（2NF）—— 消除部分依赖

**规则：在满足1NF的基础上，非主键列必须完全依赖于主键（不能只依赖于主键的一部分）。**

这条主要针对"复合主键"的场合。违反2NF的例子：

```sql
-- 错误：学生姓名只依赖于student_id，不依赖于整个主键(student_id, course_id)
CREATE TABLE enrollments (
    student_id INTEGER,
    course_id INTEGER,
    student_name TEXT,     -- 只依赖student_id！违反2NF
    course_name TEXT,      -- 只依赖course_id！违反2NF
    score REAL,
    PRIMARY KEY (student_id, course_id)
);
```

符合2NF的设计——拆成三张表：

```sql
CREATE TABLE students (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL
);

CREATE TABLE courses (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL
);

CREATE TABLE enrollments (
    student_id INTEGER NOT NULL,
    course_id INTEGER NOT NULL,
    score REAL,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (course_id) REFERENCES courses(id)
);
```

**前端类比：** 如果你的全局state是`{ user: {name, email}, cart: {items, total} }`，你不会把`cart.items`放进`user`对象里——它们各自独立。2NF就是让你把"只属于某个实体的数据"放在那个实体的表里，不要混。

### 三、第三范式（3NF）—— 消除传递依赖

**规则：在满足2NF的基础上，非主键列不能依赖于其他非主键列。**

违反3NF的例子：

```sql
-- 错误：city和province可以通过zip_code推断出来——传递依赖
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    zip_code TEXT NOT NULL,
    city TEXT NOT NULL,       -- 可以由zip_code决定
    province TEXT NOT NULL    -- 可以由zip_code决定
);
```

符合3NF的设计：

```sql
-- 拆成两张表
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    zip_code TEXT NOT NULL
);

CREATE TABLE postal_codes (
    code TEXT PRIMARY KEY,
    city TEXT NOT NULL,
    province TEXT NOT NULL
);

-- 查询时JOIN
SELECT u.name, p.city, p.province
FROM users u
JOIN postal_codes p ON u.zip_code = p.code;
```

**前端类比：** 你的应用状态里有`user.zipCode`就够了，不需要再存`user.city`和`user.province`——这些可以从邮编推导出来。如果邮编与城市的映射变了，3NF设计只需要改一条postal_codes记录，而反范式设计需要在几百个用户记录上全部更新。

### 四、反范式——什么时候该故意冗余？

范式很好，但有时矫枉过正。下面三种情况，考虑主动打破范式：

**场景一：历史快照 —— 订单要保留下单时的商品价格**

```sql
-- 订单表——故意冗余了price，因为商品价格会变
CREATE TABLE orders (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    product_name TEXT NOT NULL,   -- 冗余！但这是"下单时"的名字
    price REAL NOT NULL,          -- 冗余！但这是"下单时"的价格
    quantity INTEGER NOT NULL
);
```

如果不冗余，查订单时JOIN到products表——但商品可能已经改价甚至下架了。订单需要的是"历史快照"，此时冗余是必要的。

**场景二：高频查询 —— 用冗余换掉昂贵的JOIN**

```sql
-- 反范式：articles表直接存author_name和comment_count
CREATE TABLE articles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    author_id INTEGER NOT NULL,
    author_name TEXT NOT NULL,    -- 冗余！但省掉每次查文章列表时的JOIN
    comment_count INTEGER DEFAULT 0  -- 冗余！但省掉COUNT子查询
);
```

对于首页、列表页这种QPS高、延迟要求低的场景，适当的冗余可以大幅减少JOIN次数，提升性能。这和前端用`useMemo`缓存计算结果是一回事。

**场景三：复杂报表 —— 原始数据做ETL生成"宽表"**

数据仓库中的"大宽表"把几十个维度全拼在一张表里——严重违反范式，但查询报表时一条简单的SELECT就能出结果，不需要十几层JOIN。

```sql
-- 报表宽表（严重反范式，但查询极快）
CREATE TABLE report_daily_sales (
    date TEXT,
    category TEXT,
    product_name TEXT,
    customer_city TEXT,
    total_quantity INTEGER,
    total_amount REAL
    -- 所有维度全部拉平，不需要JOIN
);
```

### 五、范式和反范式的权衡法则

| 维度 | 范式化 | 反范式化 |
|------|--------|----------|
| 数据一致性 | 高（只在一个地方存） | 低（需同步多个副本） |
| 写入性能 | 好（只写一处） | 差（要写多处） |
| 查询性能 | 可能需要多表JOIN | 一般不需要JOIN |
| 存储空间 | 省 | 费 |
| 维护复杂度 | 改一处就行 | 改N处，容易遗漏 |

**实战原则：先范式化，再按需反范式化。**

你永远应该从规范化设计开始（至少满足3NF）。只有当实际运行的查询性能指标告诉你"这个地方慢了"，才考虑反范式化——而且必须在代码或文档中明确标注冗余字段的关系，让后来的维护者知道这些字段之间的同步规则。

## 总结

1. 1NF：列不可再分——不要把数组塞进一个字段。
2. 2NF：非主键列必须完全依赖于主键——不要把"只属于用户"的数据放在"选课记录"里。
3. 3NF：非主键列不能依赖于其他非主键列——不要通过邮编存城市名。
4. 反范式是"故意的冗余"——目的是提升查询性能或保留历史快照。
5. 设计原则：先规范化（至少3NF），再根据实际性能指标决定是否反范式化。

## 注意事项

1. 实际项目中，3NF基本够用。还有BCNF、4NF、5NF——它们处理的是更微妙的数据依赖，日常开发极少碰到，不用死记。
2. 反范式化后必须写文档——标注"author_name冗余自users.name，修改用户名时需要同步更新articles表"。没有文档的反范式就是定时炸弹。
3. 不要用"数据量小"为理由跳过范式化——SQLite里一张100行的表不范式化确实没问题，但今天100行、明天10万行，等你发现问题时已经来不及重构了。

## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 为了范式而范式，拆分过度 | 查一个用户列表要做5次JOIN，慢得离谱 | 过度范式化，把不该拆的也拆了 | 一个实体一张表，1:1关系的数据如果不常独立使用，就放一张表里 |
| 冗余字段忘记同步 | 用户改名后，他的文章里显示的仍是旧名字 | 反范式化的字段没有维护同步机制 | 用数据库触发器或应用层事件保证同步；或用物化视图（PostgreSQL支持） |
| 用逗号分隔代替关联表 | 标签字段存了"JS,Python,SQL"，查询困难 | 违反1NF，把数组塞进了单字段 | 建关联表(post_tags)，这是唯一正确的做法 |
| 把业务规则当数据冗余 | 订单存了`total = quantity * price` | 当price修改时，total不会自动更新 | 计算字段不要存，除非是"下单时的快照"；用视图或用SELECT时实时计算 |
