## 摘要
> 综合前7篇知识，实战设计一个"博客系统"的完整数据库。从需求分析到ER图、建表SQL、索引规划，手把手演示如何把业务需求转化为表结构。

## 引言

前面七篇文章，我们分别学了建表、字段类型、表关系、CRUD、索引、聚合、事务。每一篇都是数据库设计的一块拼图。现在该把这些拼图拼成完整画面了。

假设老板过来跟你说："做一个小型博客系统——用户可以注册登录、写文章发文章、其他人可以评论、用户之间可以互相关注。"你打开IDE，面对空白的SQL文件，第一步该做什么？

很多开发者一上来就写`CREATE TABLE`，这是最大的失误。正确的流程是：**需求分析 → 实体识别 → ER图 → 建表SQL → 索引规划**。中间的任何跳跃都会导致后续反复改表。

本文以一个真实可用的博客系统为例，带你走完从零到完整数据库设计的全流程。

## 基础知识储备

1. **掌握CREATE TABLE**：知道字段类型、约束、主键外键怎么写。
2. **理解三种表关系**：一对多、多对多——知道外键建在哪一端。
3. **用过博客或论坛系统**：对"用户-文章-评论"这种数据模型有直觉。
4. **了解索引的基本用法**：知道WHERE和JOIN的列上建索引。

## 正文

### 一、需求分析——先搞清楚要存什么

在动手建表之前，先列出系统的需求。我们用"用户故事"的方式梳理：

**用户故事（User Stories）：**
1. 用户可以注册（用户名、邮箱、密码、个人简介）
2. 用户可以发布文章（标题、正文、状态：草稿/已发布）
3. 用户可以对文章发表评论
4. 用户可以关注其他用户
5. 文章可以属于多个标签（如"JavaScript"、"数据库"）
6. 用户可以给文章点赞

从这些故事中提取**实体**（Entity）——实体就是需要建表的对象：

- **User**（用户）—— 注册、个人信息
- **Post**（文章）—— 内容、状态
- **Comment**（评论）—— 文章下的评论
- **Tag**（标签）—— 文章的分类标签
- **Follow**（关注）—— 用户之间的关系
- **Like**（点赞）—— 用户和文章的点赞关系

### 二、确定关系——画出ER图

在纸上或用工具画出实体之间的关系：

```
User (1) ──────< (多) Post      —— 一个用户写多篇文章
Post (1) ──────< (多) Comment  —— 一篇文章有多条评论
User (1) ──────< (多) Comment  —— 一个用户发多条评论
Post (多) >──────< (多) Tag    —— 文章和标签是多对多，需要中间表
User (多) >──────< (多) User   —— 关注是多对多（自引用），需要中间表
User (多) >──────< (多) Post   —— 点赞是多对多，需要中间表
```

在纸上画出这个ER图只需要5分钟，但能帮你避免后续80%的返工。

### 三、建表——把它翻译成SQL

现在把每一张表写成CREATE TABLE语句：

```sql
-- ======== 博客系统完整数据库设计 ========

-- 1. 用户表
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,
    email TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    bio TEXT DEFAULT '',
    avatar_url TEXT,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_users_email ON users(email);

-- 2. 文章表
CREATE TABLE posts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    title TEXT NOT NULL,
    content TEXT NOT NULL DEFAULT '',
    status TEXT NOT NULL DEFAULT 'draft'
        CHECK(status IN ('draft', 'published', 'archived')),
    published_at TEXT,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
CREATE INDEX idx_posts_user_id ON posts(user_id);
CREATE INDEX idx_posts_status ON posts(status);
CREATE INDEX idx_posts_published_at ON posts(published_at DESC);

-- 3. 评论表
CREATE TABLE comments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    post_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    content TEXT NOT NULL,
    parent_id INTEGER DEFAULT NULL,  -- 支持回复评论（嵌套）
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (post_id) REFERENCES posts(id) ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (parent_id) REFERENCES comments(id) ON DELETE CASCADE
);
CREATE INDEX idx_comments_post_id ON comments(post_id);

-- 4. 标签表
CREATE TABLE tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    created_at TEXT DEFAULT (datetime('now'))
);

-- 5. 文章-标签关联表（多对多中间表）
CREATE TABLE post_tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    post_id INTEGER NOT NULL,
    tag_id INTEGER NOT NULL,
    FOREIGN KEY (post_id) REFERENCES posts(id) ON DELETE CASCADE,
    FOREIGN KEY (tag_id) REFERENCES tags(id) ON DELETE CASCADE,
    UNIQUE(post_id, tag_id)  -- 避免重复打标签
);
CREATE INDEX idx_post_tags_post ON post_tags(post_id);
CREATE INDEX idx_post_tags_tag ON post_tags(tag_id);

-- 6. 关注表（用户对用户，多对多自引用）
CREATE TABLE follows (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    follower_id INTEGER NOT NULL,   -- 谁关注
    following_id INTEGER NOT NULL,  -- 被关注的人
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (follower_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (following_id) REFERENCES users(id) ON DELETE CASCADE,
    UNIQUE(follower_id, following_id)  -- 不能重复关注
);
CREATE INDEX idx_follows_follower ON follows(follower_id);
CREATE INDEX idx_follows_following ON follows(following_id);

-- 7. 点赞表（用户对文章，多对多）
CREATE TABLE likes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    post_id INTEGER NOT NULL,
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (post_id) REFERENCES posts(id) ON DELETE CASCADE,
    UNIQUE(user_id, post_id)  -- 每人每篇文章只能点赞一次
);
CREATE INDEX idx_likes_post ON likes(post_id);
```

### 四、索引规划——在哪建索引？

回顾一下设计中的索引决策思路：

| 索引 | 建在哪个列 | 原因 |
|------|-----------|------|
| `idx_users_email` | users.email | 登录时要查邮箱，频繁的WHERE条件 |
| `idx_posts_user_id` | posts.user_id | JOIN用户和文章时需要 |
| `idx_posts_status` | posts.status | 首页要过滤"仅显示已发布" |
| `idx_posts_published_at` | posts.published_at | 首页按发布时间倒序排列 |
| `idx_comments_post_id` | comments.post_id | 进入文章详情页加载所有评论 |
| `idx_post_tags_post` | post_tags.post_id | 查一篇文章有哪些标签 |
| `idx_post_tags_tag` | post_tags.tag_id | 查一个标签下有哪些文章 |
| `idx_follows_follower` | follows.follower_id | 查我关注了谁 |
| `idx_follows_following` | follows.following_id | 查谁关注了我 |
| `idx_likes_post` | likes.post_id | 查一篇文章有多少赞 |

每个索引都对应一个频繁的查询场景——不凭空建索引，也不漏掉高频查询。

### 五、常用查询的SQL模板

设计好表之后，写几个常用查询验证设计是否合理：

```sql
-- 博客首页：获取已发布文章列表（带作者名和评论数）
SELECT
    p.id, p.title, p.created_at,
    u.username AS author,
    COUNT(DISTINCT c.id) AS comment_count,
    COUNT(DISTINCT l.id) AS like_count
FROM posts p
JOIN users u ON p.user_id = u.id
LEFT JOIN comments c ON p.id = c.post_id
LEFT JOIN likes l ON p.id = l.post_id
WHERE p.status = 'published'
GROUP BY p.id
ORDER BY p.published_at DESC
LIMIT 20;

-- 文章详情页：查一篇文章+作者+标签+最近评论
SELECT p.*, u.username, u.avatar_url
FROM posts p
JOIN users u ON p.user_id = u.id
WHERE p.id = ?;

SELECT t.name
FROM tags t
JOIN post_tags pt ON t.id = pt.tag_id
WHERE pt.post_id = ?;

SELECT c.content, u.username, c.created_at
FROM comments c
JOIN users u ON c.user_id = u.id
WHERE c.post_id = ?
ORDER BY c.created_at DESC
LIMIT 50;

-- 用户关注列表
SELECT u.username, u.bio
FROM users u
JOIN follows f ON u.id = f.following_id
WHERE f.follower_id = ?;
```

## 总结

1. 表设计的正确流程：需求分析 → 实体提取 → ER图 → 建表SQL → 索引规划。不要跳步。
2. 多对多关系一定要中间表——文章和标签、用户关注、用户点赞，都遵循这个模式。
3. 索引跟着查询走——先确定有哪些查询场景，再根据场景建索引，不凭空建索引。
4. ON DELETE CASCADE让你删用户时自动清理关联数据——但要慎用，确认业务上确实需要级联删除。
5. 写查询验证设计——如果某个常见查询需要写三四个子查询，那表设计可能有问题。

## 注意事项

1. 不要过早优化——在建表阶段就"预留扩展字段"（field1, field2...）是技术债务。先按当前需求设计清楚，未来加字段用ALTER TABLE。
2. 命名规范很重要——统一用`snake_case`，表名用复数（users, posts），外键用`关联表单数_id`。在团队协作中，统一的命名规范比命名本身更重要。
3. 密码字段不要存明文——`password_hash`存的是哈希值，不是原始密码。这方面用bcrypt或argon2，不要自己实现哈希逻辑。

## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 跳过需求分析直接建表 | 写到一半发现表结构要大改 | 没理清实体和关系就动手，列越多偏离越远 | 花10分钟画ER图，磨刀不误砍柴工 |
| 所有字段都允许NULL | 查询结果里到处是None，前端各种判断 | 应该非空的字段没有NOT NULL约束 | 建表时想清楚：这个字段缺失时记录还有意义吗？ |
| 多对多关系不用中间表 | 文章表里用逗号分隔标签"JavaScript,Python" | 无法用SQL查"标签为Python的所有文章"——LIKE查询又慢又不准 | 多对多必须建中间表，不要用逗号分隔 |
| ON DELETE CASCADE误删数据 | 删了标签，关联的文章也被删了 | 文章-标签中间表设置了CASCADE，删除标签级联删了中间表记录（这其实是正确行为），但如果不小心对tags表也CASCADE了文章就是灾难 | 理解CASCADE的方向：只有外键所在的子表会被级联删除，主表不被影响 |
