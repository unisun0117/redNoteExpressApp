## 引言

前端开发中你肯定遇到过"部分失败"的噩梦场景。比如提交一个复杂的表单，包含了三个并发的API请求：创建订单、扣减库存、发送通知。如果第二个请求失败了，第一个请求已经创建的订单就成了"孤儿"数据。

聪明的你应该会想到"补偿"机制——失败了就回退：

```javascript
// 前端手动补偿 —— 脆弱又麻烦
async function submitOrder(data) {
  let orderId = null;
  try {
    const order = await api.createOrder(data);
    orderId = order.id;
    await api.deductStock(data.productId, data.quantity);  // 这步可能失败！
    await api.sendNotification(data.userId);
  } catch (error) {
    if (orderId) {
      await api.cancelOrder(orderId);  // 补偿：取消已创建的订单
    }
    throw error;
  }
}
```

这种手动补偿的方式不仅代码冗长，还有漏网之鱼的风险——万一补偿请求也失败了呢？

数据库的事务机制就是从根源上解决这个问题。它保证了一组操作"要么全部成功，要么全部失败"——没有中间状态，没有补偿代码。这就是事务的原子性（Atomicity）。

## 基础知识储备

1. **理解"原子操作"**：JavaScript的`Promise.all`要么全成功要么全失败，事务原理类似但更强。
2. **写过带补偿逻辑的前端代码**：知道try-catch-rollback的痛苦。
3. **熟悉SELECT和UPDATE**：事务是包裹这些操作的"容器"。
4. **理解并发的概念**：知道多个用户可以同时操作同一条数据（如同时抢库存）。

## 正文

### 一、事务的ACID四特性

事务有四个核心特性，缩写为ACID：

| 特性 | 英文 | 含义 | 前端类比 |
|------|------|------|----------|
| 原子性 | Atomicity | 要么全做，要么全不做 | `Promise.all`——一个reject全reject |
| 一致性 | Consistency | 数据从一种正确状态变到另一种正确状态 | 表单校验——提交前数据必须合法 |
| 隔离性 | Isolation | 多个事务互不干扰 | 两个Tab页操作自己的状态，互不影响 |
| 持久性 | Durability | 事务提交后数据永久保存 | localStorage——刷新后数据还在（但数据库更可靠） |

我们来写一个最经典的事务场景——银行转账：

```python
import sqlite3

def transfer_money(conn, from_account_id, to_account_id, amount):
    """转账——事务的经典案例"""
    cursor = conn.cursor()

    try:
        # 开启事务（SQLite默认自动开启）
        # 1. 检查余额
        cursor.execute(
            "SELECT balance FROM accounts WHERE id = ?",
            (from_account_id,)
        )
        row = cursor.fetchone()
        if row is None:
            raise ValueError(f"账户 {from_account_id} 不存在")
        if row[0] < amount:
            raise ValueError(f"余额不足！当前余额：{row[0]}，需要：{amount}")

        # 2. 扣款
        cursor.execute(
            "UPDATE accounts SET balance = balance - ? WHERE id = ?",
            (amount, from_account_id)
        )

        # 3. 加钱
        cursor.execute(
            "UPDATE accounts SET balance = balance + ? WHERE id = ?",
            (amount, to_account_id)
        )

        # 4. 记录流水
        cursor.execute(
            "INSERT INTO transfers (from_id, to_id, amount, created_at) "
            "VALUES (?, ?, ?, datetime('now'))",
            (from_account_id, to_account_id, amount)
        )

        # 5. 全部成功——提交事务
        conn.commit()
        print(f"转账成功：{amount}元 从 {from_account_id} → {to_account_id}")

    except Exception as e:
        # 6. 任何一步失败——回滚所有操作
        conn.rollback()
        print(f"转账失败，已回滚：{e}")
        raise

# --- 创建测试环境 ---
conn = sqlite3.connect('bank.db')
cursor = conn.cursor()

cursor.execute('''
    CREATE TABLE IF NOT EXISTS accounts (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        name TEXT NOT NULL,
        balance REAL NOT NULL CHECK(balance >= 0)
    )
''')
cursor.execute('''
    CREATE TABLE IF NOT EXISTS transfers (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        from_id INTEGER NOT NULL,
        to_id INTEGER NOT NULL,
        amount REAL NOT NULL,
        created_at TEXT
    )
''')

# 初始化测试账户
cursor.execute("DELETE FROM accounts")
cursor.execute("INSERT INTO accounts (name, balance) VALUES ('张三', 1000)")
cursor.execute("INSERT INTO accounts (name, balance) VALUES ('李四', 500)")
conn.commit()

# 测试：正常转账
transfer_money(conn, 1, 2, 300)

# 测试：余额不足（会自动回滚）
try:
    transfer_money(conn, 1, 2, 99999)
except ValueError:
    print("余额不足测试通过——事务已正确回滚")

# 验证数据一致性
cursor.execute("SELECT id, name, balance FROM accounts")
for row in cursor.fetchall():
    print(f"  {row[1]}的余额：{row[2]}")

conn.close()
```

### 二、并发问题——当多个人同时操作

事务的"隔离性"解决的是并发问题。多个用户同时操作数据库时，可能出现三种经典问题：

**脏读（Dirty Read）：** 事务A读到了事务B未提交的修改。如果B回滚了，A读到的就是"脏数据"——就像你看到了购物车里别人还没下单的商品。

**不可重复读（Non-repeatable Read）：** 事务A中两次读取同一条数据，结果不一致——因为事务B在两次读取之间修改并提交了它。就像你看了两遍超市价格标签，中间有人把标签换了。

**幻读（Phantom Read）：** 事务A两次查询同一范围的数据，第二次多出几行——事务B在中间插入了新数据并提交。

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 性能 |
|----------|------|-----------|------|------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 最快 |
| READ COMMITTED | 防止 | 可能 | 可能 | 较快 |
| REPEATABLE READ | 防止 | 防止 | 可能 | 中等 |
| SERIALIZABLE | 防止 | 防止 | 防止 | 最慢 |

SQLite默认使用SERIALIZABLE（最高级别），MySQL默认使用REPEATABLE READ。

**大多数Web应用使用READ COMMITTED就够了**——既保证了基本的数据安全，又有不错的并发性能。

### 三、事务在前端场景的实际应用

事务不只属于"转账"这种金融场景。任何需要"一组操作不可分割"的地方都该用事务：

- **用户注册**：创建用户 + 创建默认设置 + 发送欢迎消息记录 —— 三步必须原子化。
- **发布文章**：保存文章内容 + 更新作者的文章数 + 记录发布时间线 —— 缺一不可。
- **删除账号**：删除用户记录 + 删除关联文章 + 删除关联评论 + 删除缓存 —— 不能删一半留一半。

```python
def register_user(conn, username, email, password_hash):
    cursor = conn.cursor()
    try:
        cursor.execute(
            "INSERT INTO users (username, email) VALUES (?, ?)",
            (username, email)
        )
        user_id = cursor.lastrowid

        cursor.execute(
            "INSERT INTO user_settings (user_id, theme, language) "
            "VALUES (?, 'light', 'zh-CN')",
            (user_id,)
        )

        cursor.execute(
            "INSERT INTO notifications (user_id, type, message) "
            "VALUES (?, 'welcome', '欢迎注册！')",
            (user_id,)
        )

        conn.commit()
        return user_id
    except Exception:
        conn.rollback()
        raise
```

## 总结

1. 事务保证一组SQL操作的原子性——要么全成功（COMMIT），要么全失败（ROLLBACK），没有中间状态。
2. ACID是事务的四个核心属性：原子性、一致性、隔离性、持久性——每一点都对应一个实际的数据安全需求。
3. 隔离级别是在"数据正确性"和"并发性能"之间的权衡——大多数Web应用用READ COMMITTED即可。
4. 任何"多步修改一个都不许少"的场景都应该用事务，不只是转账。
5. 事务让代码更简洁——不再需要在每个地方写补偿逻辑，数据库替你保障数据完整性。

## 注意事项

1. 事务不宜太长——长时间持有事务会锁住资源，拖慢其他用户的操作。一个事务内不要放HTTP请求、文件读写等慢操作。
2. SQLite在并发写入方面较弱——它同一时间只允许一个写事务。如果你的应用高并发写入（每秒几十个写请求），考虑用PostgreSQL。
3. Python的sqlite3默认在DML语句前自动开启事务（`BEGIN`），但更推荐显式地用`conn.commit()`和`conn.rollback()`控制事务边界。

## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 忘了commit | 程序退出后数据全丢了 | SQL操作在事务中但没提交，连接关闭时回滚 | 养成写完修改操作就commit（或with语句自动commit）的习惯 |
| 异常处理中忘了rollback | 报错退出后数据库状态"卡在半路" | 只捕获了异常但没显式rollback | 使用`try...except...finally`，在finally中确保回滚未提交的事务 |
| 长事务锁表 | 其他用户访问超时或报错 | 在事务中做了耗时操作（如调用外部API、处理图片） | 外部操作放到事务外；事务只包裹数据库操作，尽量短小 |
| SQLite并发写入冲突 | 多线程同时写报"database is locked" | SQLite只支持单写者 | 加超时设置`conn.execute('PRAGMA busy_timeout = 5000')`，或用WAL模式 |
