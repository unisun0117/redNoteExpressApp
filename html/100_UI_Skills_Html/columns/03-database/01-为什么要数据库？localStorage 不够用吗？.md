# 数据库设计从0到1 第1篇：为什么要数据库？localStorage不够用吗？
@[TOC](目录)
## 摘要
> 从前端本地存储的痛点出发，解释为什么需要专业数据库。对比localStorage、IndexedDB与关系型数据库的本质差异，帮助前端开发者理解数据库的核心价值。

## 引言

很多前端开发者刚入门时，第一个接触的"存储"就是localStorage。存个用户偏好、缓存点数据，`localStorage.setItem('key', 'value')` 一行搞定，确实方便。但当项目规模逐渐变大——用户量上千、数据需要关联、多人同时操作——localStorage的短板就暴露无遗了。

你有没有遇到过这种场景：做一个小型管理系统，一开始用localStorage存用户数据，后来发现每次都要把整个JSON读出来、解析、修改、再写回去，数据一大页面就卡。更糟的是，浏览器清缓存数据全丢，换个设备更是什么都看不到。这时候，你就需要一个真正的数据库。

本文是《数据库设计从0到1》专栏的第一篇，带你理解为什么数据库是每个应用的基础设施，而不是一个"可选的高级功能"。我们会从前端开发者最熟悉的场景切入，一步步揭开数据库的面纱。

## 基础知识储备

1. **localStorage使用经验**：了解`getItem`、`setItem`、`removeItem`，以及它只能存字符串的限制。
2. **JSON序列化**：理解`JSON.stringify`和`JSON.parse`，知道为什么localStorage存对象要先转字符串。
3. **前端状态管理的基本概念**：了解React中的state、props，或Vue中的data、computed——知道"数据"在前端是如何流转的。
4. **基本网络常识**：理解前后端分离架构，知道前端运行在浏览器、后端运行在服务器。

## 正文

### 一、localStorage 够用吗？从三个真实场景说起

想象你用React写了一个笔记应用，所有数据都存在localStorage里。代码大概长这样：

```javascript
// React 笔记应用 —— localStorage 版
function App() {
  const [notes, setNotes] = useState(
    JSON.parse(localStorage.getItem('notes') || '[]')
  );

  const addNote = (text) => {
    const newNotes = [...notes, { id: Date.now(), text }];
    setNotes(newNotes);
    localStorage.setItem('notes', JSON.stringify(newNotes));
  };

  const deleteNote = (id) => {
    const newNotes = notes.filter(n => n.id !== id);
    setNotes(newNotes);
    localStorage.setItem('notes', JSON.stringify(newNotes));
  };
  // ...
}
```

这个版本在小数据量下运行良好。但三个致命问题迟早会出现：

**问题一：容量限制。** localStorage 通常只有5-10MB，存几百条笔记没问题，但如果是电商的后台管理系统，成千上万条商品数据呢？直接爆掉。

**问题二：数据丢失风险。** 用户清理浏览器缓存，或者换一台设备登录，所有数据就没了——localStorage是绑定在浏览器上的，换设备等于换存储。

**问题三：无法多人协作。** 你的笔记应用如果有两个管理员同时编辑，localStorage根本无法处理冲突。一个管理员改了数据，另一个管理员完全不知道。

这就是数据库要解决的问题：**持久化、共享、可靠、高性能**。

### 二、数据库到底帮我们干了什么？

数据库的核心能力，可以用四个词概括：

1. **持久化（Persistence）**：数据存到服务器的硬盘上，关机不丢，换设备也能访问。
2. **共享访问（Concurrent Access）**：几百个用户同时读写，数据库内部用锁机制保证数据不乱套。
3. **结构化查询（Structured Query）**：不用把整个数据表读到内存再过滤。想找"价格大于100的商品"，一句SQL就能搞定，数据库在服务端处理，只返回结果。
4. **数据完整性（Integrity）**：可以定义约束——比如用户名不能重复、年龄不能为负——数据库替你检查，比你手写JS校验可靠得多。

打个类比：localStorage就像一个记事本，翻一翻方便，但页数有限、容易弄丢、不能多人共享。数据库则像一个专业的档案室，有编号、有索引、有人管理、可以多人同时查阅。

### 三、关系型数据库 vs 非关系型数据库：该先学哪个？

数据库也分种类，前端同学可能听说过MongoDB（非关系型）和MySQL（关系型）。简单区分一下：

| 对比维度 | 关系型数据库（MySQL/PostgreSQL） | 非关系型数据库（MongoDB/Redis） |
|----------|----------------------------------|----------------------------------|
| 数据模型 | 表格（行和列），像Excel | 文档（JSON）、键值对、图等 |
| 关联方式 | 外键关联，JOIN查询 | 嵌套文档或手动关联 |
| 适用场景 | 结构化数据，如订单、用户、库存 | 灵活结构，如日志、缓存、社交动态 |
| 前端类比 | antd的Table组件，有固定列 | 嵌套的flex布局，层级自由 |

本专栏选**关系型数据库**作为主线，具体用**SQLite**作为学习工具。理由很简单：
- 关系型数据库的思维（表、列、外键）和前端组件思维互通——表格组件≈数据库表，Props类型≈字段类型。
- SQLite是零配置的——一个文件就是整个数据库，不需要安装服务器，Python内置支持。

## 总结

1. localStorage适合少量、非关键的客户端缓存，不适合作为应用的主要数据存储。
2. 数据库解决的核心问题：持久化存储、共享访问、结构化查询、数据完整性。
3. 关系型数据库以"表格"为模型，和前端数据展示天然亲和，入门门槛低。
4. 本专栏用SQLite + Python，零配置上手，专注于数据库设计思维本身。
5. 数据库思维本质上是一种"结构化思维"——把混乱的数据整理成有章可循的表格，这和前端组件化是相通的。

## 注意事项

1. 不要把数据库当成"加大号的localStorage"——它们的架构理念完全不同。localStorage是单用户、单设备的键值缓存；数据库是多用户、服务端的结构化存储。
2. 学习数据库不是为了放弃localStorage——它们各有用处。用户偏好、主题色之类的小数据用localStorage；核心业务数据（用户、订单、商品）必须存数据库。
3. 本专栏从头开始构建知识体系，即使你完全没有后端经验，跟着每篇文章的代码示例一步步来也完全能跟上。

## 坑点
| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 用localStorage当数据库 | 数据量大了页面卡死，换设备后数据消失 | localStorage是浏览器本地缓存，不支持服务端持久化 | 核心业务数据存数据库，localStorage只做UI状态缓存 |
| 把所有数据放一个localStorage key | 每次修改都要读写整个大JSON，性能极差 | 没有分片存储，读写的粒度太粗 | 按业务模块拆分成多个key，或者直接上数据库 |
| 后端返回数据直接localStorage缓存 | 用户看到过期数据，不知道自己已经在别处被删除了 | localStorage没有过期机制和同步机制 | 服务端数据不要用localStorage缓存，用React Query或SWR做请求级缓存 |
| SQLite连接不关闭 | 应用报"database is locked"错误 | SQLite同一时间只允许一个写操作，连接未释放会阻塞后续请求 | 每次操作后确保关闭连接，或使用连接池管理 |
