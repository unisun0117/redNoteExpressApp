## 正文

### 认证方案选型决策表

技术选型的核心不是"哪个技术最先进"，而是"哪个技术最适合当前场景"。以下是一张按部署架构划分的选型推荐表：

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 单体 SSR 应用（Django、Rails、Next.js Pages Router） | Session + Cookie | 同域部署，Cookie 自动携带，Session 服务端控制力强。不需要跨域，没有分布式问题 |
| 前后端分离 SPA（React/Vue + 独立 API） | JWT 双 Token | 跨域不可避免，JWT 的无状态特性让 API 服务器可以独立部署和水平扩展 |
| 微服务架构 | JWT + API Gateway | Gateway 统一验证 JWT，下游服务信任请求头中的用户信息，避免每个服务都依赖用户库 |
| 需要第三方登录 | OAuth 2.0 + 本地 JWT/Session | OAuth 只做身份验证（证明你是谁），授权完成后签发自己系统的 token |
| 移动 App（iOS/Android） | JWT 双 Token，存 Keychain/KeyStore | 移动端没有 Cookie 机制，JWT 是最自然的选择。Refresh Token 存系统安全存储 |
| 内部管理后台 | Session + Cookie | 用户量少、SSR 渲染、需要随时踢人下线，Session 的控制力更强 |
| 纯 API 产品（对外提供 API） | API Key + HMAC 签名 | 不是用户登录场景，不适用 JWT 或 Session。用 API Key 标识调用方，用签名防篡改 |

两个常见的选择误区需要单独说明。误区一："JWT 比 Session 更现代所以更好。"JWT 解决的是无状态问题，Session 解决的是控制力问题——两者是取舍，不是代际差距。如果你做的是单体 SSR 应用，Session 在安全性和实现复杂度上都占优。误区二："有 OAuth 就够了，为什么要自己做认证？"OAuth 是授权协议，被"借用"来做认证。如果你的应用只用"微信登录"，确实可以只做 OAuth。但如果你需要邮箱密码登录、需要控制用户体系、需要灵活切换第三方登录平台——那就需要自己的认证系统，OAuth 只是其中的一个登录方式。

### 面试高频 10 问

以下是认证领域面试中最常被问到的 10 个问题，以及标准回答框架。回答时遵循一个结构：一句话定义 + 关键原理 + 实际用过或遇到过什么坑，这样比背概念印象分高得多。

**Q1：JWT 的三部分分别是什么？**

Header（声明签名算法和 token 类型）、Payload（存放用户数据和声明，如 user_id、exp）、Signature（用密钥对前两部分的签名，防篡改）。三段用 `.` 分隔，Base64URL 编码。要强调一个细节：Payload 是编码不是加密，任何人都能解码——所以决不能放密码。

**Q2：JWT vs Session，怎么选？**

核心区别：Session 把用户状态存在服务端（Redis/数据库），JWT 把状态存在客户端（token 里）。Session 优势是服务端完全掌控（随时踢人、改权限即时生效），劣势是需要存储、跨域麻烦。JWT 优势是无状态（天然适合微服务和水平扩展），劣势是无法主动撤销、Payload 不能太大。选型看两点：是否跨域（跨域首选 JWT），是否需要频繁控制用户状态（需要首选 Session）。

**Q3：Access Token 过期了怎么办？怎么做到用户无感？**

用双 Token 方案。Access Token 短命（15分钟），过期后前端拦截器用 Refresh Token 换新的 Access Token，重试原请求——用户无感。如果 Refresh Token 也过期了，跳转登录页。关键实现细节：用一个 Promise 缓存防止并发 401 触发多次刷新。

**Q4：CSRF 是什么？怎么防御？**

跨站请求伪造——攻击者利用浏览器自动携带 Cookie 的机制，诱导已登录用户执行非预期的操作。防御三层：SameSite Cookie（`Lax` 或 `Strict`）禁止跨站发送；CSRF Token（表单隐藏字段，服务端验证）；验证 Referer/Origin 头确认请求来源。

**Q5：密码怎么存？为什么不直接用 MD5 或 SHA256？**

密码必须用慢哈希算法（bcrypt、argon2）加随机盐值后存储。MD5 和 SHA256 设计目标是"快"——GPU 每秒能算数十亿次，攻击者跑彩虹表瞬间碰撞出原始密码。bcrypt 故意让计算变慢（100ms+），用户登录无感，但攻击者的破解成本指数级上升。盐值确保相同密码产生不同哈希，防止批量碰撞。

**Q6：OAuth 2.0 授权码模式流程？为什么最安全？**

流程分两段。前半段浏览器跳转授权页 → 用户同意 → 重定向带回授权码（code）。后半段后端用 code + Client ID + Client Secret 向授权服务器换 Access Token。安全性的关键在于"code 和 secret 分离"：code 在浏览器中传递但单独无用（没有 secret 换不到 token），secret 只在后端服务器之间传输，浏览器接触不到。

**Q7：用户改密码后，旧 JWT 还有效吗？怎么解决？**

默认情况下仍然有效——这是 JWT 无状态特性带来的代价。解决方案有三种：缩短 Access Token 有效期到 5-15 分钟（改密码后最多等 15 分钟旧 token 失效）；维护 token 签发时间戳，用户改密码时更新 `password_changed_at`，验证时对比（需要查数据库，牺牲了无状态优势）；维护 JWT 黑名单（有状态方案，背离 JWT 设计初衷）。工程上通常用第一种 + 第二种的组合。

**Q8：前端 Token 存在哪里最安全？**

Access Token 存在 JS 内存中（闭包变量），页面刷新丢失后用 Refresh Token 重新获取。Refresh Token 存在 httpOnly + Secure + SameSite=Lax 的 Cookie 中，JS 读不到。localStorage 虽方便但能被 XSS 攻击窃取，不推荐。移动端 Access Token 存内存，Refresh Token 存 Keychain（iOS）或 KeyStore（Android）。

**Q9：RBAC 怎么设计？角色之间怎么处理继承？**

三要素：用户-角色-权限。用户不直接拥有权限，通过角色间接获取。代码中永远用 `has_permission("article:delete")` 而非 `if role == "admin"`。角色间的包含关系通过权限集合实现（admin 的权限集合是 editor 的超集），而非通过角色继承层级——这避免了"钻石继承"等复杂问题。

**Q10：从零设计一个认证系统，你会考虑哪些方面？**

从下到上五层：第一层密码安全（bcrypt + salt）；第二层认证协议（JWT 双 Token 或 Session，视场景定）；第三层第三方登录（OAuth 2.0 可选）；第四层权限管理（RBAC，路由守卫 + 数据级权限 + 前端控制）；第五层安全防护（XSS/CSRF/SQL注入/暴力破解/密钥管理）。每层说清楚选什么技术、为什么。

### 毕业项目：搭建一个完整的认证中心

作为整个专栏的毕业设计，建议你亲手搭建一个完整的认证系统，包含以下功能模块：

模块一：邮箱密码注册和登录，密码用 bcrypt 存储，登录后返回 JWT 双 Token（Access + Refresh）。

模块二：接入 GitHub OAuth 登录，授权后自动创建或绑定本地用户，签发 JWT。

模块三：RBAC 权限系统，定义 admin/editor/viewer 三种角色，实现基于权限的路由守卫和至少一个数据级权限检查。

模块四：安全防护——前端无感刷新（带刷新锁）、Cookie 安全三件套、登录频率限制（5 次失败锁 15 分钟）、API 输入校验和输出转义。

模块五：完整测试——用 pytest 写覆盖所有关键流程的测试用例（注册、登录、刷新、权限拒绝、OAuth 回调）。

这个项目的代码量不大（所有代码加起来几百行），但覆盖了一个生产级认证系统 90% 的核心知识点。把它放到 GitHub 上并写好 README，简历上关于"认证系统"的部分就有了坚实的项目支撑。

## 总结

1. 认证方案选型的核心不是"哪个更先进"，而是"哪个更适合当前场景"——Session 适合单体 SSR，JWT 适合前后端分离和微服务。
2. OAuth 是授权协议而非认证协议，它可以作为认证系统的一个登录方式，但不应该成为你整个认证体系。
3. 面试回答时遵循"定义 + 原理 + 实际经验"三段式结构，比背诵概念更有说服力。
4. 10 道高频面试题覆盖了密码存储、JWT 结构、CSRF/XSS 防御、OAuth 流程、RBAC 设计等认证领域所有核心知识点。
5. 搭建一个完整的认证中心作为毕业项目，几百行代码涵盖 90% 的核心知识，是简历上最扎实的项目经验。

## 注意事项

1. 面试时如果被问到"你用过什么认证方案"，不要只说"我用过 JWT"——要说出场景，比如"在前后端分离的 React + FastAPI 项目中用了 JWT 双 Token，前端无感刷新用 Promise 缓存解决并发 401"。
2. 技术选型不存在银弹，任何方案都有取舍——面试官更想看到你知道"这个方案不适合什么场景"，而不是只会背书"JWT 适合所有场景"。
3. 毕业项目务必真的写出来、跑起来，只看不写等于没学。

## 坑点

| 坑 | 现象 | 原因 | 解决方案 |
|----|------|------|----------|
| 面试时只说概念不说经验 | 面试官问"用过 JWT 吗"，回答"了解过，三部分 Header Payload Signature"——印象分 3/10 | 没有把知识和实际项目关联起来 | 准备一段 30 秒的项目经验描述：什么场景、用了什么方案、遇到什么坑、怎么解决的 |
| 技术选型盲从潮流 | 单体 SSR 应用硬上 JWT，Cookie 完全够用却绕了一大圈 | 只看技术博客标题，没有理解每个方案的适用条件 | 做选型前列出架构约束（同域/跨域？单体/微服务？需要踢人？），用约束反推方案 |
| 毕业项目只搭架子不写安全 | 实现了登录、JWT、OAuth，但 Cookie 没设 HttpOnly、没做频率限制 | 把安全和核心功能分开做，核心功能做完就"没时间做安全了" | 安全不是附加功能，从第一行认证代码开始就要同步考虑——把安全要求作为"完成"的定义之一 |
| 测试只覆盖正常流程 | 登录成功、刷新成功的测试都过了，但 token 篡改、权限拒绝、并发 401 的场景全漏了 | 只写了"快乐路径"测试，忽略了异常和攻击场景 | 写测试时遵循"先写异常用例，再写正常用例"——大多数安全漏洞都出现在异常路径中 |
