# Superpowers 技能参考手册

> 共 14 个技能，按工作流顺序和使用场景分类。调用方式：`/skill-name`

---

## 一、核心工作流（6 个）—— 从想法到交付

从零开始实现一个功能的标准流程：入口 → 设计 → 计划 → 执行 → 收尾。

---

### 1. `/using-superpowers`

**作用：** 一切对话的入口点。强制代理在采取任何行动之前检查是否有适用的技能。

**适用场景：** 每次新对话自动触发，任何操作之前先问"有没有对应的技能？"

**使用示例：**
```
用户: "帮我修复这个登录 bug"
代理: 自动先触发 using-superpowers → 发现 bug → 触发 systematic-debugging → 再触发 TDD → ...
```

> **核心规则：** 只要觉得某个技能可能有 1% 的概率适用，就必须调用它。

---

### 2. `/brainstorming`

**作用：** 在写任何代码之前，先探索需求、澄清意图、设计方案。

**适用场景：**
- 新功能开发
- 组件/模块设计
- 行为变更

**使用示例：**
```
用户: "给用户加个注销按钮"
代理: /brainstorming
  → 探索现有代码结构
  → 问：注销后跳转到哪个页面？是否需要确认弹窗？session 怎么处理？
  → 提供 2-3 种方案（纯前端 / 调API / 带清除缓存的完整方案）
  → 确认方案后，将设计写入 docs/superpowers/specs/
  → 然后自动衔接 /writing-plans
```

> **红线：** 设计未获批之前，不允许写任何代码。

---

### 3. `/writing-plans`

**作用：** 将确认好的设计方案，拆解成可执行的、精确的实施计划。

**适用场景：** 设计方案已确认，需要写实施计划时。

**使用示例：**
```
设计文档: 用户注销功能
↓ /writing-plans
产出实施计划（保存到 docs/superpowers/plans/2026-06-03-user-logout.md）:
  Step 1: 在 UserMenu 组件加注销按钮，写测试
  Step 2: 实现 logout() API 调用函数
  Step 3: 实现点击确认弹窗逻辑
  Step 4: 实现注销后跳转和状态清理
  ...
```

> **铁律：** 计划中不允许出现 "TODO"、"适当的"、"TBD" 等模糊词。每一步必须精确到文件路径、完整代码、预期输出。

---

### 4. `/subagent-driven-development` （推荐）

**作用：** 在当前会话中，按计划逐步执行，每个任务分派全新的子代理 + 两级审查。

**适用场景：** 有计划、任务之间相对独立时（推荐优先使用）。

**使用示例：**
```
用户: /subagent-driven-development

流程:
  任务1 → 分派子代理实现 → 规格审查 → 代码质量审查 → ✅
  任务2 → 分派子代理实现 → 规格审查 → 代码质量审查 → ✅
  ...
  全部完成 → 整体代码审查 → /finishing-a-development-branch
```

> **亮点：** 每个任务用全新子代理（无上下文污染）+ 每个任务双审查（规格 + 质量）= 高质量快速迭代。

---

### 5. `/executing-plans`

**作用：** 与子代理开发类似，但在单独会话中执行计划（无子代理支持时使用）。

**适用场景：** 当环境不支持子代理时，作为降级方案。能用于代理开发就优先用子代理开发。

**使用示例：**
```
用户: /executing-plans
代理: 加载计划 → 逐步执行 → 遇到阻塞停下来求助 → /finishing-a-development-branch
```

---

### 6. `/finishing-a-development-branch`

**作用：** 所有开发和测试完成后，决定如何整合代码（合并/PR/丢弃/保留）。

**适用场景：** 实现完成 + 测试全过之后。

**使用示例：**
```
全部测试通过 ✓
↓ /finishing-a-development-branch
代理: 检测环境 → 提供选项:
  1. 合并回基准分支
  2. 推送并创建 PR
  3. 保持分支不动
  4. 放弃本次工作
```

## 二、品质保障（5 个）—— 保证代码质量

覆盖开发过程中的每个质量环节：编写 → 调试 → 验证 → 审查 → 接收反馈。

---

### 7. `/test-driven-development`

**作用：** 严格的红-绿-重构循环，先写测试再写代码。

**适用场景：**
- 写任何新功能之前
- 修 bug 之前（先写能复现 bug 的测试）
- 重构之前

**使用示例：**
```
用户: "实现一个重试逻辑，最多重试 3 次"

RED:   写测试 test('retries failed operations 3 times')
       观察测试失败 ✓ （因功能还不存在）
GREEN: 写最简单的 for 循环，刚好通过测试
       观察测试通过 ✓
REFACTOR: 消除重复、改进命名
```

> **铁律：** 如果先写了代码再补测试 —— 删掉代码，重新来。

---

### 8. `/systematic-debugging`

**作用：** 四阶段系统调试法：根因调查 → 模式分析 → 假设验证 → 实施修复。

**适用场景：**
- 任何测试失败、生产 bug、意外行为
- 性能问题、构建失败、集成问题

**使用示例：**
```
用户: "用户登录后偶尔白屏"

阶段1 根因调查：阅读报错 → 找到稳定复现步骤 → git diff 看最近的改动
阶段2 模式分析：对比正常登录和异常登录的数据流差异
阶段3 假设验证："我认为是 token 在竞态条件下未就绪导致渲染空状态"
阶段4 修复：写复现测试 → 加防御代码 → 验证通过
```

> **铁律：** 没有找到根因之前，不允许写任何修复代码。

---

### 9. `/verification-before-completion`

**作用：** 声称"完成了"之前，必须运行验证命令并看到输出证据。

**适用场景：** 任何声称完成、修复、通过之前 —— 提交前、PR 前、任务完成前。

**使用示例：**
```
代理: "Bug 已修复 ✓"
↓ /verification-before-completion
代理: 运行修复前的复现测试 → 确认测试从"失败"变成"通过"
     运行完整测试套件 → 确认输出包含 "0 failures"
     现在可以声称修复完成
```

> **铁律：** 运行命令 → 读取输出 → 再声称结果。跳过任何一步 = 撒谎。

---

### 10. `/requesting-code-review`

**作用：** 分派独立的代码审查子代理，自动审查当前改动。

**适用场景：**
- 子代理开发中每个任务完成后（强制）
- 主要功能完成后（强制）
- 合并到主分支前（强制）

**使用示例：**
```
用户: /requesting-code-review

代理:
  获取 git diff → 分派审查子代理（用 code-reviewer.md 模板）
  → 审查子代理返回反馈:
    [关键] 缺少空值检查，可能导致崩溃
    [重要] 变量命名不够清晰
    [次要] 可提取公共函数减少重复
  → 立即修复关键问题 → 处理重要问题 → 记录次要问题
```

---

### 11. `/receiving-code-review`

**作用：** 接收代码审查反馈后的标准处理流程。不盲从、不敷衍、用技术事实说话。

**适用场景：** 收到任何代码审查反馈时（外部 PR 审查、同事审查、子代理审查）。

**使用示例：**
```
审查者: "这段代码应该用 useMemo 包裹"

正确做法:
  1. 重述: "你认为渲染结果应该被缓存以避免重复计算"
  2. 验证: grep 检查代码库中此组件的实际渲染频率
  3. 评估: 如果实际只渲染 1 次 → 合理反驳 "该组件仅在挂载时渲染一次，useMemo 没有收益"
  4. 如果确实高频渲染 → 实施修复
```

> **禁止的回复：** "你说得对！" / "好主意！" —— 用技术分析代替社交套话。

## 三、工程效能（3 个）—— 提高开发效率

---

### 12. `/dispatching-parallel-agents`

**作用：** 将多个独立任务并行分派给不同的子代理，同时解决多个问题。

**适用场景：**
- 3 个以上测试文件因不同原因失败
- 多个子系统独立损坏
- 每个问题不依赖其他问题的上下文

**使用示例：**
```
5 个测试文件失败，根因各不相同:
  /dispatching-parallel-agents
  → 代理A: 修复 user-auth.test.ts （认证中间件问题）
  → 代理B: 修复 api-cache.test.ts （缓存过期逻辑）
  → 代理C: 修复 form-valid.test.ts （校验规则变更）
  → 代理D: 修复 db-query.test.ts   （查询参数格式）
  → 代理E: 修复 upload-file.test.ts （文件大小限制）
  全部并行执行 → 汇总结果 → 运行完整套件验证
```

> **不适用时：** 失败原因相关、需要全面系统状态、代理会相互干扰。

---

### 13. `/using-git-worktrees`

**作用：** 创建隔离的工作区（Git Worktree），在新功能开发时不污染当前分支。

**适用场景：**
- 开始需要隔离的新功能时
- 执行实施计划前
- 保护当前分支不被改动影响

**使用示例：**
```
当前在 main 分支
用户: "开发用户注销功能"
代理: /using-git-worktrees
  → 创建隔离工作区 .worktrees/user-logout/
  → 自动运行项目设置 (npm install...)
  → 验证基准测试通过
  → 在新工作区中安全开发
```

---

### 14. `/writing-skills`

**作用：** 创建或修改自定义技能（就像你现在看到的所有技能，都是这样做出来的）。

**适用场景：**
- 创建新的自动化工作流
- 编辑现有技能
- 部署前验证技能是否有效

**使用示例：**
```
用户: "我想做个自动生成 API 文档的技能"

RED:    在不加载技能的情况下，让子代理"生成API文档"
        记录基准行为（格式混乱、缺参数说明...）

GREEN:  写 SKILL.md 解决这些具体问题
        重新测试，子代理输出规范了

REFACTOR: 发现子代理还有新借口 → 补到技能里堵漏洞
```

## 标准开发工作流

```
/brainstorming           ← 设计方案（不写代码）
      ↓
/writing-plans            ← 写实施计划（精确到每步）
      ↓
/using-git-worktrees      ← 创建隔离工作区（可选）
      ↓
/subagent-driven-development  ← 逐步执行 + 每个任务双审查
   (每步包含: /test-driven-development + /requesting-code-review)
      ↓
/verification-before-completion  ← 全部验证通过
      ↓
/finishing-a-development-branch  ← 合并 / PR / 清理
```
